# 100k_concurrent_server

- 원본: `source/shdkej-content/blog/100k_concurrent_server.md`
- 상태: 기준선 재구성
- 역할: 만다라트 `blog > 100k_concurrent_server` 카테고리 노드의 1:1 대응 페이지

## 한눈에 보는 요약
이 노트는 "10만 동시 접속이 가능한가"라는 단순 성능 자랑 질문보다, 한 대의 서버가 실제로 어디에서 막히는지 추적하는 실험 노트에 가깝다. 원문은 소켓, 포트, 커넥션 개념 혼선에서 출발해 ulimit, TIME_WAIT, 로컬 포트 고갈, 네트워크 대역폭, CPU, 메모리, nginx, Go 서버, Kubernetes ingress, websocket 유지 연결까지 차례로 부딪힌다. 핵심은 한계가 하나의 숫자로 정해져 있지 않다는 점이다. 동시 접속 수는 커널 파라미터, 프로세스별 FD 한계, 클라이언트 포트 수, 네트워크 조건, 서버 구현 방식, 테스트 토폴로지에 따라 전혀 다르게 보인다.

## 이 노드의 중심 질문
- 한 대의 PC로 10만 동시 접속을 유지할 수 있는가?
- 소켓, 포트, 커넥션은 각각 무엇이며 실제 병목은 어디에서 생기는가?
- 병목은 서버 프로세스, 클라이언트 포트, 네트워크 대역폭, 커널 설정 중 무엇에 먼저 걸리는가?
- echo 테스트, 유지 연결 테스트, websocket 테스트, Kubernetes 테스트는 왜 서로 다른 결과를 내는가?

## 원문 구조 재정리

### 1. 개념 혼선에서 시작한다
원문 초반은 성능 최적화보다 용어 정리에서 길게 멈춘다.

- 포트는 왜 65535인가
- 커넥션은 IP:PORT 쌍으로 어떻게 식별되는가
- 소켓과 커넥션은 어디까지 같은 말처럼 쓰이고 어디서 달라지는가
- 서버는 한 포트로 65535개 이상의 연결을 받을 수 있는가

이 구간은 미완성처럼 보이지만 중요하다. 저자는 숫자를 외우기보다, 실제 한계가 어떤 주소 공간과 자원 모델에서 나오느냐를 이해하려 한다.

### 2. 이론보다 테스트로 병목을 찾는다
원문은 바로 코드와 실험으로 넘어간다.

- Go HTTP 서버와 클라이언트로 echo 테스트
- docker 안팎 비교
- nginx worker_connections 조정
- `ulimit -n` 확장
- `watch ss -s`, `/proc/net/tcp`, `netstat`로 상태 관찰
- vagrant, 로컬, 외부 IP, Raspberry Pi, Kubernetes로 테스트 환경 분리

즉 이 노드는 부하 테스트 튜토리얼보다, "막히는 지점을 보기 위해 무엇을 바꿔가며 확인했는가"의 기록이다.

### 3. 동시 접속은 하나의 지표가 아니다
원문에서는 같은 "동시 접속"도 서로 다른 실험으로 나뉜다.

- 짧은 echo 응답을 잘 반환하는가
- HTTP 연결을 일정 시간 유지할 수 있는가
- websocket으로 장시간 유지 연결이 되는가
- ingress와 replica를 둔 Kubernetes 구조에서는 어디서 거부되는가

이 차이를 구분해야 숫자의 의미를 읽을 수 있다. 짧은 요청 처리량, 열린 소켓 수, 유지 연결 수, 실제 사용자 체감 안정성은 같은 값이 아니다.

## 핵심 논지

### 1. 10만 동접 문제는 "서버가 몇 개 받느냐"보다 "어느 층이 먼저 고갈되느냐"의 문제다
원문 전반에서 반복되는 것은 단일 원인 찾기 실패다. 처음에는 포트 한계처럼 보이지만, 곧 FD 한계, TIME_WAIT, 클라이언트 측 ephemeral port 범위, i/o timeout, CPU 과부하, 네트워크 unreachable, ingress 병목이 섞여 나온다. 즉 동시 접속 수는 하나의 절대 상한이 아니라 자원 고갈 지점들의 최소값이다.

### 2. 로컬 테스트 숫자는 서버의 절대 능력이 아니라 테스트 토폴로지의 산물이다
원문에서 특히 중요한 대목은 로컬에서 3만, 4만, 5만 근처에서 이상하게 막히는 이유를 계속 의심하는 부분이다. 클라이언트와 서버가 같은 머신에 있으면 서버만 보는 것이 아니라 클라이언트 포트 범위와 로컬 네트워크 구조까지 같이 소모한다. vagrant, 외부 IP, 별도 머신, 여러 IP 할당을 시도하는 흐름은 이 함정을 스스로 발견하는 과정으로 읽힌다.

### 3. 이 노트는 성능 수치보다 운영 감각을 축적한다
원문은 최종 정답표를 주지 않는다. 대신 다음 감각을 남긴다.

- 단순 echo와 실제 서비스는 다르다.
- 유지 연결은 요청 처리량 테스트와 다르게 봐야 한다.
- `too many open files`는 프로세스 한계 신호다.
- `cannot assign requested address`는 클라이언트 쪽 주소/포트 고갈과 연결해 읽어야 한다.
- `connection reset by peer`, `network is unreachable`, `i/o timeout`은 각각 다른 층의 병목을 가리킨다.
- nginx, ingress, websocket, multi-process는 같은 10만 동접 문제를 다른 방식으로 풀게 만든다.

## 실험 흐름에서 읽히는 큰 축

### 용어와 구조를 먼저 붙잡으려는 축
원문 초반의 길고 헷갈린 질문들은 쓸모없는 우회가 아니다. 저자는 "소켓 수를 늘리면 된다" 같은 팁으로 끝내지 않고, 포트와 커넥션 식별 구조가 실제 접속 한계와 어떻게 연결되는지 이해하려 한다. 이 덕분에 이후 실험 실패를 단순 설정 누락이 아니라 구조 문제로 다시 읽는다.

### 관측 가능한 상태를 만들려는 축
원문에는 명령어 메모가 많다.

- `watch ss -s`
- `/proc/net/tcp`
- `netstat ... TIME_WAIT`
- `iperf`
- `ulimit`
- `sysctl`

이는 단순 팁 संग्रह이 아니라, 성능 실험을 "느낌"이 아니라 상태 변화로 보려는 태도다. 숫자가 흔들릴수록 관측 도구가 중요해진다는 운영 감각이 드러난다.

### 단일 프로세스 한계에서 구조 분산으로 이동하는 축
후반부의 큰 전환은 이것이다.

- 한 포트, 한 프로세스에서 최대치를 밀어본다.
- 로컬 포트 범위와 FD 한계에 걸린다.
- 여러 IP, 여러 프로세스, nginx, Kubernetes, websocket 같은 구조적 우회를 시도한다.

즉 질문이 "한 프로세스로 가능한가"에서 "한 머신에서 어떻게 분산해 받는가"로 이동한다. 이 전환은 실제 운영 설계와 맞닿아 있다.

## 세부 해석

### 소켓, 포트, 커넥션 혼선이 왜 중요한가
원문은 이 세 용어를 깔끔하게 정의하지 못한 채 오래 붙잡는다. 하지만 바로 그 혼선이 중요하다. 10만 동접 문제에서는 추상 용어 암기보다, 누가 포트를 점유하고 누가 FD를 먹고 누가 TIME_WAIT를 남기는지 이해해야 병목을 읽을 수 있다. 이 노트의 초반부는 성능 실험 이전의 모델 만들기 단계다.

### echo 테스트가 주는 의미
echo 테스트는 가장 단순한 서버 로직에서 연결 자체가 얼마나 버티는지 보는 용도다. 여기서 1만, 2만, 3만, 4만 같은 숫자가 계속 바뀌는 것은 애플리케이션 로직보다 커널/네트워크/클라이언트 토폴로지가 더 큰 영향을 준다는 뜻이다. 즉 이 실험은 "Go가 느리다"를 말하기보다, 순수 연결 실험조차 환경 의존적이라는 사실을 보여준다.

### nginx와 ingress가 보여주는 것
원문에서 nginx는 Go 서버보다 더 높은 동접을 처리하는 사례로 나온다. Kubernetes에서는 ingress controller인 traefik이 병목 후보로 등장한다. 이 흐름은 서버 코드만 튜닝해서는 안 되고, 앞단 계층이 연결 수용 능력과 메모리 사용에 큰 영향을 준다는 점을 보여준다. 즉 동접 문제는 애플리케이션 단독 과제가 아니라 프록시와 게이트웨이 구조 문제다.

### websocket 유지 연결 테스트가 보여주는 것
10초 유지 연결과 websocket 5만 성공 메모는 중요한 전환점이다. 짧은 HTTP 요청 처리와 달리, 유지 연결에서는 연결 지속 자체가 핵심이며 메모리 사용량이 더 직접적으로 드러난다. 여기서 3GB 메모리 사용, 59980 근처 정체, 포트 범위와의 연결이 언급되는 것은 "유지 연결은 곧 상태 보존 비용"이라는 감각으로 읽힌다.

### multi-process 발견의 의미
원문 후반의 "충격적 발견"은 단순 팁 이상이다. 프로세스당 FD 한계라는 해석은 한 프로그램의 최적화 문제를 머신 자원 분할 문제로 다시 보게 만든다. 한 프로세스로 안 되면 여러 프로세스로 분산해 같은 머신에서 더 받을 수 있다는 발견은, 이 노트가 단순 벤치마크에서 운영 아키텍처 감각으로 넘어가는 지점이다.

## 보존해야 할 긴장
- 이론적으로 가능한 동접 수와 실제 안정적으로 운영 가능한 동접 수는 다르다.
- 서버 한계처럼 보이는 숫자 중 상당수는 클라이언트 테스트 환경 한계일 수 있다.
- 소켓 수를 늘리면 메모리와 CPU 문제가 다시 올라온다.
- 단일 프로세스로 밀어붙일지, 여러 프로세스나 프록시 계층으로 분산할지는 별도 판단 문제다.
- HTTP 요청 처리량, 유지 연결 수, websocket 연결 수를 같은 성능 숫자로 취급하면 해석이 틀어진다.

## 원문에서 드러나는 저자의 선호

### 1. 성능을 추상 이론보다 실험 로그로 이해한다
책이나 링크를 많이 참조하지만, 결국은 직접 열어 보고 깨지고 다시 확인한 결과를 더 신뢰한다.

### 2. 숫자 자체보다 병목 위치를 더 알고 싶어 한다
몇 만이 됐는가보다 왜 거기서 멈췄는지, 어느 층이 고갈됐는지를 더 오래 추적한다.

### 3. 운영 가능성과 재현 가능한 테스트를 중시한다
명령어, 커널 옵션, 에러 메시지, 관측 지표를 남기는 방식은 나중에 다시 실험할 수 있는 상태를 만들려는 태도다.

### 4. 완결된 결론보다 미해결 질문을 남겨 둔다
이 노트는 정답 문서보다 학습 중인 운영 메모다. 따라서 모순되거나 덜 정리된 질문을 지우기보다, 이후 [[Fundamental/Network]], [[Deep Knowledge/Monitoring]], [[Meta/Troubleshooting]]와 연결해 점진적으로 해석하는 편이 원문 성격에 더 맞다.
