## 한눈에 보는 핵심
이 노트에서 말하는 `코딩`은 문법 암기나 취향 싸움이 아니라, **변경하기 쉬운 구조를 만들고, 예외 상황을 견디게 하고, 테스트와 문서화로 동작을 고정하는 일**에 가깝다. 원문은 SOLID, OOP, DI, 테스트, 에러 처리, 코딩 컨벤션, 언어별 메모를 길게 흩어 놓고 있지만, 반복해서 드러나는 기준은 비교적 일관적이다.

1. 좋은 코드는 처음부터 완벽한 코드가 아니라 **나중에 바꾸기 쉬운 코드**다.
2. 설계 원칙은 추상 교과서가 아니라 **변경 비용, 테스트 가능성, 결합도, 복구 가능성**을 낮추기 위한 장치다.
3. 해피 패스만 통과하는 구현보다 **경계 조건, 실패 상황, 운영 중 예외**를 다루는 감각이 중요하다.
4. 언어별 철학은 다르지만, 결국 읽기 쉬움, 명시성, 단순함, 유지보수성으로 수렴한다.

## 원문을 다시 읽고 보이는 큰 구조
원문은 아래 층위가 뒤섞여 있다.

- 설계 원칙: SOLID, OOP, 상속 vs 구성, 결합도와 응집도
- 제어 구조: DI, IoC, AOP, 인터페이스, 추상화
- 품질 확보: Clean Code, TDD, 리팩토링, 각종 테스트
- 운영 안전성: 에러 처리, 빠른 리턴, 복구 가능성, 로깅
- 팀 규칙: 코딩 컨벤션, lint, 파일명/공백 스타일
- 언어 메모: Python, Golang, Dart, Javascript, Kotlin, 함수형 프로그래밍

즉 이 노트는 특정 언어 튜토리얼이 아니라, **코드를 어떻게 짜야 오래 버티는가**에 대한 기준 노트다.

## 코딩을 바라보는 중심 질문
### 1) 변경이 생겨도 국소적으로 수정할 수 있는가
원문에서 SOLID, 추상화, 인터페이스, 상속보다 구성, 패키지 목적 단일화가 계속 나오는 이유는 모두 여기에 연결된다. 좋은 구조는 "멋진 구조"가 아니라 **변경 이유가 퍼지지 않는 구조**다.

### 2) 실패가 생겨도 어디서 어떻게 다룰지 분명한가
에러 처리 메모가 길게 이어지는 것은, 이 노트의 작성자가 기능 구현 자체보다 **실패를 어디서 멈추고 어디서 복구할지**를 중요한 설계 문제로 보고 있기 때문이다.

### 3) 테스트와 문서화로 동작을 고정할 수 있는가
테스트는 품질 장식이 아니라, 리팩토링을 가능하게 하는 안전장치로 읽힌다. 특히 "외부 API가 아니라 내 구현을 테스트한다"는 문장은 이 노트의 테스트관을 잘 보여준다.

### 4) 언어의 철학을 거슬러 쓰고 있지 않은가
Python, Go, JS, Kotlin, 함수형 메모가 뒤섞여 있지만 공통 메시지는 하나다. **각 언어가 잘하는 방식으로 단순하고 읽히게 써야 한다.**

## 설계 원칙 파트의 재구성
### SOLID는 추상 교훈이 아니라 변경 비용 관리법
원문은 SOLID를 짧게 나열하지만, 실제 강조점은 다음 두 가지다.

- 한 클래스, 한 모듈, 한 패키지가 감당하는 이유를 좁힌다.
- 구체 구현보다 추상 기준에 의존해 테스트와 교체를 쉽게 만든다.

특히 DIP에 대한 관심은 외부 라이브러리 예찬보다 **테스트를 위해 경계를 세우는 습관**에 더 가깝다. 즉 추상화는 미학이 아니라 교체 가능성과 검증 가능성을 위한 도구다.

### OOP는 상속 찬양보다 역할 분리 감각에 가깝다
원문은 폴리모피즘, 캡슐화, 위임, 상속을 언급하지만, 실제 결론은 상속을 넓게 쓰기보다 **구성, 위임, 인터페이스** 쪽에 더 기울어 있다.

- 상속은 빠르게 공통 속성을 묶을 수 있다.
- 그러나 요구가 갈라지면 상속 트리가 쉽게 깨진다.
- 그래서 처음부터 무엇을 고정하고 무엇을 바꿀지 확신하기 어렵다면 구성이 더 안전하다.

이 흐름은 [[./Architecture]]에서 반복되는 "세부 결정을 늦추고 변화를 견디게 설계한다"는 감각과 맞닿아 있다.

### 결합도와 응집도는 결국 변경 반경을 설명하는 말
원문에서 결합도와 응집도는 교과서 정의보다 실전 감각에 가깝다.

- 응집도: 한 단위 내부의 데이터와 동작이 실제로 서로 강하게 관련되는가
- 결합도: 다른 단위와의 연결이 느슨해 교체와 수정이 쉬운가

즉 이 노트에서 좋은 코딩은 예쁜 함수형 문체보다도, **변경이 퍼지지 않도록 경계를 긋는 기술**로 이해하는 편이 맞다.

## 테스트 파트의 재구성
### 테스트는 리팩토링 허가서다
원문은 TDD, 유닛 테스트, 시스템 테스트, E2E, 인수 테스트, 부하 테스트를 모두 훑지만, 핵심은 한 줄로 요약된다.

> 테스트는 코드를 더 구체적으로 만들고, 나중의 변경을 덜 무섭게 만든다.

그래서 이 노트는 테스트를 "좋은 습관"보다 **변경 가능한 코드의 전제조건**으로 보고 있다.

### 테스트 층위를 구분하려는 의도가 강하다
원문은 아래 질문들을 계속 던진다.

- 단일 모듈을 검증하는가, 전체 요구사항을 검증하는가
- 기능 전반이 바뀌면 A/B 테스트나 블루그린 배포가 필요한가
- 성능 테스트, 부하 테스트, 스트레스 테스트는 무엇이 다른가
- 인수 테스트를 기준으로 유닛 테스트를 조정할 수 있는가

즉 작성자는 테스트를 하나의 기술이 아니라, **변화 크기와 위험도에 따라 다른 검증 층위를 조합하는 체계**로 이해하려 한다.

### 경계 조건 감각이 중요하다
원문 전반에 "boundary check", "해피 패스만 생각하지 않기", "최악의 상황"이 반복된다. 이는 코딩을 알고리즘 풀이보다 **예외 상황에서 무너지지 않는 시스템 설계**로 보는 시선이다.

## 에러 처리 파트의 재구성
### 실패를 어디서 처리할지 결정하는 것이 설계다
원문에서 가장 오래 붙잡는 질문 중 하나는 이것이다.

- 호출자가 처리해야 하는가
- 하위 유닛이 자체 처리해야 하는가
- 복구 가능한가, 중단해야 하는가
- 로그만 남길 것인가, 상위로 전파할 것인가

여기서 중요한 건 정답 하나를 찾는 게 아니라, **에러 처리 책임을 명확히 배치해야 한다**는 감각이다.

### Go식 에러 처리에 끌리는 이유
원문은 Go의 `error` 반환, 빠른 리턴, `Fatal`의 위치를 자주 언급한다. 이는 단순히 Go 문법을 좋아해서가 아니라,

- 실패가 눈에 잘 보이고
- 호출자가 선택권을 가지며
- 숨은 제어 흐름이 적고
- 운영 중 복구 판단을 늦게 내릴 수 있기 때문

으로 읽힌다.

즉 이 노트에서 좋은 에러 처리는 우아한 추상화보다 **관측 가능성, 선택 가능성, 복구 가능성**을 높이는 방향이다.

## 코딩 컨벤션 파트의 재구성
### 컨벤션은 취향 통일이 아니라 인지 비용 절감 장치
원문은 `-` vs `_`, `tab` vs `space`, camelCase vs underscore 등을 적어 놓지만, 결론은 단순하다.

- 회사나 팀의 컨벤션을 따르고
- lint 같은 자동 기준을 두고
- 파일과 코드의 읽는 비용을 낮춘다

즉 이 노트에서 컨벤션은 심미적 취향보다 **협업 마찰을 줄이는 운영 규칙**이다.

### 큰 util 파일을 경계하는 이유
"큰 common.js 파일이나 util.js 파일은 라이브러리 재사용성을 좋게 해주지 못한다"는 문장은 중요하다. 작성자는 중복 제거 자체보다, **무분별한 공용화가 오히려 경계를 흐리고 맥락을 잃게 만든다**는 점을 경계한다.

## 언어별 메모를 묶어서 읽는 법
원문 후반부는 Python, Golang, Dart, 함수형 프로그래밍, Javascript, Kotlin 메모가 길게 이어진다. 세부 항목은 방대하지만, 카테고리 노드로서 읽을 때는 아래 축으로 압축하는 편이 유용하다.

### Python
- 아름다움, 명시성, 단순함에 대한 선호
- 객체 모델, 복사, 참조, 메모리, 내장 함수에 대한 호기심
- 실용성이 순수성보다 앞선다는 태도

### Golang
- 명시적 에러 처리, 빠른 리턴, 단순한 패키지 목적
- 동시성을 라이브러리 내부 마법이 아니라 호출자 설계 문제로 보는 시각
- HTTP 핸들러, context, defer, 모듈 관리 같은 운영형 관심사

### Dart / Javascript / Kotlin / 함수형 메모
- 비동기와 실행 모델 이해
- 함수형 표현이 추론 가능성과 테스트 용이성을 높이는지에 대한 탐색
- 언어가 기본으로 막거나 권장하는 구조를 왜 그렇게 설계했는지에 대한 관심

결국 다양한 언어 메모는 산만한 부록이 아니라, 작성자가 **"좋은 코딩 원칙이 언어마다 어떻게 다르게 구현되는가"**를 계속 비교하고 있다는 신호다.

## 이 노트에서 반복되는 숨은 패턴
### 1. 교과서 개념을 실전 질문으로 다시 번역한다
예: SOLID를 외우는 데서 멈추지 않고, 테스트 용이성이나 변경 이유 분리로 다시 읽는다.

### 2. 구현보다 운영 시나리오를 먼저 걱정한다
예: 에러 처리, 부하 테스트, 블루그린 배포, 경계 조건, 최악의 상황 대비.

### 3. 언어 기능보다 읽는 사람의 비용을 중시한다
예: 명시적 에러 처리, 명확한 컨벤션, util 남용 경계, main에는 흐름만 드러내기.

### 4. 추상화의 목적을 교체 가능성과 검증 가능성에 둔다
예: DIP, 인터페이스, 구성, 테스트 더블, 패키지 API 고정.

## 지금 기준으로 다시 묶은 실천 체크리스트
이 노트를 실무용 기준으로 압축하면 아래 질문들로 재사용할 수 있다.

- 이 코드는 나중에 바꿀 때 수정 범위가 국소적인가?
- 인터페이스와 추상화가 멋 부리기가 아니라 테스트 가능성을 높이는가?
- 해피 패스 외의 경계 조건과 실패 시나리오를 생각했는가?
- 에러를 어디서 처리하고 어디서 전파할지 책임이 분명한가?
- 테스트가 리팩토링을 허용할 정도로 동작을 고정해주는가?
- 팀 컨벤션과 자동 검사가 읽기 비용을 줄여주는가?
- 공용화가 실제 재사용성인지, 아니면 맥락 없는 잡동사니 축적에 가까운지 점검했는가?
- 사용하는 언어의 철학을 거슬러 복잡성을 억지로 늘리고 있지 않은가?

## 후속 심화 포인트
이 페이지는 첫 강한 재구성은 끝났지만, 이후 점진 심화 여지는 남아 있다.

- Python / Go / JS 메모를 "언어 철학 비교" 관점으로 더 압축할 수 있다.
- 테스트 파트를 "개발 단계별 검증 체계"로 재정리할 수 있다.
- 에러 처리 파트를 "복구 가능성"과 "호출자 책임" 기준으로 더 선명하게 다듬을 수 있다.
- 원문 하단의 세부 언어 메모 중 반복되는 항목은 별도 패턴 묶음으로 더 정리할 수 있다.
