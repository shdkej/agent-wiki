# Tool

- 원본: `source/shdkej-content/Fundamental/Tool.md`
- 성격: 만다라트 `Fundamental` 축의 1:1 대응 카테고리 노드
- 이번 상태: 얇은 1차 대응 노트에서 기준선 수준의 구조화 페이지로 재구성

## 한눈에 보는 핵심
이 노트는 리눅스 명령어 치트시트나 툴 소개 모음처럼 보이지만, 실제로는 더 넓은 주제를 다룬다. 원문은 Linux, GIT, ETC로 나뉘어 있으나 중심 관심은 단순한 명령어 암기가 아니라 다음 네 가지에 가깝다.

1. 작업 환경은 한 번 세팅하고 끝나는 배경이 아니라, 계속 손보며 재구축 가능한 운영 자산이다.
2. 툴은 화려한 기능보다 설치, 권한, 경로, 백업, 자동화, 원격접속, 복구처럼 실제 운영 마찰을 줄이는 쪽에서 의미가 커진다.
3. 로컬 개발환경, 서버 운영, 배포, 보안, 백업, Git 사용은 분리된 주제가 아니라 하나의 작업 시스템으로 이어져 있다.
4. 좋은 툴 메모는 제품 카탈로그보다, 다시 같은 환경을 만들고 문제를 덜 겪게 해 주는 실행 기억에 가깝다.

## 핵심 논지
원본의 `Tool`은 단순한 유틸리티 목록이 아니다. 더 정확히는, 개발자 개인이 일하고 운영하고 복구하기 위해 몸에 붙여야 하는 환경 조작법과 작업 기억의 집적지다. bash 조건문, cron, ssh, backup, iptables, git, terraform, ansible, grpc, serverless, vault 같은 항목이 한 문서 안에 뒤섞여 있는 이유도 여기에 있다. 저자에게 `tool`은 특정 앱이나 명령어 이름보다, 일을 지속 가능하게 만드는 손잡이 전체를 뜻한다.

## 원본에서 먼저 잡히는 큰 구조

### 1. 출발점은 Linux를 다루는 손 감각이다
문서 맨 앞은 bash script, startup file, 한글 입력기, distro 선택, 파일 다운로드, sudo, tar backup, iptables처럼 매우 기초적이고 운영적인 메모로 시작한다. 즉 이 노드의 토대는 GUI 도구 선택보다 쉘과 운영체제의 기본 마찰을 줄이는 데 있다.

### 2. 중간부터는 개인 서버 운영 메모가 본격적으로 커진다
SSH 보안, DB 백업 cronjob, Postgresql replication, DNS, storage, NFS, FTP, TLS, 와이파이, crontab, Docker, code-server, WSL 같은 항목이 계속 이어진다. 즉 툴은 생산성 앱이 아니라 서버와 개발환경을 움직이는 운영 도구군으로 읽는 편이 맞다.

### 3. Git은 별도 대주제일 만큼 중요하게 다뤄진다
원문 후반의 큰 축 하나는 Git이다. 사용자 정보 설정, 브랜치 운영, SSH 키, hooks, rebase, reflog, stash branch, filter-branch, BFG까지 이어지는 메모는 버전 관리가 단순 저장이 아니라 복구와 협업의 핵심 도구라는 감각을 보여 준다.

### 4. ETC 구간은 잡다함이 아니라 경계 확장 기록이다
ETC에는 nginx 대안, ansible, terraform, serverless, vault, nagios, lambda cronjob, gRPC, Kafka, API gateway, 동영상 스트리밍 같은 항목이 나온다. 겉보기엔 산만하지만, 실제로는 툴 노드가 어디까지 확장되는지 보여 준다. 저자에게 툴은 편집기나 앱만이 아니라, 배포 시스템, 인프라 선언, 메시징, 비밀관리, 모니터링, 스트리밍 스택까지 포함한다.

## 이 노드가 실제로 다루는 질문
- 작업 환경을 다시 만들 수 있게 하려면 무엇을 기록해야 하는가?
- 툴 사용법보다 설치, 권한, 경로, 백업, 재시작 절차가 왜 더 중요해지는가?
- 로컬 도구와 서버 도구, 인프라 도구, 협업 도구는 어떻게 하나의 작업 흐름으로 이어지는가?
- 좋은 툴 선택은 최신 유행보다 어떤 마찰을 줄이는가로 판단해야 하는가?
- 반복 작업을 자동화할 때 어디까지 스크립트로 밀어야 하고, 어디서부터 운영 복잡도가 커지는가?
- 툴 메모를 단순 목록이 아니라 복구 가능한 운영 기억으로 남기려면 어떤 구조가 필요한가?

## 구조 재정리

### 1. 이 노드의 중심은 `기능`보다 `환경을 손에 넣는 법`이다
원문은 처음부터 설치 명령어, startup file, timezone, 한글 입력기, keyboard mapping, package 설치 여부 확인 같은 메모를 빼곡히 쌓는다. 이는 툴을 추상적 생산성 향상 수단보다, 내가 쓰는 환경을 직접 조정할 수 있게 만드는 레버로 본다는 뜻이다.

즉 이 노드의 기본 질문은 `무슨 앱이 좋은가`보다 `내가 원하는 상태를 다시 만들 수 있는가`에 가깝다.

### 2. Linux 메모는 운영체제 자체를 도구상자로 쓰는 감각을 보여 준다
bash 조건문, `set -u -e`, `find -mtime`, `tar`, `awk`, `sed`, `cut`, `xargs`, `lsof | awk | xargs kill -9` 같은 메모는 단순 스니펫 축적이 아니다. 저자는 GUI를 넘어서 운영체제를 조합 가능한 조작면으로 다루려 한다.

여기서 중요한 반복 패턴은 다음과 같다.
- 작은 명령어를 조합해 반복 작업을 줄인다.
- 파일, 프로세스, 권한, 시간, 경로를 쉘 레벨에서 직접 다룬다.
- 자동화는 거대한 시스템보다 먼저 작은 스크립트와 cron에서 시작한다.

### 3. 환경 설정 메모는 취향보다 재진입 비용 절감과 가깝다
한글 입력기, 터치패드, 키보드 딜레이, 외장 모니터, 폰트 깨짐, i3, w3m, battery, WSL, 테마, copy/paste 같은 항목은 사소해 보이지만 실제로는 중요하다. 이들은 코드를 직접 만들지 않더라도 하루 작업 리듬을 계속 방해하는 마찰이기 때문이다.

즉 이 노드는 툴을 생산성 앱 추천으로 정리하지 않고, 개발자가 오래 버틸 수 있게 해 주는 환경 마찰 제거 메모로 축적한다.

### 4. 백업과 cron 메모는 `지속 가능성`을 툴의 본질로 본다
`tar` 백업, 날짜 형식, 오래된 파일 삭제, `crontab -e`, 시스템 cron과 사용자 cron 구분, DB dump cronjob 같은 항목은 툴 메모를 운영 메모로 바꾼다. 핵심은 지금 잘 쓰는 것보다 나중에 잃지 않고 다시 돌릴 수 있는 상태를 만드는 것이다.

여기서 툴의 가치는 즉시성보다 지속성에 가깝다.
- 자동화는 귀찮음 감소이기도 하지만, 누락 방지와 복구 준비이기도 하다.
- 백업은 저장이 아니라 원복 가능성 확보다.
- cron은 편의 기능이 아니라 반복 운영의 최소 단위다.

### 5. 보안과 접근 메모는 도구를 `경계 관리`의 문제로 확장한다
SSH 설정, 포트 변경, root 접속 제한, fail2ban, sudoers, 파일 권한, iptables, TLS, ssh-keygen, github without password 같은 메모는 툴을 단순 사용성 문제가 아니라 접근 경계와 신뢰 경계 문제로 확장한다.

즉 이 노드에서는 도구를 잘 쓰는 것이 단순 빠른 작업을 의미하지 않는다. 누가 접근할 수 있는지, 어떤 키를 쓰는지, 어떤 포트를 여는지, 권한이 어디까지인지까지 함께 설계해야 한다.

### 6. 데이터베이스, DNS, 스토리지 메모는 `툴`과 `인프라`의 경계를 흐리게 한다
Postgresql 설치와 replication, Oracle DB 명령, DNS zone file, NFS, quota, LVM, VirtualBox 디스크 확장 같은 항목은 전형적인 인프라 메모다. 그런데 이 문서 안에 들어 있다는 것은 저자가 도구를 앱 레벨보다 훨씬 넓게 본다는 신호다.

여기서 툴은 다음처럼 이해된다.
- 시스템을 굴리기 위한 구성 요소
- 상태를 저장하고 전달하기 위한 기반
- 장애가 나면 직접 확인하고 고칠 수 있어야 하는 대상

즉 툴은 손에 익은 소프트웨어라기보다, 내가 다뤄야 하는 운영 표면 전체다.

### 7. Git 메모는 복구와 협업의 운영 기억 창고다
원문의 Git 구간은 특히 중요하다. 단순한 `add-commit-push`를 넘어서, rebase, merge abort, stash branch, reflog, 브랜치 이동, 민감 파일 삭제, hooks, SSH config, fast-forward 설정 문제까지 기록되어 있다.

이 구간이 보여 주는 핵심은 다음과 같다.
- Git은 코드 저장보다 히스토리 조작과 복구의 도구다.
- 브랜치는 기능 분리뿐 아니라 실수에서 되돌아오는 안전장치다.
- SSH 설정과 remote URL은 사용성보다 지속적인 접속 안정성을 좌우한다.
- 꼬인 히스토리를 푸는 능력은 일상적인 개발 생산성의 일부다.

즉 이 노드에서 Git은 협업 도구인 동시에, 자기 작업을 잃지 않기 위한 복원 도구다.

### 8. ETC 구간은 `도구를 조합해 시스템을 만드는 단계`를 보여 준다
ansible, terraform, serverless, vault, nagios, lambda cronjob, gRPC, API gateway, Kafka, 동영상 스트리밍 메모는 단일 프로그램 사용법을 넘어선다. 여기서는 도구가 서로 연결되어 하나의 서비스 운영 체인을 이룬다.

이 층위에서 중요한 반복 질문은 다음과 같다.
- 선언형으로 관리할 것인가, 수동 명령으로 둘 것인가?
- 빠르게 띄우는 것과 장기 유지 가능한 구조 중 어디에 무게를 둘 것인가?
- 비밀정보, 배포, 모니터링, 이벤트, 통신 방식을 각각 어떤 층에서 담당하게 할 것인가?

즉 `Tool` 노드는 결국 `작업 도구`에서 시작해 `운영 시스템 설계 도구`까지 확장되는 다리 역할을 한다.

## 반복되는 판단 축

### 편의성 vs 재현성
쉽게 한 번 되는 것보다, 나중에 다시 똑같이 만들 수 있는가가 더 중요하게 취급된다. dotfile, shell config, cron, terraform, ssh config가 반복 기록되는 이유다.

### GUI vs CLI
GUI는 빠르지만 기록이 남지 않기 쉽고, CLI는 번거롭지만 재실행과 자동화가 쉽다. 원문은 계속 CLI 쪽으로 무게중심이 기울어 있다.

### 단발 설정 vs 운영 기억
설치 명령 하나보다 왜 그렇게 했는지, 장애가 났을 때 무엇을 확인해야 하는지 남기는 메모가 오래 남는다. 이 문서 전체가 그런 방향으로 움직인다.

### 로컬 작업성 vs 서버 운영성
키보드, 터치패드, 폰트, WSL 같은 로컬 메모와 DB, DNS, NFS, TLS, monitoring 같은 서버 메모가 한 문서 안에 공존한다. 이는 개발자의 실제 작업이 둘로 분리되지 않기 때문이다.

### 빠른 해결 vs 구조적 유지보수
문제 해결을 위해 재설치, 직접 수정, 응급조치를 자주 쓰지만, 동시에 terraform, ansible, hooks, backup, cron처럼 구조적 재현성을 확보하려는 욕구도 강하다. 이 긴장이 이 노드의 중요한 특징이다.

## 원문에서 특히 보존해야 할 부분

### bash / shell 기초 메모
이 부분은 사소한 스니펫 모음처럼 보이지만, 전체 노드의 토대다. 작은 자동화와 상태 조작을 직접 할 수 있게 해 주는 최소 문법이 여기 있다.

### 백업, cron, 보안 설정 메모
이 구간은 툴을 단순 사용성보다 운영 지속성으로 읽게 만든다. 특히 백업과 cron은 `잃지 않기`, SSH와 iptables는 `열어 두되 통제하기`의 감각을 잘 보여 준다.

### Git의 꼬인 상황 메모
단순 명령어 설명보다 실제로 꼬였을 때 어떻게 복구할지 고민한 흔적이 중요하다. 이 부분이야말로 툴 메모가 현장 운영 기억이라는 점을 보여 준다.

### terraform / ansible / serverless / vault / nagios 메모
여기는 원문이 `개인 도구 사용법`에서 `시스템 조합 능력`으로 넘어가는 지점이다. 이후 인프라, DevOps, Monitoring 노드와 강하게 이어진다.

## 오래 남길 아이디어
- 툴은 기능 목록보다 작업 환경과 운영 상태를 직접 조정할 수 있게 해 주는 손잡이다.
- 좋은 툴 메모는 사용법보다 재설치, 재구성, 권한, 경로, 백업, 복구 정보를 남긴다.
- 작은 shell 스크립트와 cron은 거대한 플랫폼 이전에 갖춰야 할 가장 현실적인 자동화 단위다.
- Git은 저장 도구가 아니라 복구와 협업을 위한 시간 관리 도구다.
- 로컬 환경 최적화와 서버 운영 메모는 분리된 취미가 아니라 하나의 작업 시스템 양면이다.
- 툴 선택의 핵심 질문은 `최신인가`보다 `마찰을 줄이고 다시 만들 수 있게 해 주는가`에 가깝다.

저자는 툴을 멋진 신제품보다 마찰을 줄이고 다시 세팅할 수 있게 만드는 운영 표면으로 보는 경향이 강하다. 또한 명령어와 설정을 수집할 때도 단순 치트시트보다 백업, 권한, 네트워크, 자동화, 복구처럼 실패를 줄이는 정보에 더 무게를 둔다. 즉 툴을 소비하는 사람이라기보다, 툴을 조합해 자기 작업 환경을 길들이는 사람에 가깝다.

## 정리 메모
- 얇은 1차 대응 노트를 한국어 기준의 구조화된 툴 카테고리 노드로 전면 재구성했다.
- 원문 전체를 다시 읽고 Linux 기초, 환경 마찰 제거, 백업/보안, Git 복구, 운영 도구 확장이라는 흐름으로 재배열했다.
- 툴을 앱 목록이나 치트시트로 납작하게 만들지 않고, 재현 가능한 작업 환경과 운영 기억을 만드는 기반 노드로 읽히게 정리했다.
