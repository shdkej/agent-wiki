# Container

- 원본: `source/shdkej-content/Deep Knowledge/Container.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 컨테이너를 단순한 실행 포맷이 아니라, **격리된 실행 환경을 표준화하고 개발·배포·운영 흐름을 재현 가능하게 만드는 기술 묶음**으로 다룬다. 메모의 관심사는 Docker 명령 암기보다, 이미지와 볼륨의 역할 구분, Dockerfile 신뢰성, stdout 기반 로그 수집, 쿠버네티스의 추상화 계층, 그리고 핫픽스·롤백까지 포함한 운영 체계를 어떻게 설계할지에 있다.

## 이 노드가 실제로 답하려는 질문
- 컨테이너는 왜 VM보다 가볍고 이식 가능한가?
- 개발 단계와 배포 단계에서 이미지와 볼륨을 어떻게 다르게 써야 하는가?
- Dockerfile은 단순 빌드 스크립트가 아니라 어떤 운영 계약을 담아야 하는가?
- 쿠버네티스는 컨테이너 위에 어떤 복잡성을 추가하고, 또 무엇을 단순화하는가?
- 결국 중요한 것은 컨테이너 자체가 아니라, 재빌드·재배포·롤백이 가능한 운영 루프 아닌가?

## 핵심 논지
1. **컨테이너의 본질은 격리와 재현성이다.**
   - namespace, cgroups, overlayFS, chroot 같은 리눅스 기능을 바탕으로 프로세스와 자원을 분리한다.
   - VM처럼 OS 전체를 복제하지 않으면서도 실행 환경을 일정하게 묶는다.
2. **이미지와 볼륨은 목적이 다르다.**
   - 이미지는 배포 가능한 결과물이다.
   - 볼륨은 개발 중 빠른 수정과 실험을 돕는다.
   - 따라서 원문의 반복 메시지는 결국 `개발은 볼륨, 배포는 이미지`로 수렴한다.
3. **Dockerfile은 신뢰성과 추적 가능성을 보장하는 문서다.**
   - 무엇을 설치했고 무엇이 바뀌었는지 남긴다.
   - 루트 사용 최소화, 체크섬 검증, 레이어 정리 같은 습관은 보안과 배포 속도에 모두 연결된다.
4. **쿠버네티스는 컨테이너를 대량 운영하기 위한 제어 계층이다.**
   - 배포, 스케일링, 헬스체크, 서비스 디스커버리, 롤백, 권한 관리 같은 문제를 다룬다.
   - 하지만 복잡성을 제거한다기보다 더 높은 수준의 복잡성으로 재배치한다.
5. **실서비스에서 중요한 것은 "서버에서 바로 수정"이 아니라 재배포 가능성이다.**
   - 실서버 직접 수정은 편하지만 재현성과 신뢰성을 무너뜨린다.
   - 그래서 핫픽스와 롤백이 빠른 파이프라인이 필요하다는 결론으로 이어진다.

## 구조 재정리

### 1. 컨테이너의 기반 원리
원문은 컨테이너를 설명할 때 기술 스택을 아래처럼 쌓아 올린다.

- `chroot`: 루트 파일시스템을 분리한다.
- `namespace`: pid, net, mnt, ipc, uts, user 등 자원을 격리한다.
- `cgroups`: CPU, 메모리, 디스크, 네트워크 사용량을 제한한다.
- `overlayFS`: 이미지 레이어와 컨테이너 변경 레이어를 분리한다.

이 배열은 "컨테이너가 왜 가볍고 portable한가"에 대한 실제 기술적 설명이다. 즉, 이 노드는 컨테이너를 마법처럼 보지 않고 **리눅스 커널 기능을 조합한 결과물**로 이해하려고 한다.

### 2. Docker를 보는 관점
원문은 Docker를 편의 도구로만 보지 않는다.

- 장점: isolation, dependency 분리, portability, light-weight
- 한계: 버전 차이, 거대한 단일 프로세스 구조, docker.sock 위험
- 대안 흐름: OCI, runC, cri-o, podman, Buildah, skopeo

여기서 중요한 패턴은 특정 도구 충성도가 아니라 **표준화와 역할 분리**다. Docker가 시작점이지만, 결국 관심은 OCI 준수, 런타임 분리, 빌드/실행/배포 역할의 해체 쪽으로 이동한다.

### 3. 개발 환경과 배포 환경의 분리
원문에서 가장 자주 되돌아오는 실무 문제는 이것이다.

- 개발할 때는 코드를 즉시 반영하고 싶다.
- 배포할 때는 고정된 이미지로 안정적으로 올리고 싶다.
- compose와 Dockerfile, volume과 COPY, build target과 runtime target이 여기서 충돌한다.

정리하면 다음 판단으로 읽힌다.

| 상황 | 우선 수단 | 이유 |
|---|---|---|
| 로컬 개발 | volume + compose | 빠른 수정, 즉시 반영 |
| 테스트/빌드 | Dockerfile multi-stage | 재현 가능한 빌드 |
| 배포 | 고정 이미지 + 태그 | 추적 가능, 롤백 가능 |
| 운영 수정 | 서버 직접 수정 지양 | 재현성 붕괴 방지 |

이 노드의 실전 메시지는 분명하다. **개발 편의성과 배포 안정성은 같은 방식으로 얻지 않는다.**

### 4. 로그와 운영 가시성
초반 메모에서 stdout, `docker logs`, Fluentd, Elasticsearch, nginx 로그 설정이 먼저 등장하는 점이 중요하다. 이는 컨테이너를 단순 실행 단위가 아니라 **관측 가능한 실행 단위**로 본다는 뜻이다.

- 로그는 파일보다 stdout으로 내보내고 수집하도록 설계한다.
- 애플리케이션과 인프라의 경계를 나누고, 수집은 외부 시스템이 담당한다.
- 쿠버네티스에서도 이 관점은 그대로 이어져 DaemonSet 로그 수집, 모니터링 스택, 메트릭 서버, Grafana, Prometheus로 확장된다.

즉, "컨테이너를 띄운다"는 말은 사실상 "로그와 상태를 밖에서 읽을 수 있게 만든다"까지 포함한다.

### 5. Dockerfile을 보는 기준
원문은 Dockerfile을 세세하게 적어 두었는데, 반복되는 기준은 다음과 같다.

- 체크섬 검증으로 신뢰성을 높인다.
- root 대신 일반 사용자를 기본값으로 둔다.
- 잘 변하지 않는 설치 단계는 묶어서 레이어를 줄인다.
- 빌드 산출물과 런타임 산출물을 분리한다.
- `docker.sock` 노출 위험을 경계한다.

이 기준은 단순 최적화가 아니라 **보안, 속도, 재현성의 교차점**에 있다. 저자는 Dockerfile을 "이미지 만드는 파일"보다 "운영 원칙이 새겨지는 경계면"으로 다룬다.

## 쿠버네티스 파트의 의미
원문 후반의 대부분은 사실상 쿠버네티스 운영 노트다. 여기서 읽히는 핵심은 다음과 같다.

### 쿠버네티스가 제공하는 것
- 배포와 업데이트 자동화
- 스케일 조정
- 서비스 디스커버리
- 롤링 업데이트와 롤백
- 서비스, 인그레스, 로드밸런서 추상화
- RBAC, ServiceAccount 등 권한 관리
- namespace, controller, operator를 통한 구조화

### 쿠버네티스가 새로 만드는 복잡성
- 네트워크 계층 이해 필요 (Service, NodePort, Ingress, kube-proxy, netfilter)
- 저장소와 볼륨 관리 복잡성 증가
- 보안 정책과 권한 모델이 어려워짐
- 디버깅 경로가 길어짐
- 클러스터 규모가 커질수록 control plane 한계와 운영 부담이 드러남

따라서 이 노드는 쿠버네티스를 "다 해결해 주는 플랫폼"으로 보지 않는다. 오히려 **단일 서버의 수작업 문제를 더 큰 규모에서 관리 가능한 규칙으로 바꾸는 도구**로 본다.

## 반복되는 긴장 관계

### 편의성 vs 재현성
- 볼륨 마운트와 서버 직접 수정은 편하다.
- 하지만 나중에 동일 상태를 다시 만들기 어렵다.

### 추상화 vs 통제력
- 클라우드, 쿠버네티스, 서비스 메시가 많은 것을 감춘다.
- 대신 내부 동작을 깊이 이해하지 않으면 장애 원인 추적이 어려워진다.

### 단순 배포 vs 운영 체계
- docker-compose로도 시작은 가능하다.
- 그러나 실제 운영으로 가면 이미지 버전, 핫픽스 규칙, 롤백, 권한, 모니터링, 테스트 체계가 필요하다.

### 단일 클러스터 집중 vs 멀티 클러스터 분리
- 한 클러스터에 몰아 넣으면 관리가 쉬워 보인다.
- 그러나 규모가 커질수록 환경 분리와 책임 분리가 필요해진다.

## 숨은 패턴
- 이 노드는 컨테이너 자체보다 **배포 체계 설계 노드**에 가깝다.
- 메모의 질문 대부분이 "이 기술이 무엇인가"보다 "실무에서 어디서 꼬이는가"에 집중되어 있다.
- 이미지, 쿠버네티스, Istio, Helm, ArgoCD, Spinnaker, k3s, EKS 같은 키워드가 흩어져 있지만, 모두 **재현 가능한 운영 흐름을 만들기 위한 도구 후보군**으로 등장한다.
- 따라서 이 노드의 핵심 관심사는 기능 설명이 아니라 **운영 판단 능력**이다.

## 인접 노드와의 연결
- [[AI]]: AI 에이전트를 실제 서비스로 운영하려면 결국 컨테이너, 배포, 관측, 롤백 체계가 필요하다.
- [[Cloud]]: 컨테이너는 클라우드 자원 위에서 실행되며, 클라우드는 실행 기반을 제공하지만 운영 단순화의 전부는 아니다.
- [[Infra]]: 인프라는 어떤 구성 요소를 둘지의 문제이고, Container는 그 요소들을 어떻게 패키징하고 운용할지의 문제다.
- [[Devops]]: Container는 DevOps의 핵심 실행 단위이며, Devops 노드는 이 노드를 팀 프로세스와 배포 문화까지 확장한다.
- [[Web]]: 실제 웹서비스 운영에서는 ingress, TLS, CDN, API gateway와 연결되어 앞단 트래픽 구조와 만난다.

## 이후 심화 포인트
- Docker, OCI 런타임, Podman 계열의 역할 차이를 더 선명하게 정리할 수 있다.
- 쿠버네티스 파트를 `네트워크`, `배포`, `권한`, `운영 실수` 축으로 한 번 더 압축할 수 있다.
- 이미지 버저닝, 핫픽스, 롤백 규칙을 이 노드만의 운영 체크리스트로 추출할 수 있다.

## 이번 갱신 메모
- 초안형 대응 노트를 한국어 기준의 구조화된 문서로 전면 재구성했다.
- 원문의 산발적 메모를 `격리 원리 → 개발/배포 분리 → Dockerfile 기준 → 쿠버네티스 운영 → 반복 긴장` 흐름으로 다시 묶었다.
- 컨테이너를 단순 도구 설명이 아니라 재현 가능한 운영 체계를 설계하는 카테고리 노드로 읽히게 만들었다.
