# My_space

- 원본: `source/shdkej-content/Meta/My_space.md`
- 상태: 안정 디벨롭
- 역할: 원본 만다라트 노드와 1:1 대응하는 개인 운영 환경 설계 페이지

## 핵심 해석
`My_space`는 개인 서버 구축 메모가 아니라, 내가 만든 서비스와 실험을 오래 굴릴 수 있는 작은 운영 공간을 어떻게 만들지 묻는 메타 노드다. 핵심은 서버를 소유하는 것이 아니라, 앱을 쉽게 올리고, 상태를 보고, 망가지면 다시 세우고, 비용 한계 안에서 계속 재구축할 수 있는 운영 상태를 갖추는 데 있다.

원문은 Kubernetes, Oracle Cloud, DigitalOcean, ArgoCD, Terraform, SNS/SQS/Lambda, minikube, Docker Compose 같은 도구를 많이 담는다. 하지만 중심 문장은 `앱은 이미지만 만들고 배포는 인프라에서 처리한다`에 가깝다. 애플리케이션 저장소는 코드와 이미지 생성에 집중하고, 실제 배포와 복구는 `space`라는 공용 운영면이 맡아야 한다는 생각이다.

`Meta > My_space`라는 위치에서 보면 이 노드는 [Fundamental/Infra](/docs/mapped/Fundamental/Infra)의 구성 요소 지식, [Devops](/docs/mapped/deep-knowledge/devops)의 배포·복구 문법, [blog/My_space](/docs/mapped/blog/My_space)의 외부 공유용 시행착오를 한곳에서 다시 조립한다. 개인 홈랩이지만, 실제 관심은 작은 규모의 플랫폼 엔지니어링 감각이다.

## 원문 발췌 구조
- 목표 선언: `확장 가능하고 변경이 쉬운 환경`, `나만의 환경`, `Kubernetes를 직접 사용하며 깊게 이해한다`.
- 요구사항: IaC, 쉬운 접속과 배포, 빠른 재구축, 에러를 한눈에 보는 화면, 앱/배포 책임 분리.
- 진행 이력: 관리형 Kubernetes 비용 부담, DigitalOcean 저사양 한계, Oracle ARM + k3s 재시도.
- 최소 인프라: 3노드 Kubernetes, 로드밸런서, 로그 수집, 모니터링, API 서버, feedback, CI/CD 도구.
- 검증 과제: 롤링 업데이트, 롤백, 카나리, 스트레스 테스트, 오토스케일링, 페일오버, MLOps/gRPC 배포.
- 실제 워크로드: 텔레그램 봇, SNS, SQS, Lambda, note-reminder 메시징 파이프라인.
- 2026 운영면: Oracle 인스턴스 역할 분리, ArgoCD auto sync, 프로젝트 CI가 `space` 레포의 태그를 바꾸는 배포 흐름.
- 개발환경 고민: minikube와 Docker Compose 사이에서 운영 재현성과 첫 실행 접근성을 함께 비교한다.

## 판단 모델 1. 내 공간은 서버가 아니라 상태 전달 경로다
원문에서 `개인 서버 구축`은 물리 장비나 특정 클라우드 계정을 뜻하지 않는다. 더 정확히는 코드 변경이 이미지가 되고, 이미지 태그가 운영 선언으로 남고, ArgoCD가 클러스터 상태를 맞추며, 로그·모니터링·알림이 이상 징후를 되돌려 주는 경로를 뜻한다.

2026 세팅 메모는 이 변화가 가장 선명하다.

| 층 | 원문 장면 | 의미 |
| --- | --- | --- |
| 코드/이미지 | 프로젝트 레포가 CI template을 가져가 빌드한다 | 앱은 실행 가능한 산출물까지만 책임진다 |
| 운영 선언 | `space` 레포의 `argocd/apps` 태그를 커밋으로 갱신한다 | 배포 의도가 손동작이 아니라 기록으로 남는다 |
| 동기화 | ArgoCD auto sync가 설정 파일 변경을 반영한다 | 실제 상태가 선언된 상태를 따라간다 |
| 관측 | 로그, 모니터링, 상태 알림, 오버로드 알림을 붙인다 | 운영자는 현재 상태를 다시 읽을 수 있다 |
| 복구 | 롤백, 재구축, 리소스 조정, 버전 업그레이드 테스트를 둔다 | 망가진 뒤 다시 들어올 길을 남긴다 |

따라서 `My_space`의 공간성은 서버 묶음보다 상태가 지나가는 표면에 있다. 나중에 서버를 갈아엎어도 `어떤 앱을 어떤 이미지로 어떤 공간에 띄울지`가 커밋과 동기화 경로에 남아 있으면 다시 시작할 수 있다.

## 판단 모델 2. 비용과 리소스 부족은 운영 상한선을 알려 준다
이 노트는 Kubernetes를 이상화하지 않는다. Cloud 호스팅은 월 10만원 이상이라 무산되고, DigitalOcean 관리형 Kubernetes는 낮은 사양에서 제대로 돌지 않으며, EKS는 클러스터 운영비만 월 72달러 수준이라 개인 실험에는 부담스럽다. Grafana와 OpenTelemetry도 실제로 올리면 메모리를 잡아먹고, k3s 자체도 작은 인스턴스에서는 큰 고정비가 된다.

이 시행착오는 실패담이 아니라 개인 플랫폼의 상한선을 재는 데이터다.

- `.terraform` 상태 오염 때문에 access key가 맞지 않는 문제를 겪는다.
- SSH key permission, Ansible interpreter, provider 버전 차이에서 막힌다.
- ingress, DNS, Traefik, Grafana 연결을 맞추는 데 시간이 든다.
- 리소스가 부족하면 ingress controller와 관측 도구가 뜨지 않거나 오래 흔들린다.
- DigitalOcean DNS와 휴대폰 접속 확인처럼 외부에서 실제로 닿는지도 확인한다.

그래서 이 노드의 좋은 정리는 최종 성공 아키텍처만 남기는 것이 아니다. 비용, 메모리, 네트워크, DNS, provider 상태, 관측 도구 무게처럼 작은 운영 환경을 실제로 제한하는 항목을 남겨야 다음 설계의 하한과 상한을 알 수 있다.

## 판단 모델 3. DevOps 목표는 중앙 플랫폼보다 낮은 진입·복구 표면이다
원문 후반의 `docker compose 대신 minikube` 고민은 Kubernetes 학습 욕구만의 문제가 아니다. 운영 환경과 같은 문법을 쓰면 배포, 볼륨, 내부/외부 접속, yaml 설정을 실제 운영 루프와 이어서 연습할 수 있다. 하지만 과제 제출, 오픈소스 공유, 동료 온보딩처럼 처음 실행하는 사람에게는 Docker Compose가 더 낮은 진입 표면일 수 있다.

이 긴장은 데브시스터즈와 카카오의 관리페이지 사례와도 이어진다. 개발자가 중앙 자원을 쉽게 요청하고, 필요한 DB나 Kafka를 안전하게 붙이고, 배포 결과를 확인할 수 있다면 Kubernetes 자체를 몰라도 운영에 접근할 수 있다. 반대로 집에서 혼자 구축해도 같은 환경을 재현할 수 있어야 진짜 이해가 남는다는 감각도 있다.

따라서 이 노드에서 DevOps는 거대한 환경을 만들고 모두를 그 안에 가두는 역할이 아니다. 개발자가 필요한 자원을 얻고, 실행 상태를 보고, 실패하면 되돌아올 수 있게 요청 표면, 실행 표면, 복구 표면을 낮추는 인터페이스 설계에 가깝다.

## 특히 보존할 원문 앵커
- `앱은 이미지만 만들고 배포는 인프라에서 처리한다`: 앱 저장소와 운영 저장소의 책임 분리를 압축한 문장.
- Oracle ARM 2026 재시도: worker/system/실험용 역할을 나누고 k3s 기반 개인 운영면을 다시 세우려는 장면.
- ArgoCD auto sync: `argocd/apps` 변경과 태그 커밋이 실제 배포로 이어지는 GitOps형 흐름.
- SNS/SQS/Lambda/텔레그램 파이프라인: 개인 공간이 정적 서버가 아니라 알림과 자동화 워크로드가 흐르는 허브임을 보여 주는 사례.
- DigitalOcean 시행착오: provider 버전, `.terraform` 상태, SSH permission, Traefik, Grafana, DNS, 리소스 부족이 실제 운영 마찰로 남은 기록.
- minikube vs Docker Compose: 운영 재현성과 첫 실행 접근성 사이의 DevOps 경계 질문.
- EKS 업그레이드 전략: 테스트 클러스터, 워크로드 복제, healthcheck/e2e, addon 호환성, `kubectl convert`, `kubescape`로 업그레이드 완료 기준을 나누는 메모.
