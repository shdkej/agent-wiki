# About_Architecture

- 원본: `source/shdkej-content/Meta/About_Architecture.md`
- 상태: 기준선 유지 + 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 메타 아키텍처 페이지

## 한눈에 보는 요약
이 노드는 특정 아키텍처 패턴의 정답을 설명하는 문서가 아니라, 시스템을 어떻게 오래 운영 가능하게 만들 것인지에 대한 메타 설계 메모다. 핵심은 확장성 자체보다 **단일고장점 축소, 부분 교체 가능성, 기술 종속성 통제, 재구축 용이성, 모듈화의 적정선, 입력-로직-출력 구조의 일관성**에 있다. 즉 `좋은 아키텍처`를 예쁜 그림보다 **오래 버티고 다시 세우고 다른 도구로 갈아탈 수 있는 운영 구조**로 본다.

## 핵심 논지
원본에서 아키텍처는 화려한 설계도가 아니라 변화와 교체를 견디게 하는 기준선이다. 많이 분리하는 것이 늘 좋은 것도 아니고, 덜 의존하는 것이 늘 자유를 주는 것도 아니다. 대신 저자는 계속 같은 질문으로 돌아온다. **무엇을 고정하고 무엇을 바꿀 수 있게 둘 것인가, 어디까지 추상화하고 어디서 구체화할 것인가, 새로운 도구를 들여와도 전체를 여전히 내가 이해할 수 있는가.** 그래서 이 노드는 패턴 선택 메모보다 `오래 운영 가능한 시스템을 어떤 문법으로 키울 것인가`를 묻는 운영 설계 노드에 가깝다.

## 이 노드가 중요한 이유
- `Meta` 축에서 기술 선택 취향이 아니라 장기 운영 원칙을 드러내는 노드다.
- `[[Fundamental/Architecture]]`가 일반 설계 기준을 다룬다면, 이 페이지는 그 기준이 실제 개인 시스템 설계 습관으로 어떻게 번역되는지를 보여준다.
- 분산, 모듈, 라이브러리 분리, gRPC, 외부 서비스, 설치 최소화, 콘텐츠 독립성 같은 흩어진 메모들이 모두 `교체 가능하지만 통제 가능한 구조`라는 하나의 관심사로 모인다.
- `[[Meta/My_space]]`가 개인 운영 환경의 재구축을 다룬다면, 이 노드는 그 환경을 어떤 기준으로 설계해야 하는지의 상위 사고를 제공한다.

## 만다라트 위치에서 읽기
이 페이지는 `Meta` 폴더 안에서도 개별 기술 노하우보다 `설계 판단을 만드는 판단법` 쪽에 가깝다. 같은 만다라트의 `Meta/Collection`이 분화 전 감각과 비유를 모아두는 씨앗 창고라면, `Meta/About_Architecture`는 그 씨앗들 중 특히 **복잡성, 교체 반경, 공통 인터페이스, 복구 가능성**과 관련된 것들을 더 구조화해서 세우는 자리다. 그래서 이 노드는 `아키텍처 패턴 요약`보다 `어떤 구조를 오래 믿고 운영할 수 있는가`를 묻는 메타 카테고리 노드로 읽어야 한다.

## 원문 발췌 구조
- 확장성: SPOF 축소, 무상태화 지향, 묶음 단위 증설, 점진 교체 실험이 한 덩어리로 적혀 있다.
- portable / universal: 표준, 어댑터, 콘텐츠 독립성, 프레임워크 전환 가능성을 같은 가치로 묶는다.
- 모듈: 팀 규모, 인수인계 한계, 분리의 효용과 과잉 분리의 피로를 함께 적는다.
- 분산: 탈중앙화의 매력보다 관리 지점 폭증과 복잡성 상승을 같이 본다.
- 설계: 작은 실행에서 시작해 구조를 반복 증설하고, input-logic-output 같은 공통 문법을 만들려 한다.
- 외부 라이브러리/도구: 의존성 자체를 없애기보다 수정 반경을 모으고 복구를 쉽게 하려는 고민이 강하다.

## 원본에서 먼저 잡히는 구조

### 1. 확장은 트래픽 문제이기 전에 고장 반경 문제다
원본은 50명에서 10만 명으로의 확장을 상상하지만, 관심은 숫자 자체보다 장애가 어디까지 번지느냐에 있다.
- 단일고장점이 없으면 일부 실패를 감당할 수 있다.
- 서버를 하나씩 재부팅해도 전체가 유지되면 운영이 편해진다.
- 상태 저장 의존성을 줄이면 OS나 개별 장비 문제에서 조금 더 자유로워진다.
- 일부 장비만 바꿔보는 식의 점진 실험이 가능해야 한다.

즉 확장성은 더 크게 만드는 능력보다, **부분 실패를 허용한 채 운영을 이어가는 능력**에 가깝다.

### 2. 표준과 어댑터를 통해 새 기술 도입 자유를 확보하려 한다
원본은 회사 상황에 맞춘 과도한 커스터마이징보다 표준에 맞추는 방향을 선호한다.
- DB 원본은 지키고 어댑터로 감싼다.
- 새로운 기술을 배워도 기존 업무에 바로 붙일 수 있으면 좋다.
- 콘텐츠와 핵심 로직은 툴에 종속되지 않아야 한다.
- 프레임워크 전환 시 메인 로직이 아니라 연결부만 갈아끼우는 상태를 원한다.
- Gatsby에서 Next.js로 옮기는 상상을 하며, 플러그인 문법이 메인 로직을 잠식할 때 전환 비용이 급격히 커진다고 본다.

그래서 이 노드의 portability는 단순 이식성이 아니라, **새 도구를 써도 중심 자산을 보존하는 자유**에 가깝다.

### 3. 모듈화는 만능이 아니라 인간 한계를 다루는 방식이다
원본은 모듈화를 강하게 선호하지만 동시에 경계도 분명히 적는다.
- 기능이 커지면 분리해야 한다.
- 하지만 너무 잘게 쪼개면 전체를 잃는다.
- 팀이 작으면 깊어진 기능을 다시 나누기 어렵다.
- 메뉴얼이 있어도 기존 맥락을 완전히 대체할 수는 없다.
- 작은 단위를 모아 큰 시스템을 만들면 새로운 복잡성이 생긴다.

즉 모듈은 복잡도를 없애는 장치가 아니라, **인간이 감당 가능한 단위로 복잡도를 재배치하는 장치**다.

### 4. 추상화는 언어적 우아함보다 전환 비용 절감이 목적이다
원본은 언어를 넘어서는 추상화, 포팅을 쉽게 할 정도의 추상화, 외부 라이브러리 분리, 공통 입출력 형태를 반복해서 고민한다. 그러나 동시에 과한 추상화가 실체를 잃게 만든다는 걱정도 남긴다. 즉 추상화의 목적은 개념적 아름다움이 아니라 **교체 지점 축소, 영향 범위 통제, 재사용 가능한 인터페이스 확보**다.

### 5. 아키텍처는 정적 도면이 아니라 반복 가능한 성장 절차다
원본 후반의 `가장 간단한 실행 → 파일 추가 → 기능 추가 → 다시 설계` 메모는 중요하다. 이 노드에서 아키텍처는 처음부터 완성형으로 짜는 청사진이 아니라, **작게 시작하고 복잡성 신호가 보일 때 분리하는 성장 절차**다. `인프라 - 서비스 - 데이터` 구조가 다시 프렉탈처럼 반복된다는 메모도 같은 뜻이다. 저자는 구조를 한 번에 설계하기보다, **같은 분화 문법이 여러 스케일에서 반복될 수 있는지**를 본다.

### 6. 설치 최소화와 복구 가능성도 아키텍처 판단에 포함된다
원본 중간의 `외부 프로그램을 적게 쓰고 싶었다`는 고백은 취향 메모처럼 보이지만, 실제로는 이동성과 복구성을 둘러싼 설계 기준이다.
- 새 컴퓨터에서 원래 환경으로 돌아가기 쉬워야 한다.
- 특정 프로그램에 묶여 다른 선택이 막히는 상태를 싫어한다.
- 적게 설치한다고 다 기억되는 것은 아니며, 필요할 때는 과감히 쓰는 쪽으로 생각이 수정된다.

즉 이 노드에서 아키텍처는 서버나 코드만의 문제가 아니라, **도구 환경까지 포함한 재구축 가능성 설계**다.

## 이 노드를 묶는 운영 사다리
원문 전반을 다시 읽으면 흩어진 메모들이 대체로 아래 순서로 되풀이된다.

1. **작게 시작한다** — 가장 단순한 실행과 테스트 가능한 형태를 먼저 만든다.
2. **경계를 드러낸다** — input / logic / output, 모듈, 라이브러리, 데이터, API 같은 분리면을 세운다.
3. **교체 반경을 줄인다** — 어댑터, 표준, 공통 인터페이스로 특정 기술의 영향 범위를 좁힌다.
4. **증설과 실패를 견디게 만든다** — 무상태화, 묶음 단위 확장, 일부 장애 허용, 점진 교체를 가능하게 한다.
5. **다시 배우지 않고 재진입하게 만든다** — 폴더 구조, 공통 문법, 가이드, 복구 가능 상태를 남긴다.

이 사다리로 읽으면 쿠버네티스, gRPC, Hasura, Gatsby, 라이브러리 분리, 설치 최소화 메모가 서로 다른 주제가 아니라, `같은 시스템을 계속 다시 만질 수 있게 만드는가`라는 공통 질문의 변주라는 점이 더 분명해진다.

## 반복되는 판단 축

### 1. 단순함과 유연성의 동시 추구
원본은 `기본 메커니즘이 단순하고 간결하고 명확한 인프라`를 원한다. 그런데 동시에 변화에도 유연해야 한다고 말한다. 이 둘은 종종 충돌한다. 저자는 이 긴장을 해결하기보다, 플러그인 방식과 어댑터 방식으로 복잡성을 연결부에 몰아넣으려 한다.

### 2. 적은 의존성과 필요한 의존성 사이의 조정
외부 프로그램을 적게 쓰고 싶어 하지만, 나중에는 적게 깐다고 모든 문제가 해결되는 것도 아니라고 수정한다.
- 적은 설치는 복구와 이동을 쉽게 한다.
- 그러나 필요한 도구를 안 쓰는 것은 또 다른 제약이다.
- 의존성은 피할 수 없고, 문제는 의존성 자체보다 족쇄가 되는 방식이다.

이 노드는 의존성 제거보다 **의존성 관리의 문제**를 다룬다.

### 3. 분산의 이점과 관리 비용의 충돌
원본은 라즈베리파이, 분산 시스템, 블록체인, 마이크로서비스, 민주주의까지 연결해 분산을 사고한다. 그러나 장점만 보지 않는다.
- 하나가 고장나도 전체가 유지되는 장점
- 개별 노드 수가 많아질수록 관리가 번거로워지는 단점
- 작은 단위 분리가 전체 이해를 쉽게 하지 않을 수도 있다는 문제

즉 분산은 자유와 회복성을 주지만, 동시에 **관리 지점을 폭증시킨다.**

### 4. 구조 일관성에 대한 강한 집착
서비스가 달라도 같은 방식으로 동작하게 만들고 싶다는 메모, input-logic-output 구조, json 입출력, gRPC의 뼈대 재사용, CRUD 서비스 일반화 시도는 모두 같은 방향을 가리킨다. 저자가 원하는 것은 모든 기술을 하나로 통일하는 것이 아니라, **새로운 조각을 붙일 때마다 학습 비용이 폭증하지 않는 공통 문법**이다.

여기서 중요한 점은 공통 문법의 목적이 `예쁜 통일성`이 아니라는 것이다.
- 새로운 사람이 들어왔을 때 진입 시간이 덜 늘어나야 한다.
- 다른 도구로 갈아탈 때 수정 지점을 예측할 수 있어야 한다.
- 오래된 프로젝트를 다시 열었을 때 길을 잃지 않아야 한다.

즉 이 노드의 일관성 집착은 스타일 취향이 아니라 **재진입 비용 절감 장치**로 읽는 편이 더 정확하다.

### 5. 제약은 피해야 할 것만이 아니라 설계를 낫게 만드는 장치이기도 하다
원본은 외부 프로그램 수를 줄이고 싶어 하다가도, 제약을 둠으로써 더 나은 것을 만들려는 마음과 충돌한다고 적는다. 즉 이 노드에서 제약은 단순한 불편이 아니라 설계 기준을 선명하게 만드는 도구이기도 하다. 문제는 제약의 존재가 아니라, 그 제약이 **복구 가능성·이동 가능성·교체 가능성**을 높이는 쪽으로 작동하느냐다.

## 세부 영역별 해석

### 확장성 메모의 의미
확장성은 단순한 스케일 업이 아니라 장애 격리, 점진 실험, 무상태화, 묶음 단위 증설의 감각으로 적혀 있다. 쿠버네티스가 매력적으로 보이는 이유도 단순 인기 기술이라서가 아니라, 고가용성·확장성·배포·모니터링·로깅을 한 번에 다루는 운영 패키지처럼 보이기 때문이다.

### 도구 전환 메모의 의미
Gatsby에서 Next.js로 옮길 때 고통스러운 이유를 길게 적어둔 부분은 중요하다. 여기서 문제는 프레임워크 선택 자체보다, 프레임워크 고유 문법이 메인 로직까지 잠식한 상태다. 따라서 이 노드의 중요한 기준은 `무엇을 쓰는가`보다 **무엇을 분리해두면 나중에 갈아탈 수 있는가**다.

### 외부 라이브러리 분리 메모의 의미
원본은 라이브러리를 한 곳으로 모아 수정 지점을 줄이고 싶어 하지만, 그 자체가 또 다른 추상화 비용을 만든다는 점도 인식한다. 그래서 이 노드에서 라이브러리 분리는 단순 캡슐화 규칙이 아니라, **수정 반경과 구조 과잉 사이의 균형 문제**로 읽어야 한다.

### 레고식 구조 메모의 의미
입력, 로직, 출력의 형태를 반복 가능한 모듈로 만들고 싶어 하는 대목은 이 노드의 핵심 상상력이다.
- 작은 기능을 조합 가능하게 만들고 싶다.
- API 엔드포인트든 함수든 공통 입출력 형태를 갖추길 원한다.
- gRPC, gateway, serverless 같은 도구들은 이 구조를 구현하는 후보로 읽힌다.
- micro crud, 공통 CRUD, extension 메모도 결국 `일관된 기본형 + 필요한 확장`을 만들고 싶다는 욕구의 변형이다.

즉 저자는 기술 이름보다 **조립 가능한 일관된 단위**를 원한다.

### 폴더 구조와 계층 메모의 의미
트리의 깊이와 넓이, 폴더 중첩, 작은 덩어리의 과잉, cmd 구조, Java와 Go의 관습 비교 메모는 모두 정보 구조 설계 감각과 연결된다. 이 노드에서는 디렉터리 구조조차 단순 취향 문제가 아니라, **복잡도를 어느 차원에서 노출할지 결정하는 아키텍처 문제**다.

### 깊이와 넓이는 둘 다 길찾기 비용이다
원문은 폴더 중첩이 깊어지는 것과 작은 덩어리가 많아지는 것을 따로 보지 않는다. 자바식 깊은 구조가 불편하지만, Go standard layout처럼 `cmd` 아래에 메인을 다시 나누는 구조도 실제 서비스에서는 필요할 수 있다고 흔들린다. 여기서 중요한 질문은 `깊은 구조가 나쁜가, 넓은 구조가 좋은가`가 아니다. **사람이 다음 수정 지점으로 얼마나 빨리 다시 들어갈 수 있는가**다.

- 깊은 구조는 상위 맥락을 계속 따라가야 해서 진입 경로가 길어진다.
- 넓은 구조는 후보가 많아져 어디를 봐야 할지 모르게 만든다.
- 작은 함수와 모듈은 수정 반경을 줄이지만, 너무 많아지면 전체 조망을 잃게 한다.
- 처음부터 복잡성을 예상해 뼈대를 크게 만들기보다, 작은 실행이 반복되며 같은 구조가 생길 때 분리 신호로 읽는 편이 원문 감각에 가깝다.

따라서 이 노드의 구조 판단은 디렉터리 스타일 선호가 아니라 `재진입 경로 설계`다. 좋은 아키텍처는 조각을 많이 만드는 구조가 아니라, 조각이 늘어난 뒤에도 다음 사람이 원인, 책임, 호출 경로를 찾아 내려갈 수 있게 하는 구조다.

### A/S 메모의 의미
원본의 `A/S, after service` 메모는 의외로 중요하다. 여기서 A/S는 단순 고객지원이 아니라, **내가 놓친 것을 알려주고 다음 성장을 가능하게 하는 되돌아보기 루프**다. 즉 아키텍처는 만드는 순간 끝나는 구조가 아니라, 운영 후 발견되는 누락과 마찰을 다시 흡수할 수 있어야 한다. 이 점에서 이 노드는 `설계`와 `운영 후 보수`를 분리하지 않는다.

### 작은 것부터 시작하라는 메모의 의미
`Design First is Good`와 동시에 `small thing first`를 적어둔 대목은 설계를 싫어한다는 뜻이 아니다. 오히려 설계를 신뢰하되, **작은 실행으로 검증되지 않는 큰 설계는 쉽게 공허해진다**는 경계다. 그래서 이 노드의 설계 감각은 선설계와 후검증의 대립보다 `작은 실행을 계속 받을 수 있는 구조` 쪽에 가깝다.

## 이 노드에서 보존해야 할 긴장
- 단순한 구조를 원하지만 새 기술은 계속 도입하고 싶다는 긴장
- 모듈화를 원하지만 과도한 분리는 길 잃음을 낳는다는 긴장
- 의존성을 줄이고 싶지만 실제로는 필요한 도구를 써야 한다는 긴장
- 공통 인터페이스를 만들고 싶지만 그 자체가 또 다른 추상화 비용이 된다는 긴장
- 분산은 회복성을 주지만 관리점도 늘린다는 긴장
- 툴은 도구일 뿐이라 생각하면서도, 실제 구현은 툴의 제약을 강하게 받는다는 긴장
- 설계를 먼저 하고 싶지만 작은 실행 없이 큰 구조를 고정하고 싶지는 않다는 긴장

이 페이지를 오래 가치 있게 유지하려면 이 긴장들을 매끄럽게 해소하기보다, **판단이 필요한 지점으로 남겨두는 편**이 맞다.

## 원본에서 드러나는 저자의 선호

### 1. 완벽한 설계보다 복구 가능한 설계를 선호한다
처음부터 정답 구조를 만들기보다 일부 실패를 감당하고, 한 부분만 교체하고, 다시 세우기 쉬운 상태를 더 중요하게 본다.

### 2. 기술 이름보다 전환 가능성을 중시한다
새로운 기술을 배우는 것 자체를 좋아하지만, 그것이 원본 콘텐츠나 핵심 로직을 가두는 상태는 경계한다.

### 3. 큰 이론보다 손에 잡히는 구조를 선호한다
서버 재부팅, SSD 일부 교체, 폴더 구조, API 형태, 테스트 코드, 라이브러리 위치처럼 아주 구체적인 운영 장면으로 사고한다.

### 4. 아키텍처를 인간 협업 한계와 함께 본다
경험 격차, 팀 분리, 메뉴얼의 한계, 대표 구조, 많은 기기를 관리해야 하는 번거로움 등 사람의 관리 능력이 설계의 한계조건으로 계속 등장한다.

### 5. 아키텍처를 패턴 선택보다 운영 문법 선택으로 본다
무엇을 배치하느냐보다, 어떻게 시작하고 어떻게 분리하고 어떻게 되돌리고 어떻게 다른 도구로 옮길지를 더 오래 묻는다. 이 때문에 이 노드는 설계 노트이면서 동시에 **운영 철학 노트**다.

## 이번 점진 심화: 의존성은 없애는 대상이 아니라 위치를 정하는 대상이다
원본을 다시 읽으면 `외부 프로그램을 적게 쓰고 싶다`, `Gatsby 플러그인 종속이 힘들다`, `Hasura는 어디까지 종속적인가`, `라이브러리를 한 곳에서만 수정하고 싶다`는 메모가 같은 질문을 반복한다. 핵심은 의존성을 0으로 만드는 것이 아니다. 오히려 모든 연결에는 약속이 필요하다는 원문 메모처럼, 의존성은 피할 수 없고 문제는 **그 의존성이 어디에 놓여 있으며, 나중에 무엇을 다시 써야 하는지 보이는가**다.

이 기준으로 보면 원문에는 세 종류의 의존성이 섞여 있다.

1. **자산을 가두는 의존성** — 콘텐츠, 데이터, 핵심 로직이 특정 툴의 컨벤션 안에 갇혀 전환할 때 직접 옮겨야 하는 상태다. Gatsby 플러그인 문법, 블로그 도구의 카테고리 구조, UI로만 관리되는 설정이 여기에 가깝다.
2. **교체면을 만드는 의존성** — DB 원본은 지키고 어댑터로 감싸거나, Hasura처럼 데이터와 호출 인터페이스가 분리되어 있어 다른 도구로 옮길 여지가 남는 상태다. 이 의존성은 없애기보다 경계를 잘 기록해야 한다.
3. **운영 능력을 빌려오는 의존성** — Kubernetes, gRPC, feature flag, 외부 서비스처럼 고가용성·배포·관측·공통 입출력 문법을 빠르게 얻는 대신 그 도구의 사고방식을 받아들이는 상태다. 여기서는 도입 여부보다 `도구가 제공하는 능력`과 `내가 감수하는 컨벤션`을 분리해 적는 것이 중요하다.

따라서 이 노드의 도구 도입 기준은 `외부 도구를 쓰느냐 마느냐`가 아니라, **도구가 들어온 뒤에도 원본 자산, 호출 경계, 복구 절차가 따로 설명되는가**에 가깝다. 적게 설치하는 것만으로는 복구 가능성이 보장되지 않고, 많이 쓴다고 반드시 족쇄가 되는 것도 아니다. 중요한 것은 새 환경으로 옮겼을 때 무엇을 다시 설치하고, 무엇을 다시 설정하고, 무엇은 그대로 가져갈 수 있는지가 문서와 코드 구조에 남아 있는 상태다.

### state machine과 A/S 메모를 같이 읽기
원문 후반의 유한 상태 기계 메모와 A/S 메모도 같은 흐름 안에 있다. state machine은 `input -> output` 구조를 안정적으로 만들려는 시도이고, A/S는 운영 후 누락된 상태와 전이를 다시 배우는 루프다. 즉 이 노드의 레고식 아키텍처는 단순히 조립 가능한 함수 묶음이 아니라, **입력·상태·출력·사후 피드백이 반복 기록되는 운영 단위**에 가깝다.

이렇게 읽으면 `작은 실행 → 공통 입출력 → 상태 전이 확인 → 운영 후 A/S → 원천 요구사항 업데이트`라는 더 긴 순환이 보인다. 처음 설계가 완벽하지 않아도, 각 모듈이 어떤 입력을 받아 어떤 상태를 거쳐 어떤 출력을 만들었는지 남아 있으면 나중의 보수와 교체가 덜 막힌다. 반대로 레고처럼 생겼더라도 상태와 피드백이 숨겨져 있으면, 조립성은 좋아 보여도 실제 운영에서는 다시 길을 잃을 수 있다.

### 대표 인터페이스는 조각이 많아질 때의 길찾기 장치다
이번에 다시 보니 원문 곳곳의 `대표` 감각도 중요하다. 저자는 간접 민주주의, API gateway, aggregation API, 공통 CRUD, `다른 기능이라도 같은 방식으로 동작합니다` 같은 메모를 서로 다른 위치에 흩어 놓았지만, 모두 많은 조각을 직접 다 보지 않고도 시스템을 다시 호출하게 만드는 대표 표면을 찾는 시도다.

이 대표 표면은 단순한 중앙집중이 아니다. 오히려 분산과 모듈화를 유지하되, 사람이 매번 모든 하위 조각을 새로 배워야 하는 상태를 줄이려는 장치다. 그래서 좋은 대표 인터페이스는 아래 조건을 만족해야 한다.

- 내부 조각의 세부 구현을 숨기되, 어떤 상태와 데이터를 대신 책임지는지는 드러낸다.
- 공통 호출 문법을 제공하되, 모든 도메인을 억지로 같은 모델에 구겨 넣지는 않는다.
- 하위 모듈을 교체해도 호출자는 덜 흔들리게 하되, 문제가 생겼을 때 어느 조각으로 내려가야 하는지는 추적 가능해야 한다.
- 대표가 전체를 대신 말할 수 있으려면, 대표 자체도 관측·문서·테스트 가능한 운영 단위여야 한다.

이렇게 보면 원문의 API gateway, CRUD-service-extension, 라이브러리 메인 체크, aggregation API 고민은 기술 선택 목록이 아니라 **분산된 조각 위에 사람이 다시 들어갈 수 있는 현관을 어디에 만들 것인가**의 문제다. 대표가 없으면 조각이 많아질수록 길을 잃고, 대표가 너무 강하면 하위 도메인의 차이가 지워진다. 이 노드가 계속 보존해야 할 긴장은 바로 그 중간 지점이다.

### 비즈니스 로직은 추상 경계와 세부 책임을 동시에 요구한다
원문 후반의 `business logic` 메모는 이 노드의 추상화 감각을 더 현실적으로 만든다. 저자는 비즈니스 로직을 추상적인 함수로 유지하고 싶어 하지만, 실제 세부 구현까지 완전히 바깥 라이브러리에 밀어낼 수 없다는 점도 알고 있다. 그래서 질문은 `얼마나 멋지게 추상화할 것인가`가 아니라 **어떤 세부 책임을 비즈니스 로직 안에 남기고, 어떤 반복 구조를 인터페이스와 테스트로 밖에 드러낼 것인가**에 가깝다.

이 관점에서 모델 정의, CRUD 일반화, extension, serverless 배포, aggregation API 메모는 모두 같은 경계 문제로 이어진다. 모델은 시작점이지만, 모든 데이터를 `Content` 안에 구겨 넣으면 공통형은 얻어도 도메인 차이가 사라질 수 있다. CRUD 서비스는 공통 반복을 줄일 수 있지만, 호출 구분·데이터 조합·도메인 규칙의 위치를 다시 물어야 한다. 서버리스 함수는 작은 컴퓨팅 단위를 배포하기 좋지만, 모델 사이 연결과 호출 횟수 최적화가 새 설계 과제로 돌아온다. aggregation API는 여러 조각을 사용자가 읽을 수 있는 응답으로 묶어 주지만, 너무 많은 의미를 한 표면에 몰면 다시 중앙 병목이 될 수 있다.

따라서 이 노드에서 `input / logic / output`은 단순한 코드 모양이 아니다. 입력 스키마, 상태 전이, 도메인 규칙, 출력 조합이 각각 어디에 살아야 나중에 바꾸기 쉬운지를 묻는 책임 배치 언어다. 추상화는 실체를 지우지 않을 때만 도움이 되고, 구체 구현은 호출 경계와 테스트 표면이 있을 때만 오래 견딘다.

## 이번 패스에서 더 또렷해진 점
- `Meta` 만다라트 안에서 이 페이지가 개별 기술 메모가 아니라 `설계 판단을 만드는 판단법` 노드라는 점을 더 전면에 세웠다.
- 원문 전반을 `작게 시작 → 경계 드러내기 → 교체 반경 줄이기 → 실패 견디기 → 재진입 가능 상태 남기기`의 운영 사다리로 다시 묶어, 긴 메모의 반복 문법이 더 빨리 보이게 했다.
- input-logic-output, 공통 API, gRPC, CRUD 일반화에 대한 집착을 `예쁜 추상화`보다 `재진입 비용 절감` 문제로 더 선명하게 정리했다.
- 대표, API gateway, aggregation API, 공통 CRUD 메모를 `분산된 조각 위에 사람이 다시 들어갈 수 있는 대표 인터페이스` 문제로 보강했다.
- 2026-05-17 패스에서는 business logic, 모델 정의, CRUD extension, serverless, aggregation API 메모를 `추상 경계와 세부 책임의 동시 배치` 문제로 보강했다.
- 2026-05-20 패스에서는 트리의 깊이와 넓이, 작은 함수 과잉, 폴더 중첩 고민을 `재진입 경로 설계` 문제로 압축해 보강했다.
- [[Meta/Collection]] 연결을 보강해, 분화 전 메타 감각이 이 페이지에서 어떻게 구조화되는지의 관계를 더 직접적으로 드러냈다.
