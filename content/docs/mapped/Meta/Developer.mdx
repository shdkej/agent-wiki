# Developer

- 원본: `source/shdkej-content/Meta/Developer.md`
- 상태: 안정 디벨롭
- 역할: 원본 만다라트 노드와 1:1 대응하는 메타 커리어 페이지

## 핵심 해석
`Developer`는 개발자 일반론이나 이력서 초안 모음이 아니다. `나는 어떤 개발자인가`, `회사는 왜 나를 뽑아야 하는가`, `내 경험은 어떤 증거 형식으로 보여야 하는가`, `어떤 회사와 팀에서 더 잘 자라는가`를 계속 다시 묻는 자기 설명 노드다.

원문은 DevOps, 인프라, 백엔드, 포트폴리오, 면접, 회사 분석, 영어 답변 초안을 길게 섞어 둔다. 하지만 중심은 비교적 일관된다. 저자는 특정 기술 스택보다 본질 이해와 문제 해결을 중시하고, 안정적으로 운영되며 되돌릴 수 있는 변경을 선호하고, 사람과 팀이 더 쉽게 일하도록 주변 마찰을 줄이는 개발자로 자신을 설명하려 한다.

`Meta > Developer`라는 위치에서 보면 이 노드는 [About_Development](/docs/mapped/Meta/About_Development)의 개발 태도, [Teamwork](/docs/mapped/Communication/Teamwork)의 협업 표면, [Portfolio](/docs/mapped/Portfolio)의 외부 증거 표면을 한곳에서 조립하는 허브다. 따라서 좋은 정리는 경험을 더 많이 늘리는 것이 아니라, 이미 있는 경험을 재사용 가능한 자기 증명 구조로 압축하는 쪽이다.

## 원문 발췌 구조
- `Why me?`: 데브옵스 지향, 문제 해결, 일관성, 되돌릴 수 있는 변경, 성과 설명 불안이 처음부터 함께 나온다.
- `Resume`: 경험을 경력 언어로 바꾸고, 회사가 뽑을 이유가 있게 만들고, 핵심 이야기를 5개 정도로 줄이려는 자기 세일즈 층이다.
- `회사에 질문`: 조직의 문서, 커뮤니케이션, 개발문화, 인프라 깊이, 문제 해결 태도를 확인하려는 상호 탐색 층이다.
- `경험`: 대용량 메시징, Lambda/Kinesis, Kubernetes, CI/CD, ERP, 내부 도구, 최소 스택 운영 같은 실제 증거 재료가 모인 층이다.
- `나의 개발 성향`: 정리, 미니멀리즘, 사람 중심 개발, 인프라 직무 동기가 직업 철학으로 이어지는 층이다.
- `Top Personal Question`, `Technical interview`, `Interview`, `회사 분석`: 실제로 막힌 질문과 회사별 준비가 남은 면접 시뮬레이션 층이다.

## 판단 모델 1. 경험은 `증거 패킷`으로 보존한다
원문은 많은 프로젝트와 면접 질문을 나열하지만, 오래 남길 단위는 사건 자체보다 반복 조합 가능한 증거 패킷이다.

| 패킷 | 원문에서 잡히는 재료 | 다시 쓸 표면 |
| --- | --- | --- |
| 문제 | 대량 메시지 발송, CI/CD 지연, ERP 단독 운영, Kubernetes 운영 질문, 채팅 시스템 요구사항 | 이력서, 면접 첫 답변 |
| 선택 | Lambda/Kinesis, Elastic Beanstalk, Docker만 남긴 ERP, Fargate 재검토, Redis pub/sub, DynamoDB/RDS 비교 | 포트폴리오 설계 설명 |
| 운영 | 되돌릴 수 있는 변경, 공유 후 대응, 후속 조치, 근본 원인 개선, 배포 안정성 | DevOps 정체성 설명 |
| 증거 | 그래프, 수치, 아키텍처 다이어그램, 전후 비교, 실패 지점 로그 | 성과 검증 자료 |
| 성장 | 말문이 막힌 질문, 회사에 물어보지 못한 것, 태도 회고, v2/v3 설계 수정 | 다음 자기소개 버전 |

이 관점에서 `그래프는 15% 줄였다보다 규모·추세·변동성·시점을 보여준다`는 원문은 중요하다. 저자가 원하는 성과 표현은 수치 장식이 아니라, 판단과 변화가 검증 가능한 형태로 보이는 증거 포맷이다.

## 판단 모델 2. DevOps 정체성은 도구보다 운영 마찰 감소다
원문에는 Kubernetes, AWS, Lambda, Kinesis, Terraform, IaC, observability, Karpenter 같은 기술명이 많다. 그러나 반복되는 자기상은 `복잡한 시스템을 운영 가능한 형태로 정리하는 개발자`에 더 가깝다.

- 개발자가 코드와 비즈니스 로직에 집중할 수 있게 주변 환경을 단순화한다.
- 반복 작업은 스크립트화하고 자동화한다.
- 배포 시간과 안정성을 동시에 본다.
- 신규 인프라 도입은 흥미롭지만, 단순함을 먼저 두고 복잡함은 근거가 있을 때만 늘린다.
- 장애나 실패를 숨기지 않고 공유, 후속 조치, 원인 개선, 백업 기준으로 바꾼다.

특히 `백업을 하지 않아 서버와 코드를 잃은 경험`은 실패담 이상의 의미가 있다. 이 경험은 snapshot, drive backup, DB backup, source backup처럼 상태를 여러 층에서 보존해야 한다는 운영 기준으로 굳어진다. 그래서 이 노드에서 백업·복구·되돌릴 수 있는 변경은 부록이 아니라 개발자 정체성의 중심 증거다.

정리와 미니멀리즘에 대한 긴 메모도 같은 축이다. 저자는 적게 쓰는 것을 미덕으로 삼는 사람이 아니라, 더 뺄 것이 없을 때까지 구조를 이해하고 줄여 나가는 일을 좋아한다. 인프라 개발자를 `소프트웨어에서 정리하는 직업`처럼 느끼는 이유도 여기에 있다.

## 판단 모델 3. 면접은 암기장이 아니라 설명 인터페이스 테스트다
원문 후반의 실제 면접 회고와 회사별 질문은 질문 저장소처럼 보이지만, 더 중요한 것은 `어디서 내 설명이 끊겼는가`를 확인하는 테스트 로그라는 점이다.

- `왜 데브옵스에 지원했는지`, `5년 후 어떤 모습인지`에서 버벅인 것은 정체성 서사가 아직 압축되지 않았다는 신호다.
- 두나무의 Kinesis/SQS, Kubernetes 스케일링, 보안 고려 질문은 경험을 기술명으로만 기억하지 말고 선택 기준과 운영 흐름으로 말해야 한다는 신호다.
- 바로고의 upgrade, CNI, ingress/service/pod 통신, secret, Loki 파이프라인 질문은 실제 운영 경계와 관측 흐름을 더 분명히 설명해야 한다는 신호다.
- 메이크스타 회고의 태도, 어미, 회사 질문 부족은 답변 내용뿐 아니라 대화 안정감과 상호 탐색 태도를 조정해야 한다는 신호다.

채팅 시스템 설계 메모도 같은 구조다. 글로벌 유저, 무제한 트래픽, 미디어 첨부, 접속 상태 표시를 요구사항으로 받은 뒤 Lambda, Fargate, Redis pub/sub, DynamoDB, RDS를 비교하고, 제출 후 Lambda가 실시간 연결 유지에 부적합하다고 보아 v2/v3로 고친다. 이 부분은 정답 암기가 아니라 `요구사항 → 제약 설정 → 서비스 선택 → 실패 지점 관찰 → 더 나은 구조로 수정`하는 자기 증명 사례다.

EKS 과제 메모의 CIDR, bastion, routing table, IAM, node group 실패도 같은 의미로 보존해야 한다. 설치 실패 자체보다 어느 경계에서 막혔고 다음에는 어떤 완료 기준을 먼저 확인해야 하는지 남긴 점이 중요하다.

## 특히 보존할 원문 앵커
- 그래프와 수치: 규모, 추세, 변동성, 시점을 함께 보여 주는 성과 증거 포맷.
- 의사결정 맥락: `완료`보다 갈등, 데이터, A/B 테스트, 조율 근거를 남기려는 태도.
- 백업 실패: 서버와 코드 손실 경험이 snapshot, drive, DB, source backup 기준으로 굳어진 장면.
- 핵심 경험 후보: 서버리스 Lambda/SQS/Kinesis, Kubernetes/컨테이너, 개발부터 배포까지의 자동화, 대용량 전송, ERP 단독 운영.
- 실제 면접에서 막힌 질문: DevOps 지원 동기, 5년 후 모습, 회사 질문 부족, 기술 선택 비교를 말로 풀어내는 어려움.
- 채팅 시스템 v2/v3 수정: Lambda 중심 설계를 Fargate, Redis pub/sub, DynamoDB/RDS 비교로 다시 고친 설계 재검토.
- EKS 과제 실패 지점: CIDR, public/private subnet, bastion, routing table, IAM, node group처럼 운영 경계가 드러난 로그.

## 후속 정리 기준
- 면접 질문을 더 모으기보다 반복해서 막힌 질문을 핵심 경험 5개와 연결한다.
- 새로운 기술명을 추가할 때는 문제, 선택 기준, 운영 실패 가능성, 증거 포맷 중 어디에 속하는지 먼저 정한다.
- 포트폴리오와 연결할 때는 결과만 옮기지 말고 그래프, 다이어그램, 실패 로그, v2/v3 수정처럼 판단이 보이는 앵커를 함께 남긴다.
