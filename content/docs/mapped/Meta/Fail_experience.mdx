# Fail_experience

- 원본: `source/shdkej-content/Meta/Fail_experience.md`
- 상태: 안정 디벨롭
- 역할: 만다라트 `Meta` 축에서 실패를 다음 실행 조건으로 바꾸는 1:1 대응 카테고리 노드

## 핵심 해석
`Fail_experience`는 자책 메모나 사고 목록이 아니다. 복붙 확인 누락, 테스트 후 원복 실패, 코드 수정 후 미확인, 드라이브 포맷, OneTab/OneNote 유실, NVMe 호환성 오판, 음식물·벽지 같은 생활 사건이 한 파일에 붙어 있지만, 반복되는 질문은 하나다. `어떤 전제를 확인하지 않았고, 그 결과 실패가 얼마나 커졌으며, 다음에는 어떤 장치로 덜 크게 만들 것인가?`

`Meta` 아래에 놓인 이 노트는 개별 기술 지식보다 일하는 방식의 실패 조건을 다룬다. [Troubleshooting](/docs/mapped/Meta/Troubleshooting)이 이미 터진 문제를 되짚는 복구 루프라면, 이 페이지는 그 루프가 왜 필요해졌는지 보여 주는 실패 전조 지도다. [About_Development](/docs/mapped/Meta/About_Development)의 좋은 개발 기준은 여기에서 실제 손실 사례로 검증된다.

## 원문 발췌 구조
- `Before setting check state`: 시간 부족, 백업 부재, 병행 작업, 복붙, 데이터 무시, 원복 실패, 수정 후 미확인처럼 실패를 키우는 조건이 먼저 나온다.
- 생활·기기 사건: 드라이브 포맷, OneTab/OneNote 삭제, 방울토마토, WASM 테스트로 컴퓨터가 멈춘 일, 유통기한, 벽지, 음식물쓰레기 봉투가 이어진다.
- 백업 전략: 미디어 폴더 유실 뒤 최신 백업, 기간별 개발 폴더 기록, GitHub·Drive 백업, 개발환경 파일 관리가 질문으로 열린다.
- 큰 그림과 problem solving: 질문을 받고도 다시 확인하지 않은 회고, 이전 상태로 돌아가기, 하나씩 바꾸기, 정확한 원인 찾기가 실행 루프로 압축된다.
- `Uncategorized`와 2025년 메모: 쇼핑몰, 버티컬 AI, 지표, A/B 테스트, 문서 최신화, CI, Lambda, pre-commit, CKA가 실패 예방 시스템으로 번질 씨앗으로 남아 있다.

## 판단 모델 1. 실패는 `빠진 전제`에서 커진다
원문에서 가장 반복되는 실패는 고난도 지식 부족보다 확인 가능한 전제를 통과시킨 데서 생긴다.

| 빠진 전제 | 원문 장면 | 다음 확인 표면 |
| --- | --- | --- |
| 백업 | 미디어 폴더, C 드라이브, OneTab, OneNote 유실 | 변경 전 사본, 동기화 검증, 복구 테스트 |
| 호환성 | NVMe와 SATA 차이를 알면서도 어댑터를 막연히 기대함 | 실제 장비 조합, 포트, 전원, 데이터 이동 경로 |
| 상태 확인 | 복붙만 함, 코드 수정 후 확인 안 함, 테스트 후 원복 안 함 | 변경 후 결과 확인, 원복 체크, 작은 변경 단위 |
| 책임 범위 | 벽지, 음식물쓰레기, 유통기한, 떨어진 물건 | 찜찜함이 생긴 순간 바로 치우기·묻기·확인하기 |
| 목표 조건 | 알고리즘 공부를 원했지만 자료와 측정 목표를 만들지 않음 | 자료 위치, 반복 단위, 측정 기준, 다음 확인 날짜 |

이 모델에서 `막연히 되겠지`는 지식의 빈칸이 아니라 검증을 미룬 신호다. 기술 이름을 아는 것과 내 장비·내 데이터·내 운영 절차에서 실제로 되돌릴 수 있는지는 다르다.

## 판단 모델 2. 복구 가능성이 실패 비용을 결정한다
원문은 실패하지 않는 사람이 되자는 다짐보다, 실패해도 돌아올 수 있게 만드는 쪽으로 이동한다. `백업 전략을 세워야 한다`는 메모가 핵심이다. 미디어는 최신 백업만으로 충분할 수 있지만, 개발 폴더는 수시 백업과 기간별 기록이 필요하고, 워크스페이스는 GitHub와 Drive에 함께 있어야 하며, 개발환경 파일도 별도 관리 기준이 필요하다.

`problem solving` 다섯 줄도 같은 방향이다.

1. 문제를 만난다.
2. 이전 상태로 돌아간다.
3. 한 번에 하나씩 바꾼다.
4. 정확한 문제를 찾는다.
5. 그다음 해결한다.

따라서 이 노트의 운영 규칙은 자책보다 원복 경로에 가깝다. 바꾸기 전에는 백업과 롤백 지점을 확인하고, 바꾸는 동안에는 변경 단위를 줄이며, 바꾼 뒤에는 실제 결과를 다시 본다. 실패가 났다면 감정 문장만 남기지 말고 `빠진 전제`, `실패가 커진 경로`, `되돌릴 손잡이`, `다음 자동 장치`를 남겨야 한다.

## 판단 모델 3. 질문과 찜찜함은 작은 검증 게이트다
원문에서 중요한 장면은 `분명 잘 동작하는 건지 확인하는 질문을 받았음에도, 다시 확인해보지 않고 직감적으로 맞다고 얘기하고 그냥 넘어갔다`는 회고다. 질문은 방해가 아니라 이미 시스템이 보내 준 경고다. 누군가 확인을 요청했다면 그 지점은 공유된 불확실성으로 떠올랐다는 뜻이다.

생활 사건도 같은 구조다. 음식물쓰레기 봉투, 벽지, 유통기한, 방울토마토는 기술 사건보다 작아 보이지만, 공통 전조는 `내가 챙겨야 할 부분인데 그냥 넘긴 상태`다. 질문, 찜찜함, 불편한 예감은 작을 때 확인하면 검증 게이트지만, 넘기면 실패 기록이 된다.

이 관점은 [Work](/docs/mapped/Integration/Work)의 검토 게이트와 [Teamwork](/docs/mapped/Communication/Teamwork)의 진행상황 공유로 이어진다. 좋은 답변은 자신감 있는 즉답이 아니라, 지금 무엇을 확인했고 무엇은 아직 추정인지 나누어 말하는 것이다.

## 2025년 메모의 역할
후반의 2025년 메모는 주제가 흩어진 잡기장처럼 보이지만, 앞쪽 실패 조건을 줄이려는 구조적 대응으로 읽을 수 있다.

- 지표, A/B 테스트, life feedback loop는 감각적 낙관 대신 상태를 측정하려는 장치다.
- 작은 프로젝트화, 람다화, 엑셀 업로드 파이프라인은 한 번에 너무 많이 바꾸다 망하는 패턴에 대한 분해 전략이다.
- workflow 실패 알림, pre-commit, 빠른 테스트는 `수정 후 확인 안 함`을 사람의 기억에만 맡기지 않으려는 자동 검증이다.
- 문서를 최신으로 유지하는 방법, 연동 누락 확인, Redis 메모리 문제 발견 질문은 숨어 있는 상태를 빨리 드러내려는 가시화 장치다.

다만 이 구간을 전부 이 페이지에 영구 귀속하면 주제가 흐려진다. `Uncategorized`의 `1년 이상 보관하지 말자`는 문장은 이 노트가 임시 검역 대기열도 함께 품고 있음을 보여 준다. 여기서는 항목을 모두 설명하기보다 실패 예방 압력만 읽고, 관련성이 뚜렷해지면 [Product](/docs/mapped/deep-knowledge/product), [Business](/docs/mapped/Integration/Business), [Document](/docs/mapped/Communication/Document), [Devops](/docs/mapped/deep-knowledge/devops), [Work](/docs/mapped/Integration/Work)로 보낼 수 있는 씨앗으로 둔다.

## 바로 꺼낼 운영 규칙
- 수정 전에는 백업, 원복, 롤백 지점을 먼저 확보한다.
- 한 번에 하나씩 바꾸고, 문제를 섞어 만들지 않는다.
- 복붙, 테스트, 배포, 원복 뒤에는 실제 상태를 다시 확인한다.
- 애매하게 아는 영역은 감으로 넘기지 말고 실제 조합으로 닫는다.
- 질문을 받았을 때는 즉답보다 확인 표면을 다시 여는 쪽을 우선한다.
- 목표는 바람으로 두지 말고 자료 위치, 반복 단위, 측정 기준 중 하나로 내려놓는다.
- 생활 메모도 책임 범위와 마감 태도의 신호로 함께 읽는다.

## 인접 노드 연결
- [Troubleshooting](/docs/mapped/Meta/Troubleshooting): 실패 이후의 복구 루프와 상태 확인 절차를 더 구체적으로 다룬다.
- [About_Development](/docs/mapped/Meta/About_Development): 설명 가능성, 테스트, 문서화, 재진입 가능한 개발 루프와 이어진다.
- [My_space](/docs/mapped/Meta/My_space): 개인 인프라와 생활 환경을 복원 가능하게 유지하려는 태도와 연결된다.
- [Decision](/docs/mapped/Health/Decision): 조건을 세우고 감각적 낙관을 줄여 판단 품질을 높이는 습관과 맞닿아 있다.
- [Work](/docs/mapped/Integration/Work): 질문, 검토, 보고를 실제 확인 게이트로 바꾸는 업무 운영 루프와 이어진다.

## 후속 정리 기준
- 새 실패 사례가 들어오면 사건 설명보다 빠진 전제, 실패 증폭 경로, 되돌릴 손잡이, 다음 자동 장치를 먼저 채운다.
- 2025년 혼합 메모를 늘려 설명하지 말고, 실패 예방 압력이 사라진 항목은 인접 노드 후보로만 표시한다.
- `또 멍청` 같은 감정 문장은 삭제 대상으로만 보지 말고, 반복 전조를 더 빨리 알아차리게 하는 신호로 번역한다.
