# 도구

- 원본: `source/shdkej-content/Integration/Tool.md`
- 성격: 만다라트 `Integration` 축의 1:1 대응 카테고리 노드
- 이번 상태: 기존 baseline은 유지하되, 후속 판단표의 사례를 원문 명령·경계 단서 쪽으로 더 정확히 고정했다. 새 도구나 메모가 들어와도 별도 이력을 본문에 계속 쌓기보다, 기존 작업 표면을 실제로 더 잘 보이게 하거나 복구 가능하게 만드는지 먼저 본다.

## 핵심 판단 모델
이 노드의 `Tool`은 추천 앱 모음이 아니다. 맥 유틸, 터미널 툴, 업무 서비스, Tailscale, OpenClaw, Linux, Git, Terraform, gRPC, 스트리밍 메모가 한 파일에 섞여 있는 이유는, 저자가 툴을 `하루의 입력·검색·협업·원격 운영·복구·확장`을 한 몸처럼 잇는 작업 시스템으로 보기 때문이다.

| 원문 장면 | 이 노드에서의 의미 | 남길 질문 |
| --- | --- | --- |
| Karabiner, Raycast, Vimium, fzf, ripgrep, jq | 손과 검색 표면을 짧게 만드는 개인 인터페이스 | 자주 반복되는 동작이 더 가까워졌는가? |
| Asana, Slack, Confluence, GitHub, Drive | 개인 상태를 협업·배포 표면으로 꺼내는 공유 장치 | 누가 같은 상태를 다시 읽을 수 있는가? |
| Tailscale, OpenClaw, SSH, cron, backup | 원격 접속과 자동화를 끊긴 뒤에도 되살리는 운영 손잡이 | 인증·키·포트·스케줄이 끊겼을 때 복구 절차가 남아 있는가? |
| Linux, Git, startup files, known_hosts | 상위 도구를 오래 쓰게 하는 하부 조작 기억 | 막혔을 때 돌아갈 명령과 경로가 남아 있는가? |
| Terraform, Vault, API gateway, gRPC, streaming | 시스템 경계와 책임 배치를 시험하는 확장 실험 | 어느 경계에서 형식이 바뀌고 무엇이 손실되는가? |

## 만다라트 위치에서 읽기
이 페이지가 `Integration/Tool`에 놓여 있다는 점이 중요하다. 같은 `Tool`이라도 [[../Fundamental/Tool]]이 셸, 권한, 파일 경로, 복구 절차 같은 하부 감각을 붙드는 노드라면, 여기의 `Tool`은 그 하부 감각 위에 여러 표면을 실제 생활과 업무 흐름으로 엮는 통합 노드에 가깝다.

- `Fundamental/Tool`이 손에 도구를 쥐는 법이라면, 이 노드는 여러 도구를 한 작업면으로 편성하는 법에 가깝다.
- `Meta/AGENTS`가 역할·문서·협업 토폴로지를 다룬다면, 이 노드는 그 구조를 실제로 굴리게 하는 로컬/원격/협업 도구층을 다룬다.
- `Integration/Work`가 일의 흐름과 우선순위를 다룬다면, 이 노드는 그 흐름이 덜 끊기게 만드는 인터페이스와 제어면을 다룬다.

그래서 원문이 앱 추천, CLI, 원격 접속, Linux, Git, IaC, 스트리밍까지 넓게 흩어져 보여도, 실제로는 `서로 다른 도구층을 한 사람이 감당 가능한 운영 표면으로 묶는 법`을 기록한 카테고리로 읽는 편이 맞다.

## 도구 편입의 네 층

### 1. 작업 표면: 손과 검색을 가까이 둔다
`블로그 소프트웨어 탑 텐`의 맥 도구, 터미널 도구, 개발도구, 크롬 익스텐션, 앱 묶음은 취향 taxonomy보다 하루 접점의 순서를 보여 준다. Karabiner, Raycast, hidden bar, spectacle, Vimium은 손과 화면 전환을 줄이고, vim, zshrc, UltiSnips, Rclone, fzf, ripgrep, jq, Claude Code는 검색과 조합을 재실행 가능한 텍스트 흐름으로 만든다.

### 2. 공유 표면: 개인 상태를 밖으로 꺼낸다
Asana, Slack, Confluence, Google Drive, GitHub, GitHub Actions, Jenkins, Netlify, Cloudflare, Analytics는 혼자만의 생산성 앱이 아니라 여러 사람이 같은 상태를 읽는 공용 표면이다. 초반의 Traefik, Signoz, Grafana, Prometheus, Next.js, Kubernetes, k3s, ArgoCD도 구현·배포·관측을 한 사람이 이어 만지는 현실 표면으로 붙어 있다.

### 3. 복구 표면: 다시 세팅할 수 있어야 편입된다
글로벌 설치 목록을 파일로 모으려는 메모, dotfiles, startup files, backup, cron, SSH, known_hosts, Git 복구 메모는 `좋은 도구`의 기준을 설치 성공이 아니라 재진입 가능성으로 옮긴다. 원문에서 직접 남겨야 하는 장면도 이 층에 있다.

| 원문 조각 | 재사용 의미 |
| --- | --- |
| `sudo tailscale up --ssh`, `sudo tailscale up --force-reauth` | 원격 접속은 켜짐 여부보다 키 만료와 재인증 절차까지 남아야 한다. |
| `ssh -N -L 18789:127.0.0.1:18789 ubuntu@100.96.204.36` | OpenClaw dashboard는 기능 목록이 아니라 접속 경로와 포트포워딩 문법이 편입 문턱이다. |
| `openclaw models set openai/gpt-5-mini`, `clawhub install summarize` | 새 에이전트 도구는 모델·스킬·노트 접근 경로가 복원될 때 장기 도구가 된다. |
| `ssh-keygen -R <host IP>`, `git merge --abort`, `git stash branch <new-branch> stash@{0}` | 꼬인 상태에서 빠져나오는 명령이 남아야 Git과 SSH가 복구 장치가 된다. |

### 4. 확장 표면: 도구는 시스템 책임 배치로 이어진다
Ansible, Terraform, Serverless, Vault, Nagios, Lambda cronjob, API gateway, Kafka, gRPC, 동영상 스트리밍 메모는 수집벽이 아니라 시스템 조합 실험이다. Terraform vs Serverless는 책임 배치 문제이고, Vault는 보안과 운영 편의의 충돌 지점이며, gateway·gRPC·HLS는 경계에서 형식이 바뀔 때 무엇이 보존되는지 확인하는 장치다.

## 도구 의존을 관리하는 세 가지 소유 위치
이 노드를 다시 읽으면 도구 목록보다 `내가 무엇을 어디까지 소유하는가`가 더 중요하게 보인다. 회사 SaaS, 개인 서비스, 로컬 앱, 서버 운영 도구가 한 페이지에 섞여 있는 이유도 여기 있다. 각각은 편의의 층이 아니라, 상태를 어디에 맡기고 어디에서 되찾을 수 있는지를 가르는 선택지다.

### 1. 로컬에 남기는 것: 손의 속도와 즉시성
Karabiner, Raycast, Vimium, 터미널 설정, vim, fzf, jq, ripgrep은 네트워크나 조직 승인 없이 바로 꺼내 쓰는 표면이다. 이 층은 생산성의 최전방이지만, 기기 교체 때 사라질 수 있으므로 dotfiles, 설치 목록, alias, 스크립트로 외부화해야 오래 간다.

### 2. 외부 서비스에 맡기는 것: 협업과 분산 접근
Google Drive, Slack, Confluence, GitHub, Cloudflare, Netlify, Analytics, Asana는 혼자만의 속도보다 여러 장소와 사람이 같은 상태를 볼 수 있게 한다. 대신 계정, 권한, 요금, API, 서비스 종료, 내보내기 가능성이 새 비용으로 붙는다. 이 노드가 회원가입 서비스와 쓰고 있는 서비스를 따로 적는 이유도, 편의가 곧 의존 지형이 되기 때문이다.

### 3. 문서와 스크립트로 되찾는 것: 복원 가능한 운영 기억
Tailscale 재인증, OpenClaw dashboard 포워딩, cron, backup, SSH key, Git 복구, Terraform backend, Vault 고민은 `도구가 현재 잘 돌아간다`보다 `끊겼을 때 어느 절차로 다시 이어 붙일 수 있는가`를 묻는다. 이 층이 있어야 로컬 편의와 외부 서비스 의존이 단순 소비가 아니라 회복 가능한 작업 시스템이 된다.

따라서 이 노드에서 좋은 도구 선택은 `가장 강력한 기능`을 고르는 일이 아니라, 로컬 즉시성·외부 협업성·복원 가능성 사이의 소유 위치를 의식적으로 배치하는 일에 가깝다. 이 관점은 [[../Fundamental/Tool]]의 하부 조작면과 [[../Integration/Work]]의 신뢰 가능한 실행 루프를 이어 주는 중간 해석축이 된다.

## 상태 전달 인터페이스로 다시 읽기
원문 후반의 OpenClaw, Tailscale, GitHub, dashboard 포워딩, Git, Terraform, API gateway, gRPC 메모를 한 줄로 세우면 `도구를 쓴다`보다 `상태를 어디에 보이게 둘 것인가`가 더 중심에 온다. 이 노드의 도구들은 일을 대신해 주는 물건이라기보다, 현재 작업 상태를 사람·기기·서버·문서 사이로 옮기는 인터페이스다.

특히 다음 네 가지 상태 전달 경로가 반복된다.

1. **내 손으로 돌아오는 상태**: Raycast, Vimium, zshrc, fzf, jq, ripgrep은 흩어진 파일과 화면을 손 가까이 다시 불러온다.
2. **다른 사람에게 보이는 상태**: Asana, Slack, Confluence, GitHub, Drive는 작업의 현재 위치를 개인 머릿속 밖으로 꺼낸다.
3. **서버가 스스로 말하는 상태**: Grafana, Prometheus, Signoz, Nagios, dashboard 포워딩은 시스템이 정상인지 사람에게 알려 주는 관측 표면이다.
4. **끊긴 뒤 다시 이어지는 상태**: Tailscale 재인증, SSH key, backup, cron, Terraform backend, Git 복구 메모는 단절 이후에도 이전 상태로 돌아갈 손잡이를 남긴다.

이렇게 보면 [[../Integration/Work]]의 업무 가시성과 이 노드의 도구 편입은 같은 문제를 다른 층에서 다룬다. Work 노드가 `일이 어디까지 왔는가`를 사람에게 보이게 만든다면, Tool 노드는 그 상태가 로컬·SaaS·서버·스크립트 사이에서 사라지지 않도록 통로를 만든다. 따라서 새 도구를 편입할 때의 질문도 `무엇을 할 수 있는가`에서 `어떤 상태를 누구에게, 어느 표면으로, 얼마나 복구 가능하게 전달하는가`로 바뀐다.

## 편입된 도구는 계속 수명을 관리해야 한다
원문에서 Tailscale 키 만료, OpenClaw 모델 재설정, 글로벌 설치 목록 스크립트, GitHub SSH, 서비스 회원가입 목록이 반복되는 점은 중요하다. 도구는 설치 순간에 끝나는 물건이 아니라, 계정·키·버전·설정 파일·접속 경로가 계속 살아 있어야 하는 운영 대상이다.

그래서 이 노드의 도구 판단은 `써 보니 좋다`에서 멈추지 않는다. 실제 편입 여부는 시간이 지난 뒤에도 같은 작업면을 되살릴 수 있는지로 판별된다.

### 1. 인증과 키의 수명을 봐야 한다
Tailscale의 6개월 키 만료, `sudo tailscale up --force-reauth`, GitHub SSH key, Google·Naver·Apple·PayPal 회원가입 목록은 모두 `접속 권한이 언젠가 끊긴다`는 사실을 남긴다. 좋은 도구 메모는 로그인 성공보다 다음 갱신 시점과 재인증 절차를 더 오래 남겨야 한다.

### 2. 설치 목록은 취향 기록이 아니라 복원 스크립트의 재료다
brew, asdf, npm 글로벌 설치 항목을 파일로 모으려는 메모는 앱 카탈로그가 아니라 환경 재현의 씨앗이다. Karabiner, Raycast, vim, jq, ripgrep, Claude Code 같은 도구가 몸에 붙을수록, 새 기기나 서버에서 같은 작업면을 되살리는 비용도 커진다. 따라서 설치 목록은 `무엇을 좋아했는가`보다 `무엇이 없으면 손의 리듬이 끊기는가`를 알려 준다.

### 3. 새 도구는 기존 상태 전달 경로에 붙을 때만 오래 간다
OpenClaw 메모가 모니터링, cron, GitHub 노트 접근, dashboard 포워딩, 모델 변경, 스킬 설치로 이어지는 이유는 새 도구를 독립 앱으로 보지 않기 때문이다. 이 도구가 기존 서버, 노트, GitHub, 브라우저, cron 작업과 연결되어야 비로소 장기 편입된다.

### 4. 외부 서비스 의존은 편의와 복구 비용을 함께 만든다
Google Drive, Slack, Confluence, Netlify, Cloudflare, Analytics 같은 서비스는 협업과 분산 접근을 쉽게 만들지만, 계정 잠김, 권한 변경, 내보내기 제약, 요금 변화라는 복구 비용을 만든다. 그래서 `쓰고 있는 서비스`와 `회원가입 된 서비스` 목록은 정리되지 않은 잡목록이 아니라, 나의 작업 상태가 어떤 외부 표면에 흩어져 있는지 보여 주는 의존 지도다.

이 관점에서 [[../Meta/My_space]]와 연결하면, 개인 운영 공간은 서버 자원만으로 만들어지지 않는다. 로컬 앱, SaaS, 원격 접속, 문서, 스크립트의 수명이 서로 어긋날 때 어디서 다시 묶을지까지 설계해야 `내 공간`이 오래 유지된다.

## 전달 경계와 형식 변환으로 보는 후반 메모
원문 후반의 API gateway, Kong, decK, gRPC, protobuf, HTTP body, Lambda response body, ffmpeg, RTSP, WebRTC, HLS, CDN 메모는 서로 다른 기술 스택처럼 보이지만, 실제로는 같은 질문을 반복한다. `무엇을 어디까지 내부 형식으로 유지하고, 어느 경계에서 외부가 이해할 수 있는 형식으로 바꿀 것인가`라는 질문이다.

### 1. API gateway는 단순 입구가 아니라 경계 번역 장치다
원문의 API gateway 메모는 `운영서버를 개발자가 같이 써도 되는가`, `로컬에서 각자 띄워야 하는가`, `nginx도 gateway인가`, `Kong declarative 설정은 어디까지 읽을 수 있는가` 같은 질문으로 이어진다. 여기서 gateway는 트래픽을 대신 받아 주는 도구를 넘어, 개발자·운영서버·로컬 환경·외부 요청 사이의 책임 경계를 정하는 표면이다.

그래서 이 노드에서 gateway를 읽을 때의 핵심은 다음이다.
- 요청이 들어오는 표면을 어디에 둘 것인가?
- 개발자가 실험할 수 있는 경계와 운영 환경의 안정 경계를 어떻게 나눌 것인가?
- 설정은 UI에 남길 것인가, 선언형 파일로 남겨 재현할 것인가?
- 응답 body 안에 담긴 상태를 누가 읽고 판단할 것인가?

### 2. gRPC 메모는 통신 속도보다 `형식이 책임을 강제하는 방식`에 가깝다
원문은 gRPC를 HTTP/2, streaming, stub, protobuf, gateway, swagger, marshal/unmarshal, `array만으로 반환하기 어려운 문제`, `google.api.HttpBody`까지 이어서 적는다. 이는 빠른 RPC가 궁금하다는 뜻만이 아니라, 데이터가 어떤 메시지 틀 안에 들어가야 하고 클라이언트와 서버가 어떤 책임을 반복 구현해야 하는지 확인하는 과정이다.

특히 `배열을 그냥 내보내지 못하고 message 형태가 된다`, `받는 쪽에서 name을 골라야 한다`, `response body 안에 다 넣는 방식` 같은 메모는 형식이 개발 편의와 읽기 편의, gateway 호환성을 동시에 흔든다는 감각을 보여 준다. 이 부분은 [[../deep-knowledge/web]]의 전달 전략, [[../deep-knowledge/devops]]의 실행 체계와도 맞닿는다.

### 3. 스트리밍 메모는 영상 기술 목록보다 `신호가 끊기지 않게 전달되는 체인`이다
ffmpeg, 인코딩, 미디어 서버, 전송 서버, CDN, 플레이어, HLS, `.m3u8`, `.ts`, RTSP, WebRTC, SRT 메모는 영상 서버 후보 목록처럼 보이지만 중심은 `영상이 어느 단계에서 어떤 부하를 만들고 어디서 끊기는가`다. 원문은 서버를 띄우고 ffmpeg로 영상을 밀어 넣고, 서버가 다시 브라우저가 읽을 수 있는 조각으로 전달하는 체인을 계속 확인한다.

이 메모의 재사용 가치는 최신 스트리밍 서버 추천이 아니라 다음 질문에 있다.
- 입력 영상은 어디서 읽고 누가 인코딩하는가?
- 미디어 서버와 전송 서버의 책임은 어떻게 다른가?
- 실시간성, 캐싱, 화질 변환, 업로드 대역폭 중 어떤 병목을 먼저 볼 것인가?
- 브라우저가 실제로 읽을 수 있는 형식으로 언제 변환되는가?

### 4. 도구 선택은 결국 `경계에서 잃는 것`을 관리하는 일이다
API gateway는 HTTP 요청과 내부 서비스 경계를, gRPC는 메시지 형식과 호출 책임 경계를, HLS/RTSP/WebRTC는 실시간 신호와 브라우저 재생 경계를 다룬다. 따라서 이 후반 메모들은 산만한 기술 호기심이 아니라, 도구를 붙일 때마다 생기는 형식 변환 비용과 관측 가능한 경계를 찾는 기록이다.

이 관점에서 Integration/Tool은 단순히 도구를 모으는 페이지가 아니다. 손에 붙는 로컬 도구부터 원격 서버, gateway, 프로토콜, 스트리밍 체인까지 `상태가 다음 표면으로 넘어갈 때 무엇을 보존하고 무엇을 포기하는가`를 계속 묻는 통합 노드다.

## 변경 범위를 실행 단위로 줄이는 도구 감각
원문 중반의 `git diff --name-only | grep 'packages'`, 패키지 이름 추출, `BUILD_CONTEXT`와 `BUILD` 인자를 받는 Dockerfile 메모는 단순한 빌드 스크립트 조각이 아니다. 여러 패키지가 섞인 작업 공간에서 `무엇이 바뀌었고, 그 변경이 어느 실행 단위까지만 다시 만들어져야 하는가`를 좁히려는 기록이다.

이 부분은 Integration/Tool의 중심 주제와 잘 맞는다. 도구를 통합한다는 것은 모든 것을 한 번에 다시 빌드하는 일이 아니라, 변경 신호를 읽고 필요한 작업면만 다시 여는 일이다.

### 1. 변경 파일 목록은 배포 입력 신호다
`git diff --name-only`로 바뀐 경로를 뽑고, `packages` 아래의 작업 단위를 추출하려는 메모는 Git을 단순 히스토리 도구가 아니라 빌드 라우터의 입력으로 쓰는 감각을 보여 준다. 여기서 중요한 것은 파일명이 아니라 `변경이 어느 패키지의 책임인가`를 빠르게 좁히는 일이다.

### 2. Dockerfile의 `BUILD_CONTEXT`는 재사용 가능한 실행 계약이다
원문 Dockerfile은 루트의 `package.json`, `yarn.lock`, 공통 패키지, 개별 패키지의 `.next`, `public`, `node_modules`를 어떻게 복사할지 고민한다. 이는 컨테이너 이미지를 만드는 법보다, 같은 빌드 틀 안에서 어떤 패키지를 실행 대상으로 삼을지 바꾸는 계약에 가깝다.

### 3. 공통 패키지와 개별 패키지를 함께 읽어야 한다
`banadio-common`을 따로 복사하는 대목은 변경 범위를 줄이려 해도 공통 의존성을 무시할 수 없다는 신호다. 좋은 도구 통합은 `각 패키지만 빠르게`가 아니라, 공통층이 바뀌었을 때 어떤 실행 단위가 함께 영향을 받는지까지 보여 줘야 한다.

### 4. 빌드 최적화는 속도 문제가 아니라 재진입성 문제다
이 메모의 가치는 최신 Docker best practice로 고치는 데 있지 않다. 더 중요한 것은 `어떤 변경이 어떤 빌드를 다시 요구하는지`를 사람이 다시 이해할 수 있게 남긴 점이다. CI, GitHub Actions, Jenkins, ArgoCD 같은 상위 도구도 결국 이 변경 범위 판단을 자동화하려는 표면으로 이어진다.

따라서 이 노드에서 빌드 관련 조각은 개발 부록이 아니라, [[../deep-knowledge/devops]]와 [[../Integration/Work]]를 잇는 작은 관문이다. 변경 신호를 읽고, 실행 단위를 고르고, 검증 표면으로 넘기는 흐름이 있어야 도구 통합이 실제 운영 속도와 신뢰로 바뀐다.

## 후속 판단 기준
저자는 툴을 화려한 기능 소비보다 `작업 환경을 길들이는 운영 표면`으로 보는 경향이 강하다. 앱을 예쁘게 분류하기보다 실제 하루의 입력, 검색, 공유, 운영, 복구가 어떻게 이어지는지를 더 오래 본다. 그래서 새 도구 메모를 흡수할 때도 이름을 늘리기보다 아래 기준 중 하나를 실제로 선명하게 하는지 먼저 판정한다.

| 기준 | 원문에서 보이는 신호 | 본문에 남길 때 |
| --- | --- | --- |
| 상태 전달 | Raycast, fzf, ripgrep, Asana, Slack, Grafana/Prometheus, OpenClaw dashboard | 손, 동료, 서버, 문서 중 누가 어떤 표면에서 상태를 다시 읽는지 보일 때 |
| 수명 관리 | dotfiles, 글로벌 설치 목록, `sudo tailscale up --force-reauth`, GitHub SSH, backup, cron | 계정·키·버전·설정·접속 경로가 끊겼을 때의 복구 손잡이가 드러날 때 |
| 형식 경계 | Kong/declarative 설정, gRPC gateway, protobuf message, Lambda body, HLS `.m3u8`/`.ts` | 경계에서 무엇이 보존되고 무엇이 숨거나 손실되는지 더 정확해질 때 |
| 변경 라우팅 | `git diff --name-only`, `BUILD_CONTEXT`, `banadio-common`, Docker build, CI, ArgoCD | 변경 신호가 어느 실행 단위와 공통 의존성, 검증 표면으로 이어지는지 좁혀질 때 |
| 장기 편입 | OpenClaw 모델 변경, 스킬 설치, `ssh -N -L 18789:127.0.0.1:18789`, GitHub 노트 접근 | 기존 노트·서버·협업 흐름에 붙을 만큼 수명이 확인될 때 |

이 표에 들어가지 않는 새 앱 추천이나 단발 실험은 본문을 늘리기보다 원문 발췌 구조나 log에서만 다룬다. 반대로 새 메모가 기존 행 하나를 더 정확하게 바꿀 수 있으면, 별도 섹션을 추가하기보다 해당 행이나 관련 본문을 갱신한다.
