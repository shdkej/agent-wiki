# 일

- 원본: `source/shdkej-content/Integration/Work.md`
- 상태: 안정 디벨롭
- 역할: 원본 만다라트 노드와 1:1 대응하는 업무 실행 카테고리 페이지

## 한눈에 보는 요약

`Integration/Work`는 생산성 팁이나 성실성 다짐이 아니라, 흩어진 요구·시간·검토·보고·기록을 하나의 신뢰 가능한 실행 단위로 닫는 노드다. 원문은 플랜, 우선순위 3개, 한 포모도로에 하나의 업무, 요구사항 확인, KPI, 예외 처리, 약속 준수, 전문가의 암묵지, 커밋·테스트·PR, Work Log와 헤드라인을 한 파일에 둔다. 이를 묶으면 핵심 질문은 `믿고 맡길 수 있는 사람은 어떤 상태 신호를 남기며 일하는가?`가 된다.

이 페이지에서는 `일`을 네 가지 운영면으로 압축한다.

| 운영면 | 원문 단서 | 오래 남길 의미 |
| --- | --- | --- |
| 전환 게이트 | 플랜-진행-마무리, 검토 반복, 예외와 반례 | 다음 상태로 넘어가도 되는지 확인하는 업무 품질 장치 |
| 신뢰 회수 | 안심 첫 문장, 도움 요청, 약속한 시간, 헤드라인 | 상대가 현재 위치와 판단 근거를 이해하게 만드는 전달 표면 |
| 전문성 축적 | 경험, 선임의 판단, 논문, 암묵지, 경력 | 애매한 상황에서 `이렇게 해도 되나?`에 더 나은 답을 주는 사례 저장소 |
| 업무일지 루프 | 코넬식 배치, 큰 타임라인, LLM 정리, 예상/실제 비교 | 이번 일을 다음 업무 산정과 검토 순서의 입력으로 바꾸는 기록 시스템 |

## 만다라트 위치

`Integration/Work`는 루트 [[../Meta/일]]처럼 일의 가치관 전체를 다루는 칸도 아니고, [[../Communication/Teamwork]]처럼 팀 구조와 handoff를 설계하는 칸도 아니다. Integration 아래의 이 노드는 목표, 요구사항, 코드, 문서, 도구, 보고를 한 실행 단위로 통합하는 자리다.

그래서 원문의 `더 열심히`, `프로답게`, `일잘러` 같은 표현은 자기계발 문장으로 평평하게 읽으면 약하다. 이 노드의 관심은 실제로는 더 구체적이다.

- 받은 요구를 그대로 구현하지 않고 완료 기준과 영향 범위를 확인한다.
- 우선순위와 작업 단위를 좁혀 인터럽트 비용을 줄인다.
- 반례, 예외, 사이드이펙트, 실제 사용 테스트를 상태 전환마다 확인한다.
- 보고와 업무일지로 진행 상황과 판단 맥락을 보이게 만든다.
- 예상 시간과 실제 소요의 차이를 다음 업무 산정으로 되돌린다.

## 원문 구조를 업무 루프로 다시 읽기

### 1. 목표 해석

원문은 일의 시작을 `바로 구현`으로 두지 않는다. KPI 기반 목표, 요구사항 구체화, 문구 하나의 중요성, 걱정되는 부분을 먼저 테스트한다는 메모가 반복된다. 특히 `유저의 A 화면에서 B 화면으로 전환하는 클릭률을 20% 높인다` 같은 목표를 보며, 세부 태스크를 받는 사람에서 측정과 검증 방법을 고민하는 사람으로 시선이 이동한다.

여기서 좋은 시작은 다음 질문을 먼저 여는 것이다.

- 이 일의 성과는 어떤 지표나 상태 변화로 확인되는가?
- 요구사항 중 애매한 문구와 숨은 전제는 무엇인가?
- 고객·사용자·개발자 시선에서 각각 무엇을 확인해야 하는가?
- 파악, 실행, 정리에 시간이 얼마나 걸릴 것으로 보는가?

### 2. 실행 설계

원문 첫머리의 `플랜 / 진행 / 마무리`는 이 노드 전체의 뼈대다. 큰 문제를 잘게 쪼개고, 레퍼런스를 모으고, 테스트 케이스를 생각하고, 실제 소요를 다시 확인한다. `우선순위 업무 3개`와 `한 포모도로에 하나의 업무`도 같은 축에 있다. 집중은 오래 버티기가 아니라, 현재 업무 단위를 흐트러뜨리지 않는 설계에 가깝다.

실행 설계에서 특히 보존할 장면은 다음이다.

- 우선순위 외의 업무를 중간중간 끼워 넣지 않는다.
- 막혔을 때 다른 일로 전환하더라도 전환 자체를 의식적으로 관리한다.
- 하나의 일을 개발자, 고객, 사용자 세 시선으로 다시 본다.
- 팔로우업 단계를 태스크에 포함해 `나중에 해야 할 것`을 보이게 둔다.

### 3. 검토와 예외

`검토 검토 또 검토`는 마지막 확인 의식이 아니라 상태 전환 게이트다. 원문은 반례, 특수 상황, 업데이트 후 기존 동작, 사이드이펙트, 놓친 부분 확인을 반복해서 묻는다. 이는 느리면서도 놓치는 상태를 줄이려는 메모다.

| 상태 전환 | 확인할 질문 | 남길 흔적 |
| --- | --- | --- |
| 요구사항 → 계획 | 완료 기준과 애매한 문구를 풀었나? | 확인한 요구, 보류 질문, 기간 산정 |
| 계획 → 구현 | 가장 작은 실행 단위와 우선순위를 골랐나? | 태스크 분해, 레퍼런스, 테스트 후보 |
| 구현 → 검토 | 개발자·고객·사용자 시선에서 다시 봤나? | 반례, 예외, 사이드이펙트 체크 |
| 검토 → 반영 | 기존 동작과 연결 부분을 확인했나? | 검증 결과, 남은 위험, 팔로우업 |
| 반영 → 회고 | 예상과 실제 차이를 다음 일에 돌려보냈나? | 헤드라인, 한 줄 회고, 개선점 |

이 표의 핵심은 체크리스트를 늘리는 것이 아니라, 다음 사람이 안심하고 이어 받을 수 있는 상태 표시를 만드는 것이다.

### 4. 신뢰 전달

보고와 도움 요청 메모는 업무 품질의 일부다. 원문은 `안심 첫 문장`, 상대가 원하는 것 파악, 문제를 정확히 설명하기, 시도한 방법과 막힌 지점을 분리하기를 적는다. 좋은 보고는 친절한 말투보다 상대의 불확실성을 줄이는 형식에 가깝다.

도움 요청은 다음 구조로 닫힌다.

- 어떤 건으로 무엇을 이야기하려는지 첫 문장에서 밝힌다.
- 현재 문제, 시도한 방법, 막힌 위치를 분리한다.
- 정확히 어떤 부분에 도움이 필요한지 말한다.
- 일정이나 품질에 미칠 영향을 함께 드러낸다.

이 축은 [[../Communication/Teamwork]]의 handoff와 이어지지만, 이 페이지에서는 팀 문화 일반론보다 개인 업무가 신뢰 가능한 상태로 전달되는 형식에 집중한다.

### 5. 프로페셔널과 전문성

원문에는 프로페셔널, 전문가, 경력, 덕후, 아마추어가 함께 등장한다. 이 단어들을 한 서열표로 합치면 긴장이 사라진다.

- 프로페셔널: 약속한 시간, 결과, 마감 품질, 마지막 한 끗, 책임 범위를 보이게 만든다.
- 전문가: 선임 경험, 연구, 실무 사례, 암묵지로 애매한 판단을 줄인다.
- 경력: `이걸 이렇게 해도 되나?`에 답할 수 있는 상황 경험의 축적이다.
- 덕후/딜레탕트: 깊이 파는 즐거움과 검증 가능한 전문성 사이의 경계를 계속 시험한다.
- 아마추어: 노력은 있을 수 있지만 결과와 신뢰 회수까지 닫히지 않을 수 있다.

따라서 성장 루프는 `더 오래 일하기`가 아니다. 먼저 맡은 일을 예측 가능하게 닫아 신뢰를 만들고, 그 과정에서 만난 예외와 애매한 결정을 기록해 다음 판단의 전문성으로 축적하는 것이다.

### 6. 코드 변경 단위

원문 후반의 `일과 코딩`은 개발 팁 목록이 아니라 업무 판단이 코드 변경으로 내려가는 구간이다. 업무 기획 시나리오가 코드에 녹아 있어야 하고, 이전 판단이 다른 변경사항에 영향을 받을 수 있으며, 비즈니스 로직과 테크니컬 로직이 구분되어야 한다는 메모가 이어진다.

코딩 업무의 신뢰 신호는 다음처럼 읽힌다.

- 요구사항은 다이어그램, 테스트 코드, 동작하는 함수로 내려와야 한다.
- 커밋은 기능별로 작게 나누되, 동작 가능한 단위여야 한다.
- 커밋 전에는 짧게 확인 가능한 테스트를 돌리고, PR에서는 더 넓은 검증 결과가 보여야 한다.
- CI 결과는 개발 절차 부록이 아니라 `이번 변경을 믿고 넘겨도 되는가`를 보여 주는 상태 신호다.
- 검사하기 좋은 아키텍처는 취향이 아니라 변경 단위를 신뢰할 수 있게 만드는 조건이다.

이 축은 [[../Communication/Document]]의 문서화, [[../deep-knowledge/devops]]의 CI 상태 전달, [[../Integration/Tool]]의 변경 신호 라우팅과 맞닿아 있다.

### 7. Work Log

원문 하단의 Work Log는 완료 목록이 아니라 작업 인터페이스다. 큰 타임라인, 고민과 실행의 분리, 코넬식 배치, 아사나와 깃랩 연결, 작업 전중후 정리, LLM과 함께 문서화, 헤드라인 예시가 모두 여기에 들어 있다.

특히 `완료 중심`이 아니라 `의사결정 맥락 중심`으로 헤드라인을 붙이라는 메모가 중요하다. 버튼 위치를 바꿨다는 사실보다, 영업팀과 데이터팀의 요구가 충돌했을 때 어떤 실험 구조로 조율했는지가 더 오래 남는 성과라는 뜻이다.

업무일지에 남길 최소 단위는 다음과 같다.

| 기록 단위 | 남길 내용 | 다시 쓰이는 곳 |
| --- | --- | --- |
| 작업 전 | 목표, 요구사항, 기간 산정, 우려 지점 | 시작 기준과 우선순위 |
| 작업 중 | 시도, 막힘, 실제 테스트, 변경 영향 | 도움 요청과 검토 게이트 |
| 작업 후 | 완료 상태, 예상/실제 차이, 다음 개선점 | 기간 산정 보정과 회고 |
| 헤드라인 | 갈등, 판단 기준, 조율 방식, 성과 맥락 | 평가와 자기 설명 |
| LLM 정리 | 요구사항, 코드 결과물, 대화, 문서의 단일 출력 | 다음 업무의 입력 문서 |

LLM 메모도 도구 목록보다 이 루프가 핵심이다. 여러 입력을 하나의 컨플루언스 문서로 마감하고, 다시 그 문서를 다음 입력으로 쓰려는 흐름은 AI를 검색 보조가 아니라 업무 마감 루프를 묶는 정리 엔진으로 보는 관점이다.

## 반복 판단 모델

| 반복 축 | 약해지는 읽기 | 더 나은 읽기 |
| --- | --- | --- |
| 빨리 하기 | 처리량을 늘린다 | 빠르게 피드백 받되 현재 책임 범위와 남은 위험을 보이게 한다 |
| 꼼꼼함 | 모든 것을 다 본다 | 상태 전환마다 생략하면 안 되는 확인면을 정한다 |
| 열심히 | 오래 일하고 더 버틴다 | 우선순위와 작업 단위를 좁혀 결과가 남는 시간을 만든다 |
| 프로 | 멋있는 태도나 열정 | 약속 시간, 마감 품질, 마지막 한 끗으로 신뢰를 증명한다 |
| 전문가 | 많이 아는 사람 | 애매한 상황에서 더 나은 판단을 줄 수 있는 경험을 가진 사람 |
| 업무일지 | 완료 목록 | 판단 맥락과 다음 업무 산정의 입력 |

## 보존해야 할 원문 장면

- `우선순위 업무 3개`와 `한 포모도로에 하나의 업무`: 집중을 의지보다 작업 단위 설계로 다루는 장면이다.
- 개발자·고객·사용자 `3가지 시선`: 품질을 구현자의 만족으로 닫지 않는 검토 규칙이다.
- KPI와 A/B 테스트: 일을 지시받는 단위에서 측정과 검증을 설계하는 단위로 옮겨 가는 단서다.
- `안심 첫 문장`: 보고를 상대의 불확실성을 낮추는 인터페이스로 보는 문장이다.
- `문제-시도-막힌 지점` 도움 요청: 혼자 끌어안지 않고 상태를 전달 가능한 형식으로 바꾸는 규칙이다.
- `파악하는 시간 / 실행하는 시간 / 정리하는 시간`: 예상 시간을 압박 숫자가 아니라 다음 산정의 계측값으로 만드는 구조다.
- `커밋할 때는 짧게 확인, PR에서 더 진행`: 코드 변경을 업무 신뢰 신호로 닫는 개발 게이트다.
- 헤드라인 예시: 완료 사실보다 갈등과 판단 맥락이 성과 언어가 된다는 감각이다.

## 후속 흡수 기준

새 메모가 들어오면 이 페이지를 넓히기보다 아래 중 하나를 실제로 선명하게 할 때만 본문에 흡수한다.

- 업무가 어떤 상태 전환 게이트를 통과했는지 더 잘 보이게 하는가?
- 예상 시간과 실제 소요의 차이를 다음 산정에 쓸 수 있게 만드는가?
- 보고, 도움 요청, 헤드라인처럼 신뢰 전달 형식을 더 구체화하는가?
- 코드 변경 단위, 테스트, PR, CI 같은 개발 흐름의 완료 신호를 더 선명하게 하는가?
- 프로페셔널함과 전문성을 `더 열심히`가 아니라 신뢰 회수와 판단 사례 축적으로 구분하게 하는가?
