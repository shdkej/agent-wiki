# 디자인

- 원본: `source/shdkej-content/Integration/Design.md`
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트
- 상태: 기준선 재구성 후 안정 디벨롭

## 핵심 판단 모델

`Integration/Design`은 취향 보드가 아니라 **인상, 위계, 접근성, 구현 회복을 하나의 제품 언어로 묶는 노드**다. 원문은 오뚝이·나무·여유·유머 같은 이미지에서 출발하지만 곧바로 로고, 톤, 앱 이름, 설명, 디자인 시스템, 스크린리더, 색상 대비, TOC 폭, anchor offset으로 내려간다. 그래서 이 페이지의 중심 질문은 `어떤 화면이 예쁜가`보다 `어떤 의미가 여러 사용자와 환경에서 같은 제품답게 도착하는가`에 가깝다.

| 판단 칸 | 원문 단서 | 남길 기준 |
| --- | --- | --- |
| 인상 | 오뚝이, 나무, 여유, 관심, 유머, 용기, 초록·노랑·빨강·보라 | 화면이 어떤 사람처럼 보일지 먼저 정한다. |
| 위계 | 크기, 굵기, 위치, 색, 명도, 여백, 질감, 움직임, depth | 인상어를 시선 순서와 정보 레이어로 번역한다. |
| 접근성 | 스크린리더, 의미 있는 링크, 색상 의존 금지, 명암비, 평평한 길 | 보기 좋은 표면이 실제로 건널 수 있는지 검증한다. |
| 구현·회복 | 모바일 퍼스트, TOC 폭, font-size, anchor offset, 실패 복귀 버튼, 페이징·자동 로딩 | 웹의 물리 조건과 실패 후 재진입까지 디자인에 포함한다. |

## 만다라트 위치

이 노트는 `Integration` 묶음 안에 있으므로 순수 미감 자료나 CSS 팁 목록으로 읽으면 좁아진다. 같은 폴더의 [[Marketing]]이 채널과 고객 반응 언어를 다루고, [[Creator]]가 감각을 결과물로 바꾸는 제작 표면을 다룬다면, Design은 그 감각과 언어가 실제 인터페이스에서 **보이고, 읽히고, 다시 시도될 수 있는 구조**를 맡는다.

## 원문 장면별 구조

### 1. 인상과 취향

원문 앞부분의 좋아하는 이미지와 감정어는 단순 mood board가 아니다. 오뚝이, 나무, 오타니, 야마모토, 여유, 사랑, 유머, 적극성, 몰입, 편집, 정리, 밸런스, 간결 같은 단어는 화면이 가져야 할 태도 후보로 남아 있다. 색 후보도 팔레트 자체보다 어떤 에너지와 감정을 반복할지의 신호에 가깝다.

첨부 이미지 4장은 이 인상어를 눈으로 검증하는 샘플 보드다. 텍스트의 `톤은 맞추되 깊이감을 만든다`, `명암`, `그라데이션`, `미니멀하지만 납작하지 않음`이 실제로 어떤 표면을 가리키는지 붙잡아 둔다.

### 2. 위계와 뼈대

`뼈대가 먼저다`, `위계가 중요하다`는 메모가 반복된다. 디자인은 색을 입히는 마지막 단계가 아니라 무엇을 먼저 보게 하고 무엇을 배경으로 물러나게 할지 정하는 구조 작업이다. 크기, 컬러, 레이아웃, 질감, 원근, 아웃포커싱, 여백, 움직임은 각각 다른 레이어이며, `색에서 명도가 글자에서 굵기이려나`라는 질문은 색과 타이포그래피를 같은 위계 언어로 묶으려는 신호다.

이 기준에서 미니멀함과 깊이감은 반대가 아니다. 원문은 화이트·블랙 기반, 키 컬러 절제, 디터 람스식 미니멀에 끌리지만 동시에 자연의 명암, 곡선, 그림자, 원근감도 붙든다. 원하는 것은 화려한 장식이 아니라 **정돈되어 있으면서도 층이 느껴지는 화면**이다.

### 3. 디자인 시스템

`로고, 톤, 앱이름, 설명의 일관성`, `내 제품의 디자인 시스템`, `앱도 같은 테마`, `인스타에 올릴 사진에 대한 감각`은 버튼 라이브러리보다 먼저 제품 언어를 요구한다. 같은 사람이 만든 것처럼 보이려면 색상표, 글꼴, 문장 톤, 이미지 감각, 설명 방식이 따로 놀지 않아야 한다.

shadcn ui, LiftKit, 최신 CSS 같은 도구 메모는 취향을 대체하는 정답이 아니라 이미 잡은 제품 언어를 반복 가능하게 만드는 실행 표면이다. 새 도구나 레퍼런스가 들어와도 먼저 `어떤 인상 칸과 위계 칸을 더 선명하게 하는가`를 봐야 한다.

### 4. 접근성

접근성 섹션은 부록이 아니라 이 노트의 윤리 중심이다. 원문은 인터넷에서 장애인과 비장애인의 경계가 없고, 접근성은 최소기능이 아니라 필수기능이며, 벽이나 다리보다 평평한 길이어야 한다고 적는다. 따라서 접근성은 완성 후 검사하는 체크리스트가 아니라 디자인 토큰이 처음부터 따라야 할 제약이다.

| 접근성 단서 | 디자인 판단 |
| --- | --- |
| 스크린리더가 네비게이션을 반복해서 읽지 않게 하기 | 보이는 위치뿐 아니라 읽히는 순서까지 위계에 포함한다. |
| 링크 텍스트 자체가 의미를 가지게 하기 | `여기`, `자세히`처럼 떨어져 나가면 뜻을 잃는 문구를 피한다. |
| 컬러만으로 의미 전달하지 않기 | 색, 텍스트, 형태, 상태 설명이 함께 같은 의미에 도착해야 한다. |
| 글자마크와 가벼운 그림 표식 | 언어 장벽을 낮추되 대체 텍스트와 주변 문맥을 함께 둔다. |

## 색, 폰트, 다크모드

색 메모는 RGB·CMYK·색상환·보색 같은 지식 수집이면서 동시에 인터페이스 안정성 질문이다. 시스템 컬러가 있으므로 다크/화이트를 메인으로 두고 키 컬러를 살짝 넣는다는 문장은 색을 브랜드 인상과 피로도 사이에서 조절하려는 태도다.

폰트 메모는 굴림체의 역사, 기본 글꼴의 익숙함, 한글 자형과 간격 문제를 함께 본다. 다크모드는 완전한 검정보다 낮은 채도와 적절한 대비, depth를 강조한다. 둘 다 `보기 좋은가`보다 `오래 읽을 수 있는가`, `객체가 제대로 분리되는가`를 묻는다.

## 웹 구현과 회복 장치

후반의 `in WEB` 메모는 round vs non-round, shadow, font-size, line-height, TOC 폭, anchor offset처럼 구체적이다. 동시에 `웹페이지를 영화의 미장센처럼 꾸민다?`, `사진처럼 꾸민다?`라는 질문도 남아 있다. 이 조합 때문에 Design 노드는 장면 연출과 레이아웃 엔지니어링을 분리하지 않는다.

| 원문 장면 | 디자인 의미 |
| --- | --- |
| `Should have make fail return button` | 실패 화면, 취소, 되돌리기, 재시도는 미감이 아니라 사용자가 다시 시도할 수 있다는 신뢰 신호다. |
| 목록 창의 페이징과 자동 로딩 | 긴 목록에서 어디까지 훑고 어디서 멈추며 어떻게 돌아올지 정하는 탐색 리듬이다. |
| TOC 20%, screen `< 1200`, anchor offset | 장면 연출이 실제 화면 폭, 고정 헤더, 읽기 흐름에서 깨지지 않게 하는 물리 조건이다. |
| font-size 19-28, mobile 18, line-height | 폰트 취향보다 오래 읽는 피로도와 정보 밀도를 조정하는 기준이다. |

## 원문 구조 압축

| 원문 묶음 | mapped에서의 역할 |
| --- | --- |
| 디자인 아이디어 | 제품이 남길 인상어와 미감 후보를 모으는 층 |
| 내 제품의 디자인 시스템 | 인상어를 로고, 앱 이름, 설명, 사진, 화면 톤으로 반복하는 층 |
| Basic | 그림자, 간격, 곡률, 시스템 컬러, 모바일 퍼스트 같은 기본 제약 |
| 접근성 챙기기 | 모두가 같은 의미에 도착하게 하는 필수 조건 |
| Color / Font / Darkmode | 색·글자·대비를 역할과 피로도 관점에서 조절하는 층 |
| in WEB | 감각이 브라우저 폭, 헤더, 목록, 읽기 흐름에서 깨지지 않게 하는 구현 층 |
| Reference | 레퍼런스가 독립 지식이 아니라 위 판단 칸을 보강하는 보조 자료 |

## 후속 흡수 기준

새 디자인 메모는 독립 섹션을 늘리기 전에 아래 중 하나를 실제로 대체하거나 선명하게 할 때만 본문에 흡수한다.

| 새 메모 성격 | 흡수 위치 |
| --- | --- |
| 좋아하는 이미지, 색, 무드 | `인상과 취향`의 인상어 또는 이미지 레퍼런스 해석 |
| 레이아웃, 크기, depth, motion | `위계와 뼈대`의 레이어 판단 |
| 접근성, 링크, 스크린리더, 색상 대비 | `접근성` 표의 의미 도착 경로 |
| 실패, 긴 목록, 로딩, 반응형, 웹 수치 | `웹 구현과 회복 장치`의 신뢰·탐색 리듬 |
| 도구, UI kit, CSS 레퍼런스 | `디자인 시스템`의 반복 가능성 기준 |

## 드리프트 점검

대응 원본 `source/shdkej-content/Integration/Design.md`의 수정 시각은 mapped 노트보다 오래되어, 이번 실행의 직접 사용자 수정본으로 판단하지 않았다. 따라서 `human-reviews/` 복사는 생략했다. 외부 링크 소스는 사용하지 않았다.
