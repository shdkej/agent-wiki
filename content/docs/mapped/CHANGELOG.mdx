# CHANGELOG

- 원본: `source/shdkej-content/CHANGELOG.md`
- 대응 mapped: `agent-wiki/content/docs/mapped/CHANGELOG.mdx`
- 상태: 점진 심화
- 역할: 루트 만다라트에서 `개인 웹을 어떻게 계속 굴렸는가`를 붙드는 카테고리 노드

## 한눈에 보는 요약
이 노드는 기능 연혁보다 운영 압력 지도에 가깝다. 원문에는 검색 노출, 광고 실험, 화면 손질, Gatsby 기반 유지, 짧은 `update` 메모가 한 파일 안에 겹쳐 있다. 따라서 이 CHANGELOG는 `무엇을 만들었는가`보다 `무엇을 계속 다시 맞춰야 했는가`를 보여 준다.

## 핵심 해석
개인 웹은 여기서 단순한 글 저장소가 아니라 다음 네 일을 동시에 요구하는 표면으로 나타난다.

1. **보이게 만들기** — SEO, sitemap, 검색 진입면 정리
2. **붙여 보기** — Adsense, Kakao adfit 같은 현실적 수익 실험
3. **버티게 만들기** — Gatsby monorepo, starter, plugin, dependency 대응
4. **다시 만지기** — 404, popup, fragment, 자잘한 `update`와 `fix`

즉 이 파일의 중심은 제품 출시 서사보다 `발행 → 실험 → 유지 → 미세조정`의 반복 루프다.

## 왜 이 노드가 중요한가
- [[Portfolio]]가 팀 규모의 운영 능력을 보여 준다면, 이 노드는 개인 웹 표면에서 그 감각이 얼마나 생활형 유지 노동으로 나타났는지 보여 준다.
- [[Communication/Blogging]]이 `무엇을 발행할 것인가`를 붙든다면, 이 노드는 `그 발행면을 계속 살아 있게 만드는 비용`을 붙든다.
- [[Meta/My_space]]가 나중의 자기 소유 제어면 상상이라면, 이 CHANGELOG는 그보다 이른 단계의 `생활형 운영 박동`이다.

## 원문 구조를 읽는 기본 틀
원문은 `Features`와 `Bug Fixes`로 나뉘지만, 실제로는 아래 다섯 층이 겹친다.

| 층위 | 대표 흔적 | 실제 의미 |
| --- | --- | --- |
| 생활형 반영 | `update 21-04-20`, `🎸 update 21-04-23` | 큰 기능보다 자주 손보는 운영 리듬 |
| 발행면 확장 | `seo component`, `sitemap`, `google custom search` | 글을 `발견 가능하게` 만드는 작업 |
| 수익 실험 | `adsense`, `adfit`, script 삽입 | 공개 공간에 현실적 옵션을 붙여 보는 단계 |
| 표면 손질 | `update index with grid`, `update 404 page`, popup 조정 | 방문자가 직접 만지는 화면 조정 |
| 기반 상속 노동 | `gatsby monorepo`, starter/plugin/dependency 업데이트 | 선택한 프레임워크가 끌고 오는 장기 유지비 |

이 다섯 층을 먼저 잡아야, 원문이 왜 기능 목록보다 운영 장부처럼 보이는지 이해된다.

### `update`는 한 종류가 아니다
이번에 원문 전체를 다시 세어 보니 `update`라는 단어가 들어간 불릿은 229개지만, 그중 저자가 직접 날짜를 박아 남긴 생활형 `update 20-10-26`류는 50개다. 나머지는 `update gatsby monorepo`, `update dependency ...`, `update starters ...`처럼 upstream 변화 흡수나 일반 갱신 메모에 더 가깝다.

| `update` 층 | 횟수 | 의미 |
| --- | ---: | --- |
| 날짜형 생활 메모 | 50개 | 오늘 다시 열어 손본 개인 웹 운영 박동 |
| 상속/생태계·일반 갱신 메모 | 179개 | Gatsby 기반, dependency, starter가 밀어 넣는 유지 일정과 일반 갱신 |

이 구분이 중요하다. 겉보기에는 모두 같은 `update`지만, 실제로는 **직접 다시 만진 시간**과 **기반을 따라가느라 생긴 시간**이 겹쳐 적혀 있다. 그래서 이 노드는 `업데이트가 많다`로 끝내기보다, `어떤 업데이트가 내 판단에서 나왔고 어떤 업데이트가 기반 선택에서 따라왔는가`를 함께 읽어야 한다.

### `Features` / `Bug Fixes` 이름보다 더 중요한 역전
형식상으로는 기능과 버그 수정을 나누지만, 실제 분포는 거의 반대로 읽힌다.

- `Features`에도 날짜형 `update`, 광고 삽입, 검색 노출 정리, starter 상속 항목이 섞여 있다.
- `Bug Fixes`에는 단순 결함 수정만이 아니라 dependency 보안 반영, monorepo 추적, 화면 자잘한 보정이 대량으로 들어 있다.
- 그래서 이 파일의 진짜 중심은 `무엇을 새로 냈는가`보다 **얼마나 자주 다시 맞췄는가**에 있다.

### 날짜형 `update`가 몰린 구간
`update` 표기는 고르게 흩어지지 않고 몇 구간에 뭉친다.

| 구간 | 횟수 | 읽기 포인트 |
| --- | --- | --- |
| 2020-10 | 9회 | 개인 웹 표면을 자주 다시 열어 보는 초기 박동 |
| 2020-11 | 11회 | 발행면과 기반 조정이 함께 잦아지는 구간 |
| 2020-12 | 7회 | 연말에도 멈추지 않는 생활형 유지 |
| 2021-02 | 11회 | upstream 반영과 직접 손질이 다시 겹친 구간 |
| 2021-03 | 5회 | notes theme, grid, index 정리가 섞이는 전환기 |
| 2021-04 | 6회 | 큰 릴리스보다 짧은 재방문으로 끝맺는 후반 리듬 |
| 2020-11로 보이는 `11/11` | 1회 | `add google adsense, update 11/11`처럼 날짜가 축약돼 있어 별도 확인이 필요한 보조 표식 |

이 표는 `업데이트가 많다`는 인상만 주는 대신, **짧은 재방문이 실제로 언제 몰렸는지**를 다시 보게 해 준다.

### 계수 정정이 바꾸는 해석
이전 mapped 초안은 날짜형 생활 `update`를 133회로 과대 계산했다. 원문 불릿 재계수 기준은 `update` 포함 불릿 229개, 날짜형 생활 `update` 50개다. 이 정정은 해석을 약하게 만들기보다 더 선명하게 만든다.

- 생활형 재방문은 50개만으로도 충분히 반복적이다. 다만 전체의 대부분이라고 말하면 안 된다.
- 전체 `update`의 압도적 비중은 개인이 자주 만진 시간뿐 아니라 Gatsby, starter, dependency가 끌고 온 상속 시간을 함께 반영한다.
- 따라서 이 노드는 `생활형 박동` 하나가 아니라 **생활형 박동과 상속 유지비가 같은 단어로 눌려 들어간 기록**으로 읽어야 한다.

## 버전별로 압축해서 읽으면 보이는 것

### 0.1.1은 `표면 확장 + 기반 유지`가 동시에 시작되는 구간이다
원문 `Features`에는 아래가 한꺼번에 섞여 있다.
- `add an seo component`
- `add sitemap, google custom search`
- `ADD adsense script`
- `ADD kakao adfit`
- `update index with grid`
- Gatsby starter 상속 항목들

겉으로는 기능 추가지만, 실제로는 `발견 가능성`, `수익 실험`, `표면 정리`, `기반 상속`이 동시에 움직이는 초반 확장기다.

### 0.1.1의 `Bug Fixes`는 실제로 유지 노동 묶음이다
여기에는 아래가 함께 들어 있다.
- `remove unnecessary fragment`
- `update 404 page`
- `Update popup color and position`
- dependency security fix
- 반복적인 `gatsby monorepo` 업데이트

즉 이 절은 단순 버그 수정 목록이 아니라, `직접 만진 표면`과 `기반이 요구한 상속 노동`이 같은 버전 껍데기 안에서 평평해지는 구간이다.

### 0.1.2는 큰 전환보다 생활형 update의 지속을 보여 준다
마지막 버전은 사실상 `update 21-04-20`, `🎸 update 21-04-23` 같은 짧은 반영이 중심이다.
- changelog의 끝이 거대한 선언으로 닫히지 않는다.
- 이미 운영 중인 웹 표면을 계속 다시 맞추는 리듬이 유지된다.
- 이 파일의 핵심이 대형 마일스톤보다 `지속적 손질`임을 다시 확인시킨다.

## 대표 커밋 묶음으로 보면 더 선명해지는 것

### 1. 유입면을 먼저 여는 묶음
- `add an seo component that can be used between pages/templates`
- `add sitemap, google custom search`

이 묶음은 블로그나 개인 웹을 `써 두면 누가 보겠지` 수준으로 두지 않는 태도를 보여 준다. 먼저 해야 할 일은 화면 장식보다 **검색엔진과 방문자가 들어올 진입면을 정리하는 것**이다.

### 2. 현실 수익을 시험하는 묶음
- `ADD adsense script`
- `ADD kakao adfit`

광고 스크립트는 부수 기능이 아니라, 공개 공간을 취미 표면에만 두지 않고 **현실 경제성과 함께 시험해 보는 운영 선택**이다. 다만 반복 횟수는 높지 않아, 수익화 자체가 주제의 전부가 아니라 `붙여 보고 판단하는 보조 실험`으로 읽는 편이 맞다.

### 3. 사람이 바로 체감하는 표면 손질 묶음
- `update index with grid`
- `update 404 page`
- `Update popup color and position`

이 항목들은 작은 수정처럼 보이지만, CHANGELOG가 백엔드 유지 일지에만 머물지 않는다는 증거다. 저자는 **들어오는 경험, 찾지 못했을 때의 경험, 방해받는 경험**까지 운영 범위로 본다.

### 4. 기반 선택이 만든 상속 노동 묶음
- `update gatsby monorepo`
- `update starters and examples`
- 각종 starter/plugin/dependency 업데이트

이 묶음은 가장 반복적이고, 그래서 가장 중요하다. 원문의 많은 줄은 `내가 만들고 싶은 기능`보다 **선택한 기반이 계속 요구하는 따라가기 노동**으로 채워져 있다. 개인 웹이라도 framework를 택한 순간, 운영자는 창작자이면서 동시에 upstream 추적자가 된다.

## CHANGELOG 안의 두 종류의 시간

| 시간 종류 | 원문에서 보이는 표식 | 읽는 방법 |
| --- | --- | --- |
| 생활 시간 | `update 20-10-26`, `update 21-04-14` 같은 날짜형 메모 | 오늘 다시 열어 손본 생활형 운영 박동 |
| 릴리스 시간 | `0.1.1`, `0.1.2`, compare 링크, `standard-version` 문구 | 변경을 버전 껍데기 안에 묶어 두는 배포 형식 |

이 둘이 어긋난다는 점이 중요하다. 문서는 릴리스 문법을 쓰지만, 실제 본문은 생활 시간의 촘촘한 재방문으로 움직인다. 그래서 이 CHANGELOG를 읽을 때는 `버전이 어떤 기능을 뜻하는가`보다 **버전 형식이 생활형 운영을 어떻게 수습하고 있는가**를 보는 편이 더 정확하다.

## 운영 책임과 확장 반경
후반 해석은 새 섹션을 늘리기보다 아래 책임 표 하나로 접어 읽는 편이 안정적이다. 핵심은 `직접 만든 일`, `기반을 흡수한 일`, `짧게 다시 만진 일`, `그 일을 남긴 형식`을 분리하는 것이다.

| 책임 층 | 원문 표식 | 확장 반경 | 판정 질문 |
| --- | --- | --- | --- |
| 발행면 조정 | SEO, sitemap, search, 광고, grid, 404, popup | 검색엔진, 방문자, 광고 스크립트까지 닿는다 | 방문자가 마주하는 표면을 직접 바꿨는가? |
| 상속 유지비 | `**starters:**` 121회, `gatsby monorepo` 96회, starter/plugin/dependency | Gatsby 생태계와 starter 구조를 계속 따라간다 | 기반 선택이 밀어 넣은 따라가기 노동인가? |
| 공개 표면 상속 | `**gatsby:**`, `**blog:**`, `**www:**`, `**docs:**` | 빌드 도구를 넘어 블로그·문서·웹 표면까지 끌려온다 | 패키지 버전이 아니라 공개 표면의 동작을 보존하려는 변경인가? |
| 생활형 재방문 | 날짜형 `update`, 짧은 `fix`, remove/delete | 큰 마일스톤보다 자주 다시 여는 생활 시간에 묶인다 | 오늘 다시 열어 손본 운영 박동인가? |
| 기록 문법 | `0.1.1`, compare 링크, `standard-version`, 상세 upstream 제목 | 생활 시간과 외부 변경 큐를 한 릴리스 껍데기에 수습한다 | 변경의 책임 출처를 나중에 다시 구분할 수 있는가? |

이 표로 보면 `상속 로그도 한 덩어리가 아니라...`, `비용 구조와 릴리스 문법`, `루트 의미`가 같은 결론을 반복하지 않는다. 원문을 그대로 따라가면 `내 사이트를 위해 직접 한 일`과 `기반을 따라가느라 흡수한 일`이 모두 같은 `Features/Bug Fixes` 이름 아래 평평해진다. 이 노드의 가치는 그 책임 출처와 확장 반경을 다시 분리해 보여 주는 데 있다.

운영 시간축도 같은 모델로 접힌다. 초반에는 SEO, sitemap, search, 광고처럼 발행면 확장이 앞서고, 중반에는 Gatsby/starter/plugin/dependency 갱신이 길게 이어지며, 후반에는 짧은 날짜형 `update`가 운영 리듬을 남긴다. 흐름은 `발행면 확장 -> 기반 유지 장기화 -> 생활형 손질 지속`이다.

## 삭제와 정리 항목은 실패가 아니라 운영 표면을 줄이는 일이다
원문에서 `delete annoying document`, `remove resume`, `remove unnecessary fragment`, `remove comma in package.json`, `clear package`, `replaceAll to replace` 같은 항목은 크지 않아 보인다. 하지만 이 묶음은 CHANGELOG의 중요한 층이다. 개인 웹은 무언가를 계속 더하는 공간이면서, 동시에 방문자와 운영자를 방해하는 잔여물을 계속 걷어 내야 하는 공간이기 때문이다.

| 정리 항목 | 원문 흔적 | 읽는 법 |
| --- | --- | --- |
| 문서 정리 | `delete annoying document`, `remove resume` | 공개 표면에 남길 것과 치울 것을 다시 고른다. |
| 코드/구조 정리 | `remove unnecessary fragment`, `remove comma in package.json`, `replaceAll to replace` | 작은 오류나 불필요한 조각이 빌드·유지 흐름을 흐리지 않게 한다. |
| 사용자 표면 보정 | `update 404 page`, `Update popup color and position` | 사용자가 막히거나 방해받는 순간을 줄인다. |
| 기반 안정화 | `package-lock.json`, `update dependencies for security fix`, `upgrade dependencies for security fix` | 보이지 않는 의존성 상태를 다시 잠그고 안전하게 만든다. |

따라서 이 파일의 `remove/delete/fix`는 단순한 후퇴 기록이 아니다. 이미 공개된 웹 표면이 시간이 지나며 남기는 찌꺼기, 방해 요소, 잠재 위험을 줄이는 **운영 표면 축소 작업**이다. 이 관점으로 보면 `보이게 만들기`와 `붙여 보기`만큼이나 `덜어 내기`도 중요한 운영 능력이다.

이 점은 [[Communication/Blogging]]의 발행 전 편집 감각과도 연결된다. 블로그가 독자용 표면이라면, CHANGELOG의 정리 항목은 그 표면에 남은 불필요한 문서, 깨진 흐름, 어색한 UI를 다시 치우는 사후 편집이다. 또한 [[Deep Knowledge/Web]]의 흐름 보존 감각과 나란히 읽으면, 좋은 웹 운영은 기능을 늘리는 것뿐 아니라 사용자가 막히는 접점을 줄이는 일이라는 점이 더 선명해진다.

## 루트 의미와 인접 노드 역할 분담
CHANGELOG는 하위 기능 부록이 아니라 루트 노드다. 이 한 파일에 `사람이 들어오는 면`, `현실을 붙여 보는 면`, `사람이 실제로 만지는 면`, `그 표면을 살려 두는 바닥 면`이 모두 들어 있기 때문이다. 그래서 이 노드는 `개인 웹을 어떤 태도로 계속 굴렸는가`를 가장 직접적으로 압축해 보여 주는 카테고리로 남는다.

| 인접 노드 | 그 노드의 중심 | CHANGELOG의 차이 |
| --- | --- | --- |
| [[Portfolio]] | 팀 차원의 문제 해결과 운영 설계 | CHANGELOG는 개인 웹 수준의 생활형 유지 노동을 기록한다. 작은 마찰을 오래 방치하지 않는 습관이 직무 장면에서는 공용 운영 인터페이스 설계로 커진다. |
| [[Communication/Blogging]] | 무엇을 쓰고 어떻게 발행할지 | CHANGELOG는 그 발행면을 계속 살리는 실제 비용을 보여 준다. |
| [[Meta/My_space]] | 자기 소유 제어면, 배포/관측/복구 구조 | CHANGELOG는 그 상상 이전 단계의 생활형 운영 습관이다. |
| [[Communication/Document]] | 다시 들어오기 쉬운 설명 표면 | CHANGELOG는 설명 문서가 아니라 실제 diff와 유지 박동의 축적물이다. |
| [[Deep Knowledge/Web]] | 웹 표면의 흐름, 접속, 재호출 구조 | CHANGELOG는 검색엔진, 광고 스크립트, starter, dependency, upstream issue가 붙는 살아 있는 접속면을 개인 사이트의 반복 수정 기록으로 보여 준다. |

## 이 노드를 읽을 때 보존해야 할 긴장
- 개인 표현 vs 프레임워크 유지 비용
- 빠른 발행 vs 장기 안정성
- 수익화 실험 vs 사용자 경험
- 검색성 확대 vs 운영 복잡도
- 작은 사이트 감각 vs 실제 툴체인 부담

이 긴장을 지우면, 이 파일은 단순 커밋 목록으로 납작해진다.

## 안정 단계의 유지 기준
이제 이 노드는 큰 재작성보다 사실값과 책임 경계의 흔들림만 점검하는 편이 맞다. 새 해석을 더하기 전에 먼저 아래 기준을 확인한다.

| 점검 기준 | 유지 이유 |
| --- | --- |
| 날짜형 생활 `update` 50개와 전체 `update` 불릿 229개를 분리한다 | 생활형 재방문과 Gatsby·dependency 상속 갱신을 같은 시간으로 합치지 않기 위해서다. |
| `source/shdkej-content/CHANGELOG.md`와 이 mapped 페이지의 1:1 대응을 본문 메타 정보로 유지한다 | 별도 source-mapping 부재를 이유로 본문을 계속 확장하지 않기 위해서다. |
| 새 항목을 더할 때는 `운영 책임과 확장 반경` 표의 다섯 책임 중 하나에만 붙인다 | 발행면, 상속 유지비, 공개 표면 상속, 생활형 재방문, 기록 문법이 다시 중복 설명으로 갈라지는 일을 막기 위해서다. |
