# Operation

- 원본: `source/shdkej-content/Deep Knowledge/Operation.md`
- 상태: 초기 1:1 매핑
- 역할: 원본 만다라트 노드와 1:1 대응하는 운영 판단 노트

## 핵심 요약
이 노드는 운영을 배포 뒤의 잡무가 아니라 **장애를 인식하고, 보고하고, 되돌리고, 복구하고, 다시 피드백으로 회수하는 프로덕션 책임의 루프**로 다룬다. 원문은 SRE, DORA Metrics, 롤백, Postmortem, 로그 폭발, 권한 남용, 오토스케일링, 세션 클러스터링, 데이터 동기화 같은 항목을 한곳에 놓는다. 그래서 이 페이지의 중심은 특정 운영 도구가 아니라 `서비스가 깨졌을 때 사람이 어떤 순서로 상태를 이해하고 회수할 수 있는가`에 있다.

## 만다라트 위치에서 읽기
`Deep Knowledge > Operation`은 [[Devops]]나 [[Monitoring]]의 하위 부록이 아니다. Devops가 배포와 협업 흐름을, Monitoring이 관측 신호를 다룬다면, Operation은 그 신호와 배포 장치를 실제 장애 순간에 **책임 있는 행동 순서**로 바꾸는 칸이다.

원문에서 반복되는 기준은 다음 다섯 가지다.

- 되돌릴 수 있는가
- 복구할 수 있는가
- 수습이 가능한가
- 관리할 사람을 구할 수 있는가
- 종료 시점과 만료 시점을 통제할 수 있는가

이 기준 때문에 운영은 단순 안정성 목록이 아니라, 프로덕션 시스템이 실패한 뒤에도 설명 가능하고 이어받을 수 있게 만드는 설계 문제로 읽힌다.

## 이 노드가 실제로 답하려는 질문
- 장애가 났을 때 무엇을 먼저 인식하고 누구에게 어떻게 보고해야 하는가?
- 롤백과 복구는 어떤 상태 정보를 확보한 뒤 실행해야 하는가?
- DevOps의 빠른 배포와 SRE의 신뢰성 유지가 어디에서 만나는가?
- DORA Metrics는 운영 개선을 어떤 숫자로 되돌려 주는가?
- 종료 시점, 만료 시간, 오토스케일링 기준은 왜 운영을 쉽게 만드는가?
- 대규모 트래픽에서 CPU, 네트워크, 데이터 동기화 병목은 어떻게 분리해서 봐야 하는가?

## 원문 구조

### 1. 장애 대응 루프
원문 첫머리는 장애 대응을 다섯 단계로 압축한다.

- 인식
- 보고
- 롤백
- 복구
- 피드백

이 배열은 중요하다. 원문은 장애를 `고친다` 한 단어로 줄이지 않고, 먼저 알아차리고, 상황을 공유하고, 되돌릴지 판단하고, 서비스를 회복하고, 다음 운영 기준으로 남기는 흐름으로 본다. Postmortem은 이 루프의 마지막 기록 장치다.

### 2. 종료 시점과 만료 시간을 통제하는 감각
원문에는 `종료시점을 내가 통제하면 관리가 편해진다`는 문장이 있다. uvicorn의 종료 카운트, Karpenter의 `expireAfter`, JWT token 만료 시간은 서로 다른 층의 예시지만 같은 감각을 공유한다.

운영자는 무언가가 무한히 계속되는 상태보다, 언제 끝나고 언제 교체되고 언제 다시 인증해야 하는지 아는 상태를 더 잘 다룰 수 있다. 종료와 만료는 단순 제한이 아니라 **운영 가능한 시간 경계**다.

### 3. DevOps와 SRE의 역할 구분
원문은 DevOps를 빠른 배포 쪽으로, SRE를 프로덕션 신뢰성 유지 쪽으로 구분한다.

- DevOps: deploy를 빠르게 만든다.
- SRE: production reliability를 유지한다.

이 구분은 직무 정의보다 운영 균형에 가깝다. 빨리 배포할 수 있어도 장애를 빨리 이해하고 복구하지 못하면 운영은 약해진다. 반대로 안정성만 붙들고 피드백 속도가 죽으면 제품 전달이 느려진다. 이 노드는 둘 사이를 `빠르게 바꾸되, 깨졌을 때 빠르게 회수할 수 있는가`라는 질문으로 묶는다.

### 4. 성과 측정과 개선
원문은 DORA Metrics를 운영 성과 지표로 남긴다.

- Deployment Frequency
- Lead Time
- MTTR
- Change Failure Rate

여기서 중요한 것은 숫자 자체가 아니라, 배포와 장애 대응이 감각적 인상으로만 남지 않게 하는 것이다. 운영 개선은 `요즘 안정적인 것 같다`가 아니라, 배포 빈도, 변경 리드타임, 복구 시간, 실패율이 함께 움직이는지로 확인된다.

### 5. 프로덕션 속성
원문은 프로덕션에 필요한 속성을 보안, 유연한 개별 설정, 관측성, 되돌리기, 복구, 수습, 인수인계, 안정성으로 적는다. 이 목록은 운영 품질을 한 가지 값으로 보지 않는다는 신호다.

프로덕션은 기능이 실행되는 장소일 뿐 아니라, 권한이 오남용되지 않고, 설정이 상황별로 조정되고, 상태가 관측되고, 문제가 생기면 되돌아가고, 사람이 바뀌어도 이어받을 수 있어야 하는 장소다.

## 운영 판단 모델

### 신호 확보
장애가 감지되면 먼저 상태를 캡처해야 한다. 원문의 reboot report 메모는 load average를 보고 CPU bound, memory bound, disk I/O 문제를 가르고, 소프트웨어 로그와 시스템 로그를 함께 캡처해 리포트로 보낸 뒤 재부팅하는 흐름을 떠올린다.

여기서 재부팅은 무조건의 해결책이 아니라, 복구 행동 전후에 무엇을 남길지 정하는 절차다. 상태를 남기지 않은 재시작은 다음 Postmortem의 재료를 없앤다.

### 영향면 축소
롤백, 만료 시간, 오토스케일링, 다중화는 모두 영향면을 줄이는 장치다. 다만 원문은 단순히 `서버를 늘리면 된다`고 보지 않는다. CPU와 메모리가 동시에 100%가 되는 경우, 세션 전달이 필요한 로그인 구조, 데이터베이스 동기화 문제처럼 확장 장치가 새 관리 지점을 만든다는 점도 함께 본다.

### 복구와 보고
원문의 quick recovery scenario는 5xx 에러를 확인하고, 이전 버전으로 롤백하고, 에러 상황을 보고하는 흐름이다. 보고에는 link, behavior, data, code line, build package, commit source 같은 단서가 포함된다. 즉 보고는 사후 설명 문서가 아니라 복구 판단에 필요한 사건 패킷이다.

### 피드백 회수
Postmortem과 DORA Metrics는 같은 방향을 본다. 장애를 한 번 넘기는 데서 끝내지 않고, 다음 배포·모니터링·자동화·권한·스케일링 기준에 반영한다. 운영은 복구 후에야 끝나는 것이 아니라, 같은 종류의 장애를 더 빨리 인식하고 덜 크게 겪게 만들 때 닫힌다.

## 원문 발췌 구조
원문을 운영 인터페이스 관점으로 복원하면 아래 층으로 나뉜다.

1. **장애 대응**: 인식, 보고, 롤백, 복구, 피드백, Postmortem.
2. **시간 경계**: 종료 카운트, `expireAfter`, JWT 만료 시간.
3. **신뢰성 역할**: DevOps와 SRE의 차이, quick recovery scenario.
4. **성과 지표**: DORA Metrics, MTTR, Change Failure Rate.
5. **edge case**: 로그 폭발, 권한 남용, 시스템 다운, 네트워크 혼잡.
6. **프로덕션 속성**: 보안, 관측성, 되돌리기, 복구, 인수인계 가능성.
7. **운영 프로그램 흐름**: code with log/test/api, CI, CD, Monitoring.
8. **대규모 병목**: CPU, 네트워크, 데이터 동기화, 세션 클러스터링.

## 반복되는 긴장관계

### 자동화 vs 사람이 이어받을 수 있음
원문은 자동화를 원하지만, 사람이 관리할 수 있는가를 계속 묻는다. `관리할 사람을 구할 수 있는가`, `인수인계할 때 무리가 없는가`는 자동화가 블랙박스가 되면 운영 품질이 오히려 낮아진다는 경계다.

### 다중화 vs 관리 지점 증가
WAS 문제를 다중화로 해결하려면 다른 WAS를 찾아야 하고 로그인 정보를 전달해야 한다. 데이터베이스 다중화는 동기화가 어렵다. 즉 확장은 복구성을 높일 수 있지만, 세션과 데이터 상태를 새 책임면으로 만든다.

### 관습 vs 개별 설정
원문은 구성보다 관습, CoC를 언급한다. 모든 것을 설정으로 열어 두기보다 기본 관습을 따르게 하고, 설정이 필요한 때만 열어 주는 방식이다. 운영 관점에서는 선택지를 무한히 늘리는 것보다 기본 경로를 명확히 하는 편이 인수인계와 수습에 유리하다.

### 빠른 복구 vs 원인 보존
재부팅, 롤백, 스케일아웃은 빠른 복구에 도움이 된다. 하지만 load average, sar, 로그, build package, commit source를 남기지 않으면 같은 문제가 반복될 때 배울 수 없다. 이 노드는 복구 속도와 원인 보존을 함께 요구한다.

## 인접 노드와의 연결
- [[Devops]]: 배포 속도, CI/CD, 롤백 가능성, DORA Metrics가 직접 이어진다.
- [[Monitoring]]: 5xx, load average, 로그, 성능 지표를 관측 신호로 받아 장애 대응의 입력으로 쓴다.
- [[Infra]]: 프로덕션 속성, 보안, 권한, 네트워크, 다중화, 시스템 병목과 맞닿는다.
- [[Container]]: Karpenter, 오토스케일링, 노드 교체, 워크로드 수명 관리가 이어진다.
- [[../../Fundamental/Network|Network]]: 네트워크 혼잡, 세션 전달, 대규모 트래픽 병목을 더 기초적인 연결 문제로 되돌린다.

## 유지 원칙
- 이 노드는 운영 일반론 백과사전으로 확장하지 않는다.
- 새 내용이 들어오면 먼저 `장애 대응`, `시간 경계`, `성과 측정`, `프로덕션 속성`, `대규모 병목` 중 어디에 속하는지 본다.
- 도구명보다 그 도구가 어떤 복구 손잡이, 관측면, 인수인계 표면을 만드는지 남긴다.
- 장애 사례는 매끈하게 지우지 말고, 다음에 어떤 신호와 절차로 들어갈지 보존한다.

`Operation`은 프로덕션의 실패를 없애겠다는 선언보다, 실패가 왔을 때 덜 늦게 알아차리고, 덜 크게 번지게 하고, 다시 배울 수 있게 만드는 운영 책임의 노드다.
