# Monitoring

- 원본: `source/shdkej-content/Deep Knowledge/Monitoring.md`
- 상태: 안정 디벨롭
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 모니터링을 단순한 서버 상태 체크로 보지 않는다. 원문은 모니터링을 **시스템, 애플리케이션, 데이터, 웹 성능, 비즈니스 퍼널을 함께 묶어 서비스의 정상 상태를 정의하고, 이상을 더 빨리 발견하고, 다시 복구하는 피드백 루프**로 다룬다. 그래서 CPU, 메모리, 에러율 같은 전통 지표와 함께 로그인 성공 여부, 결제 가능 여부, 신규 유저와 이탈 유저, 비용 스파이크, 롤백 수, A/B 테스트 측정 가능성까지 한 노트 안에 같이 놓여 있다.

## 만다라트 위치에서 읽기
`Deep Knowledge > Monitoring`은 인프라 하위의 감시 도구 칸이 아니라, 깊은 기술 지식을 실제 서비스 운영 판단으로 되돌려 주는 **관측 피드백 칸**이다. 같은 상위 묶음의 [[Devops]]가 배포와 복구의 흐름을 묻고, [[Product]]가 제품 가설과 지표의 연결을 묻는다면, 이 노드는 그 사이에서 `무엇이 실제로 일어나고 있는가`를 확인할 수 있는 출력면을 만든다.

따라서 이 노드의 중심은 Grafana, Prometheus, Loki 같은 도구명이 아니라 다음 순서에 있다.

1. 정상 상태를 먼저 말로 정의한다.
2. 그 상태를 보여 줄 SLI와 대시보드 표면을 고른다.
3. 깨졌을 때 즉시 봐야 할 신호만 Alert로 승격한다.
4. 나머지 신호는 로그, 트레이스, 리포트, 추세 분석으로 재진입 가능하게 남긴다.

이렇게 읽으면 비즈니스 퍼널, 웹 성능, 시스템 자원, 데이터 큐, 비용 스파이크가 한 문서에 섞여 있는 이유가 선명해진다. 모두 `서비스가 살아 있다`는 말을 사용자 가치 전달, 운영 복구, 사업 지속성까지 포함해 증명하려는 서로 다른 관측면이다.

## 이 노드가 실제로 답하려는 질문
- 무엇을 봐야 `서비스가 지금 정상이다`라고 말할 수 있는가?
- 평균값보다 p95, p99 같은 꼬리 지표가 왜 더 중요한가?
- 시스템 지표와 비즈니스 지표를 따로 보지 않고 한 흐름에서 봐야 하는 이유는 무엇인가?
- 로그, 메트릭, 트레이스, 알림은 각각 어떤 역할을 맡아야 하는가?
- 알림이 많아지는 것과 관측 가능성이 높아지는 것은 왜 같은 말이 아닌가?
- 개인 또는 소규모 시스템에서도 Loki, ClickHouse, Prometheus, Grafana 같은 도구 구성이 감당 가능한가?

## 핵심 논지
1. **좋은 모니터링은 단순 수집이 아니라 정상 상태 정의에서 시작한다.**
   - 원문은 `로그인 잘 되는가`, `결제 잘 되는가`, `서버가 정상인가`처럼 서비스의 핵심 성공 조건을 먼저 묻는다.
   - 즉 지표는 많을수록 좋은 것이 아니라, 정상과 비정상을 가르는 기준점이 먼저 있어야 의미가 생긴다.
2. **모니터링 대상은 시스템만이 아니라 사용자 흐름 전체다.**
   - 신규 유저 수, 이탈 유저 수, CTA 클릭률, 캠페인 전환, 세션 수가 시스템 지표와 함께 등장한다.
   - 이는 서비스 운영을 `서버 안 죽음`이 아니라 `사용자 가치 전달이 실제로 일어나는가`로 본다는 뜻이다.
3. **관측 가능성은 예측이 아니라 측정을 통해 만든다.**
   - 원문은 `예측하지 말고 측정한다`는 태도를 분명히 남긴다.
   - 무엇이 깨질지 미리 다 상상하는 대신, 출력과 상태를 충분히 드러내 두어 예상 못 한 문제도 잡을 수 있게 하려는 방향이다.
4. **알림은 많이 보내는 것이 아니라, 정말 봐야 할 것을 놓치지 않게 설계해야 한다.**
   - 원문은 중요한 알림과 일반 안내를 분리해야 한다고 반복한다.
   - 경보 피로가 쌓이면 관측성은 높아지지 않고 오히려 무력화된다.
5. **도구 선택보다 먼저 계층과 흐름을 나눠 봐야 한다.**
   - 시스템, 앱, 데이터, 웹, 비즈니스, 비용, 배포 실패가 구분되어 적혀 있다.
   - 이는 어떤 툴을 붙일지보다 먼저 `무엇을 어떤 레이어에서 볼 것인가`를 설계하려는 감각을 보여 준다.

## 원문 구조를 다시 읽기
원문은 체크리스트, 도구 메모, 성능 개념, 실험 과제가 섞여 있지만 실제로는 아래 흐름으로 읽힌다.

### 1. 서비스의 정상 상태를 정의하려는 메모
원문 첫부분은 보안, load average, 메모리, 디스크, 네트워크, 에러 메시지 같은 시스템 상태를 적는다. 그런데 곧바로 다음 질문이 붙는다.
- 로그인이 잘 되는가
- 결제가 잘 되는가
- A/B 테스트를 측정할 수 있는가
- 신규 유저와 이탈 유저를 볼 수 있는가

즉 저자에게 정상 상태는 시스템 리소스의 범위 안에만 있지 않다. **핵심 사용자 행동이 제대로 일어나는가**까지 포함해야 한다.

### 2. 계층별 모니터링 범위를 넓히는 메모
원문은 모니터링 지표를 다음처럼 나눈다.
- 비즈니스, 마케팅, 퍼널
- 프로덕트 경험
- 시스템 자원
- 애플리케이션 API
- 데이터 레이어

이 구조는 중요하다. 모니터링을 인프라 전담 영역으로 가두지 않고, **서비스를 이루는 각 층의 신호를 한 판 위에서 연결해서 보려는 시도**이기 때문이다.

### 3. Throughput, Latency, p95/p99 같은 성능 개념 메모
원문은 서비스 수용량과 응답 속도를 단순 평균값으로 보지 않는다.
- Throughput은 얼마나 많이 처리하는가
- Latency는 얼마나 늦게 응답하는가
- p95는 대부분의 사용자가 실제로 체감하는 상한선에 가깝다
- p99는 민감하지만 이상 징후를 빨리 드러낼 수 있다

즉 저자는 모니터링을 `숫자가 있다` 수준이 아니라, **숫자를 어떤 감각으로 읽어야 하는가**의 문제로도 이해하고 있다.

### 4. Observability, Chaos Engineering, SLI/SLO 메모
후반부에서는 단순 모니터링에서 observability 쪽으로 시선이 확장된다.
- 시스템의 출력으로 내부 상태를 파악할 수 있어야 한다
- 정상 상태를 정의해야 이상 상태를 식별할 수 있다
- 비정상 상태를 일부러 만들어 보고 되돌리는 과정이 필요하다
- 지표가 있어야 목표를 세울 수 있다

이는 `보는 도구`를 넘어서, **시스템을 더 잘 드러나게 만드는 설계 작업**으로 모니터링을 이해하고 있음을 보여 준다.

### 5. Loki, ClickHouse, Prometheus, Grafana, Tempo 등의 도구 메모
도구 파트는 특정 제품 추천 목록이 아니다.
- 로그를 많이 남기고 쿼리로 검색 가능하게 하자
- 퍼널 데이터는 ClickHouse에 모을 수 있겠다
- Loki와 ClickHouse가 무거울 수 있으니 공통으로 쓰다 분리할지 고민한다
- Prometheus는 메트릭 전문, Loki는 로그 전문, Tempo는 트레이싱 전문으로 읽는다

즉 이 파트의 핵심은 `무엇이 최고 툴인가`보다, **어떤 신호를 어떤 저장/조회 체계에 태워야 하는가**를 현실 비용 안에서 고민하는 것이다.

## 원문 발췌 구조
원문이 실제로 놓고 있는 메모층을 짧게 복원하면 다음과 같다.

- **기본 시스템 체크**: 보안, load average, memory, disk, network, error message.
- **비즈니스/프로덕트 지표**: 신규 유저, 이탈 유저, CTA 클릭률, 퍼널 drop-off, 세션 수, LCP/FID.
- **앱/데이터 지표**: API latency, 400/500 에러, error log count, DB slow query, queue, DLQ.
- **웹 구간 성능**: DNS lookup, TCP 연결, SSL 연결, 첫 바이트, 전체 다운로드.
- **관측성 개념층**: observability, chaos engineering, SLI/SLO, log/trace/metric 구분.
- **운영 현실 메모**: 알림 피로, 비용 스파이크, 롤백 수, 장기 가동 열화, 벤치마크 링크.
- **도구 구성 고민**: Grafana, Prometheus, Loki, Tempo, exporter, ClickHouse, Fluentd/Promtail.
- **실제 적용 작업기록**: 메인 대시보드 구성, mock 지표 제거, 30일 기준 재설정, 비즈니스 지표 보강 필요.

이 복원 구조를 보면 원문은 `모니터링 이론`만 적은 것이 아니라 **측정 항목 설계 → 해석 기준 → 수집 도구 → 실제 배포/대시보드 적용**까지 한 노드 안에 이어 붙여 두고 있다.

## 이 노드에서 반복되는 판단 축

### 신호 수집 vs 신호 승격
- 원문은 로그를 많이 남기고 쿼리로 검색되게 하자는 쪽에 가깝다.
- 하지만 Alert 파트에서는 정말 중요한 상황에서만 알림이 와야 한다고 강하게 제한한다.
- 즉 모든 신호를 같은 강도로 울리는 것이 아니라, `기록해 둘 신호`, `대시보드에 둘 신호`, `사람을 깨울 신호`를 나누려는 감각이 있다.

### 시스템 지표 vs 서비스 경험 지표
- CPU, 메모리, 디스크, 네트워크는 중요하다.
- 하지만 로그인 실패, 결제 실패, 전환율 하락은 서버가 살아 있어도 서비스가 망가졌음을 보여 줄 수 있다.
- 원문은 둘 중 하나를 택하지 않고 함께 봐야 한다고 말한다.

### 평균값 vs 꼬리 지표
- 평균 응답 시간은 상황을 매끈하게 만든다.
- p95, p99는 소수 사용자에게 나타나는 문제를 더 빨리 드러낸다.
- 즉 저자는 예쁜 평균보다 체감 성능과 이상치를 더 신뢰하려는 편이다.

### 계측량 증가 vs 운영 부담 증가
- 로그를 더 남기면 분석 가능성은 커진다.
- 하지만 Loki, ClickHouse, tracing 스택은 무거울 수 있다.
- 따라서 이 노드는 `다 모으자`보다 `무엇을 왜 모을지 먼저 정하자`는 쪽에 가깝다.

### 빠른 알림 vs 알림 피로
- 알림을 세게 걸면 문제를 빨리 볼 수 있다.
- 그러나 무시해도 되는 알림이 쌓이면 정말 중요한 경보를 놓친다.
- 원문은 중요 알림과 일반 알림, 중요한 대화방과 아닌 방의 분리까지 떠올린다.
- 그래서 Alert는 `지표가 임계치를 넘었다`보다 `지금 사람이 끼어들지 않으면 사용자 가치나 비용이 크게 망가진다`는 조건에 가까워야 한다.

### 고정 임계치 vs 변화율 기반 이상 감지
- 절대값 기준은 직관적이다.
- 그러나 비용이나 트래픽처럼 규모가 다른 지표는 변화율로 봐야 더 정확할 수 있다.
- 원문은 `현재값 / 기준값`, `표준편차 기반 편차`까지 고민하며, 같은 숫자라도 맥락에 따라 읽는 법이 달라져야 함을 보여 준다.

## 세부 해석

### 1. `무엇을 모니터링할 것인가`보다 먼저 `무엇을 지키려는가`
원문에서 체크 항목들은 도구 관점이 아니라 운영 책임 관점으로 적혀 있다.
- 로그인
- 결제
- 신규/이탈 유저
- 서버 사용량
- 동시간대 비교, 전일 비교

즉 이 노드는 메트릭 수집 자체보다, **운영자가 무엇을 놓치면 안 되는가**를 정리한 노트다.

### 2. 비즈니스와 운영이 한 문서 안에 같이 있는 이유
마케팅 퍼널, CTA, 캠페인 전환, 세션 수와 p95, 에러율, DB slow query가 한 페이지에 섞여 있는 것은 우연이 아니다. 저자에게 모니터링은 개발 팀만 보는 기술 대시보드가 아니라, **서비스가 실제로 잘 작동하고 있는지 설명하는 공용 언어**에 가깝다.

### 3. 웹 성능 메모는 브라우저 구간까지 포함하려는 감각을 보여 준다
DNS lookup, TCP 연결, SSL 연결, 첫 바이트 시간, 전체 다운로드 시간이 따로 적혀 있다. 이는 서버 내부만 보는 것이 아니라, **사용자 브라우저에서 체감되는 경로 전체를 추적해야 한다**는 뜻이다.

### 4. 장기 실행 시스템의 열화에 주목한다
아파치 자식 프로세스 메모리 공유 감소, 오래 켜 둔 시스템의 성능 저하, 커널 jiffies 오버플로우 같은 메모는 흥미롭다. 저자는 순간 장애만이 아니라, **시간이 지나면서 천천히 나빠지는 시스템**도 관측 대상으로 보고 있다.

### 5. 모니터링 범주를 `메트릭, 로그, 트레이스, 알림, 디버깅, 트렌딩`으로 나눠 본다
이 구분은 단순 용어 수집이 아니라 역할 분리를 보여 준다.
- 메트릭은 상태를 넓게 본다
- 로그는 구체 사건을 찾는다
- 트레이스는 분산 흐름을 따라간다
- 알림은 주의를 강제로 호출한다
- 트렌딩은 추세를 읽는다

즉 저자는 `한 도구로 다 본다`보다 **각 관측 수단의 쓰임새를 분리해서 생각**하려 한다.

### 6. 벤치마크 메모의 의미
동접 수, TPS, RAM, CPU, 인스턴스 사양과 연결된 벤치마크 링크들은 실제 용량 산정 감각을 쌓으려는 흔적이다. 이는 이론적 SLO 공부와 별개로, **어느 정도 자원에서 어느 정도 부하가 가능한가**를 현실값으로 잡아보려는 시도로 읽힌다.

### 7. 작업기록이 말하는 실제 관심사
원문 맨 아래 `작업기록`은 앞선 개념 메모를 실제 운영면으로 끌어내린다.
- 기존 설정파일과 배포 흐름을 어떻게 바꿀지
- blackbox 대신 node exporter를 기본값으로 둘지
- mock 지표를 걷어내고 실제 지표를 다시 세팅할지
- 메인 대시보드에는 각 레이어에서 하나씩만 뽑아 보여줄지
- 비즈니스 지표와 목표치를 alert와 어떻게 연결할지

즉 저자는 모니터링을 공부 주제로만 붙드는 것이 아니라, **지금 운영 중인 시스템에 어떤 최소 관측면을 실제로 꽂을 것인가**라는 실행 문제로 계속 되돌린다.

## 숨은 패턴
- 이 노드는 인프라 메모처럼 보이지만 실제로는 **서비스 운영의 피드백 설계 노드**다.
- 시스템 지표, 비즈니스 지표, 성능 지표, 비용 지표, 배포 실패 지표를 한 문서에 같이 두는 이유는, 운영을 기술과 사업의 분리된 영역으로 보지 않기 때문이다.
- 저자는 평균값보다 꼬리 지표, 단순 현재값보다 비교와 변화율, 대량 알림보다 의미 있는 알림을 더 중시한다.
- 또한 observability를 도구 도입이 아니라, 정상 상태 정의와 상태 노출 설계의 문제로 이해하고 있다.

## 인접 노드와의 연결
- [[Devops]]: 빠른 피드백, MTTR, 배포 후 상태 확인이라는 운영 루프가 직접 이어진다.
- [[Infra]]: 인프라 계층의 상태를 어디까지 드러내고 표준화할 것인지 연결된다.
- [[../Fundamental/Architecture|Architecture]]: 로그, 메트릭, 트레이스가 구조 설계와 분리될 수 없다는 점에서 맞닿는다.
- [[../Meta/My_space]]: 개인 운영 환경에서도 로그, 모니터링, 알림을 기본 제어면으로 삼으려는 태도가 이어진다.
- [[Container]]: 컨테이너 환경에서는 노드, 파드, 서비스, 오토스케일링 단위의 관측이 필수로 붙는다.
- [[Product]]: 프로덕트 지표와 북극성 지표를 어디까지 기술 모니터링과 연결할지 확장해 볼 수 있다.

## 이 노드가 실제로 설계하려는 대시보드 층
원문을 평면 목록이 아니라 운영 인터페이스로 읽으면 대시보드는 대략 세 층으로 나뉜다.

### 1. 메인 상태판
한눈에 `지금 위험한가`를 보는 면이다.
- 로그인/결제 헬스
- 주요 API latency와 error rate
- CPU, memory, disk, network
- 트래픽과 세션 수
- 중요한 알림 수

### 2. 분석/진단판
문제가 생겼을 때 이유를 파고드는 면이다.
- 서비스별 로그 검색
- trace 흐름
- DB slow query와 queue/DLQ
- 브라우저 구간 성능
- 동시간대/전일 비교

### 3. 경영/제품 판단판
운영 신호가 실제 사업 결과와 어떻게 이어지는지 보는 면이다.
- 신규/이탈 유저
- 퍼널 drop-off
- 캠페인 전환
- 비용 스파이크
- 롤백 수, 빌드 실패, 고객 불만 반복

이렇게 읽으면 원문이 말하는 `메인 모니터링 판넬`, `리포트`, `퍼널`, `변화율 지표`, `알림`이 서로 다른 화면이 아니라 **같은 운영 시스템의 서로 다른 재진입 표면**으로 이해된다.

## 이 노드를 읽을 때 놓치지 말아야 할 점
- 완성된 관측 플랫폼 설계 문서라기보다, 무엇을 봐야 시스템을 이해할 수 있는지에 대한 탐색 노트다.
- 중요도는 `툴 이름`보다 `정상 상태 정의`, `계층 구분`, `비교 기준`, `알림 설계`에 있다.
- 원문에는 학습 흔적과 미완 질문이 많다. 이 미완성은 약점이라기보다 실제 운영 감각이 형성되는 과정의 일부다.
- 따라서 이 노드는 깔끔한 정답집으로 평탄화하기보다, 실무 판단 기준과 열린 질문을 함께 보존하는 편이 맞다.


## 실제 지표를 고르는 최소 규칙
원문 말미의 작업기록은 `모든 레이어 중 하나씩만 보는 메인 대시보드`, `mock 지표 제거`, `대시보드 목표치`, `비즈니스 지표 보강`을 같은 흐름에 둔다. 이는 대시보드를 많이 만드는 문제가 아니라, 각 레이어에서 **실제로 행동을 바꿀 수 있는 대표 신호**를 먼저 고르는 문제로 읽힌다.

초기 메인 화면은 넓지만 얕아야 한다. 시스템, 애플리케이션, 데이터, 사용자 행동, 비용/배포 중 하나씩만 고르면 운영자는 `어느 층이 먼저 흔들렸는가`를 빠르게 좁힐 수 있다. 반대로 처음부터 모든 지표를 같은 화면에 넣으면 관측성은 높아지는 듯하지만, 실제 장애 순간에는 어디를 봐야 할지 더 흐려진다.

따라서 이 노드의 실전 기준은 다음처럼 정리할 수 있다.

- **대표성**: 그 지표 하나가 해당 레이어의 건강 상태를 어느 정도 대변하는가?
- **행동 가능성**: 지표가 깨졌을 때 누가 어떤 조치를 시작할 수 있는가?
- **비교 가능성**: 전일, 전주, 동시간대, 기준값과 비교할 수 있는가?
- **목표 연결성**: SLO나 alert 조건으로 바꿀 수 있을 만큼 정상 범위가 정의되어 있는가?
- **실측성**: mock이나 추정값이 아니라 실제 서비스 흐름에서 나온 값인가?

이 기준을 통과하지 못한 값은 버리는 것이 아니라, 메인 대시보드가 아니라 분석판·리포트·실험판으로 내려보내는 편이 맞다. 원문이 `분석에 필요한 지표들`, `비정상 증분`, `grafana daily report`를 따로 언급하는 이유도 여기에 있다. 모든 숫자가 사람을 깨우거나 첫 화면을 차지할 필요는 없지만, 나중에 원인을 되짚을 수 있도록 적절한 위치에는 남아 있어야 한다.

## 알림은 지표가 아니라 주의 배분 정책이다
원문은 Alert를 별도 섹션으로 떼어 두고, `정말 중요한 상황에서만 와야 한다`, `피로도 관리`, `중요한 대화방과 아닌 방을 나눠야 한다`고 적는다. 이 부분은 기술 모니터링을 넘어 사람의 주의력까지 운영 자원으로 본다는 신호다. 알림은 지표의 부산물이 아니라, 어떤 문제에 지금 사람의 개입권을 줄 것인지 정하는 정책이어야 한다.

따라서 이 노드에서 Alert는 세 단계로 나눠 읽는 편이 맞다.

| 층 | 의미 | 예시 | 실패 모드 |
| --- | --- | --- | --- |
| 깨우는 알림 | 지금 대응하지 않으면 사용자 가치·돈·데이터가 훼손된다 | 결제 실패, 주요 API 장애, 비용 급증, 데이터 큐 적체 | 너무 많으면 전부 무시된다 |
| 확인 알림 | 곧 문제가 될 수 있어 담당자가 봐야 한다 | p95 악화, error rate 상승, 롤백 증가, 빌드 실패 반복 | 맥락 없이 쌓이면 불안만 만든다 |
| 기록 신호 | 지금 울릴 필요는 없지만 나중에 분석해야 한다 | 동시간대 트래픽 차이, 퍼널 drop-off, 느린 쿼리 샘플 | 저장되지 않으면 재발 분석이 막힌다 |

이렇게 나누면 `알림을 확실히 눈에 띄게 해야 한다`는 문장은 단순 UI 팁이 아니다. 중요한 알림의 수를 줄이고, 제목에서 바로 조치 방향이 보이게 하며, 일반 안내는 리포트나 대시보드로 내려보내야 한다는 운영 원칙이다. [[../Communication/Feedback]]의 피드백 회로와도 연결된다. 좋은 피드백은 많이 오는 것이 아니라, 다음 행동을 바꿀 수 있는 강도와 타이밍으로 오는 것이다.

## 오래 켜 둔 시스템을 보는 시간축
원문 중반의 아파치 자식 프로세스, 메모리 공유 감소, jiffies 오버플로우, 데이터·메모리 누적 메모는 Monitoring 노드 안에서 특히 중요하다. 저자는 장애를 `갑자기 터지는 사건`으로만 보지 않는다. 아무것도 바꾸지 않았는데 시간이 지나며 성능이 낮아지는 상태도 관측 대상으로 둔다.

이 관점은 단기 헬스체크와 다른 시간축을 요구한다.

- **즉시 상태**: 지금 요청이 성공하는가, 노드가 살아 있는가, 에러가 늘었는가.
- **추세 상태**: 메모리, 디스크, 연결 수, 비용, latency가 천천히 나빠지는가.
- **수명 상태**: 오래 실행된 프로세스, 누적 로그, 큐, 캐시, 커널·런타임 한계가 언제 재시작·정리·교체를 요구하는가.

따라서 이 노드의 모니터링은 `현재 정상`을 확인하는 데서 끝나지 않는다. 서비스가 오래 살아 있을수록 어떤 상태가 쌓이고, 어떤 자원이 조용히 줄어들며, 언제 정기 재시작·스케일 조정·보존 정책이 필요한지를 보는 장기 실행 운영면까지 포함한다. 이 점은 [[../Fundamental/Software]]의 상태 생애주기, [[Container]]의 컨테이너 운영 단위와도 이어진다.

## 후속 정리 기준
- 비즈니스, 프로덕트, 시스템, 데이터 레이어 지표를 `공통 대시보드`, `실무 대시보드`, `경보 대시보드`로 다시 나눠 볼 수 있다.
- SLI, SLO, Alert 정책을 `무엇이 정상인가`, `언제 깨졌다고 볼 것인가`, `누가 어떻게 대응할 것인가` 순서로 더 정리할 수 있다.
- 로그, 메트릭, 트레이스의 역할 분리를 실제 도구 조합과 비용 제약까지 연결해 더 선명하게 정리할 수 있다.
- `Monitoring`과 `Product`, `Devops`, `My_space`를 함께 읽으면, 저자의 운영 감각이 단순 서버 관리보다 서비스 전체 피드백 루프에 가깝다는 점이 더 선명해진다.
