# Cloud

- 원본: `source/shdkej-content/Deep Knowledge/Cloud.md`
- 상태: 원본 공란 상태 재확인, 과잉 가설 압축
- 역할: 원본 만다라트 노드와 1:1 대응하는 경계 노트

## 핵심 논지
`Cloud.md`는 현재 0줄 공란이다. 따라서 이 페이지는 클라우드 일반론이나 AWS 세부 지식을 대신 채우는 문서가 아니라, `Deep Knowledge` 안에서 `Cloud`라는 독립 카테고리 칸이 무엇을 받을 수 있는지 보존하는 관리 노트다.

지금 확실한 것은 세 가지뿐이다.

- 원본 `source/shdkej-content/Deep Knowledge/Cloud.md`는 비어 있다.
- 이 파일은 `Deep Knowledge` 아래의 독립 카테고리 노드다.
- 인접 노드에는 클라우드와 맞닿는 실행 메모가 많지만, 그것은 아직 `Cloud` 원문 자체의 서술이 아니다.

그래서 현재 단계의 목표는 내용을 늘리는 것이 아니라, 나중에 원문이 채워졌을 때 그 문장이 기존 `Infra`·`Container`·`Devops` 메모의 반복인지, 아니면 더 높은 선택 기준을 여는지 판별할 수 있게 하는 것이다.

## 원본 상태
원본 파일 전체를 다시 확인했지만 본문은 없다. 직접 발췌할 문장, 소제목, 사례, 주장 목록도 없다.

이 공란은 단순 누락일 수도 있고, 다른 노드에 흩어진 클라우드 관련 생각이 아직 하나의 상위 판단으로 수렴되지 않았다는 신호일 수도 있다. 현재로서는 어느 쪽인지 확정하지 않는다. 사용자가 나중에 실제 문장을 넣으면 그 문장과 배열을 현재 가설보다 우선한다.

## 만다라트 안에서의 위치
이 랩에서는 각 markdown 파일을 임의 문서가 아니라 8x8 만다라트의 카테고리 노드로 본다. 그런 기준에서 `Cloud`의 공란은 지워야 할 빈칸이 아니라, `Infra`, `Container`, `Devops`, `Web`과 구분되는 상위 판단 칸으로 남겨 둘 대상이다.

`Deep Knowledge` 아래에 있다는 점도 중요하다. 이 위치에서 `Cloud`는 서비스 이름 목록보다, 추상화 수용 범위, 책임 위임 방식, 비용·속도·복구 가능성의 비교처럼 시간이 지나도 다시 쓰일 판단 기준을 받을 가능성이 높다.

## 인접 노드가 던지는 질문
관련 source에는 클라우드와 닿는 메모가 이미 흩어져 있다. 다만 아래 항목은 `Cloud` 원문이 아니라 경계 판별을 돕는 보조 증거다.

| 인접 source | 보이는 메모 | `Cloud`에서 받을 수 있는 상위 질문 |
| --- | --- | --- |
| `Infra` | `모든것을 위한 클라우드는 없다`, `클라우드는 편리하지만 내부를 통제할 수 없다` | 편의와 통제의 교환비를 어디서 받아들일 것인가 |
| `Infra` | 정적 페이지 + CDN, Lambda, Fargate, NAT gateway 비용, VPC Endpoint | 구조 선택이 비용·복구성·운영 난이도에 어떤 차이를 만드는가 |
| `Infra` | IAM shared responsibility, 계정 분리, CloudWatch, VPC Flow Logs | 관리형 서비스를 써도 무엇을 직접 보고 통제해야 하는가 |
| `Devops` | 빠르고 안전한 배포, 롤백, GitOps, Terraform | 클라우드 선택이 배포 속도와 되돌리기 가능성에 어떤 기본값을 제공하는가 |
| `Container` | EKS, Fargate, Lambda 비교, AWS CNI, ingress/NLB | 실행 추상화가 높아질수록 어떤 네트워크·운영 세부가 다시 돌아오는가 |

이 표의 목적은 주변 실행 메모를 `Cloud`로 끌어오는 것이 아니다. 이미 존재하는 실행 메모가 상위 선택 기준으로 승격될 수 있는지 확인하는 임시 필터다.

## 결정 층과 실행 층 분리
빈 `Cloud` 노드에서 가장 조심할 점은 실행 세부를 상위 노드 본문처럼 옮기는 것이다. Lambda, Fargate, NAT gateway, VPC Endpoint, CloudWatch, ingress 같은 항목은 대체로 `Infra`, `Container`, `Devops`, `Monitoring`에 더 잘 머문다.

`Cloud`에는 실행 항목 자체보다 그 항목이 드러내는 결정 질문만 남긴다.

| 실행 메모 | `Cloud`에서 받을 질문 | 실행 메모가 머물 곳 |
| --- | --- | --- |
| Lambda, Fargate, 서버리스 | 관리 포인트를 줄이는 대신 지속 연결·콜드스타트·상태성 한계를 어디서 받아들일 것인가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]] |
| NAT gateway, VPC Endpoint | 숨은 네트워크 비용이 구조 선택을 언제 뒤집는가 | [[../../mapped/deep-knowledge/infra]] |
| CloudWatch, Flow Logs, 계정 분리 | 맡긴 자원이라도 무엇을 계속 관측하고 격리해야 하는가 | [[../../mapped/deep-knowledge/monitoring]], [[../../mapped/deep-knowledge/infra]] |
| Terraform, GitOps, 롤백 | 클라우드 상태의 원천이 선언 파일인지 콘솔인지, 누가 복구 경로를 소유하는가 | [[../../mapped/deep-knowledge/devops]], [[../../mapped/deep-knowledge/infra]] |

이 구분은 원문이 비어 있는 동안의 안전장치다. 실제 원문이 생기면 그 문장이 실행 절차인지 상위 선택 기준인지 다시 분류한다.

## 현재 보존할 최소 분류축
원본이 공란인 동안에는 긴 가설 설명보다 아래 분류축만 남기는 편이 안전하다.

| 분류축 | 볼 질문 |
| --- | --- |
| 선택 이유 | 관리형, 서버리스, 컨테이너, VM 중 왜 그 방식을 고르는가 |
| 책임 경계 | 무엇을 클라우드에 맡기고 무엇을 직접 관측·변경·회수하는가 |
| 비용 신호 | 숨은 네트워크 비용, 운영 복잡도, 되돌리기 비용이 구조 선택을 다시 여는가 |
| 가시성·권한 | 로그, 지표, 흐름, 권한 변경, 계정 경계가 사람이 이해할 수 있는 표면으로 남는가 |
| 회귀 가능성 | 장애·비용·이전·롤백 상황에서 다시 잡을 손잡이가 남아 있는가 |
| 시간층 | 도입 순간, 운영 중, 철수·이전 순간의 판단이 서로 어떻게 다른가 |

이 축을 통과하지 못하는 메모는 `Cloud`에 넣기보다 더 구체적인 실행 노드에 둔다.

## 인접 노드와의 관계
### [[../../mapped/deep-knowledge/infra]]
가장 가까운 이웃이다. 현재 클라우드 관련 메모 대부분은 VPC, NAT gateway, 서버리스, AWS 비용, IAM처럼 실행 레벨이 강하므로 `Infra`에 남기는 편이 안전하다. `Cloud`는 그보다 한 단계 위에서 왜 클라우드를 택하는지와 어떤 추상화를 받아들일지를 묻는다.

### [[../../mapped/deep-knowledge/container]]
컨테이너는 클라우드 위에서 워크로드를 어떻게 포장하고 이동시키는지의 문제에 가깝다. `Cloud`가 플랫폼과 책임 분배의 선택이라면, `Container`는 배포 단위와 실행 환경을 더 세부적으로 다룬다.

### [[../../mapped/deep-knowledge/devops]]
`Devops`는 빠르고 안전한 배포, 협업 문화, 피드백 루프를 중심으로 전개된다. `Cloud`는 자원·서비스·관리 모델의 선택을 묻고, `Devops`는 그 기반 위에서 조직과 배포 흐름을 운영하는 문제를 맡는다.

### [[../../mapped/deep-knowledge/web]]
정적 배포, CDN, API, 인증, 저장소 같은 웹 실행 환경은 클라우드와 맞닿는다. `Web`이 사용자에게 보이는 서비스 구조에 더 가깝다면, `Cloud`는 그 구조를 어떤 운영 모델 위에 올릴지 결정하는 배경 레이어다.

## 원본이 채워질 때 먼저 볼 질문
- 클라우드를 편의성의 문제로 보는가, 운영 전략의 문제로 보는가.
- 비용 최적화와 개발 속도 중 무엇을 더 자주 우선하는가.
- 서버리스 선호가 어디까지 유효하고, 어디서 서버·컨테이너로 되돌아오는가.
- 관측, 권한, 계정 분리, shared responsibility를 얼마나 상위 판단으로 다루는가.
- 특정 벤더 지식보다 공통 원리와 회귀 가능한 구조를 더 중시하는가.
- `Infra`에 이미 적힌 실행 메모를 반복하는가, 아니면 그 위의 선택 기준을 새로 여는가.

## 원문이 생기면 먼저 분류할 문장형
| 문장형 | 예시 역할 |
| --- | --- |
| 철학 문장 | `클라우드는 편리하지만 내부를 통제할 수 없다`처럼 선택의 태도를 직접 말한다. |
| 구조 문장 | 정적 페이지, CDN, Lambda, Fargate처럼 어떤 조합을 택하는지 드러낸다. |
| 비용 문장 | NAT gateway, 네트워크, 운영 인력처럼 표면 아래 비용을 드러낸다. |
| 가시성 문장 | 로그, 권한, 계정, 모니터링처럼 맡겼어도 직접 봐야 하는 표면을 말한다. |
| 회귀 문장 | 이전, 롤백, 축소, 벤더 의존처럼 나중에 빠져나올 손잡이를 다룬다. |

## 유지 원칙
- 원본이 비어 있는 동안에는 과도한 내용 확장을 하지 않는다.
- 인접 노드의 세부 실행 메모는 증거로만 쓰고, `Cloud` 본문처럼 옮기지 않는다.
- 외부 업계 지식, 벤더 비교, 최신 서비스명, 비용 최적화 팁은 사용자가 원문에 열어 둔 경우에만 받는다.
- 사용자가 원본을 직접 채우면 그 버전을 우선 읽고, 필요하면 `human-reviews/`에 복사해 선호 변화로 기록한다.
- 빈 노드일수록 `확실한 것`과 `현재 가설`을 분리한다.

## 후속 흡수 기준
원본이 계속 비어 있는 동안 `Cloud`에 새 내용을 더할 때는 아래 조건 중 하나가 실제로 선명해질 때로 제한한다.

| 새 신호 | `Cloud`에 흡수할 때 | 인접 노드에 남길 때 |
| --- | --- | --- |
| 선택 이유 | 관리형, 서버리스, 컨테이너, VM 중 왜 그 방식을 고르는지 반복 판단으로 굳어진다. | 특정 서비스 설정값이나 배포 절차가 핵심이다. |
| 책임 경계 | 무엇을 맡기고 무엇을 계속 관측·변경·회수할지 묻는다. | 이미 정해진 구조 안에서 권한, 네트워크, 로그를 조정한다. |
| 비용 신호 | 숨은 네트워크 비용이나 운영 복잡도가 구조 선택을 다시 열게 만든다. | 단일 요금 항목이나 절감 팁을 기록한다. |
| 시간층 | 도입, 운영 중, 철수·이전 순간의 판단 차이를 드러낸다. | 현재 플랫폼에서 문제를 해결하는 재현 절차가 중심이다. |

이 기준을 통과하지 못한 클라우드 메모는 [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/devops]], [[../../mapped/deep-knowledge/monitoring]] 중 더 구체적인 실행 노드에 남긴다. `Cloud`는 빈 원문을 대신 채우는 흡수장이 아니라, 주변 실행 메모가 상위 선택 기준으로 승격되는 순간만 받는 경계 노드로 유지한다.
