# Cloud

- 원본: `source/shdkej-content/Deep Knowledge/Cloud.md`
- 상태: 점진 심화, 원본 공란 상태 재확인
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 논지
이 노드는 아직 원본 본문이 비어 있다. 그래서 현재의 `Cloud`는 클라우드 일반론을 적어 넣는 자리보다, 저자가 `Deep Knowledge` 안에서 따로 확보해 둔 **상위 카테고리 경계**를 보존하는 노드로 읽는 편이 정확하다. 이번 보강의 목적도 내용을 임의로 채우는 것이 아니라, 공란 자체가 말해 주는 범위와 주변 노드에 이미 흩어진 클라우드 흔적을 함께 묶어 두는 데 있다.

특히 이 노드는 `AWS 서비스 모음`이나 `클라우드 입문 치트시트`로 평탄화되기 쉽다. 하지만 현재 저장소 맥락에서는 그런 일반 지식보다, **추상화를 어디까지 받아들이는지**, **운영 통제를 어디서 내려놓는지**, **비용·속도·복구 가능성을 어떤 순서로 비교하는지** 같은 상위 판단 질문을 담는 그릇으로 보는 편이 더 잘 맞는다.

## 원본 상태
- 원본 파일 전체를 다시 확인한 결과 본문이 비어 있다.
- 따라서 직접 발췌할 문장, 소제목, 사례, 주장 목록은 현재 없다.
- 이번 정리는 빈 소스를 억지로 채우는 작업이 아니라, 추후 사용자가 직접 내용을 넣었을 때 어떤 변화가 들어왔는지 더 정확히 비교·흡수할 수 있도록 해석 기준을 다듬는 작업이다.

## 만다라트 안에서의 위치 의미
이 저장소에서는 각 markdown 파일을 독립 문서가 아니라 8x8 만다라트의 **카테고리 노드**로 읽어야 한다. 그런 기준에서 `Cloud`의 공란은 단순 누락이라기보다, 아직 세부 메모가 흩어져 있더라도 이 주제가 `Infra`, `Container`, `Devops`, `Web`과는 다른 이름으로 따로 보존되어야 한다는 신호에 가깝다.

특히 이 노드가 놓인 상위 폴더가 `Deep Knowledge`라는 점도 중요하다. 여기서 `Cloud`는 서비스 이름을 훑는 입문 메모보다, **추상화 수용 범위·책임 위임 방식·복구 가능한 구조 선택**처럼 시간이 지나도 다시 써먹을 판단 기준을 담는 칸에 가깝다. 그래서 공란이어도 카테고리 위치 자체가 이미 하나의 의미를 갖는다.

즉 지금 단계의 핵심은 `Cloud`를 임의의 상식 문서로 풍성하게 만드는 것이 아니라, 앞으로 사용자가 어떤 내용을 넣더라도 그 내용을 받을 **주제 경계와 흡수 기준**을 흔들지 않는 데 있다. 이번 페이지의 역할은 지식을 대신 채우는 것이 아니라, 나중에 실제 원문이 들어왔을 때 그 내용을 어느 층위의 판단으로 읽어야 하는지 미리 고정해 두는 데 있다.

## 이 빈 노드가 시사하는 범위
현재 원문은 비어 있지만, 인접 노드에는 이미 클라우드와 직접 맞닿는 흔적이 흩어져 있다. 다만 여기서 중요한 것은 그 흔적을 그대로 옮겨 `Cloud`를 대신 채우지 않는 것이다. 관련 노트는 이 페이지의 경계를 선명하게 하는 보조 증거로만 써야 한다.

특히 `Infra` 원문에는 다음과 같은 문장 덩어리가 이미 보인다.
- `모든것을 위한 클라우드는 없다`
- `정적페이지로 최대한 만들어서 cdn 이용하기`
- `클라우드는 편리하지만 내부를 통제할 수 없다`
- `서버리스를 선호`와 그 한계에 대한 메모
- AWS 비용 항목, NAT gateway 비용, VPC Endpoint 같은 운영 메모

또한 `Devops`와 `Container` 원문을 함께 보면, 현재 저장소에서 클라우드는 단순한 호스팅 장소가 아니라 다음 층위와 접속한다.
- 개발자가 빠르고 안전하게 배포할 수 있게 하는 **운영 표면**
- 컨테이너·서버리스·관리형 서비스 중 무엇을 맡기고 무엇을 직접 쥘지 정하는 **책임 분배면**
- 비용·성능·복구 가능성의 균형을 잡는 **선택 아키텍처**

이 신호를 보면 `Cloud`는 단순히 AWS/GCP/Azure 기능을 모으는 페이지보다, 적어도 아래 층위를 다룰 가능성이 높다.
- **추상화 수용 범위**: 하드웨어 최적화 같은 내부 통제를 어디까지 포기하고 어떤 편의성을 얻을 것인가
- **구조 선택 기준**: 정적 페이지, CDN, Lambda, Fargate, DB, MQ 같은 요소를 어떤 조합 논리로 택할 것인가
- **비용 감각**: 인스턴스 요금표보다 숨은 네트워크·게이트웨이·운영 비용까지 포함해 읽는가
- **서버리스 경계선**: 관리 포인트 감소의 이점과 콜드스타트·실시간성·지속 연결 한계를 어디서 가르는가
- **책임 위임의 한계**: 관리형 서비스가 편리해질수록 무엇이 보이지 않게 되고, 그 대가를 어디서 다시 치르는가

다만 위 항목들은 현재 `Cloud` 원문에 적혀 있는 내용이 아니라, 주변 source 노드에서 읽히는 범위 가설이다. 이후 원본이 채워지면 반드시 그 직접 서술을 우선한다.

## 인접 source가 이미 던지고 있는 질문
현재 `Cloud` 원문은 비어 있어도, 관련 source를 다시 읽으면 이 노드가 장차 어떤 질문을 받아야 하는지는 어느 정도 보인다.

| 인접 source | 실제로 보이는 메모 | `Cloud` 쪽에서 받아야 할 상위 질문 |
| --- | --- | --- |
| `Infra` | `모든것을 위한 클라우드는 없다`, `클라우드는 편리하지만 내부를 통제할 수 없다` | 편의와 통제의 교환비를 어디서 받아들일 것인가 |
| `Infra` | 정적 페이지 + CDN, Lambda, Fargate, NAT gateway 비용, VPC Endpoint | 구조 선택이 비용·복구성·운영 난이도에 어떤 차이를 만드는가 |
| `Infra` | 서버리스를 선호하지만 콜드스타트·지속 연결·실시간성 한계를 의식 | 서버리스 선호가 어디서 멈추고 다른 운영 모델로 넘어가는가 |
| `Infra` | IAM shared responsibility, 계정 분리, CloudWatch/Chatbot, VPC Flow Logs | 관리형 서비스를 써도 무엇을 직접 보이고 통제해야 하는가 |
| `Devops` | 개발자가 빠르고 안전하게 배포하고 쉽게 롤백해야 한다는 목표 | 클라우드 선택이 배포 속도와 복구 가능성에 어떤 기본값을 제공하는가 |
| `Container` | EKS·Fargate·Lambda 비교, AWS CNI·IP 한계, ingress/NLB 메모 | 실행 추상화가 높아질수록 어떤 네트워크·운영 세부가 다시 돌아오는가 |

이 표의 핵심은 `Cloud`를 채우기 위해 외부 일반론을 더하는 것이 아니라, 이미 저장소 안에 흩어진 질문을 한 단계 위로 끌어올려 이 노드의 역할을 더 선명하게 만드는 데 있다.

## 주변 메모를 `Cloud` 관점으로 다시 세울 때 보이는 선택 사다리
원문 공란을 유지한 채 주변 source만 다시 읽어 보면, 이 노드에는 적어도 아래처럼 **선택이 위에서 아래로 내려오는 사다리**가 잠재해 있다.

1. **왜 클라우드를 쓰는가**  
   하드웨어 최적화나 직접 서버 관리보다 더 빨리 만들고 배포하고 되돌리기 위해 쓴다. 이 층위는 `Devops`의 배포 속도·롤백 감각과 맞닿는다.
2. **무엇을 맡기고 무엇을 쥘 것인가**  
   서버리스·관리형 서비스·컨테이너·가상서버 중 어디까지를 서비스 제공자에게 맡길지 정한다. 이때 얻는 편의와 잃는 통제의 비율이 핵심이다.
3. **숨은 비용이 어디서 튀어나오는가**  
   인스턴스 가격표보다 NAT gateway, 네트워크 왕복, 관측 체계, 권한 설계, 운영 복잡도가 뒤늦게 비용으로 돌아온다.
4. **문제가 났을 때 어디까지 보이는가**  
   관리형 서비스를 쓰더라도 CloudWatch, Flow Logs, IAM, 계정 분리처럼 직접 봐야 하는 표면은 사라지지 않는다.
5. **나중에 되돌릴 수 있는가**  
   특정 벤더나 실행 모델에 깊게 기대었을 때, 구조를 바꾸거나 다른 환경으로 옮길 여지가 남는지가 마지막 판단 기준이 된다.

이 사다리는 현재 원문에 적힌 내용이 아니라, `Infra`·`Container`·`Devops`에 흩어진 메모를 `Cloud`라는 상위 카테고리 노드에서 다시 읽었을 때 드러나는 잠재 구조다. 그래서 지금 단계의 이 페이지는 `클라우드 기술 설명서`보다 `선택 사다리를 보존하는 경계 노트`에 가깝다.

## 결정 층과 실행 층을 섞지 않는 읽기
빈 `Cloud` 노드를 유지할 때 특히 중요한 기준은 **결정 층**과 **실행 층**을 분리하는 일이다. 주변 source에는 이미 Lambda, Fargate, NAT gateway, VPC Endpoint, CloudWatch, ingress 같은 실행 메모가 많다. 그러나 이 항목들을 그대로 가져오면 `Cloud`가 금방 `Infra`의 AWS 세부 사전이 되어 버린다.

그래서 `Cloud`에는 실행 항목 자체보다 그 항목이 드러내는 선택 질문을 남기는 편이 안전하다. 예를 들면 다음과 같다.

| 주변에 보이는 실행 메모 | `Cloud`에서 받을 수 있는 결정 질문 | 실행 메모가 머물 곳 |
| --- | --- | --- |
| Lambda, Fargate, 서버리스 선호 | 관리 포인트를 줄이는 대신 지속 연결·콜드스타트·비용 한계를 어디서 받아들일 것인가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]] |
| NAT gateway 비용, VPC Endpoint | 숨은 네트워크 비용이 구조 선택을 언제 뒤집는가 | [[../../mapped/deep-knowledge/infra]] |
| CloudWatch, Flow Logs, 계정 분리 | 맡긴 자원이라도 무엇을 계속 관측하고 격리해야 하는가 | [[../../mapped/deep-knowledge/monitoring]], [[../../mapped/deep-knowledge/infra]] |
| ArgoCD, Terraform cloud, 배포/롤백 | 클라우드가 배포 속도와 되돌리기 가능성에 어떤 기본값을 제공하는가 | [[../../mapped/deep-knowledge/devops]] |

이 구분은 원문 공란을 보존하기 위한 임시 안전장치다. 사용자가 나중에 `Cloud.md`에 실제 문장을 채우면, 그 문장이 실행 절차인지 상위 선택 기준인지 다시 분류해야 한다. 지금 단계에서는 실행 세부를 덜어내고, `왜 이 실행 모델을 고르는가`라는 결정 질문만 남기는 것이 드리프트를 가장 적게 만든다.

## 공란 노드에서 특히 조심할 드리프트
`Cloud`는 원문이 비어 있기 때문에, 에이전트가 가장 쉽게 외부 지식이나 인접 노트의 세부 메모를 끌어와 과잉 작성할 수 있는 자리다. 그래서 이 페이지의 품질은 많은 내용을 담는 데서보다, **현재 확실한 것과 가설인 것을 분리하는 능력**에서 결정된다.

현재 확실한 것은 세 가지뿐이다.
- 원본 `Cloud.md`는 비어 있다.
- 이 파일은 `Deep Knowledge` 아래의 독립 카테고리 노드로 배치되어 있다.
- 인접 노드에는 클라우드와 닿는 실행 메모가 이미 많지만, 그 메모들은 아직 `Cloud` 원문 자체의 서술은 아니다.

따라서 이후 심화에서도 아래 선을 지켜야 한다.
- **인접 노트의 실행 사례는 증거로만 쓴다.** `Infra`의 NAT, VPC, 서버리스 메모를 `Cloud` 본문처럼 옮겨 쓰지 않는다.
- **외부 업계 지식은 기본값으로 들여오지 않는다.** AWS/GCP/Azure 비교, 최신 서비스명, 비용 최적화 팁은 사용자가 원문에 열어 둔 경우에만 받는다.
- **비어 있음 자체를 해석 대상으로 둔다.** 아직 정리되지 않은 주제, 따로 떼어 둔 판단 칸, 또는 나중에 채울 수 있는 상위 경계라는 가능성을 보존한다.

이 원칙 덕분에 `Cloud`는 빈 노트인데도 쓸모가 있다. 지금의 역할은 내용을 대신 생산하는 것이 아니라, 나중에 사용자가 직접 쓴 문장이 들어왔을 때 그것이 기존 `Infra`·`Container`·`Devops` 메모를 반복하는지, 아니면 더 높은 선택 기준을 여는지 판별하는 **흡수 필터**가 되는 것이다.

## 지금 이 노드에서 보존해야 할 최소 구조
원문이 비어 있어도 다음 구조는 미리 고정해 두는 편이 유용하다.
1. `Cloud`는 서비스 이름 목록이 아니라 **선택 기준을 다루는 상위 노드**일 가능성이 높다.
2. 세부 구현 메모는 [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/devops]], [[../../mapped/deep-knowledge/web]]로 흩어질 수 있다.
3. 따라서 이 페이지는 지금 당장 상세 지식을 쌓기보다, 나중에 원문이 생겼을 때 그것이 **철학·비용·구조·운영 모델 중 어디를 건드리는지** 분류하는 관문 역할을 해야 한다.

### 현재 단계에서 먼저 잡아 둘 분류축
- **플랫폼 선택**: 특정 벤더 기능보다, 관리형 서비스를 어디까지 받아들일지의 선택.
- **책임 분배**: 인프라팀/클라우드/매니지드 서비스/애플리케이션이 각각 무엇을 책임지는지.
- **비용의 보이는 층과 숨은 층**: 컴퓨팅 요금뿐 아니라 네트워크, 게이트웨이, 운영 복잡도, 복구 비용까지 포함하는가.
- **가시성·권한 경계**: CloudWatch, Flow Logs, IAM, 계정 분리처럼 무엇을 누가 보고 열 수 있는지까지 운영 설계에 포함하는가.
- **회귀 가능성**: 한번 선택한 구조에서 다른 형태로 돌아오거나 옮겨 갈 수 있는가.

### 현재 단계에서 적지 않는 편이 나은 것
- 원문에 없는 벤더 기능 백과사전식 요약.
- `Cloud`를 `Infra`의 AWS 세부 메모 복사본으로 만드는 확장.
- 최신 업계 정답처럼 보이는 일반론을 덧씌워 원래 노드의 빈 경계를 지워 버리는 정리.

## 공란이 유지될 때의 점검 질문
원본이 계속 비어 있는 동안 이 노드는 내용을 늘리는 방향보다, 다른 노드에서 넘어오려는 메모를 걸러 내는 **검문소**로 쓰는 편이 안전하다. 새 클라우드 관련 메모를 발견했을 때는 먼저 아래 질문을 통과시킨다.

| 점검 질문 | `Cloud`에 남길 가능성이 높은 경우 | 인접 노드에 남길 가능성이 높은 경우 |
| --- | --- | --- |
| 이 메모가 선택 기준을 말하는가? | 관리형/서버리스/컨테이너/VM 중 왜 그 방식을 고르는지 설명한다. | 특정 AWS 리소스 설정값이나 명령어를 기록한다. |
| 책임 경계를 바꾸는가? | 무엇을 클라우드에 맡기고 무엇을 직접 관측해야 하는지 말한다. | 이미 정해진 구조 안에서 배포·네트워크·권한을 조정한다. |
| 비용을 상위 판단으로 바꾸는가? | 숨은 네트워크 비용, 운영 복잡도, 되돌리기 비용을 비교한다. | 특정 요금 항목이나 절감 팁을 단발로 기록한다. |
| 회귀 가능성을 다루는가? | 나중에 다른 실행 모델로 옮길 수 있는지 묻는다. | 현재 플랫폼에서 문제를 해결하는 절차를 남긴다. |

이 표는 공란을 채우기 위한 대체 본문이 아니라, 공란이 유지될 때도 `Cloud`의 1:1 카테고리 경계를 잃지 않기 위한 임시 판별 장치다. 따라서 실제 원문이 생기면 이 표보다 사용자가 쓴 문장과 배열을 우선한다.

## 회귀 가능성을 계약면으로 읽기
공란 상태의 `Cloud`에서 지금 더 보강할 수 있는 안전한 축은 **되돌릴 수 있는 선택인가**다. 인접 source에는 롤백, 스냅샷, Terraform, GitOps, CloudWatch, VPC Flow Logs 같은 실행 메모가 흩어져 있지만, 이 노드가 직접 받아야 할 것은 그 도구 목록이 아니라 클라우드 선택이 나중의 복구 가능성을 어떻게 바꾸는지에 대한 계약면이다.

특히 관리형 서비스나 서버리스는 시작 속도를 높여 주지만, 문제가 생겼을 때 어떤 상태를 직접 볼 수 있고 어떤 상태는 제공자 내부에 남는지까지 함께 결정한다. 따라서 `Cloud`의 상위 질문은 `무엇을 쓰는가`보다 `장애·비용·이전·롤백 상황에서 내가 다시 잡을 수 있는 손잡이가 남아 있는가`에 가깝다.

| 선택 장면 | `Cloud`에서 남길 질문 | 실행 세부가 머물 곳 |
| --- | --- | --- |
| 서버리스·관리형 서비스 우선 | 직접 관리하지 않는 대신, 장애 시 어떤 로그·권한·상태를 볼 수 있는가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/monitoring]] |
| Kubernetes·Fargate·컨테이너 선택 | 실행 단위를 옮기거나 되돌릴 때 플랫폼 의존성이 어디까지 남는가 | [[../../mapped/deep-knowledge/container]] |
| Terraform·GitOps·수동 콘솔 작업 | 클라우드 상태의 진짜 원천이 선언 파일인지 콘솔인지, 누가 복구 경로를 소유하는가 | [[../../mapped/deep-knowledge/devops]], [[../../mapped/deep-knowledge/infra]] |
| 스냅샷·롤백·데이터 복구 | 되돌릴 수 있는 것은 코드인지 인프라인지 데이터인지, 되돌린 뒤 생긴 변경은 어떻게 합치는가 | [[../../mapped/deep-knowledge/devops]] |

이 축을 세워 두면 빈 `Cloud` 노트가 특정 서비스 추천표로 부풀지 않고, 인접 노드의 실행 메모를 `복구 가능한 계약면`이라는 상위 판단으로만 받아들일 수 있다. 나중에 원본이 채워질 때도 새 문장이 벤더 기능 설명인지, 아니면 회귀 가능성과 상태 소유권을 다루는 판단 문장인지 먼저 가를 수 있다.

## 상태 원천을 어디에 둘지 먼저 묻기
빈 `Cloud` 노드에서 인접 메모를 더 읽을 때 새로 선명해지는 축은 **클라우드 상태의 원천이 어디에 있는가**다. 같은 AWS 리소스를 다루더라도 콘솔에서 만든 상태, Terraform 코드가 선언한 상태, Kubernetes controller가 계속 맞추는 상태, GitOps 레포가 배포하는 상태는 서로 다른 복구 경로를 만든다.

이 질문은 실행 세부를 `Cloud`로 옮기기 위한 것이 아니다. 오히려 실행 세부를 인접 노드에 남겨 두면서, `Cloud`에는 선택의 상위 의미만 남기기 위한 필터다.

- 콘솔이 원천이면 빠르게 만들 수 있지만, 나중에 누가 무엇을 바꿨는지 회수하기 어렵다.
- Terraform이 원천이면 반복 생성과 복구가 쉬워지지만, 실제 데이터 리소스 삭제 위험까지 함께 관리해야 한다.
- Kubernetes controller가 원천이면 원하는 상태로 계속 수렴하지만, 클라우드 리소스와 클러스터 내부 상태가 어긋나는 지점을 따로 관측해야 한다.
- GitOps 레포가 원천이면 배포 이력과 재현성은 좋아지지만, 수동 핫픽스가 선언 상태와 어긋나지 않게 다시 합쳐야 한다.

따라서 `Cloud`의 상위 판단은 `어떤 서비스를 쓰는가`만이 아니라, **운영 중인 상태를 나중에 어느 문서·코드·컨트롤러·로그에서 다시 찾을 수 있는가**까지 포함한다. 이 축은 [[../../mapped/deep-knowledge/infra]]의 Terraform/import/VPC 메모, [[../../mapped/deep-knowledge/devops]]의 GitOps·롤백 메모, [[../../mapped/deep-knowledge/container]]의 controller·배포 단위 메모와 이어지지만, 이 페이지에서는 도구 이름보다 상태 소유권의 선택 기준만 보존한다.


## 위임을 선택할 때 남겨야 할 세 가지 손잡이
원문이 비어 있는 동안에도 `Cloud`가 안전하게 더 선명해질 수 있는 지점은, 클라우드 선택을 **편의 기능 채택**이 아니라 **위임 후에도 남겨 둘 손잡이 설계**로 읽는 것이다. 인접 노드의 실행 메모를 다시 보면, 사용자는 관리형 서비스를 쓰더라도 모든 책임이 사라진다고 보지 않는다. 오히려 무엇을 맡겼는지, 무엇을 계속 볼 수 있는지, 문제가 났을 때 어디서 다시 잡을 수 있는지를 함께 묻는다.

따라서 새 클라우드 메모가 들어오면 먼저 아래 세 손잡이 중 어디를 건드리는지 확인하는 편이 좋다.

| 손잡이 | 핵심 질문 | 드리프트 방지 기준 |
| --- | --- | --- |
| 관측 손잡이 | 맡긴 뒤에도 로그, 지표, 흐름, 권한 변경을 어디서 볼 수 있는가 | CloudWatch·Flow Logs 같은 도구명보다 `무엇이 보이지 않으면 위험한가`를 남긴다 |
| 변경 손잡이 | 리소스 상태를 콘솔, Terraform, GitOps, controller 중 어디서 다시 고칠 수 있는가 | 특정 절차를 복사하지 말고 상태 원천과 수정 경로만 남긴다 |
| 회수 손잡이 | 비용·장애·이전 상황에서 구조를 줄이거나 되돌릴 수 있는가 | 벤더 기능 추천표가 아니라 빠져나올 수 있는 경계와 비용을 묻는다 |

이 세 손잡이는 빈 `Cloud` 노트를 억지로 채우기 위한 새 지식이 아니다. 오히려 주변에 이미 많은 AWS·서버리스·컨테이너 실행 메모가 있을 때, 그것을 `Cloud` 본문으로 끌고 오지 않고도 상위 판단으로만 압축하는 필터다.

## 새 메모를 받을 때의 승격 순서
빈 `Cloud` 노드에 새 클라우드 관련 메모가 들어오면, 먼저 그것을 서비스명 기준으로 분류하지 말고 **승격될 수 있는 판단의 높이**부터 본다. 같은 AWS 메모라도 어떤 것은 `Infra`의 설정 절차에 남아야 하고, 어떤 것은 `Cloud`의 상위 선택 기준으로 올라올 수 있다.

1. **증상·명령·설정값이면 실행 노드에 둔다.** VPC 연결, NAT, IAM 적용 지연, ingress controller, ArgoCD 등록처럼 재현 절차가 핵심이면 [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/devops]]가 더 맞다.
2. **반복 선택 기준이면 `Cloud`로 승격한다.** `서버리스로 먼저 가능한지 본다`, `모든 것을 위한 클라우드는 없다`, `편리하지만 내부를 통제할 수 없다`처럼 이후 선택에도 계속 쓰일 문장은 이 노드의 후보가 된다.
3. **위임 뒤 남는 손잡이를 함께 묻는다.** 선택 기준으로 승격하더라도 관측·변경·회수 손잡이가 무엇인지 보이지 않으면 서비스 추천표로 흐르기 쉽다. 새 문장은 최소한 `무엇을 맡겼고 무엇을 계속 볼 수 있는가`까지 연결해서 받는다.

이 순서를 두면 빈 노드가 실행 메모의 흡수장이 되지 않고, 주변 노드에서 반복되는 클라우드 경험이 실제로 상위 판단으로 굳어지는 순간만 `Cloud`에 남길 수 있다.

## 비용 메모를 구조 변경 신호로 읽기
원본 `Cloud.md`가 비어 있는 동안에도, 인접 `Infra` source의 비용 메모는 이 노드의 경계를 더 선명하게 해 준다. 다만 NAT gateway, Lambda 요청당 비용, VPC Endpoint 같은 항목을 그대로 옮기면 `Cloud`가 곧바로 AWS 비용표가 된다. 이 페이지에서 받아야 할 것은 비용 항목 자체가 아니라, **비용이 어느 순간 구조 선택을 다시 묻게 만드는가**라는 신호다.

특히 인접 원문에는 `NAT gateway 비용이 은근 많이 나올 수 있음`, `S3 전용 VPC Endpoint`, Lambda의 지속 실행·사용자 증가 비용, Fargate 대안 같은 메모가 함께 놓여 있다. 이 배열은 단순 절약 팁보다, 처음에는 편해 보였던 관리형·서버리스 선택이 트래픽 모양과 상태 요구에 따라 다시 서버·컨테이너·네트워크 구조 선택으로 돌아온다는 점을 보여 준다.

| 비용 신호 | `Cloud`에서 받을 상위 질문 | 실행 세부가 머물 곳 |
| --- | --- | --- |
| NAT gateway 비용 증가 | private 구조와 외부 접근을 유지할 때 숨은 네트워크 비용이 선택을 언제 뒤집는가 | [[../../mapped/deep-knowledge/infra]] |
| S3·DynamoDB VPC Endpoint | 비용 절감이 보안·접속 경계 설계와 함께 움직이는가 | [[../../mapped/deep-knowledge/infra]] |
| Lambda 요청당 비용·지속 실행 한계 | 서버리스 기본값이 사용자 수, 지연시간, 상태성 앞에서 언제 멈추는가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]] |
| Fargate·Redis 같은 대안 | 관리 포인트를 줄이면서도 지속 실행과 브로드캐스팅을 어떻게 회수하는가 | [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/infra]] |

따라서 새 비용 메모가 들어오면 먼저 `얼마나 비싼가`보다 `어떤 구조 가정이 깨졌는가`를 본다. 이 기준을 세워 두면 `Cloud`는 비용 최적화 팁 모음으로 흐르지 않고, 관리형 편의·네트워크 경계·상태성·회귀 가능성이 함께 흔들리는 순간을 읽는 상위 판단 노드로 남을 수 있다.

## 공개/비공개 경계를 출입구 설계로 읽기
빈 `Cloud` 노드에서 인접 `Infra` 메모를 다시 볼 때 또 하나 안전하게 남길 수 있는 상위 축은 **누가 어떤 경로로 들어올 수 있는가**다. 원문 자체는 비어 있지만, 관련 source에는 public static IP, public/private subnet, NAT, VPC Endpoint, 계정 분리, 대표 로그인 계정, CloudWatch 같은 메모가 흩어져 있다. 이 항목들을 그대로 가져오면 실행 노드가 되지만, `Cloud`에서는 클라우드가 외부 접근과 내부 운영 접근을 어떤 출입구로 나누는지 묻는 기준으로만 받을 수 있다.

이 관점에서 public/private은 단순 보안 설정값이 아니다. 사용자가 들어오는 문, 운영자가 진입하는 문, 자동화 도구가 수정하는 문, 관리형 서비스가 대신 연결하는 문이 서로 다르다는 뜻이다. 경계를 잘못 잡으면 처음에는 접속이 쉬워도 나중에 비용·권한·장애 조사·계정 탈취 여파가 한꺼번에 커진다.

| 출입구 장면 | `Cloud`에서 남길 질문 | 실행 세부가 머물 곳 |
| --- | --- | --- |
| public IP·public subnet | 외부 사용자가 직접 닿아도 되는 표면은 어디까지인가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/web]] |
| private subnet·NAT·VPC Endpoint | 내부 자원이 외부와 통신해야 할 때 비용과 보안 경계를 어떻게 나누는가 | [[../../mapped/deep-knowledge/infra]] |
| 대표 로그인 계정·환경별 계정 분리 | 운영자 접근을 어디서 모으고, 탈취 시 피해 반경을 어떻게 줄이는가 | [[../../mapped/deep-knowledge/infra]] |
| CloudWatch·Flow Logs·감사 로그 | 닫아 둔 경계 안에서 실제 흐름을 어디서 다시 볼 수 있는가 | [[../../mapped/deep-knowledge/monitoring]], [[../../mapped/deep-knowledge/infra]] |

따라서 새 public/private 메모가 들어오면 `공개냐 비공개냐`로만 닫지 않는다. 먼저 그 경계가 사용자 요청, 운영자 접속, 자동화 수정, 관리형 서비스 연결 중 어떤 출입구를 설계하는지 본다. 이 기준을 두면 `Cloud`는 방화벽·서브넷 설정 사전이 아니라, 클라우드 위에서 접근성과 통제 가능성을 동시에 배치하는 상위 운영 노드로 남을 수 있다.

## 관리형 편의가 무너지는 순간을 먼저 찾기
공란 상태의 `Cloud`를 계속 다룰 때는 클라우드의 장점을 일반론으로 적기보다, **처음에는 편의로 보였던 선택이 언제 다시 운영 문제로 돌아오는가**를 묻는 편이 안전하다. 인접 노드의 서버리스, Fargate, EKS, NAT, CloudWatch, IAM 메모는 모두 관리형 서비스가 일을 덜어 주는 동시에 새로운 경계를 만든다는 사실을 보여 준다.

이 관점에서 관리형 클라우드 선택은 `운영이 사라진다`가 아니라 `운영의 위치가 바뀐다`에 가깝다. 직접 서버를 보지 않아도 지연시간, 네트워크 비용, 권한 전파, 로그 가시성, 데이터 복구, 플랫폼 종속성은 남는다. 그래서 `Cloud`에 남길 질문은 특정 서비스 추천보다, 편의가 깨지는 장면을 미리 알아보는 판별식이어야 한다.

| 편의가 무너지는 장면 | `Cloud`에서 남길 질문 | 실행 세부가 머물 곳 |
| --- | --- | --- |
| 서버리스가 콜드스타트·지속 연결·상태성 앞에서 흔들림 | 시작 속도와 운영 감소가 사용자 경험·실시간성 비용을 감당하는가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]] |
| 관리형 네트워크가 NAT·Endpoint·egress 비용으로 되돌아옴 | 보안 경계를 지키는 비용이 구조 단순성을 언제 넘어서는가 | [[../../mapped/deep-knowledge/infra]] |
| 권한·계정·감사 로그가 느리게 반영되거나 흩어짐 | 맡긴 자원이라도 누가 열고 바꿨는지 추적할 수 있는가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/monitoring]] |
| EKS·Fargate·관리형 DB가 이전과 복구를 어렵게 만듦 | 빠른 시작 대신 나중의 이전·롤백·축소 손잡이를 잃지 않았는가 | [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/devops]] |

이 섹션도 원문을 대신 채우는 내용이 아니다. `Cloud.md`가 아직 비어 있는 동안, 인접 노드의 실행 메모를 상위 판단으로만 걸러 받기 위한 필터다. 앞으로 원문에 실제 문장이 들어오면, 그 문장이 클라우드의 장점 자체를 말하는지 아니면 편의가 깨지는 조건을 말하는지 먼저 구분해야 한다.

## 선택의 시간을 세 겹으로 나누기
빈 `Cloud` 노드에서 지금 더 보강할 수 있는 또 하나의 안전한 기준은 클라우드 선택을 **도입 순간 / 운영 중 / 철수·이전 순간**으로 나눠 읽는 것이다. 같은 서비스라도 처음 선택할 때 좋아 보이는 이유와, 운영 중 계속 확인해야 하는 신호와, 나중에 줄이거나 옮길 때 문제가 되는 지점은 다르다.

이 시간축을 두면 `Cloud`가 특정 벤더 서비스 목록으로 흐르지 않고, 인접 노드의 실행 메모를 더 높은 판단 질문으로만 받을 수 있다.

| 시간층 | 처음 보이는 이점 | 나중에 확인해야 할 질문 | 연결되는 인접 노드 |
| --- | --- | --- | --- |
| 도입 순간 | 서버 관리 감소, 빠른 배포, 관리형 기본값 | 이 편의가 어떤 통제 포기를 전제로 하는가 | [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/devops]] |
| 운영 중 | 자동 확장, 관측 도구, 계정·권한 분리 | 비용·지연·권한·로그가 사람이 이해할 수 있는 표면으로 남는가 | [[../../mapped/deep-knowledge/monitoring]], [[../../mapped/deep-knowledge/infra]] |
| 철수·이전 순간 | 벤더 기능과 관리형 API의 생산성 | 데이터·상태·배포 경로를 다른 실행 모델로 옮길 손잡이가 남는가 | [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/devops]] |

따라서 새 클라우드 메모가 들어오면 먼저 `좋은 서비스인가`보다 `어느 시간층의 판단인가`를 묻는다. 도입 이유만 있으면 빠르게 시작하는 근거로 남기되, 운영 중 신호와 철수 가능성이 함께 보이지 않으면 아직 `Cloud`의 상위 판단으로 완성된 것은 아니다. 이 기준은 원문 공란을 유지하면서도 앞으로 들어올 문장을 더 정밀하게 흡수하기 위한 임시 분류축이다.

## 인접 노드와의 관계
### [[../../mapped/deep-knowledge/infra]]
가장 가까운 이웃 노드다. 실제로 현재 클라우드 관련 메모 대부분은 `Infra` 쪽에 흩어져 있다. 그러나 그 메모들은 VPC, NAT gateway, 서버리스, AWS 비용처럼 실행 레벨이 강하다. `Cloud`는 그보다 한 단계 위에서, 왜 클라우드를 택하는지와 어떤 추상화를 받아들일지를 묻는 노드가 될 가능성이 높다.

### [[../../mapped/deep-knowledge/container]]
컨테이너는 클라우드 위에서 워크로드를 어떻게 포장하고 이동시키는지의 문제에 가깝다. `Cloud`가 플랫폼과 책임 분배의 선택이라면, `Container`는 그 선택 위에서 배포 단위와 실행 환경을 어떻게 다룰지를 더 세부적으로 파고드는 노드다.

### [[../../mapped/deep-knowledge/devops]]
`Devops` 원문은 빠르고 안전한 배포, 협업 문화, 피드백 루프를 중심으로 전개된다. 따라서 `Cloud`가 자원·서비스·관리 모델의 선택이라면, `Devops`는 그 기반 위에서 조직과 배포 흐름을 어떻게 운영할지의 문제다. 둘은 겹치지만 초점이 다르다.

### [[../../mapped/deep-knowledge/web]]
정적 배포, CDN, API, 인증, 저장소 같은 웹 실행 환경은 클라우드와 직접 맞닿는다. 그러나 `Web`이 사용자에게 보이는 서비스 구조에 더 가깝다면, `Cloud`는 그 구조를 어떤 운영 모델 위에 올릴지 결정하는 배경 레이어다.

## 이 노드에 들어오면 좋은 내용 / 다른 노드에 남겨둘 내용
### `Cloud`에 들어오면 좋은 것
- 특정 벤더 서비스 자체보다, **왜 관리형/서버리스/컨테이너/가상서버를 고르는지**에 대한 선택 이유
- 비용표보다, **숨은 비용과 운영 포인트가 어느 선택에서 튀어나오는지**를 비교하는 메모
- `편리함을 얻는 대신 통제를 어디까지 포기할 수 있는가` 같은 상위 질문
- 관측·권한·계정 분리처럼 **운영 가시성과 책임 경계**를 어디서 설계할지에 대한 기준
- 나중에 구조를 바꿔야 할 때 **회귀 가능성**을 판단하는 기준

### 다른 노드에 남겨둘 내용
- VPC, IAM, Route53, CloudFront 세부 항목처럼 이미 `Infra`가 더 잘 받는 실행 메모
- Dockerfile, compose, Kubernetes 리소스처럼 `Container`가 더 잘 받는 배포 단위 메모
- CI/CD, 테스트 자동화, 조직 문화처럼 `Devops`가 더 잘 받는 팀 운영 메모
- 웹 렌더링, 프론트 흐름, 사용자 인터페이스처럼 `Web`이 더 잘 받는 서비스 표면 메모

이 구분은 현재 공란 노드를 지키기 위한 임시 경계다. 하지만 바로 이 경계 덕분에, 이후 사용자가 `Cloud` 원문을 직접 채웠을 때 그 문장을 어디에 흡수해야 하는지 더 빠르게 판별할 수 있다.

## 지금 시점의 해석
이 노드의 공란은 단순 누락일 수도 있지만, 다른 노트에 흩어진 클라우드 관련 생각이 아직 하나의 상위 판단 노드로 수렴되지 않았다는 뜻일 가능성도 크다. 지금 보이는 주변 흔적을 기준으로 하면 `Cloud`는 장기적으로 다음 역할 중 하나로 발전할 수 있다.

- 클라우드 선택 기준을 모으는 상위 판단 노드
- AWS, GCP, Azure 같은 벤더 지식을 비교하는 노드
- 서버리스, 네트워크, 데이터, 비용, 통제 가능성의 교환관계를 다루는 운영 철학 노드

현재 단계에서는 이 가능성을 억지로 확정하지 않고, 원본이 들어올 때 어느 방향으로 실제 문장이 기울어지는지 보는 편이 맞다.

## 원본이 채워질 때 가장 먼저 볼 질문
- 클라우드를 편의성의 문제로 보는가, 운영 전략의 문제로 보는가
- 비용 최적화와 개발 속도 중 무엇을 더 자주 우선하는가
- 서버리스 선호가 어디까지 유효하고, 어디서 서버·컨테이너로 되돌아오는가
- 관측, 권한, 계정 분리, shared responsibility를 얼마나 상위 판단으로 다루는가
- 특정 벤더 지식보다 공통 원리와 회귀 가능한 구조를 더 중시하는가
- `Infra`에 이미 적힌 실행 메모를 반복하는가, 아니면 그 위의 선택 기준을 새로 여는가

### 원문이 생기면 먼저 분류할 4가지 문장형
- **철학 문장**: `클라우드는 편리하지만 내부를 통제할 수 없다`처럼 선택의 태도를 직접 말하는 문장.
- **구조 문장**: 정적 페이지, CDN, Lambda, Fargate처럼 어떤 조합을 택하는지 드러내는 문장.
- **비용 문장**: NAT gateway, 네트워크, 운영 인력처럼 표면 아래 비용이 드러나는 문장.
- **가시성 문장**: 로그, 권한, 계정, 모니터링처럼 맡겼어도 계속 직접 봐야 하는 표면을 말하는 문장.

이 네 갈래로 먼저 나누면, 나중에 실제 원문이 들어와도 `Infra`의 세부 구현 메모와 섞이지 않고 `Cloud` 고유의 상위 판단을 비교적 안정적으로 복원할 수 있다.

## 원본 갱신이 들어오면 우선 볼 델타
- 새로 생긴 문장이 벤더 비교인지, 운영 철학인지, 비용/구조 기준인지 먼저 분류한다.
- `Infra`에 있는 AWS·서버리스·네트워크 메모와 겹친다면 반복인지 상위 재정리인지 구분한다.
- 사용자가 직접 쓴 표현이 현재 mapped의 가설보다 훨씬 좁거나 넓다면, 기존 해석을 유지하지 말고 `human-reviews/`에 복사 후 선호 변화로 기록한다.
- 특히 `모든것을 위한 클라우드는 없다` 같은 선택 기준 문장이 실제 `Cloud` 원문에 들어오는지, 아니면 전혀 다른 관심사로 기울어지는지를 먼저 확인한다.

## 유지 원칙
- 원본이 비어 있는 동안에는 과도한 내용 확장을 하지 않는다.
- 다만 관련 source 노드에 이미 흩어진 같은 주제 흔적은 읽어 두고, 이 페이지의 경계와 향후 흡수 기준을 더 선명하게 만드는 데만 제한적으로 활용한다.
- 사용자가 원본을 직접 채우면 그 버전을 우선 읽고 필요 시 `human-reviews/`에 복사해 선호 변화로 기록한다.
- 이후 심화는 이 페이지를 새로 쓰기보다, 실제 원문이 생긴 뒤 구조·논지·링크를 점진적으로 보강하는 방식으로 진행한다.
- 빈 노드일수록 `무엇이 아직 비어 있는지`와 `어디까지가 현재 가설인지`를 분리해 적는다.

## 후속 흡수 기준
이 페이지의 누적 편집 이력은 실행 로그로 남기고, 본문에는 앞으로의 판별 기준만 남긴다. 원본이 계속 비어 있는 동안 `Cloud`에 새 내용을 더할 때는 아래 조건 중 하나가 실제로 선명해질 때로 제한한다.

| 새로 들어오는 신호 | `Cloud`에 흡수할 때 | 인접 노드에 남길 때 |
| --- | --- | --- |
| 선택 이유 | 관리형, 서버리스, 컨테이너, VM 중 왜 그 방식을 고르는지 반복 판단으로 굳어진다. | 특정 서비스 설정값이나 배포 절차가 핵심이다. |
| 책임 경계 | 무엇을 맡기고 무엇을 계속 관측·변경·회수할지 묻는다. | 이미 정해진 구조 안에서 권한, 네트워크, 로그를 조정한다. |
| 비용 신호 | 숨은 네트워크 비용이나 운영 복잡도가 구조 선택을 다시 열게 만든다. | 단일 요금 항목이나 절감 팁을 기록한다. |
| 시간층 | 도입, 운영 중, 철수·이전 순간의 판단 차이를 드러낸다. | 현재 플랫폼에서 문제를 해결하는 재현 절차가 중심이다. |

이 기준을 통과하지 못한 클라우드 메모는 [[../../mapped/deep-knowledge/infra]], [[../../mapped/deep-knowledge/container]], [[../../mapped/deep-knowledge/devops]], [[../../mapped/deep-knowledge/monitoring]] 중 더 구체적인 실행 노드에 남긴다. `Cloud`는 빈 원문을 대신 채우는 흡수장이 아니라, 주변 실행 메모가 상위 선택 기준으로 승격되는 순간만 받는 경계 노드로 유지한다.
