# Devops

- 원본: `source/shdkej-content/Deep Knowledge/Devops.md`
- 상태: 안정 디벨롭
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 DevOps를 단순한 `배포 자동화`가 아니라 **작은 팀이 기능을 독립적으로 만들고, 프로덕션과 가까운 환경에서 검증하며, 빠르고 안전하게 배포하고, 다시 피드백을 회수하는 운영 문화와 시스템의 결합**으로 다룬다. 그래서 도구 이름보다 개발자 경험, 배포 흐름, 장애 대응, 롤백 가능성, 관측성, 팀 구조, CI/CD의 역할 분리에 반복해서 머문다.

## 이 노드가 실제로 답하려는 질문
- DevOps는 직무 이름인가, 팀 문화인가, 아니면 제품 전달 구조 전체인가?
- 개발자와 운영자의 경계를 어디까지 줄여야 빠르면서도 안전한가?
- 좋은 DevOps는 단순 자동화가 아니라 어떤 피드백 루프를 만들어야 하는가?
- CI와 CD는 무엇을 각각 책임져야 하며, 어디까지 자동화해야 하는가?
- 결국 중요한 것은 도구 선택이 아니라 `되돌릴 수 있고 복구할 수 있는 시스템` 아닌가?

## 압축 판단 모델
원문은 DevOps를 세 겹으로 겹쳐 본다.

| 겹 | 원문에서 반복되는 질문 | mapped에서 보존할 판단 |
|---|---|---|
| 문화 | 개발과 운영을 같은 제품 전달 책임 안에 묶을 수 있는가? | 배포 속도와 고객 피드백 회수 속도를 함께 본다. |
| 포지션 | 전담 DevOps가 운영을 독점하지 않고 개발자를 어떻게 돕는가? | 플랫폼 조정자, 생산성 도구, 팀 간 장벽 완화로 읽는다. |
| 시스템 | 자동화가 실패했을 때 되돌리고 설명하고 이어받을 수 있는가? | CI/CD, 관측성, 롤백, 장애 회고를 하나의 복구 루프로 묶는다. |

## 원문 구조

### 1. DevOps를 보는 기본 정의
원문 첫머리는 DevOps를 다음 축으로 설명한다.

- 기능을 작은 팀이 독립적으로 구현한다.
- 프로덕션과 유사한 환경에서 정확성을 검증한다.
- 코드를 빠르고 안전하게 배포한다.
- 자동화된 테스트와 빌드, 빠른 피드백을 확보한다.
- 개발자가 자신이 만든 것을 운영과 가깝게 느끼게 한다.

즉 이 노드에서 DevOps는 `개발과 운영이 친해지자` 수준의 구호가 아니다. **개발자가 고객과 더 가까운 피드백 루프 안에서 일하게 만드는 제품 전달 체계**다.

### 2. 문화로서의 DevOps
원문은 DevOps를 린, 에자일의 연장선으로 본다.

- 큰 조직이 오래 준비해서 한 번 출시하는 방식은 느리고 위험하다.
- 개발과 운영을 분리된 부서의 경계로 보기보다, 제품 출시를 위한 공동 책임으로 본다.
- 팀을 나누는 것보다 같은 목표 아래 협업 구조를 만드는 것이 중요하다.

여기서 중요한 것은 `운영을 개발자에게 떠넘긴다`가 아니다. 오히려 **운영을 포함한 전체 제품 전달 과정에 개발자가 더 직접 연결되게 한다**는 의미에 가깝다. 원문에서 마케팅과의 간극, 사용자와의 상호작용, 개발자 경험이 함께 언급되는 이유도 이 때문이다.

### 3. 포지션으로서의 DevOps
원문은 한편으로 DevOps를 분명한 포지션으로도 다룬다.

- 인프라와 CI/CD 파이프라인을 설계하고 관리한다.
- 개발자 생산성을 높이는 도구를 만든다.
- 팀 간 장벽을 낮추고 공통 흐름을 정리한다.
- 개발자가 운영을 감당 가능한 수준으로 다루게 만든다.

따라서 이 노드의 시선에서 DevOps 엔지니어는 `운영 담당자`보다 **개발자가 더 잘 개발하고 더 안전하게 운영하게 만드는 플랫폼 조정자**에 가깝다.

### 4. 왜 DevOps가 필요한가
원문에는 역할 분리에 대한 긴 질문이 계속 등장한다.

- 작은 조직에서는 기획, 개발, QA, DevOps를 완전히 나누기 어렵다.
- 서버가 코드화되면서 백엔드도 서버를 관리할 수 있게 되었다.
- 그렇다면 DevOps라는 별도 역할이 꼭 필요한가?
- 반대로 규모가 커지면 세분화와 전문화가 필요하지 않은가?

이 질문들은 DevOps가 고정된 직무명이라기보다, **조직 규모와 복잡도에 따라 분화되는 운영 역량의 묶음**임을 보여준다. 원문은 결국 어느 한쪽을 단정하지 않는다. 대신 다음 같은 입장을 드러낸다.

- 규모가 작을수록 통합적으로 보는 편이 빠르다.
- 규모가 커질수록 세분화와 전문화가 이점을 만든다.
- 그래도 최종 목표는 경계 강화가 아니라 흐름 최적화여야 한다.

## 개발자를 편하게 만드는 시스템이라는 관점
이 노드의 실무 감각은 매우 분명하다. 좋은 DevOps는 개발자가 다음 일을 쉽게 하게 만들어야 한다.

- 깃 폴더 안에서 작업하고 푸시하면 이미지와 파이프라인이 자연스럽게 이어질 것
- 테스트용 데이터와 연결 확인을 빠르게 할 수 있을 것
- 프론트와 백엔드가 필요한 데이터를 쉽게 주고받을 수 있을 것
- 기본 설정은 실제 운영과 최대한 가깝되, 세부 설정은 선택적으로 조정 가능할 것
- 인프라 요청과 자원 할당도 수작업이 아니라 관리 흐름으로 연결될 것

이 패턴은 결국 **개발자가 제품 흐름 안에서 막히지 않게 하는 플랫폼 설계**를 지향한다. 다만 원문은 여기서도 `편함`만 추구하지 않는다. 실제 서버 값과 유사한 테스트, 버전관리, 관리 페이지, 자원 선택, 환경 분리가 함께 나오므로, 목표는 단순 편의보다 **안전한 편의**다.

## 운영 파트의 핵심
원문 후반에서 운영은 별도 부록이 아니라 DevOps의 중심으로 나타난다.

### 장애 대응 루프
- 인식
- 보고
- 롤백
- 복구
- 피드백

이 다섯 단계는 매우 중요하다. 원문에서 DevOps는 배포를 끝내는 역할이 아니라, **문제가 났을 때 얼마나 빨리 이해하고 되돌리고 다시 배우는가**의 문제로 확장된다.

### Postmortem의 의미
사후 회고는 단순 반성문이 아니라 운영 학습 장치다.

- 무엇이 어떻게 깨졌는가
- 어떤 신호를 놓쳤는가
- 어떤 자동화가 있었으면 더 빨리 막았는가
- 다음에는 어떤 기준을 파이프라인이나 모니터링에 넣을 것인가

즉 원문은 장애를 개인 실수보다 **시스템 학습 기회**로 다루려는 경향을 갖고 있다.

### 운영에서 반복되는 기준
원문 전체에서 운영 판단 기준은 크게 네 가지로 압축된다.

- 되돌릴 수 있는가
- 복구할 수 있는가
- 수습이 가능한가
- 다른 사람이 이어받을 수 있는가

이 기준은 DevOps를 화려한 자동화보다 **운영 복원력**의 문제로 본다는 강한 신호다.

## 성과 측정과 관측성
원문은 DORA Metrics를 직접 적어 두고 있다.

- Deployment Frequency
- Lead Time
- MTTR
- Change Failure Rate

이 지표들이 중요한 이유는, DevOps를 추상적 문화가 아니라 **측정 가능한 흐름 개선 활동**으로 보기 때문이다. 또한 로그 폭발, 권한 남용, 시스템 다운, 네트워크 혼잡 같은 edge case가 함께 기록되어 있어, 원문은 숫자만 보는 것이 아니라 **실제 실패 양상까지 포함한 관측성**을 중시한다.

여기서 관측성은 `지표가 많다`는 뜻보다, 다음 질문에 빨리 답할 수 있는 상태에 가깝다.

- 어디서 병목이 시작됐는가
- 오류가 코드, 데이터, 인프라, 외부 의존성 중 어디에서 생겼는가
- 지금 롤백해야 하는가, 버텨도 되는가
- 다음에는 어떤 신호를 기본 대시보드와 알림으로 승격해야 하는가

즉 [[Monitoring]]과의 연결은 부가 기능이 아니라, DevOps가 `빠르게 배포`한 뒤 `빠르게 이해`하기 위해 필요한 중심 축이다.

## 원문 발췌 구조
원문은 대체로 아래 층으로 쌓여 있다.

1. **정의와 목적**: 작은 팀, 빠른 배포, 테스트 자동화, 피드백, 문화로서의 DevOps.
2. **역할과 필요성 질문**: DevOps가 별도 포지션이어야 하는가, 개발자가 어디까지 운영을 안아야 하는가.
3. **개발자 경험 설계**: 깃 폴더에서 바로 작업하고 테스트하고 배포로 이어지는 흐름, 테스트 데이터 접근, 관리 페이지 구상.
4. **운영과 복구**: 장애 대응, Postmortem, DORA, 롤백, 수습 가능성, 인수인계 가능성.
5. **CI 세부 메모**: 품질 게이트, 리포트, 보안 스캔, 아티팩트, Jenkins/GitHub Actions/GitLab CI 운용 팁.
6. **CD와 GitOps 고민**: ArgoCD, 인프라 레포와 코드 레포 분리, 카나리, 롤링 업데이트, 데이터 스냅샷.
7. **확장 시나리오**: ML 파이프라인, 모노레포 빌드, 장애 자동 분석, 인프라 템플릿화.

이 배열을 보면 원문은 `툴 사용법`보다 **운영 체계를 어떻게 자라게 할지에 대한 질문 보드**에 가깝다.

## CI와 CD를 읽는 방식
원문은 CI/CD를 강하게 세분해 본다.

### CI의 역할
CI는 여러 사람의 변경을 공유 공간에 통합하고, 자동 검증을 붙이는 단계다.

- 정적 분석
- 테스트
- 리뷰 보조
- 빌드
- 아티팩트 생성
- 보안 점검
- 실패 시 빠른 알림

원문에서 특히 눈에 띄는 기준은 다음이다.

- 실패를 먼저 드러낼 것
- build와 deploy를 분리할 것
- 커밋 전, PR 후, 머지 후 각 단계의 시간을 관리할 것
- 커버리지, 로그, 패키지 크기, 실행 시간 같은 리포트를 남길 것

즉 CI는 단순 체크리스트가 아니라 **팀이 무엇을 품질 기준으로 삼는지 선언하는 공간**이다.

### CD의 역할
CD는 배포 그 자체보다 배포 이후 상태까지 포함한다.

- 배포 이력 관리
- 버전 싱크 확인
- 롤백
- 핫픽스 판단
- 배포된 상태의 가시성 확보
- 인프라 코드와 애플리케이션 코드의 연결 관리

그래서 원문은 ArgoCD 같은 도구를 `배포를 해 주는 것`보다 **배포된 것을 관리하고 보이게 하는 시스템**으로 읽고 있다.

### 원문이 특히 집착하는 CD 문제들
이 노드에서 CD는 `자동 배포 여부`보다 아래의 꼬임을 어떻게 다룰지에 더 많이 머문다.

- 인프라 레포와 애플리케이션 레포를 어디까지 분리할 것인가
- 소스 변경과 인프라 변경의 감지 주체를 누구로 둘 것인가
- 카나리·롤링 업데이트 중 데이터 변경이 섞이면 어떻게 복구할 것인가
- 배치잡, Quartz 같은 장기 작업이 도는 동안 무중단 배포를 어떻게 성립시킬 것인가
- 퀵픽스를 허용하더라도 일반 플로우를 얼마나 덜 깨뜨릴 수 있는가

즉 이 원문이 보는 CD의 어려움은 `스크립트를 짜면 된다`가 아니라, **서로 다른 생명주기를 가진 것들을 한 배포 표면에서 충돌 없이 다루는 일**이다.

## 배포 흐름을 세 개의 소유권으로 나누기
원문에서 CI/CD 고민이 길어지는 이유는 `자동화할까 말까`보다, 변경이 실제 서비스에 도달하기까지 **무엇의 소유권이 어디에 있는가**가 계속 엉키기 때문이다. 특히 프로젝트 레포, 인프라 레포, ArgoCD/배포 레포를 나누는 메모는 도구 선택보다 상태 소유권 설계에 가깝다.

| 소유 위치 | 주로 맡는 것 | 실패하면 먼저 볼 질문 |
|---|---|---|
| 애플리케이션 레포 | 코드 변경, 테스트, 이미지 빌드, 버전 태그 | 이 변경이 어떤 산출물로 고정됐는가? |
| 배포/GitOps 레포 | 어떤 버전을 어떤 환경에 적용할지, 배포 이력과 싱크 | 현재 운영 환경이 어떤 선언을 따라가고 있는가? |
| 인프라 레포 | 클러스터, 네트워크, 권한, 공통 리소스 | 실행 기반 자체가 바뀐 것인가, 앱만 바뀐 것인가? |

이 구분을 세우면 원문의 `소스코드 변경 후 인프라를 다시 건드리면 안 된다`, `인프라는 인프라대로 관리되고 소프트웨어 업데이트는 소스코드에서 따로 처리되어야 한다`는 메모가 더 선명해진다. DevOps는 한 버튼으로 모든 것을 밀어 넣는 일이 아니라, **각 변경이 자기 소유 위치에서 선언되고 다른 층에는 필요한 신호만 전달되게 하는 일**이다.

또한 이 관점은 핫픽스와 롤백 판단에도 이어진다. 급한 수정이 필요해도 서버를 직접 고치기보다, 어떤 소유 위치의 선언을 바꿔야 정상 플로우를 덜 깨는지 먼저 확인해야 한다. 그래야 빠른 조치 이후에도 배포 이력, 이미지 태그, 인프라 상태, 관측 지표가 서로 어긋나지 않는다.

## 배포가 중단하기 어려운 상태를 만날 때
원문 말미의 배치잡, Quartz, DB 변경, 카나리·롤링 업데이트 고민은 DevOps를 단순히 `새 버전을 띄우는 절차`로 볼 수 없게 만든다. 애플리케이션 인스턴스는 갈아 끼울 수 있어도, 실행 중인 작업과 이미 바뀐 데이터는 그 순간부터 배포 단위 바깥의 상태를 가진다.

따라서 이 노드에서 무중단 배포의 핵심 질문은 `새 인스턴스가 떴는가`보다 다음에 가깝다.

- 이미 실행 중인 배치잡은 누가 끝까지 책임지는가?
- 새 버전과 옛 버전이 동시에 떠 있을 때 같은 Quartz 작업이 중복 실행되지 않는가?
- DB schema나 데이터 의미가 바뀌는 동안 이전 버전이 계속 요청을 처리해도 되는가?
- 카나리에서 문제가 생겼을 때 코드만 되돌리면 되는가, 데이터 스냅샷과 이후 변경분도 함께 복구해야 하는가?
- 배포 전후에 꺼야 하는 기능, 잠가야 하는 큐, 확인해야 하는 작업 상태가 사람 기억에만 있지 않은가?

이렇게 읽으면 원문의 `쿼츠를 끄고 했다`, `배포 전에 api를 쏜다?`, `스냅샷을 찍어놓고 추가된 데이터를 더한다` 같은 메모는 미완성 팁이 아니라 **중단하기 어려운 상태를 배포 흐름 안으로 어떻게 들여올지**에 대한 실전 질문이다.

여기서 필요한 설계는 세 가지다.

| 상태 종류 | 위험 | 먼저 남겨야 할 손잡이 |
|---|---|---|
| 장기 실행 작업 | 배포 중 강제 종료, 중복 실행, 작업 유실 | drain, lock, idempotency, 완료 대기 기준 |
| 데이터 변경 | 롤백 후 데이터 의미 불일치, 양방향 호환 실패 | migration 단계화, 스냅샷, backfill/rollback 계획 |
| 외부 알림·큐 | 새/옛 버전의 이벤트 중복 처리, 재시도 폭발 | dedup key, 소비자 pause/resume, DLQ 확인 |

즉 좋은 CD는 새 이미지를 적용하는 도구가 아니라, **배포 중에도 계속 살아 있는 상태를 안전하게 넘기는 절차**까지 포함해야 한다. 이 관점은 [[Container]]의 상태 소유권, [[Monitoring]]의 알림 승격, [[Infra]]의 변경 전 기준점 확보와 바로 이어진다.

## CI 리포트는 다음 행동을 여는 피드백 표면이다
원문에는 CI 도구 사용법처럼 보이는 세부 메모가 많이 흩어져 있다. `failure()`로 실패 메시지를 보내기, pull request에 코멘트를 달기, 테스트 결과를 댓글로 붙이기, artifact 만료 시간을 조정하기, CI에서 코드 변경 수·빌드 시간·패키지 크기·정적 분석 결과를 리포트로 남기기 같은 항목들이다.

이것들은 단순한 설정 조각이라기보다, **자동화가 사람에게 어떤 판단을 돌려줘야 다음 행동이 가능한가**를 묻는 메모로 읽는 편이 맞다.

| CI가 남기는 신호 | 단순 기능으로 보면 | DevOps 노드 안에서의 의미 |
|---|---|---|
| 실패 메시지와 PR 코멘트 | 알림 자동화 | 실패 위치를 작업자에게 되돌려 주는 재진입 표면 |
| 커버리지·테스트 리포트 | 품질 수치 | 머지 가능성과 추가 검토 범위를 정하는 판단 재료 |
| 빌드 시간·패키지 크기 | 성능 통계 | 개발 흐름이 느려지는 병목을 조기에 발견하는 운영 지표 |
| artifact·버전·이미지 정보 | 산출물 관리 | 어떤 결과물을 배포·재현·롤백할 수 있는지 고정하는 증거 |
| 코드리뷰봇·레포 관리 AI | 자동 리뷰 도구 | 반복 검토와 보안·문서·브랜치 관리를 놓치지 않게 하는 보조 운영자 |

따라서 이 노드에서 CI는 `통과/실패`만 결정하는 게이트가 아니다. 개발자가 다시 들어갈 위치, 리뷰어가 봐야 할 범위, 배포자가 신뢰할 산출물, 운영자가 관측할 변화량을 한 번에 보여 주는 **작업 상태 전달 인터페이스**다.

이 관점은 원문 말미의 `Devops Agent` 메모와도 이어진다. 장애 상황을 1분 안에 파악하고, 시급도를 분류하고, 분석 과정을 정리하고, 컨텍스트를 축적하려는 구상은 CI 리포트의 확장판이다. 평소에는 PR과 파이프라인의 상태를 사람에게 되돌려 주고, 장애 시에는 운영 신호를 모아 대응 가능한 설명으로 돌려주는 것이다.

## 이상적인 흐름에 대한 욕망
원문 중간의 긴 `이상적인 개발 - 배포 환경` 인용은 이 노드의 방향감을 잘 보여 준다. 저자가 원하는 것은 거대한 중앙 통제 시스템보다, 다음이 자연스럽게 이어지는 상태다.

- 오늘 할 일이 자동으로 보인다.
- 작업 상태가 흐름에 맞게 갱신된다.
- 코드 리뷰와 피드백이 늦지 않게 도착한다.
- 머지 후 테스트와 리포트가 자동으로 이어진다.
- QA와 스테이징, 프로덕션 반영이 같은 사슬 안에서 관리된다.
- 실패와 품질 정보가 사람의 기억이 아니라 시스템에 남는다.

이상적인 DevOps는 결국 `배포 자동화`보다 **일의 상태가 끊기지 않는 전달 인터페이스**에 가깝다.

## 확장 노드로서의 DevOps
원문 후반에는 전형적 웹서비스 운영을 넘어서는 메모가 섞여 있다.

### ML까지 확장되는 흐름
`Devops for machine learning` 메모는 이 노드의 범위가 단순 서버 배포에서 끝나지 않음을 보여 준다.

- 데이터 소스와 ETL/ELT도 소스 관리 대상이 된다.
- 모델 학습과 평가도 CI/CD의 일부가 된다.
- 배포 후에는 성능뿐 아니라 재학습 전략까지 운영 대상이 된다.

즉 DevOps는 애플리케이션 코드만이 아니라, **실험-학습-배포-재학습 루프 전체를 다루는 운영 문법**으로 확장될 수 있다.

### 에이전트화 욕망
마지막의 `Devops Agent` 메모도 중요하다. 저자는 장애 대응을 단순 알람 수신이 아니라,

- 장애 규모를 빠르게 파악하고
- 시급도를 분류하고
- 분석 과정을 정리해 축적하고
- 다음 대응 품질을 높이는

반자동 운영 조수의 문제로 보고 있다. 이는 [[../Meta/AGENTS|AGENTS]] 노드와 맞닿으며, DevOps가 점점 `사람이 일일이 클릭하는 운영`에서 `맥락을 이어받는 운영 오케스트레이션`으로 가고 있음을 드러낸다.

## 원문 고유 시나리오
원문 말미의 `상황`과 `TODO`는 DevOps를 추상 문화론에서 실제 설계 문제로 끌어내린다.

| 원문 장면 | 직접 남길 질문 | 이 노드에서의 의미 |
|---|---|---|
| 개인 프로젝트로 Go web server를 원격 서버에 배포하다 에러가 난 상황 | 브랜치, 정적 테스트, 배포, 릴리즈 실패, 리소스 부족, 외부 라이브러리 문제를 한 흐름에서 어떻게 돌려받는가? | 작은 프로젝트에서도 릴리즈 결과가 바로 개발자에게 피드백되어야 한다. |
| 10인 개발팀이 compose로 개발하고 ArgoCD로 배포하는 상황 | 권한, 휴먼 에러, 비슷한 템플릿, 개발팀과 경영팀의 정보 경계를 어떻게 나누는가? | DevOps는 편의뿐 아니라 팀별 접근권과 반복 가능한 운영 틀을 함께 설계한다. |
| Terraform으로 실제 서버 3대를 구성하고 카나리/롤링 업데이트를 돌리는 상황 | 서버별 접속 링크, 접속 현황, 실제 테스트, 오토스케일링 기준을 어떻게 검증 표면으로 만드는가? | 인프라 변경은 선언만으로 끝나지 않고 실제 트래픽과 테스트 관측까지 이어져야 한다. |
| 인프라용 cookiecutter 구상 | CI/CD pipeline, ansible, Grafana/Prometheus, fluentd, Traefik, Kubernetes yaml을 어떻게 한 번에 시작 가능한 기본값으로 묶는가? | 새 프로젝트를 0에서 반복 세팅하지 않고, 기본 테스트와 추가 설정이 공존하는 운영 템플릿으로 승격하려는 욕망이다. |

## 반복 판단 축과 숨은 패턴

| 긴장 | 원문이 기울어지는 쪽 | 보존할 패턴 |
|---|---|---|
| 자동화 vs 통제 | 자동화로 속도를 얻되 배포 시점, 승인, 카나리, 핫픽스는 통제한다. | 빠른 배포는 안전장치 제거가 아니라 판단 지점 외부화다. |
| 개발자 자율성 vs 중앙 관리 | 개발자가 직접 피드백을 받되 공통 인프라와 배포 규칙은 플랫폼이 관리한다. | 편한 개발 경험과 재현성은 같이 설계되어야 한다. |
| 통합 팀 vs 전문 역할 분화 | 현재 규모에서 감당 가능한 구조를 찾는다. | 작은 팀의 통합과 큰 조직의 분화를 고정 정답으로 만들지 않는다. |
| 빠른 수정 vs 일반 플로우 유지 | 서버 직접 수정이나 단계 생략보다 일반 플로우 안의 빠른 조치를 선호한다. | 긴급 대응 뒤에도 이력, 이미지, 인프라, 지표가 어긋나지 않아야 한다. |
| 스테이징 vs 카나리 | 실제 환경에서 검증하고 싶어 하지만 데이터 영향 격리와 롤백 검증을 요구한다. | DevOps의 핵심 불안은 `어디서 꼬일까`이며, 개발 편의와 복구 가능성을 동시에 붙든다. |

## 인접 노드와의 연결
- [[Infra]]: DevOps가 실제로 다루는 기반 자원과 운영 환경의 범위를 넓혀 준다.
- [[Container]]: 컨테이너는 DevOps 흐름에서 재현 가능한 빌드와 배포의 핵심 실행 단위다.
- [[Monitoring]]: 빠른 피드백과 MTTR 개선은 관측성 없이는 성립하지 않는다.
- [[Cloud]]: 인프라 자동화와 자원 관리의 많은 전제가 클라우드 환경과 연결된다.
- [[../Fundamental/Architecture|Architecture]]: 배포 단위, 서비스 경계, 롤백 가능성은 결국 아키텍처 결정과 맞물린다.
- [[../Fundamental/Coding|Coding]]: CI 기준과 테스트 전략은 코드 작성 습관, 컨벤션, 실패 처리와 직접 연결된다.
- [[../Meta/AGENTS|AGENTS]]: 에이전트 기반 운영 자동화가 현실화될수록 DevOps는 파이프라인과 도구 자동화에서 더 넓은 운영 오케스트레이션으로 확장될 수 있다.

## 후속 정리 기준
- CI 메모는 새 도구 이름을 늘리기보다 `품질 게이트`, `리포트`, `보안 검사`, `개발 편의` 중 어느 피드백 표면을 바꾸는지 확인한 뒤 흡수한다.
- CD와 GitOps 메모는 배포 자동화 자체가 아니라 배포 이력, 선언 위치, 롤백, 데이터 변경 관리 중 하나를 더 선명하게 할 때만 본문에 더한다.
- `DevOps vs SRE` 구분은 역할명 비교보다 배포 속도와 신뢰성 운영의 책임 경계가 실제로 달라질 때 별도 비교 프레임으로 다듬는다.
- 조직 사례 링크는 사례 수집으로 늘리지 않고, 코드 관리, 리뷰/승인, 카나리, 장애 복구, 배포 봇처럼 반복 가능한 운영 패턴을 보여 줄 때만 묶는다.
- 작업 이력과 패스별 보강 내역은 본문에 누적하지 않고 `agent-wiki/log.md`로 보낸다.
