# Infra

- 원본: `source/shdkej-content/Deep Knowledge/Infra.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 인프라 카테고리 노드

## 핵심 요약
이 노드는 인프라를 단순한 AWS 서비스 목록으로 보지 않는다. 원문은 인프라를 **서비스 규모에 따라 필요한 구성요소를 조절하고, 클라우드의 편의와 통제력 손실을 같이 감수하며, 서버리스와 서버 기반 방식을 상황에 맞게 섞고, 네트워크·권한·비용·복구 가능성을 함께 판단하는 운영 설계 문제**로 다룬다. 그래서 `정적 페이지 + CDN + S3 + Lambda` 같은 최소 조합에서 시작해, 규모가 커질수록 데이터베이스, 캐시, 오토스케일링, 메시지 큐, VPC, IAM, 관측, 테스트 전략이 붙는다.

## 이 노드가 실제로 답하려는 질문
- 지금 서비스 단계에 필요한 최소 인프라는 어디까지인가?
- 관리 포인트를 줄이기 위해 클라우드와 서버리스를 얼마나 받아들일 것인가?
- 어떤 작업은 Lambda에 맞고, 어떤 작업은 결국 서버나 컨테이너가 더 낫나?
- VPC, NAT, Endpoint, IAM 같은 기반 제약은 비용과 보안에 어떤 영향을 주나?
- 인프라 지식은 서비스 선택 암기보다 어떤 운영 판단으로 축적되어야 하나?

## 핵심 논지
1. **인프라는 처음부터 완전체로 설계하는 것이 아니라, 단계별로 필요한 층을 붙여 가는 구조다.**
   - 원문은 최소 인프라, 안정화를 위한 추가 인프라, 더 탄탄한 인프라를 나눠 적는다.
   - 이는 인프라를 고정 정답보다 성장 단계에 맞춘 증설 문제로 본다는 뜻이다.
2. **클라우드는 관리 포인트를 줄여 주지만, 내부 통제력과 하드웨어 감각 일부를 포기하게 만든다.**
   - 원문은 클라우드의 편리함을 인정하면서도 캐시, CPU, 디스크 최적화 같은 저수준 통제는 줄어든다고 적는다.
   - 즉 편의성은 곧 추상화 수용이며, 그만큼 다른 종류의 판단이 중요해진다.
3. **서버리스는 기본 선호이지만, 지속 연결·낮은 지연·상태 유지가 중요해지면 한계가 선명해진다.**
   - Lambda는 관리 부담이 적고 모듈화에 유리하지만, 콜드스타트와 상태 관리, 비용 급증 문제가 반복된다.
   - 원문은 결국 채팅 같은 실시간 시스템에서는 Fargate, Redis, 별도 데이터 계층이 더 현실적일 수 있다고 본다.
4. **네트워크와 권한은 인프라의 부속 이슈가 아니라 비용, 보안, 접속 가능성 전체를 좌우하는 중심 층이다.**
   - NAT Gateway 비용, VPC Endpoint, Route53, CloudFront, subnet, route table, IAM 권한 지연 등이 길게 남아 있다.
   - 이는 인프라를 컴퓨팅 선택보다 `경계와 연결의 설계`로 이해하고 있음을 보여 준다.
5. **좋은 인프라 메모는 서비스 이름 나열보다 증상, 제약, 수정, 재검증의 운영 기억을 남겨야 한다.**
   - 원문 후반의 EKS, Beanstalk, EC2 OS 패치, RDS 접근, DNS, ingress, CNI 메모는 거의 전부 실제 문제 해결 기록이다.
   - 이 노드의 가치는 이론 요약보다, 어떤 상태를 확인했고 무엇이 병목이었는지에 있다.

## 원문 구조를 다시 읽기

### 1. 규모별 인프라 층위
원문 첫머리는 인프라를 세 단계로 끊어 본다.
- 최소한의 인프라: 정적 페이지, CDN, S3, Lambda, API Gateway
- 규모가 커질 때 추가되는 것: 데이터베이스, 캐시, 오토스케일러
- 더 탄탄한 단계: 메시지 큐

이 구조는 중요하다. 저자는 인프라를 처음부터 거대한 플랫폼으로 상상하지 않고, **서비스 성격과 트래픽 변화에 따라 필요한 구성요소가 달라지는 계단형 구조**로 이해한다.

### 2. 클라우드와 서버리스에 대한 기본 태도
원문은 `모든 것을 위한 클라우드는 없다`고 적는다. 여기서 핵심은 벤더 비교 자체보다 선택 기준이다.
- 비용
- 사용자
- 시간
- 공간
- 회귀 가능한 요소

또한 가능한 한 정적 페이지를 만들고 CDN을 활용하자는 메모가 반복된다. 이는 **상태를 최대한 앞단에서 줄이고, 동적인 부분만 뒤로 밀어 운영 복잡도를 통제하려는 태도**에 가깝다.

### 3. 서버리스 선호와 그 한계 학습
원문은 서버리스를 먼저 고려하는 편이다.
- 관리 포인트가 줄어든다.
- 모듈화와 서비스 연결이 쉽다.
- 가볍고 실행 보장이 좋다.

하지만 동시에 분명한 한계도 적는다.
- 지속 연결이 필요하면 어렵다.
- 빠른 응답이 중요하면 어렵다.
- 콜드스타트가 있다.
- 사용자가 많아질수록 비용이 커질 수 있다.
- 웹소켓 같은 상태 관리가 어렵다.

그래서 이 노드에서 서버리스는 정답이 아니라 **먼저 시도해 볼 수 있는 기본값**이며, 실시간성·상태성·고부하가 커질수록 서버나 컨테이너로 되돌아오는 판단이 중요해진다.

### 4. AWS 기반 지식의 실제 무게중심
원문 중간 이후는 AWS 자격증 요약처럼 보일 수 있지만, 실제로는 서비스 소개보다 운영 포인트가 더 많이 남아 있다.
- EC2 추가 비용 항목: public static IP, bandwidth
- NAT Gateway 비용 폭탄 가능성
- S3, DynamoDB용 VPC Endpoint 활용
- VPC, subnet, route table, IGW, NACL, security group 관계
- IAM 정책과 principal의 구분
- Route53, CloudFront, 캐시 무효화, 지리 제한
- SQS 실패 큐, Lambda 연계

즉 이 노드는 AWS 서비스 백과사전이 아니라, **실제 운영에서 자주 마주치는 경계면과 숨은 비용의 메모 집합**으로 읽는 편이 맞다.

### 5. 핸즈온과 장애 메모의 비중
후반부에는 이론보다 실무 메모가 훨씬 많이 남아 있다.
- Beanstalk를 private VPC에 붙일 때 IGW, NAT, Endpoint를 어떻게 볼지
- EKS subnet, ingress controller, NLB/ALB 전환, external-dns, aws cni 설정
- EC2 OS 패치 중 Docker 재시작 위험과 LB 분리 필요
- RDS private subnet 접근 문제
- IAM 적용 지연
- keyv cronjob, Redis 세션 저장, lifecycle policy, 인증서 캐시 문제

이 부분은 인프라를 `제품 설명`으로 평탄화하면 놓치기 쉽다. 원문은 인프라를 배경지식보다 **실제 운영 중 부딪히는 마찰과 복구 루프의 축적**으로 다루고 있다.

## 원문 발췌 구조
원문을 실제 메모층 기준으로 다시 접으면 대략 아래 순서가 드러난다.

- **최소 인프라 도식**: 정적 페이지, CDN, S3, Lambda, API Gateway에서 출발하는 기본형.
- **클라우드 선택 기준**: 비용, 사용자, 시간, 공간, 통제 가능 범위를 함께 따지는 상위 질문.
- **서버리스 판단 메모**: Lambda 선호, 콜드스타트, 웹소켓/실시간성 한계, Fargate·Redis 같은 대안.
- **AWS 기반기 메모**: VPC, IAM, Route53, CloudFront, Storage, SQS, 비용 최적화.
- **핸즈온/트러블슈팅 기록**: Beanstalk, EKS, EC2 패치, RDS 접근, IAM 지연, 인증서·DNS·캐시 문제.
- **테스트/레퍼런스 층**: stress test, 외부 인프라 사례, 채팅 아키텍처, 모범 사례 링크.

이 배열을 보면 `Infra`는 서비스 카탈로그보다 **선택 기준 → 기본 구조 → 경계 설계 → 문제 해결 → 외부 사례 흡수**의 흐름으로 읽는 편이 더 정확하다.

## 반복되는 판단 축

### 최소 구성 vs 운영 안정성
- 시작은 단순할수록 좋다.
- 하지만 데이터, 캐시, 오토스케일링, 큐는 규모가 커질수록 빠르게 필요해진다.
- 즉 단순함은 영구 상태가 아니라 현재 단계에 맞춘 선택이다.

### 서버리스 vs 지속 실행 서버
- 서버리스는 관리가 쉽고 가볍다.
- 하지만 실시간성, 상태 유지, 비용 예측이 어려워지면 불리하다.
- 이 노드는 둘 중 하나를 신념처럼 택하지 않고, 워크로드 성격으로 가른다.

### 관리 편의 vs 세부 통제력
- 클라우드는 하드웨어 운영 고민을 줄여 준다.
- 반대로 내부 최적화, 자원 배치, 네트워크 경계의 감각을 덜 직접 통제하게 만든다.
- 원문은 이 손실을 불만이라기보다 선택의 대가로 이해한다.

### 기능 구현 vs 숨은 비용
- 인스턴스 요금만 보면 싸 보일 수 있다.
- 하지만 공인 IP, 트래픽, NAT, 로깅, 관리성 비용이 뒤늦게 붙는다.
- 저자는 기능보다 비용 누수 지점을 계속 기록해 둔다.

### 빠른 연결 vs 안전한 경계
- 외부 접근을 열면 테스트는 쉬워진다.
- 하지만 subnet, security group, 인증서, DNS, LB 구조가 엮이면 보안과 운영 위험이 커진다.
- 이 노드는 연결 가능성을 항상 경계 설계와 함께 본다.

## 세부 해석

### 1. 정적 우선 사고방식
`스테이트리스한 것은 정적으로`라는 문장은 이 노드 전체를 관통한다. 가능한 한 정적 파일, CDN, 캐시 계층에서 해결하고, 상태가 필요한 지점만 서버로 보내려는 태도다. 이는 인프라 복잡도를 줄이는 동시에 배포와 확장도 단순하게 만든다.

### 2. 채팅 예시가 드러내는 판단법
채팅 시스템 메모는 서버리스 한계를 가장 선명하게 보여 준다.
- 실시간성
- 웹소켓 상태 관리
- 사용자 증가에 따른 비용 급증
- 콜드스타트

그래서 Fargate, Redis pub/sub, DynamoDB, RDS 조합을 검토한다. 중요한 것은 특정 조합이 아니라, **워크로드 특성 때문에 아키텍처가 바뀌는 판단 과정**이 노출된다는 점이다.

### 3. 네트워크는 거의 모든 운영 기억의 교차점이다
VPC, NAT, Endpoint, Route53, CloudFront, ENI, subnet, NLB/ALB 전환, ingress, external-dns 메모는 서로 흩어져 있지만 결국 같은 질문으로 모인다. `어디를 공개하고 어디를 숨기며, 어떤 경로로 연결하고, 그 대가로 무엇을 내는가?` 이 노드에서 네트워크는 별도 하위 과목이 아니라 인프라 전체의 연결 문법이다.

### 4. IAM 메모는 추상 권한론보다 실제 운영 지연과 구조를 중시한다
IAM 설명은 교과서적 정의도 있지만, 핵심은 최소 권한 원칙, 그룹 기반 관리, resource-based policy와 identity-based policy의 차이, 그리고 `적용에는 시간이 걸린다` 같은 실무 감각이다. 즉 권한은 문서상 구조보다 **실제 배포와 운영에서 바로 체감되는 제약 조건**으로 기록되어 있다.

### 5. 테스트 전략도 인프라의 일부로 본다
후반의 stress test, 벤치마크 링크, locust, redis I/O test, curl loop 메모는 인프라를 구축에서 끝나는 것이 아니라 **어느 정도까지 버티는지 확인하는 실험 시스템**까지 포함해서 본다는 신호다. 이는 [[Monitoring]]과도 직접 이어진다.

## 인프라 운영 사다리
이 노드에서 반복되는 실제 운영 순서는 대략 다음 사다리로 읽힌다.

1. **먼저 정적으로 줄일 수 있는가를 본다.**
   - CDN, S3, 정적 페이지, API Gateway 같은 최소 조합으로 시작한다.
2. **그다음 상태와 실시간성이 정말 필요한지 확인한다.**
   - 서버리스로 충분한지, 지속 연결과 빠른 응답이 필요한지 가른다.
3. **상태가 생기면 경계 설계를 먼저 다룬다.**
   - VPC, subnet, route table, NAT, Endpoint, 보안그룹, IAM이 중심으로 올라온다.
4. **운영이 길어지면 숨은 비용과 복구 절차가 전면으로 나온다.**
   - NAT 비용, public IP, bandwidth, lifecycle policy, 패치 절차, LB 분리, 인증서 캐시 문제가 등장한다.
5. **마지막에는 테스트와 외부 사례로 기준을 다시 갱신한다.**
   - stress test, 당근 사례, 채팅 아키텍처 레퍼런스가 단순 참고가 아니라 다음 선택 기준을 보정하는 재료가 된다.

즉 이 노드는 `무엇을 쓸까`보다 **어떤 단계에서 어떤 문제가 새로 전면에 올라오는가**를 기억하는 운영 사다리다.

## 실무 메모를 상태 소유자별로 읽기
후반의 EKS, Terraform, ArgoCD, LB, DNS, CNI 메모는 길고 이질적이지만, 실제로는 `현재 상태를 누가 소유하고 있는가`를 계속 묻는 기록이다. 이 질문을 세우면 산만한 항목들이 더 잘 묶인다.

| 상태 소유자 | 원문에서 보이는 장면 | 확인해야 할 질문 |
|---|---|---|
| AWS 콘솔/관리형 서비스 | RDS 버전 업데이트, Beanstalk, NAT, 인증서, Route53 | 콘솔에서 바꾼 상태가 코드와 충돌하지 않는가? |
| Terraform | subnet import, route table, 변수/locals/output, DB 삭제 위험 | 코드가 실제 리소스를 소유하는가, 아니면 참고만 하는가? |
| Kubernetes controller | ingress, external-dns, aws cni, lb controller, statefulset | 원하는 상태를 어떤 controller가 계속 복원하는가? |
| GitOps/ArgoCD | 앱 생성, namespace 변경, repository token | Git에 남은 상태와 클러스터 상태가 같은가? |
| 사람의 운영 절차 | EC2 OS 패치, LB 분리, JumpServer 접속, Docker 재시작 | 수동 절차가 끊겨도 복구 순서가 남아 있는가? |

이 관점에서 `Deep Knowledge/Infra`는 [[../../mapped/Fundamental/Infra]]보다 더 실행면에 가깝다. Fundamental 쪽이 어느 인프라 층을 선택할지 묻는다면, 이 노드는 이미 선택된 층들이 서로 다른 소유자를 가질 때 **충돌을 어떻게 발견하고, 어디에서 고치며, 무엇을 다시 검증할지**를 다룬다.

## 운영 변경을 안전하게 다루는 순서
원문 후반의 EC2 OS 패치, Terraform import, RDS 업그레이드, EKS CNI, ingress, CDN 캐시 메모는 서로 다른 사건처럼 보이지만, 실제로는 **변경을 넣기 전후에 어떤 손잡이를 남길 것인가**를 묻는 같은 루프다. 인프라 변경은 `적용`보다 `적용 전 상태 고정`, `영향면 분리`, `적용 후 재검증`이 더 중요하게 반복된다.

1. **현재 상태의 원천을 먼저 확인한다.**
   - 콘솔에서 만들어진 subnet·route table인지, Terraform이 소유하는 리소스인지, Kubernetes controller가 계속 복원하는 상태인지 가른다.
   - 이 확인 없이 코드를 지우거나 콘솔에서 고치면 DB 삭제, ingress 잔존, route table 충돌처럼 원치 않는 동작이 생긴다.
2. **되돌릴 기준점을 남긴다.**
   - EC2 OS 패치 전 AMI/스냅샷을 만들고, 운영 인스턴스는 LB에서 빼고, Docker 재시작 가능성을 따로 본다.
   - RDS나 Redis처럼 상태가 큰 리소스는 Terraform 코드의 삭제 동작과 실제 데이터 손실 위험을 분리해서 본다.
3. **영향 범위를 좁혀서 적용한다.**
   - subnet, security group, NAT, VPC Endpoint, 인증서, DNS, ingress class처럼 연결 경계를 하나씩 확인한다.
   - IAM 적용 지연, CloudFront 캐시, Route53/인증서 전파처럼 `수정했는데 바로 안 보이는 시간`도 변경 계획에 포함한다.
4. **재검증 표면을 명시한다.**
   - `last reboot`, 버전 확인, 컨테이너 상태, Flow Logs, CloudWatch, ALB/NLB 응답, DNS 매핑, Redis 메모리, CNI IP 할당 같은 확인 지점을 남긴다.
   - 재검증이 없으면 변경 성공과 단순 대기, 캐시 잔존, controller 재생성, 권한 전파 지연을 구분하기 어렵다.

이 순서를 세우면 이 노드의 많은 실무 메모가 단순 팁 모음이 아니라, **변경 전 기준점 → 영향면 축소 → 적용 → 재검증 → 다음 운영 기준 갱신**의 반복으로 보인다. 특히 인프라에서는 `무엇을 바꿨는가`보다 `그 변경이 어느 층에 남고, 누가 다시 복원하며, 실패하면 어디까지 되돌릴 수 있는가`가 더 오래 남는 지식이다.

## 변경이 보이는 시간과 실제 적용 시간을 분리하기
원문에는 `IAM 적용에는 시간이 걸린다`, Route53 인증서 캐시, CloudFront 캐시 무효화, EKS controller 재생성, ArgoCD namespace 변경 뒤 잔존 리소스처럼 **수정 명령과 체감 결과 사이가 벌어지는 장면**이 반복된다. 이 간격을 단순 대기 시간으로만 보면 같은 문제를 여러 번 건드리게 된다.

인프라 변경을 읽을 때는 세 시간을 나눠야 한다.

1. **선언 시간**: Terraform, 콘솔, kubectl, ArgoCD, AWS API에서 변경을 요청한 시점.
2. **전파 시간**: IAM, DNS, 인증서, 캐시, controller, CNI처럼 여러 계층이 새 상태를 받아들이는 시간.
3. **검증 시간**: 사용자가 보는 응답, 로그, 대시보드, Flow Logs, `last reboot`, pod 상태처럼 실제 결과가 확인되는 시간.

이 구분은 [[Monitoring]]과도 직접 이어진다. 모니터링이 없다면 전파 중인지, 캐시에 남은 것인지, controller가 되돌린 것인지, 실제 장애인지 구분하기 어렵다. 또한 [[Container]]의 헬스체크·readiness·rollout 상태와 [[Devops]]의 배포/롤백 루프는 모두 `선언은 끝났지만 운영 상태가 아직 끝나지 않은 시간`을 다루는 장치다.

따라서 이 노드의 실무 메모는 `명령어를 무엇으로 쳤는가`보다 **변경이 어느 표면에 선언되고, 어느 계층에서 늦게 반영되며, 어떤 관측 지점에서 완료로 인정할 것인가**를 남기는 쪽으로 유지해야 한다.

## 앞단 경계는 사용자에게 보이는 계약이다
원문에서 CloudFront, Route53, ALB/NLB, 인증서, CDN 캐시, proxy protocol, `x-forwarded-for`, CSP, 지리적 제한, signed URL/쿠키 메모는 모두 서비스 앞단에 놓인다. 겉으로는 네트워크·보안 설정 조각처럼 보이지만, 실제로는 사용자가 어떤 주소로 들어오고, 어느 지역에서 접근할 수 있고, 누구의 IP로 기록되며, 어떤 콘텐츠가 캐시된 상태로 보이는지를 정하는 **사용자 가시 계약**에 가깝다.

| 앞단 장치 | 원문에서 보이는 메모 | 운영 질문 |
|---|---|---|
| Route53·인증서 | A/CNAME 선택, 인증서 캐시, 400 에러 | 사용자가 보는 주소와 인증 상태가 실제 배포 상태를 정확히 가리키는가? |
| CloudFront·CDN | 캐시 무효화, query/header/cookie 캐시 정책, price class | 어떤 사용자에게 어떤 버전의 콘텐츠가 보이고, 언제 새 상태로 인정할 것인가? |
| ALB/NLB·ingress | NLB에서 ALB를 바라보는 문제, ingress class, 리다이렉트 | 공개 진입점과 내부 서비스 경계가 분리되어도 추적 가능한가? |
| proxy protocol·X-Forwarded-For | 클라이언트 IP 전달, security group 조건 | 실제 사용자의 출처를 보존하면서도 접근 제어가 깨지지 않는가? |
| CSP·signed URL | 콘텐츠 제한, private origin, signed URL/쿠키 | 앞단에서 무엇을 허용하고 무엇을 막는지 정책이 설명 가능한가? |

그래서 이 노드에서 앞단 경계 메모를 다룰 때는 `로드밸런서를 무엇으로 바꿨다`보다, 그 변경 뒤에 **사용자가 보는 주소, 캐시된 상태, 인증서, 원본 IP, 접근 정책이 일관되게 맞는지**를 확인하는 쪽이 더 오래 남는 지식이다. 특히 CloudFront와 DNS는 수정 후 바로 체감되지 않을 수 있으므로, 앞의 `선언 시간 / 전파 시간 / 검증 시간` 구분과 함께 읽어야 한다.

이 축은 [[../../mapped/Fundamental/Network]]의 네트워크 기본기와도 이어지지만, 여기서는 프로토콜 설명보다 운영 계약에 초점을 둔다. 사용자는 내부 서비스 구조를 보지 못한다. 사용자가 보는 것은 주소, 응답, 인증서, 속도, 차단 여부, 캐시된 화면이다. 따라서 앞단 인프라는 내부 배치도가 아니라 **서비스가 외부에 약속한 얼굴을 유지하는 층**으로 읽어야 한다.

## 숨은 패턴
- 이 노드는 AWS 서비스 공부 노트처럼 보이지만 실제로는 **운영 판단 로그**에 더 가깝다.
- 저자는 인프라를 화려한 아키텍처보다, 비용 누수와 연결 경계, 권한 구조, 복구 가능성을 다루는 현실적인 조정 작업으로 본다.
- 서버리스 선호가 분명하지만, 현실 제약이 드러나면 기꺼이 서버·컨테이너·캐시·큐로 이동한다.
- 실무 메모가 많은 이유는 이 노드의 가치가 완결된 정답보다 `다음 비슷한 문제를 더 빨리 풀 수 있는 운영 기억`에 있기 때문이다.

## 인접 노드와의 연결
- [[Cloud]]: 클라우드 추상화의 장단점을 더 상위 선택 기준으로 다룬다.
- [[Devops]]: 배포, 롤백, CI/CD, 운영 복원력은 인프라 선택과 바로 이어진다.
- [[Monitoring]]: 인프라 계층의 정상 상태와 이상 징후를 어떻게 드러낼지 연결된다.
- [[Container]]: 서버리스 한계를 넘는 순간 컨테이너, ECS, EKS, 이미지 운영 문제가 같이 붙는다.
- [[../../mapped/Fundamental/Network]]: VPC, Route53, LB, 보안그룹, 서브넷 이해의 기초 층을 받쳐 준다.
- [[../../mapped/Fundamental/Architecture]]: 단계적 확장, 저결합, 복구 가능성은 결국 아키텍처 판단과 겹친다.
- [[../../mapped/Meta/Troubleshooting]]: 실제 핸즈온 메모 대부분은 증상 확인, 수정, 재검증 루프로 읽히며 트러블슈팅 노드와 강하게 맞닿는다.

## 이 노드를 읽을 때 놓치지 말아야 할 점
- 서비스 이름 암기보다 어떤 조건에서 어떤 선택이 깨지는지 읽어야 한다.
- AWS 자격증 범위 메모도 시험 대비보다 운영 지형을 넓게 잡아 두려는 흔적으로 보는 편이 맞다.
- 미완 질문과 TODO가 많다는 점은 약점이 아니라, 실제 인프라 학습이 계속 열린 상태였다는 증거다.
- 특히 후반의 실무 메모는 `잡다함`이 아니라, 인프라 지식이 실제로는 증상 확인·수정·재검증의 루프로 축적된다는 사실을 보여 준다.
- 따라서 이 페이지는 깔끔한 인프라 백과사전보다, 실무에서 반복 재사용할 판단 축과 운영 기억을 보존하는 노드로 유지하는 것이 적절하다.

## 이후 심화 포인트
- 서버리스 판단 기준을 `적합한 작업`, `한계가 선명한 작업`, `대체 조합`으로 더 세분화할 수 있다.
- AWS 실무 메모를 `네트워크`, `권한`, `배포/런타임`, `데이터`, `비용` 축으로 재분류하면 탐색성이 더 좋아진다.
- 후반의 EKS·LB·CNI 메모는 별도 하위 운영 패턴으로 압축해도 가치가 있다.
- 당근 인프라/채팅 사례처럼 외부 레퍼런스가 이미 붙어 있는 부분은 추후 `운영 레퍼런스에서 읽은 기준` 섹션으로 정리할 수 있다.

## 이번 갱신 메모
- 이번 런에서는 원본 전체와 기존 mapped 전체를 다시 읽고, 기존 baseline을 유지한 채 `앞단 경계는 사용자에게 보이는 계약이다`를 추가했다. CloudFront, Route53, ALB/NLB, 인증서, CDN 캐시, proxy protocol, CSP 메모를 사용자에게 보이는 주소·캐시·인증·원본 IP·접근 정책의 일관성 문제로 묶었다.
- 이전 갱신에서는 `변경이 보이는 시간과 실제 적용 시간을 분리하기`를 추가했다. IAM 적용 지연, DNS·인증서·CloudFront 캐시, ArgoCD/Kubernetes controller, CNI 같은 메모를 선언 시간·전파 시간·검증 시간의 차이로 묶었다.
- 이전 갱신에서는 `운영 변경을 안전하게 다루는 순서`와 `실무 메모를 상태 소유자별로 읽기`를 보강했다. EC2 OS 패치, Terraform import, RDS/EKS/CNI, DNS·인증서·캐시 메모를 변경 전 기준점 확보 → 영향면 축소 → 적용 → 재검증 → 다음 기준 갱신의 루프로 읽고, AWS 콘솔/관리형 서비스, Terraform, Kubernetes controller, GitOps/ArgoCD, 사람의 운영 절차가 각각 어떤 상태를 소유하는지 분리했다.
- [[../../mapped/Fundamental/Infra]]와의 역할 차이를 유지해, 이 노드가 인프라 선택 기준보다 운영 중인 리소스의 소유자·수정 지점·재검증 경로를 다루는 실행면이라는 점을 선명하게 했다.
