# Open_Source

- 원본: `source/shdkej-content/Communication/Open_Source.md`
- mapped: `agent-wiki/content/docs/mapped/Communication/Open_Source.mdx`
- 역할: 원본 만다라트의 `Communication/Open_Source` 노드와 1:1 대응하는 관리 노트
- 상태: 안정 디벨롭 · 공개 협업 인터페이스 정리

## 핵심 논지

이 노트에서 오픈소스는 `코드를 공개한다`는 이념보다, 낯선 사람이 프로젝트에 들어오고 다시 나갈 수 있게 만드는 공개 협업 인터페이스다. 원문은 기여 전 코드베이스 읽기, 라이선스와 공개 범위, 공개의 장단점, 버저닝, 발표와 레퍼런스를 함께 놓는다. 따라서 중심 질문은 `무엇을 열 것인가`보다 **처음 보는 사람이 무엇을 믿고, 어디까지 이해하고, 어떤 단위로 기여하거나 사용할 수 있는가**에 있다.

## 만다라트 위치

`Communication/Open_Source`는 라이선스 사전이나 무료 도구 목록이 아니라 Communication 카테고리 안의 공개 협업 노드다. [Communication/Document](/docs/mapped/Communication/Document)가 문서군의 입구와 복구면을, [Communication/Teamwork](/docs/mapped/Communication/Teamwork)가 내부 협업의 handoff 구조를, [Communication/Feedback](/docs/mapped/Communication/Feedback)이 반복 질문과 반응을 조정하는 회로를 맡는다면, 이 노드는 그 구조가 회사 바깥의 낯선 기여자와 사용자에게 열릴 때 필요한 표면을 맡는다.

그래서 이 페이지의 경계는 명확하다. 오픈소스 철학 일반론으로 넓히지 않고, 원문에 있는 `기여 진입`, `공개 경계`, `릴리즈 신뢰`, `설명 재진입` 네 축 안에서만 다룬다.

## 원문에서 유지한 구조

- 오픈소스 기여하기: 작은 부분에서 시작, 흐름 파악, 브레이크포인트와 호출 스택, 구조 그리기, 작성자 습관 읽기, 레퍼런스 확인
- Open-source: 모두가 기여할 수 있지만 라이선스가 있고, 거의 free일 수 있어도 모든 것이 free는 아님
- License: MIT, Apache, GPL, LGPL, AGPL, 사용 명시 요구
- 공개의 단점과 장점: 핵심기술 노출, 취약점 노출, 사용자 증가, 예상 밖의 확장
- 버저닝: `1.2.x`, patch 호환성, 실제 기능 동작 단위의 버전 상승
- 추천 사이트와 Reference: opensourcestash, free-for.dev, 라이선스 분쟁, 릴리즈 팁, 구조 파악, 컨트리뷰터 찾기, 개발기
- 발표: 40분 구조화, 10분마다 포인트, 공감형 강의, 초 단위보다 분 단위 리듬

## 1. 코드베이스 진입

원문 첫머리에서 가장 구체적인 부분은 라이선스가 아니라 `오픈소스 기여하기`다. 작은 부분에서 시작하고, 먼저 흐름을 파악하고, 디버그와 브레이크포인트로 호출 스택을 확인하고, 필요한 부분부터 보며, 구조를 그려 보라는 조언이 이어진다. 이 노트의 출발점은 공개 철학보다 낯선 시스템에 안전하게 들어가는 읽기 기술이다.

좋은 기여자는 새 코드를 바로 밀어 넣기보다 프로젝트의 기존 질서를 먼저 읽는다. 원문은 작성자의 if문 사용 방식, 함수 분해 방식, 호출 방식, 테스트 방식을 확인하라고 적는다. 이는 코드 스타일 취향이 아니라 `이 프로젝트는 어떤 방식으로 판단하고 변경을 받아들이는가`를 읽는 과정이다.

첫 기여를 작은 패키지로 만들 때 특히 중요한 조건은 네 가지다.

- **범위**: 고치려는 파일과 행동 변화가 좁아야 한다.
- **문법**: 함수 분해, 조건문, 테스트 방식이 기존 프로젝트 습관과 맞아야 한다.
- **검증**: 호출 스택, 브레이크포인트, 테스트, 재현 절차 중 무엇으로 확인했는지 보여야 한다.
- **선례**: 비슷한 코드나 레퍼런스를 확인해 `왜 이 방식인가`를 프로젝트 안의 언어로 말할 수 있어야 한다.

이 관점에서 오픈소스 기여는 `내가 얼마나 많이 만들 수 있는가`보다, 메인테이너가 내 변경을 얼마나 적은 비용으로 신뢰할 수 있는가의 문제에 가깝다.

## 2. 공개 범위와 사업 판단

원문은 MIT, Apache, GPL, LGPL, AGPL을 짧게 나열하지만, 핵심은 이름 암기가 아니다. 오픈소스는 아무 조건 없는 무료 공간이 아니라 공개 범위와 공유 의무가 정해진 협업 공간이다.

| 원문 단서 | 이 노드에서의 의미 |
| --- | --- |
| MIT, Apache | 사용과 확산의 마찰을 낮추는 선택 |
| GPL | 사용한 프로그램 전체 공개를 요구할 수 있는 강한 환류 조건 |
| LGPL | 개선한 소스에 대한 공유 요청 |
| AGPL | 서버에서 사용하더라도 공유 책임을 묻는 조건 |
| 사용 명시 요구 | 가져다 쓰는 쪽의 설명 책임 |

따라서 라이선스는 법률 부록이 아니라 공개 협업의 말투다. 많이 쓰이게 하고 싶은지, 개선분이 다시 돌아오게 하고 싶은지, 서버 사용까지 공개 책임을 묻고 싶은지에 따라 프로젝트가 외부와 맺는 관계가 달라진다.

원문의 `핵심기술은 비공개로 하고 API만 개방하면 괜찮을까`, `B2B에서는 개방화를 할 필요가 없을까`라는 질문도 중요하다. 이는 오픈소스를 순수한 선의나 홍보 수단으로만 보지 않는다는 신호다. 공개 전략은 저장소 visibility 문제가 아니라 사용자, 기여자, 고객, 경쟁자가 각각 어떤 표면을 보고 무엇을 믿게 할지 정하는 설계다.

공개 범위는 세 층으로 나눠 볼 수 있다.

- **학습과 기여의 표면**: 구조, 이슈, 테스트, 릴리즈 흐름은 충분히 열려야 낯선 사람이 작은 기여를 준비할 수 있다.
- **제품 신뢰의 표면**: 버전, 호환성, 라이선스, 보안 대응이 보여야 사용자가 도입 위험을 계산할 수 있다.
- **사업 차별화의 표면**: 핵심 알고리즘, 운영 데이터, 내부 자동화는 API나 문서화된 계약으로만 노출하는 선택도 가능하다.

오픈소스는 소유권 포기가 아니라, 진입 가능한 외부 표면과 유지해야 할 내부 판단권을 동시에 설계하는 방식이다.

## 3. 릴리즈 신뢰

원문 후반의 버저닝 메모는 이 노트의 운영 감각을 잘 보여 준다. patch 버전은 하위호환을 유지한 채 오류 수정을 전달해야 하고, 사용자는 `1.2.x`처럼 안전하게 받을 수 있어야 한다. 또한 버전은 커밋하거나 PR했다고 오르는 것이 아니라 실제 기능이 동작할 때 오르는 편이 납득된다고 적는다.

여기서 버전은 개발자 내부 히스토리가 아니라 외부 사용자를 위한 신뢰 신호다. 사용자는 `이 변경을 받아도 되는가`, `업그레이드하면 깨지는가`, `버그 픽스만 받을 수 있는가`를 버전과 릴리즈 노트로 판단한다. 그래서 [Portfolio](/docs/mapped/Portfolio)와 이 노드는 직접 이어진다. 변경 기록은 내부 작업표가 아니라 외부 협업자가 위험을 계산하는 표면이다.

릴리즈 신뢰는 다음 순서로 닫혀야 한다.

- **발견**: 문제나 개선점이 개인 대화가 아니라 공개 이슈, 레퍼런스, 재현 조건으로 나온다.
- **제안**: PR은 코드 묶음만이 아니라 의도, 선례, 검증 경로, 위험 범위를 함께 전달한다.
- **수용**: 메인테이너는 기존 질서, 공개 경계, 라이선스, 사업 판단과 충돌하지 않는지 확인한다.
- **배포**: 버전과 릴리즈 노트는 사용자가 무엇을 안전하게 받아도 되는지 알려 준다.
- **재설명**: README, 발표, 추천 링크, 릴리즈 팁은 다음 사람이 같은 길을 덜 헤매게 만든다.

즉 좋은 오픈소스 변경은 `좋은 PR 하나`로 끝나지 않고, 발견·검토·배포·설명까지 이어지며 다음 기여자의 길잡이로 남는다.

## 4. 설명 재진입

원문 마지막의 발표 메모는 주제 밖의 발표 기술처럼 보이지만, 실제로는 오픈소스 운영의 일부다. 40분 발표라면 전체를 구조화하고 10분마다 흥미 포인트를 배치할 수 있다는 메모, 계몽주의식 일방 강의보다 공감이 이어지는 형식이 좋다는 메모, conference에서는 초 단위보다 분 단위가 중요하다는 메모가 이어진다.

이는 README, 릴리즈 노트, 발표가 모두 같은 온보딩 표면이라는 감각이다. 공개 프로젝트를 설명할 때 모든 초를 통제하려 하기보다, 낯선 사람이 목적·위험·사용 장면을 자기 언어로 다시 붙잡을 수 있게 큰 구조와 반복 포인트를 배치해야 한다.

원문의 추천 사이트와 Reference도 단순 북마크가 아니다. `opensourcestash`, `free-for.dev`, 라이선스 분쟁 글, 릴리즈 팁, 구조 파악 글, 컨트리뷰터 찾기 자료는 오픈소스 생태계에서 들어갈 곳과 조심할 곳을 찾는 탐색 인터페이스다. 어떤 프로젝트가 활발한지, 어떤 라이선스가 기업 사용과 충돌하는지, 어떤 문서가 초심자의 진입을 돕는지, 어떤 릴리즈 문법이 사용자 신뢰를 만드는지를 비교하게 한다.

## 핵심 판단 축

| 축 | 판단 |
| --- | --- |
| 진입 속도 vs 구조 이해 | 빠른 수정 전에 흐름, 호출 스택, 작성자 습관을 먼저 읽어야 좋은 기여가 된다. |
| 자유 개방 vs 조건부 개방 | 오픈소스는 무조건 무료가 아니라 라이선스로 경계가 정해진 협업 공간이다. |
| 외연 확장 vs 차별화 보호 | 공개는 사용자 증가와 외부 기여를 부르지만 핵심 메리트와 취약점도 노출할 수 있다. |
| 내부 커밋 흐름 vs 사용자 체감 릴리즈 | 버전은 개발 히스토리보다 사용자가 안전하게 받을 수 있는 변경 단위로 설계되어야 한다. |
| 코드 작성 vs 공공 설명 책임 | 구현뿐 아니라 README, 릴리즈 노트, 발표, 레퍼런스가 함께 있어야 공개 자산이 된다. |
| 기여자 준비 vs 프로젝트 응답성 | 기여자는 먼저 읽어야 하고, 프로젝트도 구조·검증·선례·응답 기준을 추적 가능하게 남겨야 한다. |

## 반복 패턴

- 오픈소스는 학습용 거울이다. 낯선 구조를 읽고, 설계 습관을 추적하고, 레퍼런스를 비교하는 과정 자체가 개발자 성장 방식으로 남는다.
- 기여는 먼저 맞춰 보는 행동이다. `나라면 어떻게 할지`를 생각하되, 기존 프로젝트 질서와 비교해 들어가야 한다.
- 운영 규칙은 신뢰 비용을 줄인다. 라이선스, 버저닝, 릴리즈 규칙은 자유를 제한하려는 장치가 아니라 협업 상대가 안심하고 쓸 수 있게 하는 약속이다.
- 질문은 작은 문서 업데이트의 씨앗이다. 반복될 질문은 이슈, FAQ, README, 릴리즈 노트, 발표 자료 중 하나로 옮겨져 다음 사람의 시작점을 낮춰야 한다.
- 프로젝트의 친절함은 빠른 답변보다 추적 가능한 구조에서 나온다. 디렉터리, 테스트, 과거 결정, 릴리즈 기준을 직접 따라갈 수 있어야 커뮤니티가 한 사람의 기억에만 기대지 않는다.

## 인접 노드 연결

- [Communication/Document](/docs/mapped/Communication/Document): README, 릴리즈 노트, 발표, Reference를 온보딩과 재진입 표면으로 다루는 점에서 직접 연결된다.
- [Communication/Teamwork](/docs/mapped/Communication/Teamwork): 내부 협업의 handoff가 공개 이슈·PR·릴리즈로 확장될 때 이 노드가 된다.
- [Communication/Feedback](/docs/mapped/Communication/Feedback): 반복 질문을 개인 응답으로 소모하지 않고 공개 기록과 문서 씨앗으로 돌리는 감각이 이어진다.
- [Fundamental/Coding](/docs/mapped/Fundamental/Coding): 호출 구조, 테스트, 변경 단위, 기존 코드 문법을 읽는 태도와 연결된다.
- [Integration/Business](/docs/mapped/Integration/Business): 핵심 기술 비공개, API 공개, B2B/B2C 공개 범위 판단과 이어진다.
- [Integration/Tool](/docs/mapped/Integration/Tool): 무료·오픈소스 도구를 기능 목록이 아니라 유지 방식과 책임 경계까지 포함해 보는 선택 표면으로 연결된다.
- [Portfolio](/docs/mapped/Portfolio): 버전과 릴리즈를 외부 협업 신호로 다룬다는 점에서 연결된다.

## 선호 신호

저자는 오픈소스를 선행 문화나 유명 프로젝트 참여 욕망보다, 낯선 구조를 읽는 법을 배우고 협업 규칙을 이해하며 공개 범위와 릴리즈 약속을 판단하는 실전 개발 훈련장으로 보는 경향이 강하다. 특히 무턱대고 크게 기여하기보다 작은 진입, 구조 파악, 작성자 습관 이해, 레퍼런스 비교처럼 기존 운영자의 시간과 질서를 존중하는 접근을 선호한다.

## 후속 정리 기준

- 새 오픈소스 메모가 들어오면 먼저 `코드베이스 진입 / 공개 범위와 사업 판단 / 릴리즈 신뢰 / 설명 재진입` 중 어느 축에 붙는지 확인한다.
- 현재 원본 대응은 이 문서 상단의 `원본`/`mapped` 메타 정보와 `maintenance/source-mapping`의 개별 행으로 함께 고정하고, 본문 보강은 새 원문 증거가 있을 때만 한다.
- 라이선스 자료는 법률 사전으로 확장하지 말고, 어떤 공개 의무와 채택 마찰을 만드는지 중심으로만 흡수한다.
- 도구 추천 링크는 무료 도구 목록으로 늘리지 말고, 유지 책임·도입 리스크·커뮤니티 응답성을 판독하는 표본으로만 쓴다.
- 발표 메모는 발표술 일반론으로 보내지 말고, 공개 프로젝트를 낯선 사람이 다시 이해하게 만드는 재진입 표면으로 유지한다.

## 드리프트 점검

대응 원본 `source/shdkej-content/Communication/Open_Source.md`의 최근 git 이력은 초기 동기화 시점에 머물러 있어 이번 실행의 직접 사용자 수정본으로 보이지 않았다. 따라서 `human-reviews/` 복사는 생략했다. 외부 링크 소스는 이번 압축과 직접 관련성이 낮아 사용하지 않았다.
