# Teamwork

- 원본: `source/shdkej-content/Communication/Teamwork.md`
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트. `Communication/Teamwork.md` 바깥의 협업 일반론이나 source-mapping 보조 문서로 주제를 넘기지 않는다.
- 상태: 안정 디벨롭 · 협업 상태 전달/재진입 표면 정렬

## 핵심 논지

이 노트에서 팀워크는 친화력이나 분위기 관리가 아니다. 서비스 구조, 조직 배치, 리더의 완충, 업무 할당, 문서화, 코드 리뷰, 온보딩, 비동기 소통이 함께 만드는 **협업 운영 시스템**이다. 핵심은 모두가 모든 맥락을 아는 이상이 아니라, 맥락이 끊겨도 다시 들어올 수 있는 경로를 남기는 것이다.

원문 전체를 네 축으로 압축하면 다음과 같다.

| 축 | 원문에서 보이는 장면 | 남길 판단 |
| --- | --- | --- |
| 리더 완충 | 맥락 반복 전달, 우선순위 변경 흡수, 팀원 성장 버퍼, 안전감 | 리더는 답을 독점하기보다 외부 혼선을 걸러 팀원이 판단 가능한 상태로 일하게 한다 |
| 조직 handoff | 기능 조직, 목적 조직, TF, 피자 2판 팀, basecamp | 좋은 구조의 기준은 이름이 아니라 아이디어·우선순위·책임이 덜 새는 전달 비용이다 |
| 상태 패키지 | 왜/기대 전달, 진행상황 불안, `OK/할게`의 한계, 명시적 호출 | 협업 메시지는 확인보다 시간·소유자·위치·대안이 보이는 작은 맥락 패키지여야 한다 |
| 재진입 기록 | 문서, 핸드북, 코드리뷰, changelog, GitHub workflow, 온보딩 빈칸 | 질문과 실수는 개인 대화에서 사라지지 않고 다음 사람이 찾을 수 있는 표면으로 환류되어야 한다 |

## 원문 흐름 요약

원문은 먼저 리더가 맥락을 반복 전달하고 우선순위 변경을 완충해야 한다는 문제의식에서 출발한다. 이어 기능 조직, 목적 조직, TF, 피자 2판 팀, crew/basecamp 사례를 통해 서비스 구조와 팀 구조가 서로 닮아 가는 장면을 넓게 비교한다. 중반 이후에는 진행상황 공유 불안, 바텀업/탑다운 긴장, PL 역할 고민, 감독·PM·팀장 비유가 이어지며 중심점과 자율의 균형을 묻는다. 후반의 커밋 규칙, changelog, 코드 리뷰, API 문서, GitLab식 비동기 소통, 핸드북 환류, GitHub workflow는 협업이 사람의 기억에만 머물지 않고 다음 사람이 다시 들어올 수 있는 기록 표면으로 남아야 한다는 결론으로 모인다.

## 1. 리더 완충

원문에서 리더는 일을 지시하는 사람보다 **맥락을 반복해서 전달하고 외부 혼선을 완충하는 사람**에 가깝다. `말하고자 하는 바가 백프로 전달되는 경우는 잘 없다`는 전제가 있으므로, 좋은 리더십은 한 번의 설명으로 끝나지 않는다. 같은 방향과 배경을 여러 번 전달하고, 팀원이 다시 물어볼 수 있게 해야 한다.

리더가 고려해야 하는 저울도 단순하지 않다. 고객 만족, 기간, 시간, 리소스, 비용이 있고, 동시에 팀원이 이 일을 통해 어떤 성장을 할 수 있는지도 봐야 한다. 원문은 이 저울이 약간 팀원의 좋은 결과물과 성장 쪽으로 기울어 있다고 적는다. 즉 업무 할당은 생산량 배분이 아니라 **성장 가능성과 납기 위험을 함께 조정하는 설계**다.

리더는 우선순위 변경과 타 부서 협의도 그대로 팀원에게 흘려보내지 않는다. 외부 상황을 적절히 잘라 주고, 팀원은 자신의 작업과 성장에 집중하게 만드는 역할이 반복해서 나온다. 이때 리더는 모든 답을 직접 주는 사람이 아니라, `무엇을 만들자`보다 `어디로 가야 하는가`를 알려 주는 방향 번역자다.

### 리더가 남겨야 하는 것

- 목적과 배경: 왜 이 일을 하는지, 고객과 팀에는 어떤 의미인지
- 판단 경계: 팀원이 직접 결정해도 되는 범위와 반드시 확인해야 하는 범위
- 확인 표면: 기능이 잘 됐는지 말로만 듣지 않고 무엇을 같이 볼 것인지
- 성장 여지: 이 일이 팀원의 다음 능력과 어떻게 이어지는지
- 안전감: 실패와 질문이 숨겨지지 않는 피드백 방식

공개된 자리에서 바로 면박을 주지 않고, 잘한 일을 명시적으로 말해 주고, 실패를 공유 가능한 학습으로 바꾸려는 메모는 모두 이 축에 속한다. 팀워크의 반대말은 갈등 자체가 아니라, 사람들이 질문과 실패를 숨기게 되는 상태다.

원문 후반의 PL 고민과 `촉매자` 메모는 이 리더상을 더 좁힌다. 관리자는 상대 대신 문제를 풀어 주는 사람이 아니라, 조언을 구한 사람이 자기 힘으로 문제를 다시 정의하고 탐구하게 돕는 사람이다. 그래서 면담 말미의 `뭐 필요한 거 없어요?` 같은 질문은 단순한 친절이 아니라, 막힌 지점·필요 자원·다음 실험을 표면으로 끌어내는 작은 트리거다. 판을 깔아 주고 시작 트리거를 만들어 주되, 실행 판단을 모두 빼앗지 않는 것이 이 노트의 리더십 감각에 가깝다.

## 2. 조직 Handoff

원문 첫머리의 `서비스의 구조와 팀의 구조가 닮는다`는 문장은 이 노트 전체의 중심이다. 기능별 서비스는 기능별 팀을 부르고, 고객 문제 중심의 일은 직군을 섞은 목적 조직을 부른다. 그래서 팀워크는 사람 성향만의 문제가 아니라, 시스템을 어떻게 쪼개고 다시 연결할지의 문제다.

원문은 기능 조직, 목적 조직, TF, 피자 2판 팀을 하나의 정답으로 고르지 않는다. 대신 각 구조가 어떤 handoff 비용을 만드는지 묻는다.

| 구조 | 얻는 것 | 잃기 쉬운 것 |
| --- | --- | --- |
| 기능 조직 | 서버, 웹, 디자인처럼 전문성 깊이를 쌓기 쉽다 | 다른 기능의 아이디어를 실행하기 어렵다 |
| 목적 조직 | 광고, 커리어처럼 고객 문제 앞에서 빠르게 뭉친다 | 기반 전문성이나 장기 유지 책임이 흐려질 수 있다 |
| TF/타이거팀 | 긴급 문제에 빠르게 반응한다 | 겉만 파고 끝나는 임시 조직이 될 수 있다 |
| 피자 2판 팀 | 작은 단위의 자율성과 집중을 높인다 | 팀 사이 전달 인터페이스가 약하면 중복과 단절이 커진다 |

따라서 조직 선택의 기준은 `기능별이 좋은가 목적별이 좋은가`보다, 아이디어·우선순위·문서·공통 라이브러리가 다른 팀으로 얼마나 덜 손상되고 넘어가는가다. 원문의 리멤버 사례처럼 기능조직과 목적조직, crew와 basecamp가 함께 등장하는 이유도 이 때문이다. 깊이와 속도를 둘 다 얻으려면 한 구조만으로는 부족하고, 서로 다른 구조 사이의 전달면을 설계해야 한다.

리멤버 사례에서 원문이 남긴 직접 단서는 `테크 스택`, `코드 리뷰`, `테스트 코드`, `오버 커뮤니케이션`, `문서화`, 그리고 기능조직/목적조직의 병행이다. 여기서 `crew`는 고객 문제를 향해 움직이는 목적 조직이고, `basecamp`는 서버·웹·디자인 같은 기능 전문성을 유지하며 여러 crew를 받치는 기반 조직으로 읽힌다. 이 조합은 조직을 한 번 잘 나누면 끝난다는 뜻이 아니라, 목적 조직이 빠르게 달릴 때도 코드 품질, 테스트, 리뷰, 공통 스택, 문서화 같은 장기 기준을 잃지 않게 하는 보정 장치다.

후반의 바텀업/탑다운 메모도 같은 질문을 다룬다. 전체를 보는 중심점은 필요하지만, 고객 이해와 현장 아이디어는 구성원에게서 올라온다. 지나친 중앙통제는 실패하고, 중심 없는 자율도 흩어진다. 이 노트가 찾는 것은 강한 통제나 완전 자율이 아니라 **중심점은 두되 현장의 신호를 죽이지 않는 배치**다.

## 3. 상태 패키지

원문에는 `하려는 것과 왜 하려는지와 기대하는 것을 같이 전달해야 서로 오해가 줄어든다`는 문장과, `진행상황은 보고 하고 싶은데 진행상황 정리가 잘 안됨`이라는 불안이 함께 있다. GitLab식 소통 팁의 `OK`, `할게`, `나중에 할게` 비판도 같은 문제를 가리킨다. 팀워크에서 중요한 것은 말을 많이 하는 것이 아니라, 다음 사람이 판단할 수 있을 만큼의 맥락을 작게 포장해 넘기는 일이다.

### 상태 패키지의 최소 구성

| 장면 | 꼭 들어가야 하는 정보 | 줄이는 비용 |
| --- | --- | --- |
| 업무 할당 | 목적, 판단 경계, 확인 표면, 공유 형식 | 일을 받은 사람이 무엇을 임의로 판단해도 되는지 모르는 비용 |
| 진행 공유 | 현재 위치, 막힌 지점, 다음 행동, 기한 영향 | 리더가 일일이 캐묻거나 팀원이 보고를 미루는 비용 |
| 응답/호출 | 시간, 소유자, 위치, 대안 | 상대가 `읽었는지`, `누가 할지`, `어디서 이어갈지` 추측하는 비용 |

이 관점에서 좋은 비동기 소통은 느린 소통이 아니다. 상대가 기다리는 동안에도 일을 잃어버리지 않게 하는 상태 전달 규칙이다. 질문은 많이 해도 되지만, 1:1 채팅에서 사라지기보다 이슈나 공개 채널에 남아야 한다. 참조와 요청도 분리되어야 한다. 단순히 누군가를 멘션한 것과, 어떤 행동이 필요한지 명시한 것은 다르다.

커뮤니케이션 불안은 개인 성격 문제로만 읽기 어렵다. `너무 바쁜 리더`, `일일이 물어보기 죄송`, `진행상황 정리가 잘 안됨`은 질문 경로와 공유 형식이 충분히 마련되지 않았다는 설계 신호다. 좋은 팀워크는 더 용감한 사람을 요구하기보다, 덜 미안하게 질문하고 더 짧게 상태를 공유할 수 있는 구조를 만든다.

## 4. 재진입 기록

후반의 커밋 규칙, changelog, 코드 리뷰, API 문서, 핸드북, GitHub workflow 메모는 기술 잡항목이 아니라 팀워크의 기억 형식이다. 사람이 바뀌어도 같은 방식으로 읽고, 만들고, 검토하고, 배포할 수 있어야 협업 비용이 줄어든다.

원문의 기술 협업 흐름은 다음처럼 묶인다.

| 단계 | 원문 메모 | 팀워크 관점 |
| --- | --- | --- |
| 이해 | 그림 그리기, 레퍼런스 확인, 코드 구조 파악 | 일을 시작하기 전에 같은 문제를 보고 있는지 맞춘다 |
| 실행 | 구현, 분기 위치 결정, 필요한 라이브러리 찾기 | 개인 판단이 팀 구조 안에서 길을 잃지 않게 한다 |
| 공유 | 커밋, PR, 코드 리뷰, 진행상황 보고 | 변경 의도와 현재 상태를 다음 사람에게 넘긴다 |
| 환류 | changelog, 핸드북 PR, API 문서, 온보딩 개선 | 반복 질문과 실수를 다음 사람의 진입로로 바꾼다 |

특히 GitLab식 소통 팁은 이 노트의 비동기 협업 원칙을 선명하게 보여 준다. 답이 핸드북에 없다면 질문에 답한 뒤 바로 문서 PR로 환류한다. 누군가에게 설명했다면 링크도 같이 남긴다. 내부 데이터는 개인 파일보다 공유 가능한 이슈 댓글로 남긴다. 즉 좋은 답변은 그 순간의 사람 한 명을 돕는 데서 끝나지 않고, 다음 비슷한 질문의 비용을 낮춘다.

`정보 불균형`, `라이브러리 찾기`, `github workflow` 메모도 같은 축이다. 조직이 커지면 모든 사람이 전체 맥락을 갖는 것은 불가능하다. 그래서 목표는 정보 비대칭 제거가 아니라 재진입 설계다. 빠진 사람이 목적, 결정, 다음 행동, 레포 위치, 액션 상태를 다시 복원할 수 있어야 한다.

## 유지 체크리스트

| 체크 | 보존할 원문 장면 | 판단 기준 |
| --- | --- | --- |
| 고객 만족과 팀원 성장 | 시간 버퍼, 과제 난도, 팀원의 좋은 결과물에 기운 저울 | 빠른 결과와 학습 기회를 함께 조정했는가 |
| 촉매자 역할 | PL 고민, 조언 요청, 문제 재정의, 면담 질문 | 막힌 지점과 필요한 자원이 표면에 남았는가 |
| 중심점과 현장 자율 | 바텀업/탑다운, 감독 없는 팀, 방향 제시 | 중심은 두되 현장 아이디어와 질문을 죽이지 않는가 |
| 조직 구조와 전달 비용 | 기능 조직, 목적 조직, TF, 피자 2판 팀, crew/basecamp | 깊이와 속도를 모두 얻기 위해 handoff 면을 설계했는가 |
| 응답과 상태 전달 | `OK/할게`의 한계, 마감시간, 명시적 호출, 링크 첨부 | 시간·소유자·위치·대안이 보여 상대의 대기열을 줄였는가 |
| 실패와 공개 기록 | 실패 공유, 공개 지적 주의, 핸드북 PR, 온보딩 빈칸 | 질문과 실수를 다음 사람이 찾을 수 있는 표면으로 되돌렸는가 |

## 후속 정리 기준

새 메모가 들어오면 먼저 `리더 완충 / 조직 handoff / 상태 패키지 / 재진입 기록` 중 어디에 붙는지 본다. GitLab식 소통 규칙은 소통 표면 묶음과 연결하되 Teamwork 본문에서는 응답 규칙의 최소 형태만 남긴다. 진행상황 공유 양식은 운영 구조 묶음과 맞추되, 이 페이지에서는 개인 생산성 템플릿보다 협업 상태 전달 기준으로 유지한다.

특히 `상태 패키지`는 지금 진행 중인 일이 상대의 대기열에서 사라지지 않게 하는 현재형 신호이고, `재진입 기록`은 같은 질문·실수·결정이 다음 사람에게 다시 설명되지 않게 남기는 장기 표면이다. 새 항목이 둘 다 건드릴 때는 먼저 현재 대기열을 줄이는지, 아니면 다음 진입 비용을 줄이는지로 배치한다.

평가나 자동 선택 과정에서 대응 관계를 확인할 때는 이 페이지와 `source/shdkej-content/Communication/Teamwork.md`의 source-mapping 행을 함께 본다. 핸드북, GitHub workflow, 코드리뷰, 온보딩 메모는 별도 기술 노드로 분리하기보다 Teamwork 안에서 협업 상태가 전달되고 다시 진입되는 표면으로만 다룬다.
