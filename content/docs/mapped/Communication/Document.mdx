# Document

- 원본: `source/shdkej-content/Communication/Document.md`
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트
- 상태: 점진 심화 2026-05-09

## 핵심 논지
이 노트에서 문서는 단순 기록물이 아니다. 팀의 질문, 판단, 사용 경험, 운영 절차, 문제 해결 이력을 한곳에 모으고 다시 꺼내 쓸 수 있게 만드는 작업 시스템이다. 좋은 문서는 최신 상태를 유지하고, 바로 찾을 수 있고, 바로 고칠 수 있어야 하며, 사람 사이의 설명 비용을 줄이면서도 실제 사용 맥락을 잃지 않아야 한다.

## 만다라트 위치에서 읽는 해석
`Communication/Document`는 글을 예쁘게 쓰는 기술 메모보다, **사람 사이에 지식이 어떻게 건너가고 다시 재사용되는가**를 다루는 카테고리 노드다. 같은 `Communication` 아래의 [[Blogging]]이 외부 독자를 향한 발행 표면을 다룬다면, 이 노트는 그보다 안쪽에서 팀 위키·README·히스토리·트러블슈팅·제품 소개처럼 서로 다른 문서 표면을 설계하는 운영 중심축에 가깝다. 즉 이 노드의 핵심은 문장력보다 `질문이 들어오고, 맥락이 붙고, 다음 사람이 다시 움직일 수 있게 만드는 문서 인터페이스`다.

## 이 노트가 실제로 다루는 질문
- 좋은 문서는 무엇을 만족해야 하는가?
- 정보 입력 창구와 출력 창구는 어떻게 설계해야 하는가?
- 팀 문서, 제품 문서, 기술 문서, 히스토리 문서는 어떻게 연결되어야 하는가?
- 공식 문서가 주지 못하는 경험 정보는 어디에 남겨야 하는가?
- 새 프로젝트가 시작될 때 왜 늘 0부터 다시 시작하는 느낌이 드는가?

## 오래 남길 아이디어
- 문서는 최신성, 검색성, 수정 용이성이 동시에 있어야 가치가 생긴다.
- 좋은 문서는 "설명 잘한 문장"이 아니라 "같은 문제를 다시 덜 겪게 하는 구조"에 가깝다.
- 질문 창구가 흩어지면 팀의 관심도 흩어지고, 기록도 흩어진다.
- 반대로 창구를 하나로 모으면 노이즈와 매몰 문제가 생기므로 태그, 인덱스, 링크 구조가 필요하다.
- 문서는 개발자용, 고객용, 운영용, 온보딩용처럼 독자별 목적을 구분해야 한다.
- 자동 문서화는 API, 테스트 시나리오, 모니터링 포인트처럼 구조적 정보에 강하다.
- 수동 문서화는 의도, 판단 이유, 주요 시나리오, 놓친 부분, 데모 맥락처럼 경험 정보에 강하다.
- README, 제품 소개 페이지, 사용 가이드, 히스토리 문서는 서로 다른 진입 지점을 맡는다.
- 공식 문서는 기준과 정확성을 주지만, 개인 경험 문서는 시행착오와 실제 선택의 감각을 보충한다.
- 히스토리 문서는 사건 순서와 문제 해결 경로를 남겨서 온보딩과 재현 가능성을 높인다.

## 구조 재정리

### 1. 문서는 검색 가능한 집적 장치여야 한다
원문은 계속해서 "모아두는 것"보다 "잘 찾고 바로 쓰는 것"을 강조한다. 데이터나 코드가 이미 있어도 찾기 어렵고 쓰기 어렵다면 자산처럼 작동하지 않는다. 그래서 문서화의 핵심은 축적 자체보다 접근성과 재사용성이다.

### 2. 문서는 대화 수준을 맞추는 공통 바닥이다
TDD 얘기가 나오면 잘 정리된 링크 3개 이내를 바로 전달할 수 있어야 한다는 대목은 중요하다. 문서는 모든 것을 길게 쓰는 일이 아니라, 대화가 같은 수준에서 시작되도록 기준점을 제공하는 일이다. 좋은 문서는 토론 전에 공통 맥락을 빠르게 맞춘다.

### 2-1. 좋은 문서 묶음은 `최소 공유 패키지`로 작동한다
원문에는 TDD 링크 3개, Banksalad의 `awesome style`, 사내 템플릿 레포, 프로젝트 한 페이지 같은 예가 반복된다. 이들은 모두 많은 자료를 전부 던지는 방식이 아니라, 특정 대화를 시작할 수 있을 만큼만 골라 묶는 **최소 공유 패키지**에 가깝다. 문서 운영의 핵심은 자료를 무한히 축적하는 것이 아니라, 필요 순간에 `이 세 개부터 보면 된다`, `이 페이지에서 다시 시작하면 된다`, `이 템플릿을 복제하면 된다`처럼 다음 행동이 바로 열리는 작은 묶음을 만드는 데 있다.

이 관점에서 문서화는 저장소를 키우는 일보다 큐레이션에 가깝다. 너무 큰 위키는 질문자를 다시 헤매게 만들고, 너무 작은 문서는 판단 맥락을 잃는다. 따라서 이 노드가 선호하는 문서 단위는 `짧지만 빈약하지 않고`, `공유하기 쉽지만 맥락을 잃지 않는` 재사용 가능한 묶음이다. 이는 [[Teamwork]]의 handoff 비용 절감, [[Open_Source]]의 공개 진입면 설계와도 이어진다.

### 3. 하나의 입력창구는 집중을 만들지만, 분류 체계가 없으면 곧 묻힌다
사내 위키를 잘 쓰는 법에 대한 고민은 단순 툴 선택의 문제가 아니다. 질문과 정보의 입구를 통일하면 팀 전체의 주의가 모이지만, 동시에 정보량이 많아지면서 과거 기록이 사라진다. 그래서 입력 통합과 태그/필터/인덱스 설계는 같이 가야 한다.

### 4. 문서는 독자별 진입면을 따로 가져야 한다
원문에는 `developer/client`, README, 제품 소개, 사용 가이드, 트러블슈팅, 히스토리 문서가 한꺼번에 나온다. 흩어져 보이지만 사실은 같은 시스템의 서로 다른 진입면이다.
- README: 구조를 한눈에 보여 주고 빠른 시작점을 준다.
- 제품 소개 페이지: 무엇을 하는 도구인지, 왜 흥미로운지, 데모와 실사용 장면으로 관심을 만든다.
- 사용 가이드: 실제 사용 순서와 최소 예제로 첫 성공 경험을 만든다.
- 트러블슈팅 문서: 막혔을 때 증상과 해결 경로를 다시 따라가게 한다.
- 히스토리 문서: 시간이 지나도 왜 이런 판단이 쌓였는지 맥락을 복원하게 한다.

즉 좋은 문서화는 한 문서에 모든 것을 몰아넣는 일이 아니라, 같은 대상을 독자와 순간에 맞는 여러 입구로 배치하는 일이다.

### 4-1. 이 노드에서 문서군은 분업 구조로 읽어야 한다
원문 후반으로 갈수록 README, 제품 소개, 프로그램 안의 사용 가이드, 공식 문서, 블로그형 경험 글, 히스토리 문서가 뒤섞여 보이지만, 실제로는 서로 대체 관계가 아니다.
- **README**는 "여기서 무엇을 먼저 보면 되는가"를 정하는 재진입 표면이다.
- **제품 소개**는 도구가 실제로 어떤 장면을 바꾸는지 상상하게 만드는 관심 유도 표면이다.
- **사용 가이드**는 가장 작은 성공 경로를 손에 쥐여 주는 실행 표면이다.
- **트러블슈팅**은 실패했을 때 어디서부터 다시 확인할지 알려 주는 복구 표면이다.
- **히스토리**는 시간이 지나 흐릿해진 판단 배경과 사건 순서를 되살리는 시간축 표면이다.
- **공식 문서 바깥의 경험 글**은 왜 그런 선택을 했는지, 무엇이 실제로 걸렸는지를 보충하는 현장 표면이다.

이렇게 읽으면 원문이 문서 종류를 늘어놓는 것이 아니라, 하나의 프로젝트를 여러 진입 각도에서 다시 열 수 있게 하는 `문서군 운영 설계`를 고민하고 있다는 점이 더 선명해진다.

### 5. 문서에는 맥락과 경험이 들어가야 한다
컨플루언스, 노션, README, 제품 소개, 히스토리 문서에 대한 메모를 묶어 읽으면, 저자는 단순 사실 목록보다 "왜 이런 선택을 했는지", "어떤 상황에서 썼는지", "실제로 어떻게 동작하는지"를 중요하게 본다. 외부 문서 첨부, 디자인 목업, 회의 맥락, 고객 인터뷰, 데모, 실사용 시나리오가 모두 이 요구를 보여준다.

### 6. 프로젝트는 문서 묶음으로 기억되어야 한다
한 프로젝트를 한 페이지에 관리하고, 관련 링크와 채팅 맥락, 아키텍처, 드라이브, 이슈 도구, 운영 환경을 함께 모으려는 발상은 프로젝트를 단일 산출물이 아니라 연결된 작업 기억으로 다루는 태도다. 새 프로젝트가 늘 0부터 시작하는 느낌이 들지 않게 하려면, 이전 프로젝트의 히스토리와 문서 묶음이 재사용 가능해야 한다.

### 7. 코드 집적 문제는 결국 문서 문제의 연장선이다
원문은 "문서는 집적 가능한데 코드는 어떻게 집적하고 확인하고 사용할 수 있을까"를 따로 묻는다. npm package, modules, pip, GitHub, 추천 알고리즘, IPFS까지 나오는 이유는, 코드도 결국 `찾을 수 있어야 쓸 수 있다`는 같은 고민 위에 있기 때문이다. 이 노드에서 문서화는 글쓰기 스킬보다 조직 기억의 검색성과 재사용성 전체를 다루는 상위 문제다.

### 7-1. 자동 문서화와 수동 문서화는 서로 다른 빈칸을 채운다
원문은 Swagger, API, test scenario, monitoring point 같은 자동 문서화 항목과, 주요 시나리오, 아키텍처, 모델, 데모, 히스토리 같은 수동 문서화 항목을 함께 둔다. 이 둘을 `자동화하면 좋은 문서`와 `사람이 써야 하는 문서`로만 나누면 약하다. 더 정확하게는 자동 문서화가 **현재 시스템이 드러내는 구조적 사실**을 붙잡고, 수동 문서화가 **사람이 어떤 의도와 제약 속에서 그렇게 만들었는지**를 붙잡는다.

그래서 이 노드의 문서화는 다음 분업을 요구한다.
- **자동 문서화**: API 스펙, 테스트 시나리오, 모니터링 포인트처럼 코드와 설정에서 반복 추출 가능한 정보를 최신 상태로 유지한다.
- **수동 문서화**: 왜 이 시나리오를 중요하게 봤는지, 어떤 부분을 신경 썼고 무엇을 아직 못 봤는지, 실제 사용자가 어떤 장면에서 막히는지 남긴다.
- **연결 문서**: 자동으로 나온 사실과 수동으로 남긴 판단을 README, 히스토리, 트러블슈팅, 제품 소개가 서로 호출하게 만든다.

이 분업이 있어야 문서는 `정확하지만 건조한 스펙`이나 `경험은 풍부하지만 곧 낡는 글` 중 하나로 치우치지 않는다.

### 8. 문서는 `무엇부터 알게 할 것인가`를 설계해야 한다
원문에는 `새로운 툴에 대해 아는 순서`로 `이게 무엇인지 → 왜 필요한지 → 어떻게 쓰는지`가 짧게 적혀 있다. 이 메모는 사소한 듯하지만, 문서를 단순 정보 저장소가 아니라 **이해 순서를 설계하는 인터페이스**로 본다는 점을 잘 보여 준다.
- 정의가 먼저 없으면 사용법이 떠다닌다.
- 필요성이 먼저 납득되지 않으면 기능 설명은 읽혀도 남지 않는다.
- 사용법은 최소 예제로 빨리 확인되어야 한다.

즉 이 노트에서 좋은 문서는 사실을 많이 담는 문서보다, **독자의 이해 순서를 덜 헤매게 만드는 문서**에 가깝다.

### 9. 메타 문서함은 많이 모으기보다 다시 찾기 위한 보조 장치다
원문 후반의 `용어 사전 / 유의어 사전 / 년도별 묶음 / 인덱스 페이지` 메모는 본문보다 부차적인 부록처럼 보이지만, 실제로는 문서군 전체의 재진입성을 받치는 장치다. 문서가 늘어날수록 본문 하나만 잘 써서는 충분하지 않고, 검색어가 달라도 같은 내용에 도달하게 만드는 우회로가 필요해진다.

그래서 이 노드에서 메타 문서함은 정리 취미가 아니라 다음과 같은 운영 보조층으로 읽는 편이 맞다.
- **용어 사전**: 팀마다 다른 말로 부르는 대상을 같은 개념으로 묶는다.
- **유의어 사전**: 검색 실패를 줄인다.
- **연도별 묶음**: 시간축 회고와 히스토리 탐색을 쉽게 만든다.
- **인덱스 페이지**: 어디서 시작해야 할지 모를 때 첫 진입점을 준다.

### 10. 트러블슈팅 문서는 에러명을 적는 것보다 접수 구조가 먼저다
원문은 `에러명을 적고 설명을 적으면` 같은 방식만으로는 같은 에러·다른 원인, 다른 에러·같은 원인을 충분히 처리하기 어렵다고 본다. 그래서 중요한 것은 더 긴 에러 백과를 만드는 일이 아니라, 에러가 들어오는 창구와 필수 맥락을 먼저 통일하는 일이다.

이 노드에서 트러블슈팅 문서는 다음 네 가지 빈칸을 가져야 한다.
- **출처**: 어느 환경, 어느 화면, 어느 로그, 어느 사용자 흐름에서 들어왔는가.
- **증상**: 사용자가 본 문장과 시스템이 남긴 신호를 분리해 적었는가.
- **가설/원인**: 비슷한 증상이라도 원인이 달라질 수 있음을 열어 두었는가.
- **처리/재검증**: 무엇을 고쳤고, 어떤 방식으로 다시 확인했는가.

이렇게 보면 에러 리포팅, 댓글 방식, 윈도우 리포트, GitLab 방식에 대한 짧은 메모들은 모두 `문서만 보고 동일 문제를 다시 처리할 수 있게 만드는 접수 인터페이스`를 찾는 흔적이다. 이는 [[../Meta/Troubleshooting]]의 증상-상태 확인-수정-재검증 루프와 직접 이어진다.

## 핵심 판단 축

### 최신성 vs 작성 부담
문서는 최신이어야 하지만, 너무 무거우면 아무도 갱신하지 않는다. 원문이 수정 용이성을 같이 강조하는 이유다.

### 중앙집중 vs 정보 매몰
입구를 하나로 모으면 팀의 시선이 합쳐지지만, 흐름이 빨라지면 과거 정보가 묻힌다. 중앙화만으로는 해결되지 않고 구조화가 필요하다.

### 표준화 vs 맥락 보존
표준 문서 양식은 전달 효율을 높이지만, 실제 판단 이유와 현장감이 빠지면 쓸모가 줄어든다. 저자는 표준을 원하지만 맥락 없는 템플릿은 원하지 않는다.

### 자동화 vs 서술형 설명
Swagger 같은 자동 문서화는 강력하지만, 사용 의도와 경험적 판단을 대신하지는 못한다. 자동과 수동은 대체 관계보다 분업 관계에 가깝다.

### 공식성 vs 경험성
공식 문서는 신뢰의 기준점이고, 개인 블로그나 사례 문서는 실제 시행착오의 감각을 준다. 이 노트는 둘 중 하나를 버리기보다 둘을 연결하려 한다.

### 문서 축적 vs 프로젝트 재시작 비용
원문 후반은 새 프로젝트를 시작할 때마다 `0부터 다시 시작하는 느낌`이 들면 안 된다고 적는다. 즉 문서화의 진짜 효용은 보기 좋은 정리보다, 다음 프로젝트가 이전 프로젝트의 시행착오·절차·판단을 이어받게 만드는 데 있다.

## 반복해서 나타나는 패턴

### 문서는 "설명"보다 "재진입"을 돕는다
문서를 읽고 같은 문제를 다시 다룰 수 있어야 한다는 감각이 강하다. 온보딩, 트러블슈팅, 사용 가이드, 프로젝트 히스토리 모두 재진입 비용을 줄이는 장치다.

### 문서는 객체보다 흐름을 담아야 한다
정의, 사양, 기능 목록만으로는 부족하다. 문제를 어떻게 인지했고, 어떤 선택을 했고, 어떻게 해결했는지가 남아야 다음 사람이 따라갈 수 있다.

### 히스토리 문서는 일지와 매뉴얼 사이의 다리다
원문 인용부에서 중요한 것은 `일지 → 히스토리 → 서브 문서`로 발전하는 흐름이다. 당일 체크리스트와 스크린샷은 살아 있는 작업 흔적이고, 히스토리 문서는 그것을 시간축으로 묶어 주며, 서브 문서는 개별 방법을 재사용 가능한 매뉴얼로 분리한다. 그래서 히스토리 문서는 회고용 부록이 아니라 온보딩과 운영 전수를 가능하게 하는 중간 층이다.

### 문서는 독자에 따라 같은 사실을 다른 표면으로 번역해야 한다
개발자용, 고객용, 내부 위키용, README, 제품 소개, 에러 리포트는 각각 다른 언어와 밀도를 요구한다. 이 노트는 문서의 정답 형식 하나를 찾기보다, 같은 대상을 여러 표면으로 번역하는 설계를 고민한다.

### `정의 → 필요성 → 사용법` 순서가 반복해서 등장한다
툴 소개, 제품 소개, 퀵스타트, 실제 사례 메모를 함께 보면 저자는 설명의 첫 단추를 꽤 일관되게 본다. 먼저 이것이 무엇인지, 왜 필요한지, 실제로 어떻게 쓰는지를 차례로 보여 줘야 사람의 이해가 끊기지 않는다. 이 패턴은 README, 제품 페이지, 사용 가이드까지 함께 관통하는 작은 문법이다.

### 문서는 도구보다 운영 방식의 문제다
컨플루언스, 노션, GitHub, 패키지 관리, 추천 알고리즘, IPFS까지 언급되지만, 핵심 고민은 언제나 같다. 이미 있는 지식을 어떻게 발견하고, 어떤 기준으로 공유하며, 다시 사용할 것인가.

### 문서는 느낌을 원칙으로 바꾸는 변환기다
원문 후반의 `느낌은 전수되지도 않고 일관되지도 않다 / 느낌이 아닌 원칙을 만들어 나가면 전수받고 같이 성장할 수 있다`는 문장은 이 노드의 숨은 기준에 가깝다. 문서화는 개인이 어렴풋이 아는 요령을 예쁜 문장으로 남기는 일이 아니라, 다음 사람이 같은 상황에서 다시 판단할 수 있는 **전달 가능한 원칙**으로 바꾸는 일이다.

이 관점에서 신입 가이드, 테크스펙, 코드 수정·정기 배포 절차, 비상상황 행동 방침은 모두 같은 역할을 한다.
- **신입 가이드**는 암묵지의 첫 진입 순서를 정한다.
- **테크스펙**은 구현 전 판단 기준과 합의된 제약을 드러낸다.
- **배포 절차**는 반복 작업의 불안을 줄이고, 누가 해도 같은 확인면을 갖게 한다.
- **비상상황 행동 방침**은 당황한 순간에도 각자의 감각이 아니라 합의된 기준으로 움직이게 한다.

즉 좋은 문서군은 `잘 아는 사람이 알아서 처리하는 상태`를 `덜 아는 사람도 원칙을 따라 안전하게 들어올 수 있는 상태`로 바꾼다. 이는 [[Teamwork]]의 handoff 비용 절감, [[../Meta/About_Development]]의 개발 지식 외부화, [[../Meta/Troubleshooting]]의 증상-상태 확인-수정-재검증 루프와 직접 맞닿아 있다.

### 문서는 시간층을 나눠 보관해야 다시 쓰인다
원문에는 고객 인터뷰, 요구사항, 회의, 릴리스 계획, 히스토리 문서, 프로젝트 한 페이지가 함께 나온다. 이들은 모두 같은 문서함에 들어갈 수 있지만 같은 시간층은 아니다. 요구사항과 회의록은 **결정 전의 흔들림**을 남기고, README와 사용 가이드는 **현재 실행법**을 남기며, 히스토리 문서는 **이미 지나간 판단의 이유**를 남긴다.

이 시간층을 구분하지 않으면 문서가 최신화될수록 과거 판단 근거가 사라지고, 반대로 히스토리를 모두 본문에 붙들면 현재 사용자가 필요한 빠른 진입이 흐려진다. 따라서 이 노드에서 좋은 문서군은 `현재 진입면`과 `과거 판단층`을 분리하되, 서로 링크로 왕복할 수 있게 만드는 구조에 가깝다.

### 좋은 문서는 질문이 들어오는 순간의 모양까지 설계한다
원문은 질문하지 않으려는 문제와 질문이 많아져 묻히는 문제를 동시에 본다. 이 때문에 문서 운영은 작성 완료 뒤의 정리가 아니라, 질문·에러·요구사항·회의 맥락이 처음 들어오는 순간부터 시작된다. 질문이 처음부터 공개 채널, 태그, 필수 맥락, 관련 링크를 갖고 들어오면 나중에 문서로 바꾸는 비용이 크게 줄어든다. 반대로 접수 순간의 모양이 흐리면, 아무리 좋은 위키 도구를 써도 기록은 개인 기억이나 채팅 조각으로 흩어진다.

### 질문이 없어서 비는 문제와 너무 많아서 묻히는 문제를 함께 다룬다
원문 중 "없어서 문제가 생기는 부분은 질문이 없을 리가 없으므로 유도를 해야 한다"와 "많아서 문제가 생기는 부분은 태그로 필터링할 수 있도록 하면 될까"라는 메모는 특히 중요하다. 이 노드는 문서를 `잘 정리된 저장소`로만 보지 않고, **질문을 끌어내는 장치이자 질문 과부하를 걸러내는 장치**로 함께 본다. 즉 문서 운영의 어려움은 축적량 자체보다 `침묵 문제`와 `매몰 문제`를 동시에 다뤄야 한다는 데 있다.

### 한 페이지 프로젝트 문서는 운영 제어면에 가깝다
원문 말미의 `프로젝트 한 페이지에 관리` 메모는 단순 링크 모음이 아니다. 개요, 아키텍처, 저장소, 드라이브, 작업도구, 채팅, 클라우드 링크를 한 화면에 모으려는 이유는 프로젝트를 설명하기 위해서이기도 하지만, 더 깊게는 **다시 들어가서 움직일 수 있는 제어면**을 만들기 위해서다. 이 감각은 문서를 정적 저장소보다 작업 재개 인터페이스로 보는 태도를 잘 보여 준다.

### 문서의 좋은 단위는 `다 읽어야 하는 것`이 아니라 `바로 넘길 수 있는 것`이다
TDD 링크를 3개 이하로 추리고, 사내 행동강령을 `awesome style` 레포로 모으고, 프로젝트 관련 링크를 한 페이지에 모으려는 메모는 모두 같은 신호다. 저자는 문서를 작성자의 지식 과시물이 아니라 **다른 사람에게 곧바로 건넬 수 있는 전달 단위**로 본다. 그래서 좋은 문서는 길이보다 전달 가능성이 중요하다. 누군가 질문했을 때 바로 공유할 수 있고, 받은 사람이 어디서부터 읽을지 알 수 있으며, 이후 대화가 같은 바닥 위에서 시작되어야 한다.

## 특히 보존할 부분

### 히스토리 문서의 가치
원문 후반의 히스토리 문서 인용은 이 노트의 핵심 중 하나다. 일지에서 히스토리로, 히스토리에서 문서로 발전하는 흐름은 사건 기록을 제도화하는 과정이다. 이것이 있으면 새로 합류한 사람이 과거 판단과 문제 해결 경로를 빠르게 배울 수 있다.

### README와 제품 소개의 차이
README는 구조와 빠른 진입을, 제품 소개는 관심 유도와 사용 상상을 담당한다. 이 둘을 같은 것으로 취급하지 않는 감각이 보인다.

### 공식 문서와 경험 문서의 분업
원문은 공식 문서가 주는 정확성과, 블로그·튜토리얼·사례 문서가 주는 경험성을 동시에 붙든다. 저자는 둘 중 하나를 택하기보다, 공식 문서에는 없는 시행착오와 선택의 이유를 별도 문서 층으로 보충해야 한다고 보는 쪽에 가깝다.

### 문서와 코드 자산 관리의 연결
문서는 집적 가능한데 코드 자산은 어떻게 찾고 쓰게 만들 것인가라는 질문은, 문서화가 결국 재사용 가능한 조직 기억 전체를 겨냥한다는 신호다.

### 시각화는 장식이 아니라 빠른 이해 장치다
README의 아키텍처 한눈에 보기, 제품 소개의 데모 이미지, 인포그래픽 참고 메모는 모두 같은 방향을 가리킨다. 저자는 문서를 텍스트만 잘 쓰는 문제로 보지 않고, 구조와 흐름을 더 빨리 읽게 하는 시각적 인터페이스까지 문서화의 일부로 취급한다.

### 국제화·UTM·시각화 메모도 결국 `배포 후 이해`의 일부다
원문 말미의 국제화, UTM, 인포그래픽, 이미지 설명 위치 메모는 주제가 튄다고 느껴질 수 있다. 하지만 이들을 함께 보면 저자는 문서를 작성하는 순간만이 아니라, **배포된 뒤 누가 어떤 경로로 들어와 어떻게 읽고 오해 없이 이해하는가**까지 문서화의 범위로 보고 있다.
- 국제화 메모는 언어 표면을 바꾸는 문제다.
- UTM 메모는 유입 경로를 추적해 전달 결과를 보는 문제다.
- 이미지·인포그래픽 메모는 시각 표면을 다듬는 문제다.

즉 이 노드의 문서화는 작성 행위에서 끝나지 않고, 배포·이해·재유입까지 이어지는 커뮤니케이션 운영으로 확장된다.

## 원문 발췌 구조
- **문서의 기본 조건**: 최신성, 검색성, 수정 용이성
- **지식 집적 고민**: 입력/출력 창구 통일, 태그, 위키, 컨플루언스
- **코드·데이터 재사용 고민**: 패키지, 저장소, 추천, 분산 탐색
- **실패 복구 문서**: 에러 리포팅, 출처 기록, 단일 접수 창구
- **재진입 표면 설계**: README, 제품 소개, 사용 가이드, 데모
- **운영 기억 축적**: 히스토리 문서, 프로젝트 한 페이지, 이전 프로젝트 재사용
- **이해 순서 설계**: 이것이 무엇인지, 왜 필요한지, 어떻게 쓰는지
- **표현 보조 장치**: 시각화, 국제화, UTM, 이미지/인포그래픽

이 배열을 보면 원문은 문서 종류를 병렬로 정리한 것이 아니라, `기록 → 발견 → 사용 → 복구 → 온보딩 → 재시작 방지`로 이어지는 운영 흐름을 여러 문서 표면으로 분산 저장하고 있다.

## 인접 노드 연결
- [[Readability]]: 문서를 쉽게 읽히게 만들되 의미를 납작하게 만들지 않아야 한다는 점에서 직접 연결된다.
- [[Blogging]]: 내부 위키와 외부 발행을 가르는 편집 감각이라는 점에서 같은 `Communication` 축의 바깥 표면이다.
- [[../Deep Knowledge/Product]]: 제품 소개, 데모, 사용 시나리오, 감동 설계에 대한 감각과 이어진다.
- [[../Human/Evolve]]: 많은 양의 정보를 계속 관리하고 갱신 가능한 상태로 두려는 운영 감각과 맞닿아 있다.
- [[../Meta/Collection]]: 흩어진 정보와 가치 기준을 한데 모으려는 메타적 태도와 연결된다.
- [[../Meta/Troubleshooting]]: 같은 문제를 다시 만났을 때 문서만 보고 처리할 수 있게 하려는 문제 해결 축을 공유한다.
- [[../Meta/About_Development]]: 설명 가능성, 온보딩, 문서 기반 협업 감각을 개발 메타 차원에서 이어 준다.

## 선호 신호
저자는 문서를 형식적 산출물이 아니라 팀과 개인의 재사용 가능한 작업 기억으로 보는 경향이 있다. 예쁘게 정리된 문서보다, 실제 질문에 바로 답하고, 새 사람을 빠르게 온보딩하고, 문제 재발 시 바로 재진입할 수 있게 만드는 문서를 선호한다. 또한 자동화된 구조 정보와 사람이 남기는 경험 정보를 함께 유지하려는 성향이 강하다.
