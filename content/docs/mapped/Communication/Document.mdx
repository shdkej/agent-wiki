# Document

- 원본: `source/shdkej-content/Communication/Document.md`
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트
- 상태: 안정 디벨롭 2026-06-15, 1:1 대응 및 인접 링크 독자용 표시명/절대 경로 확인

## 핵심 논지
이 노트에서 문서는 예쁘게 정리한 글이 아니라 질문, 오류, 요구사항, 사용 경험, 판단 이력을 다음 사람이 다시 움직일 수 있는 상태로 바꾸는 작업 인터페이스다. 좋은 문서는 최신이고, 찾기 쉽고, 고치기 쉬우며, 독자가 지금 무엇을 먼저 보고 어디서 다시 시작해야 하는지 알려 준다.

## 만다라트 위치에서 읽는 해석
`Communication/Document`는 `Communication` 안에서 말과 피드백보다 오래 남는 전달 표면을 맡는다. [Talk](/docs/mapped/Communication/Talk)가 순간의 수신 상태를 열고 [Feedback](/docs/mapped/Communication/Feedback)이 돌아온 반응을 조정한다면, Document는 그 접점에서 나온 질문과 판단을 팀이 다시 쓸 수 있는 구조로 고정한다. 따라서 이 노드는 문서 형식 목록이 아니라 **접수 / 배치 / 전달 / 재호출 / 신뢰 신호**를 설계하는 카테고리 노드로 읽어야 한다.

## 원문 흐름 요약
원문은 좋은 문서화의 조건으로 최신성, 검색성, 접근성, 수정 용이성을 먼저 놓는다. 이어 TDD 링크 3개처럼 바로 전달할 수 있는 자료 묶음, 개발자용·고객용 문서 구분, Swagger 같은 자동 문서화, 사내 위키와 Confluence의 입력 창구 문제로 이동한다.

중반에는 코드와 데이터도 결국 찾고 쓸 수 있어야 자산이 된다는 질문, 트러블슈팅 접수 구조, README와 제품 소개 페이지, 사용 가이드, 공식 문서의 한계가 이어진다. 후반에는 자동/수동 문서화, 메타 문서함, 신입 가이드·테크스펙·배포 절차·비상 행동 방침, UTM·국제화·히스토리 문서·프로젝트 한 페이지·시각화가 붙으면서 문서가 정보 저장소를 넘어 운영 기억과 배포 후 이해 환경을 설계하는 문제로 확장된다.

## 이 노트가 실제로 다루는 질문
- 좋은 문서는 최신성, 검색성, 수정 용이성을 어떻게 동시에 만족하는가?
- 질문·오류·요구사항은 처음 들어올 때 어떤 형태로 받아야 나중에 문서가 되는가?
- README, 제품 소개, 사용 가이드, 트러블슈팅, 히스토리 문서는 각각 어떤 진입면을 맡는가?
- 자동 문서화와 사람이 남기는 경험 문서는 어떻게 분업해야 하는가?
- 새 프로젝트가 늘 0부터 시작하지 않으려면 어떤 작업 기억을 남겨야 하는가?

## 중심 모델: 접수에서 재호출까지

| 단계 | 원문 단서 | 문서 운영에서 맡는 역할 |
| --- | --- | --- |
| 접수 | 입력창구 통일, 고객 인터뷰, 요구사항, 회의, 에러 리포팅 | 질문·오류·요구가 들어올 때 출처, 상황, 관련 링크를 잃지 않게 한다. |
| 배치 | README, 제품 소개, 사용 가이드, 트러블슈팅, 히스토리, 메타 문서함 | 들어온 조각을 독자와 사용 순간에 맞는 표면으로 나눈다. |
| 전달 | TDD 링크 3개, awesome style, 템플릿 레포, 프로젝트 한 페이지 | 많은 자료를 전부 던지지 않고 바로 공유 가능한 최소 패키지로 줄인다. |
| 재호출 | 태그, 용어 사전, 유의어 사전, 연도별 묶음, UTM, 히스토리 문서 | 과거 판단과 유입 경로를 나중에 다시 찾고 설명할 수 있게 한다. |
| 신뢰 신호 | 최신 버전, 데모, 최소 예제, 실제 사용 시나리오, 소스 버전관리 연동 | 독자가 이 도구나 절차를 믿고 시작해도 되는지 판단하게 한다. |

이 모델로 보면 문서화의 완료 기준은 `작성했다`가 아니라 `다음 재사용이 쉬워졌다`다. 최신성, 검색성, 수정 용이성은 이 순환을 유지하기 위한 조건이고, 문서 종류의 분화는 같은 정보를 서로 다른 순간에 다시 열기 위한 인터페이스 분업이다.

## 문서군의 분업

| 문서 표면 | 맡는 일 | 주의할 점 |
| --- | --- | --- |
| README | 구조를 한눈에 보여 주고 첫 실행 경로를 만든다. | 모든 히스토리를 넣으면 빠른 진입성이 떨어진다. |
| 제품 소개 | 무엇을 하는 도구인지, 왜 필요한지, 실제로 어떻게 움직이는지 보여 준다. | 마케팅 문구보다 데모, 시나리오, 비교, 현재성 신호가 중요하다. |
| 사용 가이드 | 최소 예제로 첫 성공 경험을 만든다. | 불필요한 의존성이나 주변 도구를 과하게 요구하면 시작 신뢰가 깨진다. |
| 트러블슈팅 | 같은 문제를 다시 만났을 때 출처, 증상, 가설, 처리, 재검증을 따라가게 한다. | 에러명 백과보다 접수 구조와 필수 맥락이 먼저다. |
| 히스토리 | 일지와 매뉴얼 사이에서 사건 순서와 판단 배경을 복원한다. | 현재 사용법과 과거 판단층을 섞으면 둘 다 흐려진다. |
| 메타 문서함 | 용어 사전, 유의어 사전, 연도별 묶음, 인덱스로 재진입성을 받친다. | 정리 취미가 아니라 검색 실패를 줄이는 보조층이어야 한다. |
| 프로젝트 한 페이지 | 개요, 아키텍처, 저장소, 드라이브, 작업 도구, 채팅, 운영 환경을 한 화면에 모은다. | 링크 모음에 그치지 말고 다시 들어가 움직일 수 있는 제어면이어야 한다. |

원문에 문서 종류가 많이 나오는 이유는 한 문서에 모든 것을 넣으려는 뜻이 아니다. 같은 프로젝트를 처음 보는 사람, 막힌 사람, 운영하는 사람, 나중에 이어받는 사람이 각자 다른 표면으로 들어올 수 있게 하려는 분업 설계다.

## 핵심 판단 축

| 축 | 보존할 판단 |
| --- | --- |
| 최신성 vs 작성 부담 | 문서는 최신이어야 하지만 갱신이 무거우면 곧 죽는다. 수정하기 쉬운 구조가 최신성의 전제다. |
| 중앙집중 vs 정보 매몰 | 입력창구를 하나로 모으면 팀의 관심이 모이지만, 태그·필터·인덱스가 없으면 과거 정보가 묻힌다. |
| 표준화 vs 맥락 보존 | 표준 양식은 전달 효율을 높이지만 판단 이유와 현장감이 빠지면 실제 재사용성이 떨어진다. |
| 자동 문서화 vs 수동 문서화 | Swagger, API, 테스트, 모니터링 포인트는 구조적 사실을 맡고, 주요 시나리오·아키텍처·데모·히스토리는 의도와 제약을 맡는다. |
| 공식성 vs 경험성 | 공식 문서는 정확성의 기준점이고, 블로그·튜토리얼·사례 문서는 실제 시행착오와 선택 이유를 보충한다. |
| 축적 vs 재시작 비용 | 문서화의 효용은 보기 좋은 저장소보다 다음 프로젝트가 이전 판단과 절차를 이어받게 하는 데 있다. |

| `새 프로젝트를 시작할 때 다시 0부터` | 문서가 조직 기억으로 작동하지 못할 때 생기는 실제 비용을 보여 준다. |
| UTM·국제화·읽는 방향·인포그래픽 | 문서 작성 뒤에도 유입 경로, 언어, 읽기 관습, 시각 이해까지 배포 후 해석 환경을 설계해야 한다는 단서다. |

## 자동과 수동의 연결
자동 문서화는 현재 시스템에서 반복 추출 가능한 구조를 최신으로 붙잡는다. API 스펙, 테스트 시나리오, 모니터링 포인트, 소스 버전관리 연동이 여기에 가깝다. 수동 문서화는 왜 이 시나리오가 중요한지, 어떤 부분을 신경 썼는지, 무엇을 아직 못 봤는지, 실제 사용자가 어디서 막혔는지를 남긴다.

둘은 대체 관계가 아니라 연결되어야 한다. 자동으로 나온 사실은 README, 사용 가이드, 트러블슈팅에서 호출되어야 하고, 사람이 남긴 판단은 히스토리와 제품 소개에서 독자가 맥락을 이해하게 해야 한다. 그래야 문서가 `정확하지만 건조한 스펙`이나 `경험은 풍부하지만 곧 낡는 글` 중 하나로 치우치지 않는다.

## 배포 후 이해 환경
원문 후반의 UTM, 국제화, 읽는 방향, 말풍선 배치, 인포그래픽 메모는 주제가 튄 것이 아니라 문서가 독자에게 도착한 뒤의 환경을 묻는 부분이다. 문서는 작성자가 아는 내용을 담는 것으로 끝나지 않고, 독자가 어떤 경로로 들어와 어떤 언어와 관습으로 읽으며 어디서 오해할 수 있는지까지 고려해야 한다.

- **언어 표면**: 표시 문자열을 분리해 다른 언어와 수정 흐름을 받을 수 있는가.
- **읽기 관습**: 좌우·상하 흐름, 이미지와 설명 위치가 독자의 습관과 충돌하지 않는가.
- **유입 표면**: UTM처럼 어떤 경로로 들어왔는지 나중에 확인할 수 있는가.
- **시각 표면**: 데모 이미지와 인포그래픽이 장식이 아니라 이해 순서를 줄여 주는가.

## 외부 산출물 연결
`source/external-links/marketing/2026-06-14-agentic-plg-outcome-docs.md`는 Document 노트에 새 마케팅 이론을 더하기보다, 문서가 기능 설명을 넘어 `좋은 결과 / 판단 기준 / 다음 행동`을 함께 보여 주어야 한다는 신뢰 신호를 선명하게 한다. 특히 AI 제품이나 에이전트가 읽는 문서에서는 README와 제품 소개가 "무엇을 할 수 있는가"만 말하면 부족하고, 사용자가 받은 출력이 왜 성공인지와 그다음에 저장·검증·반복 중 어디로 가야 하는지를 사람이 읽어도 기계가 읽어도 남겨야 한다.

이 연결은 Document의 기존 다섯 칸 모델 중 `전달`과 `신뢰 신호`에만 붙인다. 외부 자료를 성장 전략으로 확장하지 말고, 제품 소개·사용 가이드·히스토리 문서가 첫 성공 결과를 재호출 가능한 문장으로 남기는지 점검하는 보조 렌즈로 쓴다.

- [Meta/Troubleshooting](/docs/mapped/Meta/Troubleshooting): 같은 문제를 다시 만났을 때 증상·상태 확인·수정·재검증을 따라가게 하는 복구 문서 축을 공유한다.
- [Meta/About Development](/docs/mapped/Meta/About_Development): 개발 지식과 판단을 외부화해 다음 사람이 재사용하게 만드는 개발 메타 감각과 연결된다.
- [Integration/Marketing](/docs/mapped/Integration/Marketing): UTM, 제품 소개, 독자 유입 경로처럼 배포 후 반응을 읽는 표면과 닿아 있다.

## 후속 정리 기준
새 메모를 붙일 때는 문서 유형 목록을 더 늘리기보다 먼저 다섯 칸 중 어디에 들어가는지 판정한다.

| 새 메모 유형 | 먼저 붙일 위치 |
| --- | --- |
| 질문, 오류, 요구사항, 회의, 고객 인터뷰, 채팅 맥락 | `접수` |
| README, 제품 소개, 가이드, 트러블슈팅, 히스토리, 메타 문서함 | `배치` |
| 링크 묶음, 템플릿, 프로젝트 한 페이지, 공유용 레퍼런스 | `전달` |
