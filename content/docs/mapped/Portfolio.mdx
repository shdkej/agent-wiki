# Portfolio

- 원본: `source/shdkej-content/Portfolio.md`
- 상태: 안정 기준선 이후 압축 디벨롭
- 역할: 원본 만다라트 노드와 1:1 대응하는 루트 카테고리 페이지

## 한눈에 보는 요약
이 노드는 프로젝트 자랑 목록보다, 저자가 인프라 문제를 어떻게 발견하고 운영 표면으로 다시 설계하는지 보여주는 자기 소개형 기준점이다. 핵심은 기술 도입 자체가 아니라 마찰 감소, 안전한 자동화, 비용 감각, 관찰성, 문서화 가능한 플랫폼화다.

## 중심 해석
저자는 인프라와 DevOps를 "더 많은 도구를 붙이는 일"보다 "사람이 매번 애써서 버티던 운영 문제를 구조로 옮기는 일"로 이해한다. 원본의 프로젝트는 메시징, EKS, Slack 배포, 비용 최적화, 로그 추적, 백엔드 개발로 퍼져 있지만, 반복되는 문법은 일정하다.

1. 병목이나 반복 마찰을 현실 제약 언어로 잡는다.
2. 개인 숙련이나 수동 절차에 기대던 흐름을 공용 제어면으로 옮긴다.
3. 속도를 올리면서 누락 방지, 배포 안전성, 재시작, 권한, 멱등성 같은 보호 장치를 붙인다.
4. 수치, 문서, 다이어그램, 절차로 다음 사람이 다시 들어올 수 있는 운영 기억을 남긴다.

## 만다라트 루트 자리
`Portfolio`는 루트 레벨에서 기술 정체성과 실행 스타일을 가장 직접적으로 드러내는 노드다. 이 페이지를 읽으면 주변 노드도 도구 숙련도보다 "팀이 덜 막히도록 어떤 운영 표면을 만드는가"라는 질문으로 다시 읽힌다.

- [Fundamental/Infra](/docs/mapped/Fundamental/Infra)·[Deep Knowledge/Devops](/docs/mapped/deep-knowledge/devops)는 개별 기술 지식 노드라면, `Portfolio`는 그 기술들이 실제 조직 장면에서 어떤 책임으로 묶이는지 보여 주는 상위 해석면이다.
- [Communication/Document](/docs/mapped/Communication/Document)·[Communication/Teamwork](/docs/mapped/Communication/Teamwork)는 협업 표면 노드라면, `Portfolio`는 문서화와 handoff가 왜 운영 성과인지 직무 장면에서 증명하는 근거 노드다.
- 삭제된 CHANGELOG 원본과 함께 보면 작은 개인 웹 운영에서도 같은 유지 노동, 정리, 배포 리듬이 반복됐다는 점이 보인다.

최근 평가가 지적한 반복을 줄이기 위해 이 노드는 여러 해석 문단을 네 축으로 압축해서 유지한다.

| 축 | 원본 증거 | 읽는 질문 |
| --- | --- | --- |
| 입력면 | 외부 API 한도, 소켓 부족, 수동 배포, 신규 사이트 반복, 로그 추적 어려움 | 어떤 요청, 장애, 비용, 대기 시간이 팀 흐름을 막고 있었는가? |
| 제어면 | Lambda 파이프라인, Kinesis, Slack 명령, GitOps, CronJob, Helm | 그 마찰을 사람이 기억하는 절차에서 어떤 공용 조작면으로 옮겼는가? |
| 안전면 | 누락 최소화, 배치 중단 확인, 신규 환경 생성, CNI/인증서/정책 조정, 멱등성 | 빨라진 흐름이 덜 깨지도록 어떤 보호 장치를 붙였는가? |

이 네 축으로 보면 큰 인프라 프로젝트와 작은 자동화 사례가 같은 문법 안에 놓인다. 단순히 `인프라를 잘한다`가 아니라, 팀이 덜 막히도록 운영 표면을 설계한다는 점이 이 노드의 핵심이다.

## 외부/실행 산출물 연결
Control Center 계열 실행 산출물은 이 포트폴리오의 최신 실행형 변주로 읽을 수 있다. `build-05`의 editable CMS 설계는 "수정이 핵심"이라는 피드백을 곧장 범용 CMS나 배포 버튼으로 키우지 않고, editable surface schema, 필드 단위 anchor, `preview.write: "none"`, approval-needed publish 경계, change-log 모델로 쪼갰다. 이후 `build-07`과 `build-08`은 Next.js·Supabase·shadcn/ui 기반 CMS, 트리형 Status 구성, Activity 감사 로그, service-role-only 경계를 실제 웹 조작과 배포 검증까지 밀어 넣었다.

중요한 점은 `build-09`에서 정적 Control Center 페이지가 제거되자, 이전 publish/rollback target을 계속 밀지 않고 live CMS(`https://cms.oracle.shdkej.com`)를 새 기준면으로 다시 잡았다는 것이다. 원본의 Slack 배포 봇·버티컬 사이트 관리와 같은 문법이 여기서 반복된다. 반복 수정 요청을 개인 숙련이 아니라 공용 제어면으로 옮기되, 실제 공개 정본 변경·권한·롤백은 승인 경계로 분리하고, build/pass·rollout·HTTPS 200·API mutation·activity log 같은 검증 흔적으로 설명면을 남긴다.

## 프로젝트별 증거

### 대용량 메시지 아키텍처
가장 강한 사례는 메시지 발송 시스템이다. 단일 Lambda 내부 비동기 호출을 밀어붙이는 대신, Lambda가 Lambda를 재호출하는 파이프라인으로 실행 모양을 바꿨다. SQS FIFO와 Kinesis를 비교했고, 1000개 데이터를 1개로 압축해 전송 비용을 낮췄으며, 트리거 튜닝으로 누락을 줄였다.

이 사례의 핵심은 `분당 1만건 -> 5만건`이라는 숫자만이 아니다. 외부 API 한도, 소켓 부족, 순서 보장, 비용, 누락 방지를 하나의 흐름 문제로 묶어 시스템 모양을 다시 설계했다는 점이다.

### EKS 업그레이드와 플랫폼 마이그레이션
EKS 1.20에서 1.27로 가는 작업은 단순 버전 변경이 아니라 운영 플랫폼 전환에 가깝다. 원본은 AWS CNI, LoadBalancer 이름 제한, ArgoCD 리소스 삭제, 인증서, VPC CNI, 보안 그룹, RDS/캐시 통신, S3 접근 정책을 한꺼번에 다룬다.

따라서 이 프로젝트는 최신 버전 적용보다, 숨어 있던 통합면을 다시 맞추고 위험 반경을 줄이는 능력을 보여준다. 해결 과정 문서화가 성과로 들어간 것도 같은 맥락이다.

### Slack 배포 자동화와 버티컬 사이트 관리
Slack 배포 봇과 버티컬 사이트 관리 시스템은 묶어 읽을 때 더 선명하다. 하나는 테스트 배포 진입 비용을 줄이고, 다른 하나는 신규 사이트 추가 절차를 표준화한다. 둘 다 특정 운영자만 아는 설정 변경을 팀이 사용할 수 있는 플랫폼 기능으로 바꾼다.

여기서 좋은 플랫폼 작업은 내부 전문성을 과시하는 일이 아니라, 다른 사람이 스스로 움직일 수 있게 만드는 일이다. `배포 시간 70% 단축`, `신규 사이트 추가 1일 -> 2시간`은 그 흐름 변화가 실제였다는 증거다.

### 비용 최적화와 관찰성 개선
야간 파드 스케줄러, CloudFront 비용 분석, TraceID 로그 구축은 모두 계속 아픈 지점을 줄이는 작업이다.

- 유휴 시간은 운영자의 기억이 아니라 CronJob으로 다룬다.
- 기능 제안은 가능성만 보지 않고 트래픽, 호출량, 월 비용으로 계산한다.
- 로그는 쌓는 것보다 TraceID로 다시 찾을 수 있게 만드는 것이 핵심이다.

CloudFront Functions 검토는 실제 도입되지 않았지만 중요하다. 실행하지 않은 일도 비용과 보류 이유가 남으면 다음 제안의 기준점이 된다.

### 백엔드 개발과 작은 자동화
알림 시스템, 마이그레이션, 소셜 로그인, 보안 대응, 해커톤 프로젝트는 인프라 바깥의 별도 자랑이 아니라 경계 확장 증거다. 특히 알림 시스템의 멱등성, 확장성, 성능 트레이드오프는 기능 구현보다 "다음에도 버틸 구조"를 먼저 보는 태도를 보여준다.

엑셀 데이터 -> HTML 변환, 배송 상태 처리 API 자동화 같은 작은 사례도 같은 문법을 반복한다. 큰 문제에서는 아키텍처를 바꾸고, 작은 문제에서는 반복 마찰을 절차·스크립트·제품 인터페이스로 치환한다.

## 사람별 수혜자
이 포트폴리오는 시스템만 설계하는 문서가 아니라 세 사용자 층의 마찰을 동시에 줄이는 문서다.

| 수혜자 | 줄어드는 마찰 | 원본 사례 |
| --- | --- | --- |
| 개발자 | 테스트 배포, 신규 환경 진입, 빌드·배포 대기, 로그 추적 | Slack 배포 봇, 버티컬 사이트 관리, 빌드 최적화, TraceID |
| 운영자 | 누락, 장애 분석, 야간 자원 관리, 업그레이드 위험, 수동 확인 | 메시징 파이프라인, 배치 안전 배포, EKS 업그레이드, 파드 스케줄러 |
| 의사결정자 | 도입 전 비용 불확실성, 처리량 근거 부족, 구조 설명 비용 | CloudFront 비용 계산, 처리량 수치, 단축률, 문서화, 다이어그램 |

이 3층을 함께 보면 `Portfolio`는 기술 성과집보다 사람별 운영 마찰을 다루는 설계 문서에 더 가깝다. 같은 자동화라도 `무엇을 만들었나`보다 `누가 덜 막히게 되었나`를 함께 읽을 때 중심이 선명해진다.

## 원본 문장 문법
원본은 기술명을 먼저 쌓기보다 문제와 배경을 먼저 적는다. `업무 요약` 뒤에는 대체로 `문제`, `도전`, `해결`, `성과`가 이어지고, 성과는 기능 추가보다 처리량, 시간 단축, 비용 절감, 안정화, 문서화로 귀결된다.

### 기술명보다 동사가 먼저 보인다
기억에 남는 것은 특정 제품명보다 `구축했다`, `최적화했다`, `자동화했다`, `개선했다`, `문서화했다` 같은 작업 동사다.

- IaC는 인프라 선언형 관리 능력으로,
- CI/CD는 배포 흐름 자동화 능력으로,
- AWS 서비스들은 워크로드에 맞는 자원 선택 능력으로,
- Bash·Python·Node.js는 반복 운영을 스크립트화하는 손발로 제시된다.

즉 이 포트폴리오는 도구 카탈로그보다 `운영자가 어떤 변화를 끝까지 책임졌는가`를 설명하는 형태에 가깝다.

### 수치와 이미지는 장식보다 검증 흔적이다
분당 5만건, 70% 단축, 75% 단축, 20% 절감, 50% 이상 단축 같은 수치는 자랑보다 흐름이 실제로 바뀌었다는 증거다. Slack 배포 봇, 야간 스케줄러, 버티컬 사이트, 로그 추적 시스템에 붙은 다이어그램과 예시 이미지도 마찬가지다. 성과는 `구현`, `수치`, `도식화 가능성`이 한 묶음으로 패키징된다.

## 역할 스펙트럼
원본 제목은 `DevOps & 인프라 엔지니어링`이지만, 실제 본문은 인프라 안에만 머물지 않는다.

- 메시징 파이프라인에서는 애플리케이션 호출 구조와 외부 API 한계까지 함께 다룬다.
- 배포 자동화에서는 Slack 인터페이스, ArgoCD API, Helm 배포 구조를 한 번에 묶는다.
- 로그 추적과 알림 시스템에서는 운영 문제를 백엔드 설계와 데이터 흐름 변경으로 이어서 푼다.

강점은 깊은 한 기술보다 여러 경계를 엮는 조정력에 있다. AWS, Kubernetes, Node.js, Spring, 로그, 배포, 비용, 트래픽 경험은 따로도 의미 있지만, 더 중요한 것은 서로 다른 층위의 제약을 동시에 조정한다는 점이다.

## 외부 설명으로 옮길 때의 압축
포트폴리오를 자기소개나 협업 설명으로 바꿀 때는 프로젝트 수를 늘어놓기보다 상대가 다시 움직일 수 있게 된 표면으로 말하는 편이 원본에 가깝다.

- 개발자에게는 배포·테스트·로그·신규 환경 진입의 마찰을 줄여 개발 흐름을 보호하는 사람으로 설명한다.
- 운영자에게는 장애, 업그레이드, 비용, 누락 위험을 되돌릴 수 있는 절차와 관측 가능한 상태로 바꾸는 사람으로 설명한다.
- 의사결정자에게는 기능 도입 전후의 비용, 처리량, 위험 반경, 보류 이유를 숫자와 문서로 남겨 선택을 가볍게 만드는 사람으로 설명한다.

실제 문장 순서는 `문제 현장 -> 바뀐 제어면 -> 재진입 가능성`이 좋다.

1. 누가 어떤 반복 마찰, 장애, 비용, 대기 시간 때문에 막혀 있었는지 먼저 말한다.
2. Slack 명령, GitOps, TraceID, 스케줄러, 비용 계산처럼 그 마찰을 어떤 공용 표면으로 옮겼는지 말한다.
3. 처리량·절감률 같은 결과 뒤에, 다음 사람이 같은 상태로 다시 들어올 수 있게 남은 문서·도식·절차를 붙인다.

## 보존해야 할 긴장
- 단순화와 과도한 추상화 사이의 긴장
- 배포 속도와 롤백 안전성 사이의 긴장
- 개발자 자율성과 플랫폼 통제 사이의 긴장
- 클라우드 편의성과 비용 규율 사이의 긴장
- 자동화의 레버리지와 숨은 운영 결합도 사이의 긴장

이 긴장을 남겨두는 것이 중요하다. 원본은 무조건 더 자동화하자는 문서가 아니라, 통제된 단순화를 통해 운영 품질을 높이려는 문서다.

## 인접 노드 연결
- 삭제된 CHANGELOG 원본: 개인 웹의 작은 변경 기록에서도 운영 유지, 정리, 배포 리듬이 반복됐다.
- [Fundamental/Infra](/docs/mapped/Fundamental/Infra): 인프라 선택과 운영 출입구 설계를 더 깊게 읽는 배경 노드다.
- [Deep Knowledge/Devops](/docs/mapped/deep-knowledge/devops): 개발자 자율성과 운영 복원력을 함께 설계하는 공통 운영 감각이 가장 직접적으로 이어진다.
- [Communication/Document](/docs/mapped/Communication/Document): 문서화와 다이어그램이 성과 검증 흔적으로 기능하는 지점과 연결된다.
- [Communication/Teamwork](/docs/mapped/Communication/Teamwork): handoff 비용과 팀 재진입성을 줄이는 플랫폼 표면과 연결된다.
- [Meta/About_Development](/docs/mapped/Meta/About_Development): 개발자의 기본기, 요구 해석, 재진입 가능한 작업 상태라는 기준과 이어진다.
