# My_space

- 원본: `source/shdkej-content/blog/My_space.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 `내가 만든 서비스들을 쉽게 띄우고 다시 세울 수 있는 개인 운영 공간`을 만들기 위한 실험 기록이다. 관심사는 단순히 쿠버네티스를 써 보는 것이 아니라, **적은 비용 안에서 배포를 인프라로 분리하고, IaC로 재구축 가능하며, 모니터링과 롤백이 가능한 개인 서버 운영 체계**를 만드는 데 있다.

## 이 노드가 실제로 답하려는 질문
- 개인이 감당 가능한 비용으로 얼마나 확장 가능한 운영 환경을 만들 수 있는가?
- 앱 수정과 배포 책임을 어떻게 분리해야 서버가 덜 지저분해지는가?
- 쿠버네티스는 개인 환경에서도 정말 가치가 있는가, 아니면 운영비와 복잡도를 과하게 올리는가?
- 모니터링, 알림, 롤백, CI/CD까지 포함해야 비로소 `내 공간`이 되는가?
- 특정 클라우드 벤더보다 더 중요한 것은 재구축 가능성과 관측 가능성 아닌가?

## 핵심 논지
1. **이 노드의 출발점은 소유 가능한 실행 환경에 대한 욕구다.**
   - 원문은 `월 5천원`, `원할 때 접속`, `다시 쌓아올리기` 같은 표현을 반복한다.
   - 즉 목표는 거대한 플랫폼이 아니라, 내가 통제할 수 있는 최소 운영 기반이다.
2. **배포는 애플리케이션이 아니라 인프라가 맡아야 한다.**
   - 앱은 이미지를 만들고, 실제 배포·복구·확장은 인프라 계층이 담당해야 한다는 생각이 선명하다.
   - 이 분리가 있어야 서버가 수동 수정으로 오염되지 않는다.
3. **쿠버네티스는 기능보다 운영 훈련장으로 읽힌다.**
   - 롤링 업데이트, 카나리, 오토스케일링, 페일오버, 모니터링, ArgoCD 같은 키워드는 모두 운영 연습 과제처럼 등장한다.
   - 즉 쿠버네티스 채택은 유행 추종보다 `운영 감각을 체득하기 위한 장치`에 가깝다.
4. **현실의 병목은 기술 선택보다 리소스와 비용이다.**
   - DigitalOcean, AWS, GCP 비교도 결국 사양 대비 비용과 메모리 한계로 수렴한다.
   - Grafana보다 k3s 자체가 무겁다는 메모는 이 노드의 현실 감각을 잘 보여 준다.
5. **개인 인프라도 관측, 알림, 복구 루프가 있어야 의미가 있다.**
   - 로그 수집, 모니터링, 알림, 롤백 실험이 반복되는 이유는 띄우는 것보다 `망가졌을 때 다시 다루는 능력`이 더 중요하기 때문이다.

## 원문 구조를 다시 읽기
원문은 산발적으로 보이지만 실제로는 아래 흐름으로 읽힌다.

## 원문 발췌 구조
- 출발 선언: `확장 가능하고 변경이 쉬운 환경`, `배포는 인프라에서 처리`
- 비용 탐색: AWS/GCP/DigitalOcean, CPU/RAM/네트워크/디스크, `3 nodes` 비용 계산
- 운영 검증 과제: 롤링 업데이트, 롤백, 카나리, 오토스케일링, 페일오버, MLOps/gRPC 배포
- 실제 시행착오: Terraform provider 변경, `.terraform` 상태 오염, SSH permission, Ansible interpreter, Traefik/Grafana/ingress/DNS 연결
- 최소 인프라 상: 3노드 Kubernetes, 로드밸런서, 로그 수집, 모니터링, API, feedback, CI/CD
- 서비스 후보층: note server, wiki, searching, n8n, tailscale

이 배열을 보면 원문은 `클라우드 비용 메모`에서 끝나지 않는다. 비용 탐색 → 운영 실험 → 실제 막힘 → 최소 공통 플랫폼 상상으로 점점 올라가는 구조다.

### 1. 비용 안에서 가능한 호스팅 탐색
- AWS, GCP, DigitalOcean 비교
- CPU, RAM, 네트워크, 디스크 비용 확인
- `3 nodes` 비용 계산
- `한달 5유로` 같은 예산 감각

이 부분은 단순 가격표가 아니라, 개인이 운영 환경을 지속할 수 있는 하한선을 찾는 작업이다.

### 2. 쿠버네티스 운영 실험 과제 정의
- 롤링 업데이트, 롤백, 카나리
- 스트레스 테스트, 오토스케일링
- 페일오버, 이중화
- MLOps 서버, gRPC 서버 배포

즉 이 노드는 인프라 구축 일기이면서 동시에 `무엇을 검증해야 진짜 운영 경험이 되는가`를 적어 둔 체크리스트다.

### 3. DigitalOcean 기반 시행착오
- API token, Spaces, endpoint, region
- Terraform provider 설정 변화
- SSH permission, Ansible interpreter 문제
- 로드밸런서, DNS, ingress, Traefik, Grafana 연결 문제

이 대목은 이상적인 설계보다 실제 삽질 노트에 가깝다. 그래서 이 노드는 기술 소개 문서보다 `운영 문제를 어디서 많이 맞는가`를 보존하는 가치가 크다.

### 4. 최소 인프라의 구성 요소 정의
- 3노드 쿠버네티스
- 로드밸런서
- 로그 수집
- 모니터링
- API
- feedback
- CI/CD

이 목록은 개인용 장난감 클러스터가 아니라, 장기적으로 서비스를 올릴 수 있는 최소 제어면을 상정한다.

### 5. 재구축 가능한 개인 서버라는 기준
- IaC여야 한다
- 원할 때 접속/배포가 쉬워야 한다
- 지저분해지면 원하는 상태로 바로 복구돼야 한다
- 쿠버네티스 안에서 에러를 한눈에 봐야 한다

결국 `My_space`는 물리적 서버 공간이 아니라 **복원 가능한 운영 상태**를 의미한다.

## 이 노드에서 반복되는 판단 축

### 확장성 vs 개인 예산
- 확장 가능한 구조를 원한다.
- 하지만 개인이 감당할 비용 한계가 강하게 작동한다.
- 그래서 관리형 쿠버네티스의 편의와 DIY/k3s의 비용 절감이 계속 충돌한다.

### 재현성 vs 즉흥 수정
- 직접 서버에 들어가 빠르게 고치는 것은 쉽다.
- 하지만 원문은 점점 `이미지는 앱이, 배포는 인프라가`라는 분리로 이동한다.
- 이는 즉흥 수정보다 재현성을 더 높은 가치로 보기 시작했다는 신호다.

### 학습 욕구 vs 운영 현실
- 쿠버네티스를 깊게 이해하고 싶다.
- 동시에 작은 리소스 환경에서는 관측 도구조차 뜨지 않는 현실을 겪는다.
- 그래서 이 노드는 기술 학습 욕구와 운영비 현실 사이의 긴장을 그대로 보존한다.

### 구축 자체 vs 운영 루프 완성
- 클러스터를 띄우는 것만으로는 충분하지 않다.
- 상태 확인, 알림, 롤백, 배포 자동화까지 연결돼야 비로소 쓸 만한 공간이 된다.

## 세부 해석

### 1. `배포는 인프라에서 처리한다`의 의미
이 문장은 이 노드의 중심 문장에 가깝다.

- 애플리케이션 저장소는 코드와 이미지 생성에 집중한다.
- 실제 반영은 ArgoCD, Terraform, 클러스터 설정 같은 인프라 계층에서 처리한다.
- 이렇게 해야 여러 앱을 같은 공간에 일관되게 올리고, 다시 세우고, 추적할 수 있다.

즉 개인 서버라도 수작업 서버가 아니라 `플랫폼처럼 다루는 서버`를 지향한다.

### 2. 모니터링과 알림이 먼저 등장하는 이유
원문에서는 `로그 수집`, `모니터링`, `상태 추적`, `오버로드 알림`, `바로 롤백`이 매우 일찍 등장한다. 이는 저자가 서버 운영의 핵심을 성능 튜닝보다 **이상 징후를 빨리 감지하고 되돌리는 능력**으로 보고 있다는 뜻이다.

### 3. 쿠버네티스를 선택하는 이유와 흔들림
- 선택 이유: 확장 가능, 변경 용이, 직접 써 보며 깊게 이해 가능
- 흔들리는 이유: 메모리 부담, 관리형 서비스 비용, 추가 컴포넌트의 무게

그래서 이 노드는 `쿠버네티스 예찬`이 아니다. 오히려 `개인 환경에서 어디까지 쿠버네티스를 감당할 수 있는가`를 시험하는 노드다.

### 4. DigitalOcean 메모의 의미
DigitalOcean 관련 부분은 단순 공급자 기록이 아니라 중요한 운영 교훈을 담고 있다.

- 문서만 보고는 안 되는 provider/버전 차이
- `.terraform` 잔존 데이터 같은 상태 오염 문제
- SSH permission, interpreter 설정 같은 사소하지만 치명적인 막힘
- ingress와 DNS가 붙어야 비로소 외부 접속이 완성된다는 점

즉 이 노드는 `클라우드 인프라는 클릭 몇 번으로 끝나지 않는다`는 사실을 몸으로 학습한 기록이기도 하다.

### 5. 블로그 노드로서의 성격
같은 `My_space`라도 [[../Meta/My_space]]와 이 페이지의 결은 조금 다르다.

- Meta 쪽이 장기 운영 기준과 최근 제어 구조를 더 직접적으로 잡아 준다면,
- blog 쪽은 `왜 이런 공간을 원했는가`, `어디서 실제로 막혔는가`, `개인 예산에서 무엇이 과했는가`를 더 서사적으로 보존한다.

그래서 이 페이지는 설계 명세보다 **운영 욕구가 시행착오를 통해 어떤 기준으로 굳어졌는가를 보여 주는 외부화된 실험 기록**으로 읽는 편이 맞다.


## 블로그 글로 읽을 때의 독자 가치
이 페이지가 `blog` 아래에 있다는 점도 중요하다. 같은 주제를 [[../Meta/My_space]]처럼 내부 운영 설계로만 읽으면, 비용 계산과 시행착오의 서사가 지나치게 압축된다. 하지만 블로그 글로 보면 이 노드는 독자에게 다음 질문을 대신 검증해 주는 글감이다.

- 개인 프로젝트가 늘어날 때 서버를 손으로 고치는 방식은 언제 한계에 부딪히는가?
- 관리형 Kubernetes의 편의는 개인 예산 안에서 어느 지점부터 과한 선택이 되는가?
- k3s, Grafana, Traefik, ArgoCD 같은 도구는 `깔 수 있음`과 `지속 운영 가능함` 사이에 어떤 리소스 차이를 만드는가?
- 실패한 설정과 비용 계산을 공개하면, 다른 사람에게도 `작은 플랫폼을 어디까지 감당할 수 있는가`라는 판단 재료가 되는가?

따라서 이 노드는 성공한 홈랩 튜토리얼보다 **운영 욕구가 현실 비용과 리소스 제한을 만나며 어떤 기준으로 정제되는지 보여 주는 실험기**에 가깝다. 독자가 얻는 것은 완성 아키텍처가 아니라, 개인 인프라를 시작하기 전에 반드시 계산해야 할 운영 흡수 비용이다.

## 운영 흡수 비용으로 다시 보기
원문에 등장하는 막힘은 대부분 기술 오류처럼 보이지만, 실제로는 새 도구를 들일 때 함께 들어오는 유지비를 드러낸다.

| 원문 단서 | 겉보기 문제 | 실제로 남는 운영 질문 |
|---|---|---|
| 관리형 Kubernetes 비용 | 가격이 비싸다 | 개인 프로젝트가 이 편의를 매달 감당할 만큼 가치가 있는가? |
| k3s 메모리 0.9G | 리소스가 부족하다 | 관측 도구까지 올릴 최소 사양을 먼저 계산했는가? |
| `.terraform` 상태 오염 | 설정이 꼬였다 | IaC도 상태 파일과 초기화 경로를 관리하지 않으면 재현성이 깨지는가? |
| SSH permission / Ansible interpreter | 사소한 설정 오류 | 자동화 도구는 로컬 전제와 원격 런타임 전제를 함께 고정해야 하는가? |
| Traefik / ingress / DNS | 접속이 안 된다 | 앱 배포는 컨테이너 실행이 아니라 외부 이름·라우팅·인증까지 이어져야 완료되는가? |

이 표를 기준으로 보면 `My_space`의 핵심은 어떤 도구를 선택했는지가 아니라, **도구를 들였을 때 생기는 숨은 운영면을 감당 가능한 크기로 줄이는 능력**이다.

## 숨은 패턴
- 이 노드는 개인용 인프라 메모처럼 보이지만, 실제로는 **작은 규모에서 플랫폼 엔지니어링 감각을 익히는 노드**다.
- 비용 계산, 리소스 관찰, 배포 자동화, 관측 도구, 롤백 실험이 모두 한 방향을 가리킨다.
- 저자는 화려한 멀티클러스터보다 `다시 만들 수 있고, 상태를 볼 수 있고, 앱을 일관되게 올릴 수 있는 환경`을 더 중요하게 여긴다.
- 따라서 핵심은 쿠버네티스 자체보다 `복구 가능한 개인 운영 시스템`이다.

- [[../Deep Knowledge/Container]]: 앱 이미지를 만들고 배포 책임을 분리한다는 관점이 직접 이어진다.
- [[../Deep Knowledge/Infra]]: 어떤 구성 요소가 필요한가를 묻는 노드라면, 이 노드는 그것을 개인 환경에서 어떻게 실제로 굴려 볼지를 묻는다.
- [[Deploy_ERP_server_story]]: 더 작은 스케일에서 `docker-compose`로 운영 환경을 만들던 감각이 이후 개인 플랫폼 설계로 확장된 흐름으로 읽을 수 있다.

## 이 노드를 읽을 때 놓치지 말아야 할 점
- 완성된 아키텍처 문서가 아니라 비용, 실험, 막힘, 복구 욕구가 섞인 운영 설계 노드다.
- `내 공간`은 서버 한 대가 아니라 재구축 가능성과 배포 일관성의 다른 이름이다.
- 기술 선택 메모보다 `무엇을 자동화하고 무엇을 관측해야 하는가`의 기준이 더 중요하다.
- 작은 규모에서도 이미 플랫폼적 사고가 시작되고 있다는 점이 이 노드의 핵심 가치다.

## 이후 심화 포인트
- 이후 source에 후속 메모가 들어오면 `개인 홈랩 욕구`가 `작은 플랫폼 운영 문법`으로 바뀌는 전환점을 더 시간축으로 정리할 수 있다.
- 알림, 롤백, GitOps, 메시징 시스템을 하나의 운영 루프로 압축해 체크리스트화할 수 있다.
- `개인 서버`와 `개인 플랫폼`의 차이를 다른 노드와 연결해 더 분명하게 만들 수 있다.

## 이번 갱신 메모
- 기존 baseline은 유지한 채 `블로그 글로 읽을 때의 독자 가치`를 추가해, 이 노드가 완성형 홈랩 튜토리얼보다 개인 인프라의 비용·리소스·실패 조건을 공개적으로 검증하는 실험기에 가깝다는 점을 보강했다.
- `운영 흡수 비용으로 다시 보기` 표를 추가해 관리형 Kubernetes 비용, k3s 메모리, Terraform 상태 오염, SSH/Ansible 전제, ingress/DNS 문제를 각각 재사용 가능한 운영 질문으로 압축했다.
- [[../Meta/My_space]], [[../Deep Knowledge/Container]]와의 연결을 유지하면서도 실제 편집 범위는 이 blog 노드 안으로 제한했다.
