# Journal

- 원본: `source/shdkej-content/Idea/Journal.md`
- 성격: `Journal` 만다라트 카테고리 노드의 대응 페이지

## 한 줄 요약
`Journal`은 일기를 모아두는 장소가 아니라, 하루의 경험·실패·할 일·글감·스크랩이 나중에 다시 읽히고 다른 노드로 이동하며 새 산출물로 바뀌게 하는 기록 운영 노드다.

## 만다라트 위치
`Journal`은 `Idea` 아래에 있다. 그래서 이 노드는 감정 배출형 일기보다 **아이디어가 시간축에서 들어와 주제축으로 다시 번역되는 입력면**에 가깝다. `Reading`이 외부 텍스트를 질문으로 바꾸고, `Article`이 웹 입력을 보관하며, `Information`이 정보의 재등장 방식을 묻는다면, `Journal`은 그날의 경험과 미완성 생각이 처음 붙잡히는 표면이다.

`Meta` 계열 노트가 시스템 원리 자체를 더 직접 다룬다면, `Journal`은 실제 기록이 쌓이다가 시스템 설계 질문으로 번지는 자리다. 연도별 아카이브, 글쓰기 규칙, 태그·카테고리 실험, todo, 위키 전환, 제텔카스텐 메모가 한 파일에 함께 있는 이유도 여기에 있다.

## 원문 경계
원문 첫 문장은 이 노트의 출처 경계를 정한다. `AI에게 물어보고 대답 받은 지식은 최대한 옮기지 않는다`는 말은 외부 지식을 배척한다는 뜻보다, **몸에 새겼거나 몸에 새길 수 있는 것만 자기 기록이 된다**는 기준에 가깝다.

- AI 답변과 외부 정보는 원문 경험을 대체하지 않는다.
- 남의 문장은 `내 언어`, `내 사례`, `다음에 다시 볼 표면`을 얻은 뒤에야 기록 자산이 된다.
- 이 노트의 신뢰성은 최신 지식량보다, 무엇이 실제 경험에서 검증되었는지 구분하는 데 있다.

## 원문의 큰 구성
1. `All / 2025 / 2024 ...`: 연도별 기록을 한 입구에 묶되, 단순 연표로 끝내고 싶어 하지는 않는다.
2. `Writing technique`: 쉬운 글, 짧은 문장, 결론 먼저, 첫 줄 요약, 큰 그림에서 작은 그림으로 내려가는 문서 감각을 세운다.
3. `내가 노트에 바라는 것`: 전체를 그림처럼 보고, 실패를 다시 반복하지 않으며, 변경 추적과 재사용성을 확보하려는 기대가 나온다.
4. `Keep specific note to yearly archive / Note Category / 분류 구조`: 날짜 기록, 태그, 카테고리, 링크, 인덱스의 역할을 계속 가른다.
5. `Todo-list / Template`: 기록이 오늘의 실행 표면으로 다시 올라오게 만드는 방식을 찾는다.
6. `WIKI / 위키를 분리해야할까 / 노트 쪼개기 / 편집가능성`: 큰 문서와 작은 메모, 정리글과 카드, 아카이브와 살아 있는 위키 사이의 긴장을 다룬다.
7. `ETC`: duration, frequency, guide, 입력은 하나로 출력은 여러 개, 10000자 제한, 블로그/프로젝트/회고 전환 같은 운영 질문을 남긴다.

## 중심 모델: 입력에서 출력까지

### 1. 입력
하루 기록, 실패 경험, 인터넷 페이지, 사진, 독서·영화 기록, 업무 질문, 할 일, 짧은 생각이 먼저 흩어진 형태로 들어온다. 원문은 입력을 너무 빨리 제한하려 하지 않는다. `봤던 인터넷 페이지, 작성한 파일, 소스코드`를 단축키 하나로 넣고 싶다는 메모처럼, 처음에는 마찰을 낮추는 것이 중요하다.

다만 입력이 곧 지식은 아니다. 이 노트는 복붙 저장소가 아니라 `내가 몸에 새길 수 있는 것`을 그림처럼 다시 볼 수 있게 만드는 장소다.

### 2. 표식
입력에는 나중에 다시 잡을 수 있는 최소 손잡이가 붙어야 한다.

- 날짜: 당시 맥락과 시간성을 보존한다.
- 태그: 횡단 검색과 재분류를 돕는다.
- 링크: 개별 문서 사이의 맥락을 만든다.
- 카테고리: 큰 주제의 뼈대를 세운다.
- 제목·첫 줄 요약: 후루룩 봤을 때 무엇인지 바로 떠올리게 한다.

원문은 태그를 많이 달지 엄선할지, `:tag:`와 `[[tag]]` 중 무엇을 쓸지, root tag와 sub-tag를 어떻게 볼지 계속 실험한다. 이 고민은 예쁜 분류 체계보다 `찾지 못해서 지나치는 기록`을 줄이려는 시도다.

### 3. 재호출
원문에서 가장 큰 문제는 쓰기가 아니라 다시 찾기다. 날짜별 기록은 시간이 지나면 찾아볼 엄두가 나지 않고, 태그가 없으면 묻히며, 태그가 너무 많으면 우선순위가 흐려진다.

그래서 검색, 언태그드 목록, todo 검색, 연도별 아카이브, 리마인드, 인덱스 파일, 대시보드가 반복해서 등장한다. 특히 `duration`과 `frequency` 메모는 모든 기록을 같은 방식으로 보관하면 결국 아무것도 다시 보지 않게 된다는 문제의식이다.

- 오늘 다시 볼 기록: todo, Pomodoro 전 점검, 바탕화면 리스트, inbox 개수
- 주기적으로 다시 볼 기록: 독서노트, 미완성 글, 월간·연말 회고용 느낌
- 주제가 떠올랐을 때 볼 기록: 태그가 붙은 스크랩, 오래된 판단 기준, 정보성 메모
- 출력으로 승격될 기록: 블로그, 프로젝트 문서, 결정 기준, 정리된 글

따라서 핵심 질문은 `어디에 저장할까`보다 `언제 어떤 표면으로 다시 나타나야 하는가`다.

### 4. 재분류
날짜 기록만으로는 오래 못 가고, 주제 기록만으로는 당시 맥락이 사라진다. 원문은 둘 중 하나를 고르기보다 날짜별 입력을 주제별 위키로 옮기는 변환 절차를 찾는다.

여기서 큰 문서와 작은 메모의 긴장이 생긴다. 큰 문서는 한 번에 보기 좋지만 너무 커지고, 작은 메모는 연결하기 좋지만 존재를 잊기 쉽다. 원문이 제텔카스텐과 카드식 메모에 끌리는 이유는 작은 메모를 버리지 않으면서도, 나중에 잘 작성된 정리글로 엮을 수 있기 때문이다.

기록의 수명도 이 단계에서 갈린다.

- 보존: 당시 생각, 실패 경험, 감정 반응처럼 과거 버전 자체가 의미 있는 기록
- 갱신: 언어별 메모리 구조, 툴 사용법, 노트 시스템 규칙처럼 현재 기준과 맞춰야 하는 기록
- 폐기·압축: 검색 가능한 단순 스크랩, 중복 후보, 이미 출력물로 흡수된 임시 메모

### 5. 출력
기록은 다시 읽히는 데서 끝나지 않는다. 원문은 `입력은 하나로, 출력은 여러 개`라고 정리한다. 같은 기록이 하루 회고, 블로그 글, 업무 일지, 프로젝트 문서, todo, decision check list, 검색 기록 템플릿으로 달라질 수 있어야 한다.

이때 가이드는 강제 규칙이 아니라 포장도로다. 템플릿, 체크리스트, 카테고리, 태그, 다시 보기 주기는 기록이 멈추지 않게 돕는 구조여야 한다. 가이드가 기록을 막으면 다시 줄여야 한다.

## 글쓰기 기준은 노트 시스템 기준이다
원문 초반의 글쓰기 규칙은 문체 팁으로만 읽으면 좁다. 쉬운 글, 짧은 문장, 결론 먼저, 첫 줄 요약, 큰 그림에서 작은 그림으로, 읽다가 막히지 않는 흐름은 노트 구조에도 그대로 적용된다.

- 문장에서는 불필요한 수식과 모호한 표현을 줄인다.
- 문서에서는 목차, 요약, 사례, 반례, 정리로 독자의 위치를 잡아 준다.
- 위키에서는 전체 그림, 인덱스, 링크, 태그가 같은 역할을 한다.

저자에게 좋은 글은 정보를 많이 담은 글보다 `읽는 사람이 목적지까지 편하게 도달하는 글`이다. 그래서 `Journal`의 글쓰기 기준은 `Readability`와 연결되지만, 이 노드에서는 특히 다시 읽히고 다시 쓰이는 기록 표면의 기준으로 작동한다.

## todo는 별도 앱 문제가 아니다
원문 후반의 todo 메모는 할 일 앱 후보처럼 보이지만 실제로는 `기록이 다음 행동을 다시 부르는가`라는 질문에 붙어 있다.

- 현재 처리할 것, 프로젝트성 작업, backlog, 반복할 것, 행동 방침을 구분하려 한다.
- `[ ]` 검색, Pomodoro, kanban, 바탕화면 리스트, inbox 개수는 실행 표면을 다시 여는 장치다.
- 문제는 위치보다 우선순위, 기간, 진행 상태, 다시 보기 표면이다.

그래서 todo는 기록과 분리된 실행 목록이 아니라, 기록이 행동으로 바뀌는 재진입 인터페이스다. 행동 방침은 `Principle`로, 선택 조건은 `Decision`으로, 프로젝트 진행은 작업 문서로 흘러갈 수 있지만, 처음 붙잡히는 표면은 여전히 `Journal`에 가깝다.

## 후반 메모가 맡는 운영 표면
원문 후반의 `Template`, `WIKI`, `ETC`는 남은 잡항목이 아니라 앞의 중심 모델을 실제 사용 화면으로 내려보는 구간이다. 검색 기록, 하루 회고, 업무 일지, 프로젝트 진행 템플릿은 모두 기록을 다시 꺼내는 모양을 미리 정해 둔다. 위키 분리, 노트 쪼개기, 카드, 타임라인, 언태그드 목록, 할 일 모아보기는 저장 구조보다 **다시 보이는 표면**을 계속 조정하는 질문이다.

| 원문 구간 | 맡는 표면 | 보존할 질문 |
| --- | --- | --- |
| `Template` | 검색 기록, 하루 회고, 업무 일지, 프로젝트 진행 중 모아둘 제목 | 같은 입력을 어떤 출력 형식으로 다시 열 것인가 |
| `WIKI` | 점진적 발전, 태그 관계, 백링크, 작은 메모와 정리글 분리 | 작은 메모를 잊지 않으면서 큰 흐름으로 엮을 수 있는가 |
| `vimwiki` | 링크, todo, 태그, 아카이빙, 맞춤법 검사 | 기록 도구의 문법이 재호출과 유지보수를 실제로 돕는가 |
| `ETC` | duration, frequency, guide, category, inbox, 10,000자 제한 | 어떤 기록을 오늘·주기·검색·출력 중 어디에 다시 띄울 것인가 |

따라서 후반부는 새 주제 확장이 아니라 `입력 → 표식 → 재호출 → 재분류 → 출력` 모델의 검증장이다. 이 구간을 읽을 때는 항목 수를 줄이기보다, 각 메모가 어느 재등장 표면을 시험하는지 붙잡는 편이 드리프트를 줄인다.

## 특히 보존할 원문 압력
- `메모를 남기는 건 쉽다. 근데 어떻게 남겨야 찾기 쉽고 후루룩 봤을 때 원하는 걸 찾을 수 있을까.`
- 실패 경험을 적어도 같은 실수를 반복한다는 문제의식
- 날짜 기록은 당시 맥락을 살리지만 오래 지나면 찾을 엄두가 나지 않는다는 감각
- 작은 메모는 그대로 두고, 이를 정리한 잘 작성된 글을 따로 만들고 싶다는 제텔카스텐식 끌림
- `가이드는 작성을 편하기 위한 포장도로를 까는 것이고 꼭 지킬 필요는 없다`
- `입력은 하나로, 출력은 여러 개`
- 검색 기록, 하루 회고, 업무 일지, 프로젝트 진행 템플릿처럼 같은 입력을 여러 출력 표면으로 바꾸려는 시도
- `노트 10,000자를 안넘게 하고 5,000자는 넘길 수 있도록 하자`

## 판단 축

| 축 | 한쪽 위험 | 다른 쪽 위험 | 이 노드의 방향 |
| --- | --- | --- | --- |
| 날짜 기록 / 주제 기록 | 다시 찾기 어렵다 | 당시 맥락이 사라진다 | 날짜 입력을 주제 위키로 재분류한다 |
| 큰 문서 / 작은 메모 | 비대해지고 연결이 약해진다 | 산발적이고 잊힌다 | 작은 메모와 정리글의 역할을 나눈다 |
| 태그 / 링크 / 카테고리 | 하나로 통일하면 역할이 흐린다 | 너무 많으면 관리가 어렵다 | 검색, 맥락 연결, 큰 뼈대를 분업시킨다 |
| 입력 편의 / 장기 유지 | 아무 데나 쌓여 묻힌다 | 규칙이 무거워 기록이 막힌다 | 입력은 낮은 마찰, 재호출은 최소 표식으로 관리한다 |
| 보존 / 갱신 / 폐기 | 과거를 지워 버린다 | 낡은 기록이 지도를 흐린다 | 기록마다 수명과 재등장 방식을 다르게 둔다 |
